# Home Dashboard 与上下文面板优化设计

## 背景

Studio 已完成第一轮视觉精修，但当前还有两个明显问题：

1. Home 首页仍然带有传统 dashboard 的平均铺开倾向，信息不少，但首个动作不够明确。
2. 全局右侧上下文侧边栏长期压缩主舞台宽度，导致 Home、Chat、System、Terminal 等页面的聚焦感和呼吸感不足。

本轮优化不引入新的产品域，而是在现有 Studio 平台壳层下，重新定义首页信息架构与全局上下文面板机制。

## 目标

### 目标
- 把 Home 从“平均铺开的 dashboard”收敛为“风险优先 + 工作续做 + 趋势判断”的控制台首页。
- 保留仪表盘能力，但让图表退居辅助判断层，不再主导首页第一视觉层。
- 把全局右侧上下文区从常驻布局骨架改为按需出现的上下文面板系统。
- 恢复主舞台宽度与聚焦感，减少整站长期被右栏挤压的视觉问题。

### 非目标
- 不在本轮设计中扩展新的业务能力。
- 不把未成熟的 Room / Workflow 强行提升为首页第一屏核心内容。
- 不把 Home 做成 BI 分析后台或运维大屏。
- 不要求所有页面都使用统一的双栏布局。

## 设计原则

1. **风险优先**：首页先回答“现在先处理什么”。
2. **工作续做优先**：在没有更高优先级风险时，首页要帮助用户最快回到当前上下文。
3. **趋势辅助判断**：图表用于看趋势、看异常、看变化，不负责首页首个动作决策。
4. **主舞台优先**：辅助上下文不应长期压缩主内容区域。
5. **按真实能力显示**：有数据才显示，无数据就降级或隐藏。
6. **蓝图完整，落地分期**：Room / Workflow 继续保留在全局蓝图内，但首页当前实现只依赖已经稳定存在或已形成真实记录的能力。

## 范围

### 当前可直接服务的首页能力
- System
- Terminal
- Agents
- Channels
- Cron
- Sessions
- Chat
- Runtime / monitoring 相关状态与事件

### 蓝图保留但不前置依赖的能力
- Room
- Workflow
- 后续协作 / 编排类模块

这些模块可以保留入口、建设中占位或在有真实最近记录时动态出现，但不进入首页首屏核心区。

## Home 首页角色重定义

Home 不再是“所有信息平均摆放”的总览页，而是一个控制台首页。它只承担三层职责：

1. **暴露风险与待处理**：让用户先判断有没有必须立即处理的事项。
2. **恢复当前工作**：让用户在确认风险后，可以最快回到最近上下文。
3. **提供趋势与态势概览**：帮助用户理解平台整体变化，而不是只看当前瞬时状态。

因此，Home 的定位是“风险优先的工作恢复页”，不是模块启动器，也不是单纯图表大盘。

## Home 信息架构

从上到下，Home 固定为六个区块：

### 1. 顶部总览条
内容：
- 当前环境 / 运行状态
- 关键告警计数
- 待处理数
- 活跃 session / workflow 摘要（如果有真实数据）
- 快捷全局操作入口

作用：
- 给出瞬时整体状态判断。
- 这一层应当很薄，不抢主舞台。

### 2. 风险与待处理主舞台
内容：
- 高优先级告警
- 失败 workflow / 阻塞任务（仅在当前已有真实数据时）
- 待确认 / 待处理事项
- 需要立刻进入处理的事件

作用：
- 首页第一主区。
- 用户进入后先决定有没有必须立即处理的内容。

### 3. 继续当前工作区
内容：
- 最近会话
- 最近可恢复任务
- 最近访问页面
- 最近对象 / 最近上下文
- 一键继续入口
- Room / Workflow 仅在已有真实最近记录时动态出现

作用：
- 在确认没有更高优先级风险后，帮助用户立即回到工作流。

### 4. 趋势仪表盘区
内容：
- 异常 / 告警趋势
- Workflow / Task 趋势（仅在当前有真实数据时）
- Session / Agent 活跃度
- 系统资源趋势

作用：
- 回答“整体走势发生了什么”。
- 是判断层，不是第一动作层。

### 5. 平台资源与系统概览区
内容：
- 资源卡片
- 模块状态摘要
- 系统健康状态
- 能力分布 / 服务可用性概览

作用：
- 提供平台级态势感知。

### 6. 最近变化与入口区
内容：
- 最近事件流
- 最近变更
- 常用模块入口
- 推荐进入点

作用：
- 作为页面底部的补充信息与导航层。
- 模块入口不再是首页第一主角。

## Home 图表策略

Home 保留 dashboard 属性，但图表退居第三层辅助判断。

### 建议保留的图表
1. 告警 / 异常趋势
2. Workflow / Task 运行趋势（仅在已有真实数据时）
3. Session / Agent 活跃度
4. 系统资源趋势

### 建议删除或降级的旧式图表
- 纯装饰性大仪表盘
- 只有累计值、没有趋势意义的图
- 重复表达同一件事的多张图
- 过度 BI 化、读图成本过高的复杂图表

### 图表规则
- 首屏或首屏下半区只保留 2-4 张关键图。
- 每张图必须绑定一个明确问题：是否恶化、是否积压、是否异常升高、是否持续高压。
- 每张图旁都要有摘要值与跳转入口。
- 没有趋势数据时，退化为摘要统计卡，而不是显示空图表。

## Home 组件策略

### 保留并降权的组件
- 统计摘要卡
- 系统状态类卡片
- 最近变化 / 最近事件流

这些组件继续保留，但需要调整到更轻、更短的位置，不再占据主舞台。

### 需要重做的组件
- 旧式 dashboard 图表卡
- 模块入口卡墙
- 大而平均的资源总览区

这些组件的问题不是“不能要”，而是旧形态已经不适合作为首页第一层结构。

### 需要新增的组件
- 风险主舞台组件组
- 继续当前工作组件组
- 趋势仪表盘组件组
- 动态模块占位组件

## Home 数据来源与显示规则

### 数据分级

#### A. 常驻数据
- 系统状态
- 基础 runtime 状态
- 告警计数
- 待处理计数
- session / agent 基础活跃摘要
- 最近事件流（若已具备）

这些数据构成首页稳定骨架。

#### B. 条件显示数据
- 最近会话
- 最近恢复任务
- 最近访问页面
- 最近 Room / Workflow 上下文
- 失败任务卡
- 某类趋势图表

规则：有真实内容才显示，没有就隐藏或降级。

#### C. 未来能力 / 预留数据
- Room 深度协作概览
- Workflow 编排分析视图
- 尚未稳定的数据域联动

规则：保留蓝图位置，但不进入当前首页主流程。

### 空态与降级规则
- 无高优先级风险：显示轻量安全态，不用红色空壳卡。
- 无最近工作记录：显示常用入口或推荐起点。
- 无趋势数据：显示统计摘要，不显示空图。
- 未开发模块：显示入口或建设中占位，但不占据首页主区。

### 数据新鲜度规则
- 首屏优先显示轻量、快速、当前的数据。
- 图表与次级区块可以晚于首屏摘要加载。
- 不为了“完整”牺牲首屏响应速度。

## 响应式布局

### 桌面端
- 顶部总览条横向铺开。
- 风险主舞台使用全宽或偏宽主区。
- 继续工作区紧跟风险区，可并列但视觉权重更轻。
- 趋势仪表盘区采用 2 x 2 网格。
- 平台概览与最近变化位于下半区。

核心目标是形成“风险 → 续做 → 趋势 → 全局”的纵向阅读节奏，而不是平均分栏。

### 窄屏端
- 所有主区单列堆叠。
- 风险区优先展开。
- 继续工作区紧随其后。
- 趋势图表纵向排列。
- 模块入口下沉到底部。

窄屏端强调动作顺序，而不是完整复制桌面信息密度。

## 全局上下文面板系统重定义

当前右侧上下文栏的核心问题是：
- 长期占据横向宽度
- 页面需求不统一
- 信息重要性不稳定
- 持续干扰主舞台聚焦感

因此，本轮将它从“全局常驻右侧边栏”改为“全局按需调用的上下文面板系统”。

### 新职责
1. 展示当前页的补充上下文
2. 提供临时深入查看
3. 承载跨页通用辅助能力

前提是：只有在需要时才展开，不再默认长期占位。

## 上下文面板的三种形态

### 1. 悬浮侧板
适用：桌面端默认形态。

特征：
- 从右侧滑出
- 覆盖在主内容之上
- 不永久压缩主内容宽度
- 适合查看详情与补充上下文

### 2. 页内可折叠 Inspector
适用：天然需要双栏关系的页面。

特征：
- 作为页面局部结构存在
- 可以折叠 / 收起
- 不上升为全站统一骨架

当前优先适用于 Chat，未来可扩展到 Workflow。

### 3. Sheet / Drawer
适用：窄屏端。

特征：
- 以底部 sheet 或全屏详情层承载内容
- 不再坚持右侧挤压式布局

## 分页面策略

### 取消常驻右栏
- Home
- Agents
- Channels
- Cron
- System

这些页面只在查看详情、对象信息或补充上下文时，临时打开侧板。

### 保留可折叠 Inspector
- Chat
- Workflow（未来）

这些页面可以保留更强的 inspector 能力，但必须支持折叠与按需展开。

### 禁用全局右栏，改用临时浮层
- Terminal
- 沉浸式全屏页面

这些页面最需要横向空间，不应被右侧栏打断。

## 上下文面板交互规范

### 默认关闭
页面进入时，上下文面板默认不展开，保证主舞台优先成立。

### 明确触发
只通过明确动作打开，例如：
- 点击对象详情
- 点击查看上下文
- 点击页面的 inspector/context 按钮
- 在 Chat 中点击会话信息入口

### 保持局部连续性
面板打开后，用户应始终明确：
- 当前查看的对象是谁
- 面板内容属于哪个页面上下文
- 关闭后会返回哪里

### 支持快速关闭
必须支持：
- 关闭按钮
- Esc 关闭
- 覆盖式面板点击外部关闭（适用时）
- 窄屏的关闭动作

### 支持对象切换
对于详情型面板，应允许在不关闭面板的情况下切换对象，以保持管理页的浏览连续性。

## 上下文面板视觉规范

- 面板视觉权重应弱于主舞台。
- 宽度必须受控，不能再次形成双主区。
- 内容密度应高于装饰密度。
- 顶部结构建议统一为：对象标题、类型/状态标签、少量主操作、关闭按钮。
- 桌面端动效以轻量右滑为主，不强推挤布局。
- 窄屏端优先使用接近原生应用的 sheet / 全屏详情层过渡。

## 适合与不适合放进面板的内容

### 适合
- 选中对象详情
- 状态说明
- 元信息
- 最近事件
- 关联操作
- 当前筛选 / 上下文说明

### 不适合
- 整页主流程
- 大型表单主编辑
- 大面积图表墙
- 需要长时间专注的大型操作
- 本应占据主舞台的核心内容

一句话总结：面板适合“看清楚”，不适合“做整件事”。

## 预期收益

完成本轮优化后，预期得到：
- Home 更像真正的平台首页，而不是平均铺开的卡片页。
- 首页能先帮助用户判断风险，再恢复工作，再理解趋势。
- 图表重新回归辅助判断层。
- 主舞台宽度明显恢复。
- 右侧上下文从“常驻噪音”变成“按需能力”。
- 全站页面层级更清楚：主内容是主内容，辅助上下文是辅助层。

## 实施优先级建议

1. 先改全局上下文面板机制
2. 再重做 Home 信息架构
3. 最后补图表与动态模块显隐规则

原因：全局右侧面板机制先调整后，Home 与其他页面才能真正释放主舞台宽度。

## 验收标准

### Home
- 首页首屏先看到风险 / 待处理，再看到继续工作区。
- 图表不再占据首页第一视觉层。
- 无数据时不出现空图、空卡、空模块占位。
- Room / Workflow 不会在无真实记录时成为首屏核心内容。
- 桌面端与窄屏端都能保持明确的阅读节奏。

### 上下文面板
- 默认进入页面时不再全站常驻展开。
- Home、System、Agents、Channels、Cron 不再被固定右栏长期压缩。
- Chat 仍保留可折叠 inspector 能力。
- Terminal 默认无右栏。
- 窄屏上下文查看改为 sheet / drawer / 全屏详情层。
