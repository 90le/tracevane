# OpenClaw Studio 总体重构设计文档

## 1. 背景与目标

OpenClaw 已升级到 2026-04-10 版本，Studio 扩展项目位于 `extensions/openclaw-studio/apps`。本次工作不是简单补字段或局部新增页面，而是基于 OpenClaw 4.10 的真实能力、真实配置和真实模型，对 Studio 进行一次覆盖全域的产品与架构级重构。

Studio 的目标不是做 OpenClaw 的 1:1 页面镜像，而是成为 OpenClaw 的高级管理平台与功能扩展层：允许重新组织信息架构、重新定义页面布局与交互方式、对底层能力做更高级的抽象与聚合，但所有抽象都必须完整覆盖 OpenClaw 的真实配置字段与能力边界，并在保存时准确回写到 OpenClaw 对应配置与能力结构。

本设计文档的目标是建立一份 Studio 总蓝图，作为后续分阶段开发、验证、提交和与 OpenClaw 版本同步升级的总依据。

## 2. 产品定位

Studio 的定位定义为：

- 面向 OpenClaw 全域能力的高级管理平台
- OpenClaw 的高级功能扩展层
- 以资源总览为首页入口的中控台
- 以平台化壳层承载共性能力、以模块化子应用承载领域能力的体系

Studio 对齐的是 OpenClaw 的真实能力与真实配置语义，而不是字段命名、页面结构或底层源码目录结构。Studio 可以对字段重新命名、分组、聚合和重构交互，只要这些高级抽象最终可以无损映射回 OpenClaw 的真实模型。

## 3. 设计原则

### 3.1 能力与语义对齐，而非字段名镜像

Studio 不需要做字段名级别的 1:1 镜像，但必须：

- 覆盖 OpenClaw 的真实字段与能力面
- 保证保存时准确回写
- 不因抽象造成配置语义丢失
- 不隐藏或削弱 OpenClaw 已有能力

### 3.2 平台壳层 + 模块化子应用

Studio 采用平台化壳层与模块化子应用结构，而不是单体统一控制台。平台壳层负责跨模块共性能力；各业务域作为相对独立的子应用存在，在统一设计系统、导航体系、状态体系和映射合同之下独立演进。

### 3.3 资源总览型首页

首页主入口采用资源总览导向，以系统状态、风险、待处理事项、关键资源聚合态、最近变化和推荐动作作为主视图，而不是仅做模块跳转页。

### 3.4 双核并重

Studio 同时承担两类职责：

- 运维与治理中枢
- 日常操作与工作台入口

因此顶层设计必须兼顾管理域与运行域，而不能只偏后台配置，也不能只偏实时运行态。

### 3.5 可维护、可替换、可升级

模块化不是导航分组层面的拆分，而是为了让后续维护、调整、替换、开发切换和版本升级同步都更方便。要求做到业务联动、代码解耦。

## 4. 总体架构

Studio 的总体架构分为两层：

### 4.1 平台壳层

平台壳层负责以下全局共性能力：

- 全局导航
- 全局搜索
- 命令面板
- 告警中心
- 待处理中心
- 最近变更流
- 全局运行状态栏
- 主题切换
- 语言切换
- 实例 / 环境上下文
- 统一布局系统
- 统一空态 / 错误态 / 加载态
- 跨模块资源跳转

平台壳层不承载具体业务配置逻辑，而是作为所有子应用共享的操作底座与视觉底座。

### 4.2 模块化子应用层

模块化子应用覆盖以下全域模块：

- Dashboard / 总览
- Config Center
- Agents Center
- Channels Center
- Skills Center
- Cron Center
- Chat / Sessions
- Room / Workflow
- System Center
- Terminal

每个子应用都应在统一壳层之下保持自身职责清晰、边界独立、可单独开发与替换。

## 5. 信息架构

顶层信息架构建议按用户任务组织为四类一级域组，而不是按底层源码结构组织：

### 5.1 总览

- Dashboard
- Alerts
- Pending
- Recent Changes

### 5.2 运营

- Chat / Sessions
- Room / Workflow
- Terminal

### 5.3 管理

- Config
- Agents
- Channels
- Skills
- Cron

### 5.4 系统

- System
- Runtime
- Logs / Events
- Diagnostics

这一组织方式面向用户日常任务：看状态、找异常、改配置、管理资源、推进流程、查看会话、执行运维介入。

## 6. 平台壳层设计

### 6.1 壳层职责

平台壳层不仅是页面外壳，而是 Studio 的统一操作底座，负责：

- 全局状态感知
- 全局资源发现
- 跨模块跳转
- 告警与待处理聚合
- 统一主题与 i18n
- 统一视觉语言与操作节奏

### 6.2 布局模型

建议采用“轻壳层 + 重内容区 + 右侧上下文面板可开启”的结构：

- 左侧：主导航轨道，承担一级域组切换与主入口
- 顶部：全局控制条，承载搜索、命令面板、环境上下文、状态摘要、主题与语言切换
- 中部：主内容区，承载首页与各子应用工作区
- 右侧：上下文侧栏，可折叠，承载告警、待处理、最近变更、当前资源上下文

目标是减少传统后台那种厚重卡片和边框切碎布局，提升整体流动性和高级感。

### 6.3 首页结构

首页不是等权卡片堆叠，而是运营总览画布，分为四层：

1. 全局态势：实例状态、Gateway / Runtime 状态、风险数量、待处理数量、配置差异摘要
2. 核心关注区：失败任务、失效渠道、异常 Skill、Cron 失败等
3. 资源总览区：Agents、Channels、Skills、Cron、Sessions、Workflows、Rooms 等聚合态
4. 最近变化与推荐动作区：配置变化、资源变更、建议下一步动作、快捷入口

## 7. 模块分层与职责

模块按职责类型分为三类：

### 7.1 配置型模块

以字段映射、保存回写、差异预览和校验为核心：

- Config Center

### 7.2 运行型模块

以实时状态、事件流、运行介入、流程推进为核心：

- Chat / Sessions
- Room / Workflow
- System Center

### 7.3 混合型模块

既具备配置治理，又具备运行态与操作能力：

- Dashboard
- Agents Center
- Channels Center
- Skills Center
- Cron Center
- Terminal

### 7.4 模块内部四层边界

每个子应用应拆成四层：

1. Module UI：页面、布局、组件
2. Module Application Layer：页面编排、交互状态、视图模型
3. Module Domain Contract：领域模型、资源定义、动作定义、筛选与搜索语义
4. Mapping / Gateway Adapter：与 OpenClaw 真实模型之间的映射与持久化层

这样可以支持后续模块级独立维护、独立重构、独立升级。

## 8. 重点模块定位

### 8.1 Config Center

Config Center 是 Studio 与 OpenClaw 真实配置之间的总映射中枢，不是简单表单页。它应承担：

- 统一配置总览
- 字段映射
- 配置分组编辑
- 原始配置与结构化配置双视图
- 差异检测
- 保存预览
- 回写校验

### 8.2 Agents Center

Agents Center 是混合型模块，负责：

- Agent 列表与详情
- Agent 配置编辑
- 默认模型与能力设置
- 生命周期操作
- 与 Sessions / Workflow / Skills 的关联查看

### 8.3 Channels Center

Channels Center 是连接资源管理中心，负责：

- Channel 列表
- 账号 / 连接 / 认证
- Channel 配置项
- 健康检查与状态异常
- 与 Routing / Skills / Sessions 的关联

### 8.4 Skills Center

Skills Center 是能力市场与运行依赖管理中心，负责：

- Skill 列表
- 安装状态
- 依赖与来源
- 分类与标签
- 配置与启停
- 与 Agents / Workflow / Cron 的绑定关系

### 8.5 Cron Center

Cron Center 是自动化任务控制中心，负责：

- 定时任务列表
- 任务定义
- 调度表达式
- 最近执行记录
- 失败诊断
- 关联资源跳转

### 8.6 Chat / Sessions

Chat / Sessions 是运行型模块，负责：

- Chat 会话
- Sessions 检索
- 历史恢复
- 中断补偿
- 会话状态追踪
- 与 Agent / Channel / Workflow 的上下文联动

### 8.7 Room / Workflow

Room / Workflow 是运行型协作域，定位为 Studio 的高级能力区之一。当前总蓝图只定义其模块位置和运行型属性。进入详细设计或实施前，必须先与用户进行一轮深入需求沟通，不得直接按默认理解定稿。

### 8.8 System Center / Terminal

System Center 负责：

- 系统状态
- Runtime 状态
- 健康检查
- 事件 / 日志
- 诊断入口

Terminal 负责：

- 运维入口
- 调试入口
- 快速执行
- 与资源上下文联动

System 偏观察诊断，Terminal 偏主动操作。

## 9. Studio 与 OpenClaw 的映射架构

这是整个 Studio 蓝图的技术底座。数据体系分为三层：

### 9.1 OpenClaw Source Model

OpenClaw 4.10 的真实模型，包括：

- 真实配置字段
- 真实 schema
- 真实资源结构
- 真实运行时状态
- 真实能力边界
- 真实 API / route / payload

这是唯一真相源。

### 9.2 Studio Domain Model

Studio 为实现高级抽象和更优交互而建立的领域模型。允许：

- 重命名
- 重分组
- 多字段聚合
- 一个 UI 面板映射多个底层字段
- 一个底层字段在多个视图中被不同语义解释

但不允许凭空发明无法回写的核心配置语义，也不允许抽象掉 OpenClaw 已有能力。

### 9.3 Mapping / Persistence Layer

映射层负责：

- OpenClaw -> Studio 的读取与映射
- Studio -> OpenClaw 的回写与持久化
- 差异比对
- 合法性校验
- 覆盖审计
- 升级兼容

它不应分散到各页面中，而应作为独立的架构层存在。

### 9.4 每个域的映射责任拆分

每个重要域都应具备四类职责：

- Source Adapter：从 OpenClaw 读取真实数据
- Domain Mapper：转换成 Studio 域模型
- Save Transformer：把 Studio 的修改回写为 OpenClaw 可接受的真实结构
- Diff Validator：进行差异预览、合法性校验、影响分析和遗漏检测

### 9.5 映射层应暴露为产品能力

Studio 应内建以下能力：

- 配置差异预览
- 原始视图 / 结构化视图双视图
- 未覆盖字段提醒
- 能力映射审计

## 10. 升级同步友好性设计

本次重构必须显式解决未来 OpenClaw 版本升级与 Studio 同步更新的便利性。Studio 架构必须对上游变化敏感、可盘点、可局部更新。

### 10.1 覆盖清单机制

每个域应有清单，记录：

- OpenClaw 当前真实字段 / 能力
- Studio 已覆盖内容
- 只读内容
- 可编辑内容
- 未接入内容

### 10.2 映射合同机制

每个模块应定义 mapping contract，明确：

- 数据来源
- 映射后的 Studio 模型
- 回写目标
- 组合映射字段
- 透传映射字段

### 10.3 能力差异检测机制

不只看字段，还看：

- 新资源
- 新操作
- 新状态源
- 新 schema / help / metadata

### 10.4 模块局部更新机制

上游升级后应尽量做到：

- Config 变更主要落在 Config mapping
- Agents 变更主要落在 Agents contract
- Channels 变更主要落在 Channels adapter
- 平台壳层与其他模块尽量不受牵连

### 10.5 固定同步工作流

后续 OpenClaw 升级时的建议流程为：

1. 拉取新版本源码
2. 盘点 schema / routes / docs / capability 变化
3. 更新覆盖清单
4. 标出受影响模块
5. 更新 mapping contract
6. 更新 UI / 交互 / 展示
7. 做整体验证与提交

## 11. 实施分期

### Phase 0：升级同步基线层

目标：建立长期同步基础设施。

产出：

- 模块覆盖清单
- 字段 / 能力映射清单
- mapping contract
- 版本差异盘点机制
- 受影响模块识别机制

### Phase 1：平台壳层与设计系统

目标：先替换 Studio 底座。

范围：

- 新导航系统
- 首页总览骨架
- 全局搜索 / 命令面板入口
- 告警 / 待处理 / 最近变更框架
- 主题 token
- i18n 基础结构
- 浅色 / 深色双主题基础
- 统一布局规范

### Phase 2：管理域重构

范围：

1. Config Center
2. Agents Center
3. Channels Center
4. Skills Center
5. Cron Center

关键要求：每个中心都必须同时解决字段覆盖、保存回写、差异预览和运行态关联。

### Phase 3：运行域重构

范围：

- Chat / Sessions
- Room / Workflow

重点：实时运行态、历史恢复、运行轨迹、跨模块联动。

注：Room / Workflow 在进入详细设计前必须先与用户做深入需求沟通。

### Phase 4：系统域与运维域重构

范围：

- System Center
- Runtime Status
- Logs / Events
- Diagnostics
- Terminal

重点：统一底层状态、事件、诊断与运维入口。

### Phase 5：全域联动与体验收口

范围：

- 跨模块资源跳转
- 全局搜索打通
- 命令面板打通
- 统一空态 / 错误态 / 加载态
- 中英文一致性
- 浅深色一致性
- 首页推荐动作与告警联动
- 视觉质感统一提升

## 12. 非功能约束

这些约束与功能同级，必须写入架构设计：

### 12.1 i18n 基础设施

不是后期翻译层，而是架构层能力。需要从一开始覆盖：

- 页面标题
- 简短标题
- 描述文案
- 状态术语
- 动作文案
- 搜索别名
- 空态 / 错误态文案

### 12.2 主题系统

不是事后皮肤，而是设计令牌系统。应包含：

- 基础 token
- 语义 token
- 组件 token
- 场景 token

必须从一开始支持：

- 浅色 / 深色
- 高质感视觉表达
- 优雅、克制、高密但不拥挤的界面气质

### 12.3 性能与数据加载策略

默认以快速数据加载和低缓存依赖为导向：

- 首页分层加载
- 首屏只拿关键状态
- 明细按需加载
- 重型模块延迟加载
- 长列表虚拟化
- 运行态增量更新
- 避免首页全域数据一次性全量拉取

### 12.4 并行开发能力

平台壳层、映射层、共享设计系统、共享资源协议和各中心模块应保持清晰边界，支持后续分轮、并行、模块化开发。

### 12.5 提交策略兼容性

架构边界应天然支持按轮次开发与提交。每轮开发应具备：

- 明确目标
- 独立验证范围
- 独立中文提交
- 不混合多个逻辑轮次

### 12.6 风险控制

禁止以下做法：

- 页面直接硬绑 OpenClaw 原始字段
- 用复制旧页面方式长期堆叠新版本
- 把主题 / i18n 放到最后补
- 在未与用户详细沟通前直接定稿 Room / Workflow 细节

## 13. 结论

本次 Studio 总体重构的目标是：把 Studio 建成一个面向 OpenClaw 全域能力的高级控制平台，以平台壳层承载共性能力，以模块化子应用承载领域深度操作，以映射层保证对真实配置与真实能力的完整同步、准确回写和可持续升级。

这份蓝图不是一次性全量重写的执行清单，而是后续分阶段实施、逐轮提交、持续同步 OpenClaw 版本升级的总设计依据。
