# Studio Chat Surface IM 化与当前会话记录检索设计

**日期：** 2026-04-11

## 目标

为 Studio Chat 完成一轮以可读性、稳定性和检索能力为核心的收口，重点解决以下问题：

1. 会话列表过于混乱，在 PC、iPad、手机三端都缺乏稳定的信息层级。
2. 待发送队列语义虽已基本形成，但 UI 仍不够清晰，且需要和主聊天线程彻底解耦。
3. 搜索和筛选职责混杂，用户难以判断“在找会话”还是“在搜聊天内容”。
4. 当前会话缺少 IM 风格的聊天记录入口，无法对完整历史做稳定检索和定位。
5. 加载、空态、错误态和响应式行为不统一，容易继续出现“白屏、空白、点不到、看不懂”的体验问题。

本设计只覆盖 Studio Chat surface 本身，不扩展到外部 IM 聊天记录导入，也不引入新的独立记录中心产品线。

## 已确认边界

- 左侧主视图采用 `全部 / 文件夹 / 归档` 三分区，不再把多种结构混在一个列表里。
- 会话列表要更像 IM 最近会话列表，但保留文件夹与归档能力。
- 左侧顶部搜索框默认只负责“找会话”，不承担消息全文搜索。
- 当前会话完整历史检索由聊天窗口内的“聊天记录”入口承载。
- 消息队列保持严格串行：上一条未结束，后续全部进入 FIFO 队列等待。
- 队列默认紧凑折叠展示，可展开编辑和删除。
- 已进入队列的消息必须立刻在 UI 中可见，而不是等真正发出后才出现。
- 聊天记录面板的检索条件刷新保留，Studio 重启不保留。
- 第一阶段不做跨所有会话全文搜索，不做外部 IM 记录导入，不做独立聊天记录中心页面。

## 约束

- 不破坏现有 Studio 自管 session 与 OpenClaw 宿主的会话安全边界。
- 不让附属面板打断主聊天线程、流式渲染、草稿输入或队列状态。
- 不继续把队列、历史列表、optimistic message、stream delta 混在同一份状态里。
- 手机端不能再依赖简单缩放解决布局问题，必须按交互层级重排。

## 方案选择

### 方案 A：在现有页面上继续打补丁

- 优点：实现快，改动面小。
- 缺点：会继续堆叠更多局部逻辑，列表、搜索、队列、记录浏览仍然缠在一起。
- 缺点：移动端和状态管理问题会继续反复出现。

### 方案 B：以 IM 会话流为主线，重构 chat surface 信息架构

- 优点：能同时收口会话列表、搜索、筛选、队列、聊天记录入口。
- 优点：符合用户真实心智，后续也能自然接入已接入 channel 的记录能力。
- 缺点：需要重新梳理前端状态边界和部分后端检索接口。

### 方案 C：先做独立“聊天记录中心”，再反向改 chat surface

- 优点：记录能力扩展性强。
- 缺点：会把当前最痛的 chat 主路径问题继续拖后。
- 缺点：在没有稳定会话列表与队列语义前，记录中心价值有限。

**结论：采用方案 B。**

## 总体信息架构

### 1. 左侧会话区重构为 IM 化主列表

左侧区域不再同时承担“管理台 + 筛选器堆叠区 + 最近会话列表”三种角色，而是回到更接近 IM 的主结构：

- 顶部：搜索框与少量高频入口。
- 主切换：`全部 / 文件夹 / 归档`。
- 主内容：最近会话列表或对应分区内容。
- 次级动作：筛选、批量管理、移动文件夹、重命名、归档等。

阅读优先，管理次之。默认态先让用户稳定看清“最近有哪些会话”，而不是先看到一排操作控件。

### 2. 搜索与历史检索彻底分层

系统中明确存在两种搜索：

- 会话级搜索：在左侧顶部搜索框内，目标是“找到某个会话”。
- 当前会话历史检索：在聊天窗口的“聊天记录”面板内，目标是“找到这段会话里说过什么”。

两种搜索入口、反馈、结果展示都不同，不共用同一个输入框和结果区。

### 3. 聊天窗口增加当前会话记录入口

聊天区内新增一个明确的“聊天记录”入口：

- 桌面端：放在聊天区动作栏，打开右侧记录抽屉。
- 平板端：优先使用覆盖式侧板。
- 手机端：放入底部动作栏，打开全屏或大底部面板。

该入口只针对当前会话，不跳出 chat 主路径。

### 4. 队列保持为聊天窗口的附加层

待发送队列不进入左侧 rail，也不侵入消息主线程，而是保持在 composer 上方的独立“待发送队列栏”：

- 默认折叠，显示队列数量、首条摘要、等待原因。
- 展开后显示完整队列项。
- 编辑、删除都在这里完成。

队列是当前聊天行为的补充层，不是第二个消息列表。

## 会话列表设计

### 1. 三分区主结构

左侧主切换固定为：

- `全部`
- `文件夹`
- `归档`

含义如下：

- `全部`：按最近活跃时间排序的 IM 风格最近会话流。
- `文件夹`：仅展示文件夹结构与其中会话，不和最近会话流混排。
- `归档`：单独展示已归档会话。

这能避免一个列表既像时间流、又像文件管理器、又像筛选结果页的混乱状态。

### 2. 列表项结构

每条会话行默认只强调高价值信息：

- 头像
- 标题
- 一行摘要预览
- 最近时间
- 轻量状态标记

会话行不默认摊开复杂管理动作。管理动作统一收进尾部菜单或“管理模式”。

### 3. 管理模式与移动端策略

桌面端：

- 更多菜单承担移动文件夹、重命名、归档等次级操作。
- 批量操作仅在显式进入管理模式后出现。

移动端：

- 以长按和底部动作面板为主。
- 避免在窄屏上同时显示行内按钮、筛选入口和菜单堆叠。

## 搜索与筛选设计

### 1. 左侧顶部搜索只搜索会话元数据

搜索范围限定为：

- 会话标题
- 最近一条摘要预览
- Agent
- 来源频道或来源标签
- 文件夹
- 归档状态
- 最近活跃时间相关标签

它不搜索完整消息历史，也不跨会话全文检索。

### 2. 筛选独立于搜索

筛选面板负责结构化条件，例如：

- Agent
- 文件夹
- 归档状态
- 来源频道
- 最近活跃时间

搜索和筛选结果共同作用于会话列表，但职责明确：

- 搜索：快速缩小范围。
- 筛选：精确定义范围。

### 3. 结果反馈

搜索和筛选结果区必须始终显示：

- 当前结果数
- 已启用的筛选 chips
- 一键清空搜索或筛选入口

无结果时给明确空态，例如：

- 清空筛选
- 返回全部会话
- 新建会话

不允许出现“看起来搜了，但不知道搜的是什么范围”的情况。

## 消息队列设计

### 1. 严格串行语义

当当前 session 已有活动 run 时：

- 后续消息全部进入待发送队列。
- 上一条未彻底结束前，不允许下一条真实发出。
- 当前条目结束后，队列头部自动发出。

队列发送始终为 FIFO，一次只放行一条。

### 2. 紧凑折叠的队列栏

composer 上方显示独立队列栏：

- 无队列时不显示。
- 有队列时默认折叠。
- 折叠态显示：`待发送 N 条`、首条摘要、当前等待原因。
- 展开态显示全部队列项。

### 3. 队列项可见性

已进入队列的消息必须立即在 UI 中存在：

- 队列栏内可见。
- 主线程中可通过轻量“待发送”映射提示建立感知。

禁止再次出现“用户发了消息，但看起来像没发出去也没排队”的感知断裂。

### 4. 队列项操作

第一阶段支持：

- 编辑
- 删除

第一阶段不支持：

- 拖拽排序
- 批量移动顺序

这样可以保持状态机简单、可预测。

## 当前会话聊天记录面板设计

### 1. 定位

聊天记录面板是“当前会话的完整历史浏览与检索工具”，不是第二个聊天窗口，也不是全局记录中心。

### 2. 能力范围

第一阶段支持：

- 当前会话完整历史全文检索
- 按时间分组浏览
- 按角色筛选
- 按内容类型筛选
- 跳转到原消息
- 复制结果片段
- 导出当前会话

第一阶段不支持：

- 跨会话全文搜索
- 跨来源统一档案中心
- 批量删除记录
- 外部 IM 本地记录导入

### 3. 结果展示

结果按时间分组展示，每条结果至少包含：

- 匹配片段
- 角色信息
- 时间
- 跳转入口

跳回主线程后：

- 自动滚动到原消息附近
- 对目标消息做短暂高亮

### 4. 状态保留

聊天记录面板中的以下状态在刷新后保留：

- 最近一次关键词
- 最近一次筛选条件
- 当前结果集定位

Studio 重启后不保留。

## 状态系统与错误恢复

### 1. 四类基础状态

以下区域都必须支持统一的四类状态：

- 会话列表
- 聊天主线程
- 队列栏
- 聊天记录面板

状态类型：

- `loading`
- `empty`
- `error`
- `ready`

### 2. 渐进渲染

UI 不等待全部数据准备好再渲染整体框架：

- 会话列表先渲染骨架和基础壳体。
- 聊天记录面板先打开容器与搜索框，再填充结果。
- 队列栏先显示局部状态，再更新具体条目。

这样可以避免再次出现整块空白或白屏。

### 3. 局部失败局部降级

错误恢复规则：

- history 拉取失败，只影响聊天记录面板，不影响主聊天。
- 队列接口失败，保留本地待发送草稿并允许重试。
- 会话列表筛选失败，自动回退到无筛选列表。

原则是“局部坏了，局部降级”，不能让单点失败拖垮整个 chat surface。

## 响应式策略

### 1. 桌面端

- 左侧会话栏固定显示。
- 聊天记录使用右侧抽屉。
- 队列展开在 composer 上方内联展示。

### 2. 平板端

- 左侧栏保留，但收窄密度。
- 聊天记录优先使用覆盖式侧板。
- 筛选与管理操作进入更集中、更易点击的面板。

### 3. 手机端

- 顶部只保留必要信息，不继续堆叠操作按钮。
- 会话列表使用全屏侧板或底部面板进入。
- 筛选使用全宽底部面板。
- 聊天记录使用全屏或大底部面板。
- 队列展开也使用底部 sheet，而不是继续挤压消息区。

这不是单纯缩放，而是按交互层级重排。

## 前端状态边界

前端状态拆为三块：

### 1. `session list state`

负责：

- 会话列表数据
- `全部 / 文件夹 / 归档`
- 会话维度搜索
- 筛选条件

### 2. `chat runtime state`

负责：

- 当前 session 消息流
- 流式回复状态
- composer 草稿
- 队列状态
- 会话级控制项

### 3. `record browser state`

负责：

- 当前会话完整历史检索
- 检索条件
- 检索结果集
- 跳转定位状态

`record browser state` 不直接复用当前已加载消息列表，避免和 optimistic message、stream delta、分页历史互相污染。

## 后端接口边界

### 1. 会话列表接口

会话列表接口继续承担：

- 会话元数据列表
- 文件夹关系
- 归档状态
- 会话维度搜索与筛选所需字段

### 2. 队列接口

延续现有 session 级队列接口，保证：

- 当前 session 队列可查询
- 队列项可编辑
- 队列项可删除
- 自动续发状态可被前端感知

### 3. 当前会话 history 检索接口

新增或扩展专用 history 查询能力，要求：

- 查询基于完整 history，而不是当前已加载的少量消息
- 支持关键词
- 支持角色筛选
- 支持内容类型筛选
- 返回稳定可跳转的消息定位信息

这部分必须独立于当前聊天消息流接口。

## 第一阶段交付范围

### 必做

- 会话列表重构为 `全部 / 文件夹 / 归档`
- 左侧搜索与筛选职责拆分
- 待发送队列栏紧凑化与可展开编辑
- 当前会话聊天记录入口与完整历史检索面板
- PC / iPad / 手机三档响应式重排
- 加载、空态、错误、重试的统一状态系统

### 明确不做

- 跨所有会话全文搜索
- 外部 IM 记录导入
- 队列拖拽排序
- 记录批量删除
- 独立聊天记录中心页面

## 测试与验收

### 1. 行为验收

- 当前有活动 run 时，连续发送多条消息，全部进入队列并严格串行发送。
- 已入队消息立即可见，且可编辑、删除。
- 左侧搜索只影响会话列表，不影响聊天记录面板结果。
- 聊天记录面板可以检索当前会话完整历史，而不是仅检索当前已加载消息。
- 从聊天记录面板跳回主线程时能稳定定位并高亮目标消息。

### 2. 响应式验收

- PC、iPad、手机三端下，会话列表、筛选面板、队列展开、聊天记录面板均可操作。
- 手机端不再出现顶部按钮堆叠、无法点击筛选项、面板超出屏幕、主消息区被过度压缩的问题。

### 3. 状态验收

- 任一区域加载失败时，整体页面仍然可用。
- 任一区域空数据时，提供清晰空态与下一步动作。
- 刷新后保留队列与当前会话记录检索状态；Studio 重启后不保留这些临时状态。

## 风险与后续扩展

### 1. 当前风险

- 如果 history 检索仍复用现有 message list 状态，记录面板将继续出现结果不完整和定位不稳定。
- 如果移动端继续依赖桌面布局缩放，筛选、列表和记录面板的可点击性问题会反复出现。

### 2. 后续扩展方向

本设计为后续能力预留接口，但不在第一阶段实现：

- 跨会话全文搜索
- 已接入 channel 的统一记录浏览
- 会话记录中心
- 归档与文件夹的更完整批量管理
