# Model Gateway Detailed Goal

> Updated: 2026-06-22
> Status: active execution goal
> Owner surface: `/model-gateway`
> Design authority: `docs/prototypes/pages/model-gateway.html`, `docs/prototypes/Aurora设计体系.md`, `docs/模型网关信息架构契约.md`

## Goal Statement

把 Model Gateway 做成 Tracevane 的模型路由控制中心，而不是 OpenClaw 管理后台、Provider CRUD 表格、配置字段仓库或低价值概览页。

最终状态必须满足：

- 用户能清楚知道当前 Codex、Claude Code、OpenCode、OpenClaw 或默认客户端实际会路由到哪个 Provider、endpoint 和模型。
- 用户能以清晰、低风险、可回滚的方式创建 API Provider、登录 Codex 账号、编辑 Provider、管理 endpoint、管理模型 alias、管理账号池、应用或回滚客户端接入。
- 用户能从状态、检查结果、确认弹窗和结果证据里判断一次操作是否成功、失败、可重试或需要进入其它域处理。
- 页面视觉、布局、组件语义、密度和交互必须严格服从 Aurora 原型；缺数据时使用空态、占位或后续任务，不允许改成自创设计。
- 任何写操作都必须通过后端 Gateway 管理 API，不允许前端直接写本地客户端配置、直接暴露明文密钥、静默生成不可回滚状态或伪造成功。

## Hard Requirement: Overview Must Not Create

概览页只负责状态、路由事实、风险提示和下钻导航，不承担创建入口。

明确禁止：

- 不在概览页放 `新建 Provider`、`登录 Codex`、`新增 endpoint`、`新增模型`、`新增账号` 这类创建按钮。
- 不在概览页打开 Provider 创建表单。
- 不在概览页直接写客户端配置。
- 不在概览页做删除、停用、清冷却、回滚等危险动作。
- 不用概览页承载“创建向导”或“全功能控制面板”。

概览允许：

- 展示 Gateway 当前状态。
- 展示 active routes。
- 展示未应用客户端接入、Provider 异常、路由异常等待处理项。
- 跳转到 Provider、客户端接入、账号池或模型页。
- 对已解析的 Provider 做只针对当前路由的检查动作，前提是后端已有明确测试 API 和结果证据。

## Product Boundary

Model Gateway owns:

- Provider registry。
- Endpoint profiles。
- Model catalog and alias。
- Account-backed Provider account pool。
- Route scopes and active provider/model resolution。
- Client App Connection for applying Gateway routing into local clients。
- Gateway usage summary by model and route。

Model Gateway does not own:

- Agent workspace, persona, permission, native session or task runtime. These belong to CLI Agents.
- IM channel binding, message replay or transport delivery. These belong to IM Channels / Channel Connectors.
- OpenClaw generic config, plugin, skill, agent, channel or host doctor CRUD. These belong to Platform / OpenClaw.
- File editing, Git staging, terminal launch or preview-time source edit. These belong to Workspace IDE.
- Generic external OAuth/MCP management. These belong to External Connections or Platform support.

If a workflow crosses domains, Model Gateway must show a clear link or route handoff instead of duplicating the other domain's controls.

## Primary Views

### 1. Overview

Purpose:

- Answer "what routes where now?"
- Answer "what needs attention now?"
- Provide navigation into the owning child flow.

Required content:

- Gateway availability and listener/status timestamp from live API.
- Active route rows: scope, resolved Provider, endpoint, model, state, check action when supported.
- Attention rows: route warning, Provider problem, client connection not applied.
- Client connection summary with a management entry into the App Connection child page.

Forbidden content:

- Decorative KPI cards that do not change the user's next action.
- Hard-coded prototype incidents.
- Fake health failures.
- Global OpenClaw service status unless directly relevant to Gateway routing.
- Create Provider or login account entry points.

### 2. Provider

Purpose:

- Be the primary object list for Provider management.
- Let users find a Provider, check it, configure it, or enter account pool when applicable.

Required content:

- Header title and short help text in Chinese.
- Primary actions area: `登录 Codex`, `新建 API Provider`.
- Tool area: search, filter, `探测选中 Provider`.
- Provider table rows with identity, default model/count summary, type, status and common actions.
- Row actions: `配置`, `检查`, `账号池` only for account-backed Providers.

Forbidden content:

- Permanent right-side inspector.
- Protocol/source/latency/date field dumps in the row.
- Delete button in row.
- Endpoint/model full editing inside list row.
- Account pool controls for non-account Providers.

### 3. Models

Purpose:

- Show all Provider model entries in one place.
- Let users inspect capability/usage and edit alias safely.

Required content:

- Search by model id, alias and Provider.
- Model rows with model identity, Provider, capability summary and usage.
- Inline alias edit that writes back to the owning Provider catalog.
- Capability/price panel for selected model.

Forbidden content:

- Provider base URL/auth editing.
- Account login.
- Large raw JSON editor.
- Bulk mutation across hidden rows without explicit scope and confirmation.

### 4. Usage

Purpose:

- Show current model request and token totals.
- Show lightweight distribution and latency evidence.

Required content:

- Request count.
- Total/input/output token counts.
- Distribution by model.
- Latency summary when available.

Forbidden content:

- Reintroducing old archive, pagination, CSV, cost estimation, provider/account filters or recent request detail unless a new product goal explicitly reopens that scope.

## Child Flows

### Provider Config

Entry:

- Provider row `配置`.
- Provider issue row `处理`.
- Provider page `新建 API Provider`.

Sections:

- `基础`: id, name, category, source type, baseUrl, API format, auth strategy, enabled, API key input.
- `Endpoint`: endpoint id, name, baseUrl, API format, auth strategy, enabled toggle, add/remove endpoint.
- `模型`: model id, alias, default model, add/remove model.
- `高级`: timeout, first byte timeout, idle timeout, proxy, TLS verify, reasoning/thinking, metadata, app scopes, danger zone.

Rules:

- Create mode starts from an empty draft.
- Edit mode starts from selected Provider data.
- Advanced section is not visible on initial open.
- API key is write-only from frontend perspective; edit mode empty input means no secret change.
- Delete appears only in advanced danger zone and requires dialog confirmation.
- Save validates id, name, baseUrl, endpoint, model list, duplicate model id, duplicate alias and dangling default model before mutation.

### Codex Account Login

Entry:

- Provider page `登录 Codex`.
- Account pool `登录新账号`.

Rules:

- It is not an API Provider create form.
- It starts `/api/model-gateway/account-providers/codex/login/start`.
- It must show verification URL, user code and login id in evidence sheet.
- It must not reuse a selected ordinary API Provider as Codex account target.
- After login, user refreshes or returns to account pool to inspect state.

### Account Pool

Entry:

- Account-backed Provider row `账号池`.

Rules:

- Only account-backed Providers can enter.
- Show masked account identity, plan/state, last success, expiry/cooldown and enabled state.
- Actions are refresh, clear cooldown, enable/disable.
- Removing accounts requires a future confirmation flow; do not add casual row delete until designed.

### Client App Connection

Entry:

- Overview `客户端接入`.

Ownership:

- Belongs to Model Gateway because it applies Gateway route config into Codex, Claude Code, OpenCode and similar local clients.
- Does not own Agent workspace/persona/permissions/session.

Rules:

- Show client, target config path, selected route profile/model/protocol, configured state and actions.
- Selecting a client opens an inline App Connection workbench instead of the generic evidence sheet.
- The workbench must show target path, target exists/format, endpoint, protocol, model, launch hint, last backup, blocking issues and a redacted config preview.
- The workbench can save the per-client model override through `/api/model-gateway/app-connections/profile`; saving the profile does not write the client config file.
- The workbench can copy the target path, refresh evidence, jump to Provider configuration and jump to the model catalog.
- `应用` opens confirmation dialog and must mention target file and backup behavior.
- `回滚` opens confirmation dialog and must mention backup/restore behavior.
- Successful apply/rollback reports result evidence inline and refetches app connection data.
- The generic sheet is reserved for long external previews elsewhere; App Connection preview itself must stay in the Gateway page context.
- Frontend never writes client config files directly.

## Allowed Work

The implementation may:

- Refactor Model Gateway page state and components to match this IA.
- Add small local components only when they reduce repeated JSX and preserve Aurora styling.
- Add backend API fields if the current frontend needs real state that cannot be safely inferred.
- Add system tests for page source contracts, mutation guards and route ownership.
- Add Playwright smoke checks for desktop and mobile.
- Add empty/loading/error states for every data area.
- Add confirmation sheets/dialogs around writes.
- Add backend validation to prevent malformed Provider, endpoint, model or app connection mutations.

## Forbidden Work

The implementation must not:

- Restore a permanent Provider side inspector.
- Put every Provider field into the Provider list or initial config screen.
- Put creation flows on Overview.
- Create a separate top-level Client App Connection route unless a later IA decision moves it out of Model Gateway.
- Duplicate CLI Agents runtime controls.
- Duplicate OpenClaw platform management.
- Add new dependencies without explicit approval.
- Expose secret plaintext in list/query responses.
- Write local client config from the browser.
- Make optimistic UI changes for write actions unless rollback is explicitly implemented.
- Claim a workflow is complete based only on static source tests.
- Use hard-coded fake Provider failures as live-looking status.

## Detailed Execution Tasks

### Phase A: IA Lock And Regression Contracts

- Keep `docs/模型网关信息架构契约.md` and this document aligned.
- Add tests that fail if primary viewbar exceeds four views.
- Add tests that fail if Overview contains create/login buttons.
- Add tests that fail if Provider row contains delete controls.
- Add tests that require Provider config section navigation.
- Add tests that require App Connection to be a child flow.
- Add tests that require destructive writes to call `openDialog`.

Acceptance:

- Source contract tests pass.
- Docs describe the same ownership as the code.

### Phase B: Provider Workbench

- Keep Provider header split into primary actions and tool actions.
- Make search/filter predictable and mobile safe.
- Keep row fields minimal.
- Ensure account Providers expose account pool and ordinary Providers do not.
- Ensure selected Provider state is only used for row/action context, not as an always-visible inspector.
- Ensure `探测选中 Provider` never acts without selected Provider.

Acceptance:

- Desktop and mobile show Provider list without overflow.
- Row actions are visible and do not wrap into unreadable fragments.
- No raw protocol/source/latency/date dump appears in Provider rows.

### Phase C: Provider Create/Edit

- Keep create and edit as explicit draft states.
- Add or preserve validation for required fields and model catalog invariants.
- Ensure section navigation starts at `基础`.
- Ensure `Endpoint`, `模型`, and `高级` sections are reachable.
- Ensure delete stays in advanced danger zone.
- Ensure save refetches Provider, status, usage and app connection data as needed.

Acceptance:

- Creating an API Provider does not inherit selected Provider health, endpoint evidence or delete action.
- Editing a Provider does not require re-entering API key unless secret changes.
- Empty/invalid drafts fail before API call.

### Phase D: Codex Account And Account Pool

- Keep Codex login separate from API Provider creation.
- Ensure login start targets an account-backed Codex provider or creates/uses the Codex account provider id.
- Show login evidence in sheet.
- Keep account pool actions scoped to selected account Provider.
- Add confirmation before any future account removal.

Acceptance:

- Ordinary Provider selection cannot hijack Codex login.
- Account pool is hidden for non-account Providers.

### Phase E: Client App Connection

- Keep App Connection as Overview child page.
- Show configured/pending counts.
- Show target path and route profile.
- Ensure preview/apply/rollback use existing backend APIs.
- Ensure apply/rollback dialogs mention target path and backup/restore.
- Ensure result evidence sheet opens after mutation.

Acceptance:

- User can preview before write.
- User can cancel write.
- User can see result evidence after write.
- Frontend does not write any local config file directly.

### Phase F: Overview Recut

- Remove all low-value KPI cards.
- Show only route status, attention and navigation.
- Keep route check actions tied to real resolved Provider.
- Avoid creating fake incidents when backend data is missing.
- Add empty states for no routes, no attention and no client connection data.

Acceptance:

- Overview can explain current routing without offering create controls.
- It never looks operational based on hard-coded sample failures.

### Phase G: Backend/API Support

- Add backend fields only when needed to remove unsafe frontend inference.
- Keep Provider write endpoints validating shape and secret handling.
- Keep App Connection write endpoints responsible for backup/rollback.
- Keep usage endpoint minimal unless a new goal reopens detailed usage.
- Keep route status aligned with actual request routing.

Acceptance:

- Frontend does not infer hidden backend state in a way that can mislead the user.
- Backend tests cover added contracts.

### Phase H: Verification And Release Gate

Required before claiming Model Gateway frontend complete:

- `npm run typecheck:web`
- `npm run typecheck:api`
- `npm run build:web`
- `npm run build:api`
- Targeted React Aurora Model Gateway system test.
- Targeted Model Gateway backend/service tests when backend changed.
- `git diff --check`
- Dev restart.
- Live `/api/model-gateway/providers`, `/status`, `/usage`, `/app-connections` checks.
- Playwright desktop smoke.
- Playwright mobile smoke around 390px width.
- Manual or automated check that Overview does not contain create buttons.
- Manual or automated check that Provider row does not contain delete.

## Completion Definition

The detailed Model Gateway goal is complete only when:

- The IA document, detailed goal document, architecture doc and progress doc agree.
- Overview is status/navigation only.
- Provider list is simple and usable.
- API Provider creation works through Provider child flow.
- Codex account login works through dedicated account flow.
- Provider edit supports basic, endpoint, model and advanced sections.
- Provider delete is confirmation-backed in danger zone.
- Account pool works only for account Providers.
- Client App Connection preview/apply/rollback are evidence-backed.
- Models page alias editing writes real Provider catalog data.
- Usage page stays minimal and does not resurrect retired usage features.
- Desktop and mobile smoke pass.
- Source, typecheck, build and targeted system tests pass.

Until every item above is true, Model Gateway is not considered fully done; it is only partially aligned.
