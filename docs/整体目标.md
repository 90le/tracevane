# Tracevane 整体目标

> 状态：Active baseline
> 更新：2026-06-25
> 目的：统一推翻旧前端后重新建立的产品目标，避免 IDE、文件管理、终端、CLI Agent、Gateway、IM、Platform 再次混成一个半成品控制台。

## 1. 一句话目标

Tracevane 是一个 **本地优先的 AI Agent Workspace 与连接控制层**。

它不是通用 OpenClaw 后台，不是普通文件管理器，也不是把终端、聊天、Provider 表单堆在一起的 admin console。它要让用户在一个项目工作区里完成：

```text
选择上下文 → 编辑/运行/预览 → 调用 Agent → 审查 diff/命令/证据 → 验证 → 回滚或交付
```

## 2. 产品支柱

| 支柱 | 作用 | 归属 |
| --- | --- | --- |
| Workspace | 本地项目工作区：文件、编辑器、预览、终端、Git、证据、Agent handoff。 | 核心主入口 |
| Model Gateway | Provider、模型、协议、账号池、客户端接入、用量与路由证据。 | 连接层 |
| IM Channels | IM 平台账号、bot 凭据、绑定路由、投递会话、守护诊断。 | 连接层 |
| CLI Agents | Codex / Claude Code / OpenCode 等 CLI Agent runtime readiness、配置状态、Agent Run 生命周期和结果证据。 | Agent 运行层 |
| Platform | 第三方平台目录与平台原生管理，OpenClaw 是第一个平台。 | 系统/平台层 |
| Agent 会话 | 统一承载 Web / IM / CLI / OpenClaw / 未来平台触发的所有 Agent 对话、工具事件、审批和证据，可 handoff 到 Workspace。 | 工作层 |

## 3. 最重要的边界

### 3.1 CLI Agent 不是终端

必须严格区分：

| 概念 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| Terminal | shell session、cwd、命令输入输出、task runner、dev server、ports、terminal evidence。 | Codex/Claude/OpenCode runtime 管理。 |
| CLI Agents | CLI Agent 安装/版本/配置/readiness、Gateway 连接状态、Agent Run 生命周期、Agent 结果证据。 | 普通 PTY、shell tab、终端输入、终端 split/delete/rename。 |

Workspace 可以把 terminal 输出作为 Agent 上下文或证据，但普通 terminal session **不等于** CLI Agent Run。

### 3.2 文件管理属于 Workspace

新版本不再让 `/files` 与 `/ide` 平级抢职责。

- 高频文件管理：Workspace Files / Explorer / File Manager。
- 审计型文件材料：Evidence / Artifacts。
- 如果保留 `/files`，它只能是证据/Artifacts 入口，不能再以“文件管理器”名义只读。

### 3.3 Workspace 是项目工作面，不是 VS Code 仿品

Tracevane 不追求复刻完整桌面 IDE。它追求比普通 IDE 更懂 Agent 工作流：

- 文件和 diff 是 Agent 修改的审查对象。
- 预览、终端输出、日志、截图是 Agent 上下文和验证证据。
- 高风险写入必须可审查、可回滚、可追踪。

## 4. 目标用户循环

### 4.1 本地项目修改循环

1. 打开 workspace/root。
2. 浏览或搜索文件。
3. 编辑文件或查看 diff。
4. 运行 task/terminal。
5. 预览结果和问题。
6. 保存证据。

### 4.2 Agent 协作循环

1. 选择文件、代码片段、终端输出、预览截图、Git diff 或 IM 消息作为上下文。
2. 进入统一 Agent 会话，选择 Agent / 模型 / 工作目录 / 权限。
3. CLI Agents 记录 Agent Run 状态；模型网关提供模型路由；IM/平台只作为来源标签和投递证据。
4. Workspace 审查 Agent plan、patch、命令和验证结果。
5. 用户 approve/reject/apply/revert。

### 4.3 IM 触发循环

1. IM Channels 收到消息。
2. Binding route 选择 Agent runtime 和默认目录。
3. CLI Agent 执行任务。
4. 结果证据回到 IM 和 Workspace/Evidence。

## 5. 响应式目标

Tracevane 必须同时是 PC 专业工作台和手机任务控制器。

| 设备 | 形态 | 目标 |
| --- | --- | --- |
| PC / Wide | 多栏 IDE：Explorer + Editor + Inspector + Bottom Panel。 | 高密度、可长时间工作、可 resize、快捷键/右键完整。 |
| Tablet | 双栏/抽屉混合：主舞台 + 可抽屉侧栏。 | 保持操作完整，减少同时可见区域。 |
| Mobile | 单栏任务流：Files / Editor / Terminal / Agent / Evidence 模式切换。 | 无横向溢出，所有 hover/右键能力有触摸替代。 |

## 6. 当前问题判断

现状是半成品的原因不是“缺几个按钮”，而是产品模型没有重建完成：

- `/ide` 像 VS Code 壳，但 Agent / Search / Problems / Output / Diff Review 都未闭环。
- `/files` 名为文件，实际只读证据，和设计目标冲突。
- 终端首屏会暴露 stale session 错误，缺少稳定 runtime 设计。
- CLI Agents 与 terminal 语义曾经混淆，必须重新切开。
- 旧三域文档不再能表达 Workspace-first 目标。

## 7. 重设计判定

本轮 Workspace/IDE/文件管理不是现有 UI 的增量美化，而是对现有错误模型的替换。后续文档和实现必须优先遵守：

1. 文件管理、编辑/预览、终端、CLI Agents 四个 owner 分区。
2. 文件管理不能依赖终端表达核心能力；终端不能承担文件管理生命周期。
3. CLI Agent 不是终端；普通 PTY 不是 Agent Run。
4. 窗口系统必须支持打开、收起、折叠、拖拽 resize、布局记忆。
5. Markdown 是 Editor Stage 内的一等编辑对象，必须支持编辑/预览/源码模式。
6. 历史前端实现不作为架构迁移权威；当前原型与用户任务优先。

详见 `Workspace重设计总纲.md`。

## 8. 成功标准

1. 用户 3 秒内知道当前页面主对象是什么。
2. PC 无横向溢出；移动端无被挤压多栏。
3. 文件操作真实可用，不靠“去另一个页面”的说明文字。
4. 终端 session 可恢复、可重连、可清理 stale、可移动端使用。
5. Agent 改动必须形成 plan/diff/命令/验证/证据链。
6. CLI Agents 不管理普通 terminal；Terminal 不伪装 Agent Run。
7. 文档、导航、后端 API 和前端页面使用同一套边界语言。
8. OpenClaw 不拥有独立对话中心；所有人机对话统一进入 Agent 会话。
