# Tracevane Agent 会话 Design

> 状态：运行态参考
> 更新：2026-06-25
> 规则：本文只保存当前 Chat 架构结论；历史性能/布局细节看 git。

## 1. 定位

Tracevane Agent 会话是 Tracevane 统一的人机交互层，承载 Web、IM、CLI、OpenClaw 平台和未来平台触发的所有 Agent 会话。它不是 OpenClaw 专属对话页，不是 IM 投递页，也不是 Gateway raw protocol 的浏览器客户端。

```text
Web / IM / CLI / Platform
  -> Tracevane typed Agent Session API / WS
  -> Tracevane adapter
  -> Codex / Claude Code / OpenCode / OpenClaw / custom agent
  -> Model Gateway where model routing is needed
```

## 2. 当前决策

- 浏览器不直连 OpenClaw Gateway。
- OpenClaw 不拥有独立对话中心；OpenClaw Agent 只是 Agent 会话的一种来源/运行时。
- IM 渠道不复刻聊天 UI，只把消息投递到 Agent 会话并显示投递证据。
- 前端不消费 Gateway raw frame。
- Tracevane 后端负责把 Gateway / OpenClaw 事件归一成 typed contract。
- `tracevane_managed` 会话可写；`observed_external` 默认只读；`system_internal` 不进普通用户工作区。
- 会话 UI 采用 `Tracevane前端重设计原型.md` 的 Primary Stage + Context Strip + Runtime Inspector Drawer，不使用固定三栏。
- 会话列表行只承担“打开会话”的主动作；管理动作通过右键上下文菜单和行尾更多菜单承载，避免把重命名/归档/删除铺成列表按钮。
- 会话管理分层：默认显示可管理会话，只读/外部观察会话用独立标签进入；organizer 文件夹在列表内以树状行展示，支持展开/折叠、多级目录、右键菜单、文件夹 CRUD、文件夹移动/排序和会话移动。
- Chat 保留单人模式优先，不引入多人/群聊持久事件总线。
- 统一 Agent 会话不承载旧 OpenClaw WebChat 的主机管理控制页签；不同 Agent 的权限/审批应在各自运行器或具体风险动作中处理，不在 Chat 默认拦截。
- Chat 合同不再提供 per-session `controls`、`session.controls` stream event 或 `/api/chat/sessions/:key/controls`；旧 host-exec 权限属于 OpenClaw/具体运行器安全域，不属于统一 Agent 会话。
- Chat 合同不再提供 `/api/chat/sessions/:key/slash-gateway` 或 `requestSlashGateway` 这类 OpenClaw 管理代理；模型列表、技能状态、OpenClaw 原生配置、exec approvals、session patch/compact 等平台管理能力迁入 Platform/OpenClaw API。
- 不引入 Clawith 式独立数据库会话模型。

## 3. 前端边界

当前主实现集中在：

- `apps/web/src/features/chat/ChatWorkbenchPage.tsx`
- `apps/web/src/features/chat/views/SessionListView.tsx`
- `apps/web/src/features/chat/views/ConversationView.tsx`
- `apps/web/src/features/chat/views/RuntimeInspectorView.tsx`
- `apps/web/src/features/chat/**`

要求：

- 保持单目录边界。
- 不恢复版本化/平行 chat 前端目录。
- Composer、Session Rail、Conversation Pane、Inspector、Record Browser、Slash UI 和富消息渲染都走同一 Chat Shell。

## 4. 后端边界

### 4.1 Chat 保留的通用能力

当前后端范围：

- `/api/chat/health`
- `/api/chat/bootstrap`
- sessions list/create/patch/delete/reset
- history page/search
- send/abort/queue
- upload/resource/media resolve
- composer fileRefs send
- stream / websocket replay

后端职责：

- 做 typed adapter / BFF。
- 维护 session policy、权限、runtime snapshot 和错误映射。
- 将 raw tool / agent / chat 事件归一为前端可消费事件。

### 4.2 Chat 必须删除/迁出的旧 OpenClaw 能力

这些能力来自旧 OpenClaw WebChat，不属于统一 Agent 会话；如果继续留在 Chat，会导致后续 Codex / Claude Code / OpenCode / 其它平台混成 OpenClaw 语义：

| 能力 | 处理方式 | 归属 |
| --- | --- | --- |
| per-session host-exec controls | 已从 Chat 合同、前端 query、stream、持久状态和 session-state SQLite schema 中删除；旧 schema 启动时迁移为 queue-only | OpenClaw/具体 runtime 安全域 |
| `/api/chat/sessions/:key/slash-gateway` | 已删除 Chat 路由和 ChatService 方法 | `/api/platforms/openclaw/gateway` |
| exec approvals get/set/node set | 不在 Chat 运行详情中全局暴露 | OpenClaw 平台配置/安全页面或具体工具调用确认 |
| models.list / skills.status / agents.list / config.schema.lookup | 不作为 Chat slash/raw proxy 暴露 | Platform/OpenClaw 原生管理，或未来 Runtime Adapter typed API |
| sessions.patch / compact / steer | Chat 只保留 typed reset/abort/send/queue；OpenClaw 原生命令在平台域处理 | Platform/OpenClaw gateway proxy |

迁移原则：Chat 只接收“用户要和哪个 Agent 会话说什么、附件是什么、运行状态是什么”；平台管理、原生配置、权限策略和 raw RPC 都不放进 Chat。


## 4.3 Runtime Adapter 边界

Chat 后端现在必须先走 `ChatRuntimeAdapter`，再进入具体运行器。当前默认 adapter 仍然封装现有 OpenClaw Gateway `chat.send`，行为不变；但 `performDirectSend` 不再直接把 OpenClaw raw response 当成 Chat 业务合同。

```text
ChatService.performDirectSend
  -> ChatRuntimeAdapter.send(input)
  -> OpenClaw Gateway adapter: chat.send / chat.abort / sessions.reset / sessions.delete
  -> Native CLI adapter: Codex CLI / Claude Code / OpenCode through Channel Connector runner
  -> ChatSendAck + typed stream/runtime projection + durable mirror history
```

Adapter 暴露通用 `send`、`abort`、`reset`、`deleteSession`、`listSessions` 与 `readHistory`：`send` 入参包含 `sessionKey`、消息文本、thinking、deliver、idempotencyKey、attachments，返回 `status`、`runId`、`raw`，并允许 adapter 返回归一化的 terminal state、assistant text、error 和 native session id。OpenClaw adapter 继续封装现有 gateway；native-cli adapter 复用 IM Channel Connector runner，负责把 Codex / Claude Code / OpenCode 的 CLI 参数、gateway endpoint/key、模型、工作目录和权限模式翻译掉。前端与 ChatService 主流程不能依赖 provider/raw CLI 字段。

Native CLI 会话不再读取 OpenClaw 本地 transcript 作为 canonical source；assistant 回复写入 Tracevane durable mirror，后续 resume 用 registry 内的 `runtimeSession` 保存 native session id / Codex thread id。

2026-06-26 外部核对：OpenAI Codex CLI 非交互/exec 文档确认 `codex exec` 与 resume/JSON 输出适合自动化运行；Anthropic Claude Code CLI 官方文档确认 print/headless 运行、stream-json、permission mode 与 resume 是稳定 CLI 合同；OpenCode CLI 文档确认非交互运行支持 JSON/非交互参数。因此 Chat native-cli 首阶段复用 Channel Connector runner，而不是重新实现一套 CLI 运行器。

### 4.4 会话列表与文件夹范围修正

2026-06-26 交互核对：WAI-ARIA APG Menu Button Pattern（https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/）把菜单按钮定位为动作/命令列表，适合承载会话重命名、移动、归档、删除等二级动作；Microsoft Fluent Tree View（https://learn.microsoft.com/en-us/fluent-ui/web-components/components/tree-view）与 Windows TreeView（https://learn.microsoft.com/en-us/windows/apps/design/controls/tree-view）文档都把树视图定位为可展开/折叠的层级列表，适合文件夹组织；Apple HIG Outline Views（https://developer.apple.com/design/human-interface-guidelines/outline-views）也说明 outline/list 能在 split view 前导区域呈现层级文本内容。基于这些合同，Session Rail 采用：

- 会话主行动作仍然是“打开会话”；重命名/移动/归档/删除放右键菜单和行尾更多菜单。
- 默认显示可管理会话；只读/外部观察会话进入“只读”标签，不污染主工作列表。
- 文件夹范围明确区分“全部会话”和“未分组”，不再把 `all` 当成未分组，避免用户以为会话丢失。
- 可管理判定使用 typed permissions：`writable`、`canSend`、`canDelete`、`canReset`，而不是依赖某一个可能缺失的旧字段。
- 会话运行目标是会话元数据的一部分，支持后续编辑 Agent / 模型 / 工作目录 / 权限模式；保存后从下一次发送生效，避免用户只能创建时选择、后续无法切换。新建会话默认面向 native CLI（Codex CLI），OpenClaw 作为平台运行时选项保留，避免 Chat 默认回到 OpenClaw WebChat 语义。
- Composer 文件能力分两类：本地上传继续走 `/api/chat/sessions/:key/upload` 生成 session-local `fileRef`；工作区已有文件走文件管理域 `/api/files/summary` + `/api/files/browse` 选择项目文件，并以 `workspace:<path>` / `@path` 形式进入 Chat send contract，避免 Chat 复制一套文件浏览后端。
- Native CLI runner 的 progress events 必须映射到 Chat typed stream / runtime projection / observability：assistant progress 进入临时 assistant 文本与 timeline，tool progress 进入 temporary.tool、run overlay 和 toolCards，避免 Codex / Claude Code / OpenCode 只在最终结果出现。

## 5. 实时与恢复

- 服务端 stream event 使用单调 `streamSeq`。
- WebSocket/SSE/Gateway attach 支持 `lastStreamSeq`。
- 前端短 TTL 保存 cursor，用于刷新/短断线后的窗口补偿。
- replay 是短窗口补偿，不是多人持久事件系统。

## 6. 当前不做

- 多人 room / workflow。
- 把生产频道会话默认写入 Tracevane Agent 会话。
- 浏览器直接解析 provider/model/tool raw block。
- 新建独立数据库作为官方会话真相。

## 8. 2026-06-26 重新审视：旧 OpenClaw Chat 能力与统一 Agent Chat 边界

### 8.1 现状判断

旧 Chat 的历史中心是 OpenClaw WebChat：浏览器发消息给 OpenClaw gateway，Chat 页面顺手承载 host-exec 控制、OpenClaw slash/raw 管理、OpenClaw session key 展示和平台诊断。现在产品目标已经改变：Chat 是统一 Agent 会话层，运行目标可以是 Codex CLI、Claude Code、OpenCode、OpenClaw 平台 Agent 或未来平台 Agent。因此 Chat 只能保留“会话、消息、附件、运行投影、队列、运行目标选择”这些跨 Agent 的共同语义。

### 8.2 必须删除或迁出的能力

| 旧能力/代码气味 | 为什么会混淆 | 处理结论 |
| --- | --- | --- |
| OpenClaw WebChat 专属 controls / host-exec 默认拦截 | Codex / Claude Code / OpenCode 各自有权限模型；统一 Chat 不应用 OpenClaw 的主机执行开关解释所有 Agent | 从 Chat 默认 UI 和通用 contract 删除；如果某 runtime 需要确认，做成具体工具调用确认或平台安全配置 |
| Chat 内置 OpenClaw slash-gateway/raw 管理代理 | 模型、技能、OpenClaw config、compact/patch 等不是“聊天”动作 | 迁到 Platform/OpenClaw typed API；Chat 不暴露 raw RPC |
| 固定展示 OpenClaw session key / webchat 字段作为标题 | 对用户是乱码，对其它 Agent 没意义 | 标题只用 customLabel/derivedTitle/最近消息；原始 key 放诊断详情 |
| permanent 三栏：会话列表 + 聊天 + 证据/控制 | 像管理后台，不像真实 Chat；右侧证据会让 OpenClaw 控制域常驻 | 保留 IM 式 session rail + conversation；运行详情改为按需抽屉，且命名为运行检查器 |
| Chat 后端直接依赖 OpenClaw transcript 作为唯一真相 | native CLI 没有同一种 transcript；会导致所有 Agent 被迫模拟 OpenClaw | Chat 自有 durable mirror 作为统一可读历史；adapter 只负责投递/读取运行器必要状态 |
| native runtimeSession 在切换 Agent/模型/目录后继续复用 | 会把新运行目标 resume 到旧 Codex/Claude/OpenCode 线程，属于严重串会话风险 | runtimeTarget identity 变化后清理 registry.runtimeSession，下一次发送创建新 native session |
| OpenClaw 平台 Agent、OpenClaw 原生配置、OpenClaw Channel/Bot/Binding 混在 Chat | 平台管理是 Platform 域，不是 Chat 域 | Chat 只可选择 OpenClaw 作为一种 runtimeTarget；配置管理留在平台域 |

### 8.3 后端目标形态

```text
/api/chat/*
  只提供统一 Agent Session BFF：
  - session create/list/patch/delete/reset/archive/folder
  - send/abort/queue
  - history/search/durable mirror
  - upload/resource/fileRefs
  - runtime summary / observability / typed stream

ChatRuntimeAdapter
  - native-cli: Codex CLI / Claude Code / OpenCode
  - openclaw-gateway: 兼容 OpenClaw 平台 Agent
  - future-platform: 未来平台 Agent

/api/platforms/openclaw/*
  只提供 OpenClaw 平台管理：Agent / Channel / Bot / Binding / Config / Skills / Logs / Diagnostics
```

### 8.4 前端目标形态

- Chat 首页就是可用的对话工作台，不再是介绍页或管理卡片页。
- 左侧 session rail 负责搜索、文件夹、右键菜单、新建/移动/归档/删除；不可管理/外部只读会话单独标签显示。
- 中间 conversation 是主舞台，铺满内容区域；不再显示顶栏冗余导航和页面边距。
- 新建/编辑运行目标使用弹窗：Agent、模型、工作目录、权限模式都来自可用列表或明确输入，保存后从下一轮发送生效。
- 右侧运行检查器只在用户主动打开时出现，包含概览、队列、诊断，不提供 OpenClaw 专属控制默认开关。
- 附件分两类：上传文件与工作区文件引用；工作区文件复用文件管理域 API，Chat 不再复制文件浏览后端。

### 8.5 近期代码清理顺序

1. 继续保留 `openclaw-gateway` adapter，但把它标记为兼容运行目标，不作为 Chat 默认语义。
2. 把 `session-policy`、测试 fixture 中的 `webchat` 命名逐步收敛为 legacy fixture；不要再把新功能写成 webchat 专属判断。
3. 删除/跳过已经无效的 host-exec/controls Playwright acceptance，避免后续误以为 controls 还属于 Chat。
4. 后续新增 runtime 只能接 `ChatRuntimeAdapter`；不得在 `ChatWorkbenchPage` 或 ChatService 主流程里直接分支 provider/raw OpenClaw 字段。
