# Tracevane Agent 会话 Design

> 状态：运行态参考
> 更新：2026-06-25
> 规则：本文只保存当前 Chat 架构结论；历史性能/布局细节看 git。

## 1. 定位

Tracevane Agent 会话是 Tracevane 统一的人机交互层，承载 Web、IM、CLI、OpenClaw 平台和未来平台触发的所有 Agent 会话。它不是 OpenClaw 专属对话页，不是 IM 投递页，也不是 Gateway raw protocol 的浏览器客户端。

```text
Web / IM / CLI / Platform
  -> Tracevane typed Agent Session API / WS
  -> Tracevane adapter
  -> Codex / Claude Code / OpenCode / OpenClaw / custom agent
  -> Model Gateway where model routing is needed
```

## 2. 当前决策

- 浏览器不直连 OpenClaw Gateway。
- OpenClaw 不拥有独立对话中心；OpenClaw Agent 只是 Agent 会话的一种来源/运行时。
- IM 渠道不复刻聊天 UI，只把消息投递到 Agent 会话并显示投递证据。
- 前端不消费 Gateway raw frame。
- Tracevane 后端负责把 Gateway / OpenClaw 事件归一成 typed contract。
- `tracevane_managed` 会话可写；`observed_external` 默认只读；`system_internal` 不进普通用户工作区。
- 会话 UI 采用 `Tracevane前端重设计原型.md` 的 Primary Stage + Context Strip + Runtime Inspector Drawer，不使用固定三栏。
- 会话列表行只承担“打开会话”的主动作；管理动作通过右键上下文菜单和行尾更多菜单承载，避免把重命名/归档/删除铺成列表按钮。
- 会话管理分层：默认显示可管理会话，只读/外部观察会话用独立标签进入；organizer 文件夹在列表内以树状行展示，支持展开/折叠、多级目录、右键菜单、文件夹 CRUD、文件夹移动/排序和会话移动。
- Chat 保留单人模式优先，不引入多人/群聊持久事件总线。
- 统一 Agent 会话不承载旧 OpenClaw WebChat 的主机管理控制页签；不同 Agent 的权限/审批应在各自运行器或具体风险动作中处理，不在 Chat 默认拦截。
- Chat 合同不再提供 per-session `controls`、`session.controls` stream event 或 `/api/chat/sessions/:key/controls`；旧 host-exec 权限属于 OpenClaw/具体运行器安全域，不属于统一 Agent 会话。
- Chat 合同不再提供 `/api/chat/sessions/:key/slash-gateway` 或 `requestSlashGateway` 这类 OpenClaw 管理代理；模型列表、技能状态、OpenClaw 原生配置、exec approvals、session patch/compact 等平台管理能力迁入 Platform/OpenClaw API。
- 不引入 Clawith 式独立数据库会话模型。

## 3. 前端边界

当前主实现集中在：

- `apps/web/src/features/chat/ChatWorkbenchPage.tsx`
- `apps/web/src/features/chat/views/SessionListView.tsx`
- `apps/web/src/features/chat/views/ConversationView.tsx`
- `apps/web/src/features/chat/views/RuntimeInspectorView.tsx`
- `apps/web/src/features/chat/**`

要求：

- 保持单目录边界。
- 不恢复版本化/平行 chat 前端目录。
- Composer、Session Rail、Conversation Pane、Inspector、Record Browser、Slash UI 和富消息渲染都走同一 Chat Shell。

## 4. 后端边界

### 4.1 Chat 保留的通用能力

当前后端范围：

- `/api/chat/health`
- `/api/chat/bootstrap`
- sessions list/create/patch/delete/reset
- history page/search
- send/abort/queue
- upload/resource/media resolve
- composer fileRefs send
- stream / websocket replay

后端职责：

- 做 typed adapter / BFF。
- 维护 session policy、权限、runtime snapshot 和错误映射。
- 将 raw tool / agent / chat 事件归一为前端可消费事件。

### 4.2 Chat 必须删除/迁出的旧 OpenClaw 能力

这些能力来自旧 OpenClaw WebChat，不属于统一 Agent 会话；如果继续留在 Chat，会导致后续 Codex / Claude Code / OpenCode / 其它平台混成 OpenClaw 语义：

| 能力 | 处理方式 | 归属 |
| --- | --- | --- |
| per-session host-exec controls / 全局 Chat 宿主管理开关 | 已从 Chat 合同、前端 query、stream、持久状态、插件配置 schema 与验收测试中删除；旧 schema 启动时迁移为 queue-only | 具体 runtime 安全域或平台安全配置 |
| `/api/chat/sessions/:key/slash-gateway` | 已删除 Chat 路由和 ChatService 方法 | `/api/platforms/openclaw/gateway` |
| exec approvals get/set/node set | 不在 Chat 运行详情中全局暴露 | OpenClaw 平台配置/安全页面或具体工具调用确认 |
| models.list / skills.status / agents.list / config.schema.lookup | 不作为 Chat slash/raw proxy 暴露 | Platform/OpenClaw 原生管理，或未来 Runtime Adapter typed API |
| sessions.patch / compact / steer | Chat 只保留 typed reset/abort/send/queue；OpenClaw 原生命令在平台域处理 | Platform/OpenClaw gateway proxy |

迁移原则：Chat 只接收“用户要和哪个 Agent 会话说什么、附件是什么、运行状态是什么”；平台管理、原生配置、权限策略和 raw RPC 都不放进 Chat。


## 4.3 Runtime Adapter 边界

Chat 后端现在必须先走 `ChatRuntimeAdapter`，再进入具体运行器。当前默认 adapter 仍然封装现有 OpenClaw Gateway `chat.send`，行为不变；但 `performDirectSend` 不再直接把 OpenClaw raw response 当成 Chat 业务合同。

```text
ChatService.performDirectSend
  -> ChatRuntimeAdapter.send(input)
  -> OpenClaw Gateway adapter: chat.send / chat.abort / sessions.reset / sessions.delete
  -> Native CLI adapter: Codex CLI / Claude Code / OpenCode through Channel Connector runner
  -> ChatSendAck + typed stream/runtime projection + durable mirror history
```

Adapter 暴露通用 `send`、`abort`、`reset`、`deleteSession`、`listSessions` 与 `readHistory`：`send` 入参包含 `sessionKey`、消息文本、thinking、deliver、idempotencyKey、attachments，返回 `status`、`runId`、`raw`，并允许 adapter 返回归一化的 terminal state、assistant text、error 和 native session id。OpenClaw adapter 继续封装现有 gateway；native-cli adapter 复用 IM Channel Connector runner，负责把 Codex / Claude Code / OpenCode 的 CLI 参数、gateway endpoint/key、模型、工作目录和权限模式翻译掉。前端与 ChatService 主流程不能依赖 provider/raw CLI 字段。

Native CLI 会话不再读取 OpenClaw 本地 transcript 作为 canonical source；assistant 回复写入 Tracevane durable mirror，后续 resume 用 registry 内的 `runtimeSession` 保存 native session id / Codex thread id。

2026-06-26 外部核对：OpenAI Codex CLI 非交互/exec 文档确认 `codex exec` 与 resume/JSON 输出适合自动化运行；Anthropic Claude Code CLI 官方文档确认 print/headless 运行、stream-json、permission mode 与 resume 是稳定 CLI 合同；OpenCode CLI 文档确认非交互运行支持 JSON/非交互参数。因此 Chat native-cli 首阶段复用 Channel Connector runner，而不是重新实现一套 CLI 运行器。

### 4.4 会话列表与文件夹范围修正

2026-06-26 交互核对：WAI-ARIA APG Menu Button Pattern（https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/）把菜单按钮定位为动作/命令列表，适合承载会话重命名、移动、归档、删除等二级动作；Microsoft Fluent Tree View（https://learn.microsoft.com/en-us/fluent-ui/web-components/components/tree-view）与 Windows TreeView（https://learn.microsoft.com/en-us/windows/apps/design/controls/tree-view）文档都把树视图定位为可展开/折叠的层级列表，适合文件夹组织；Apple HIG Outline Views（https://developer.apple.com/design/human-interface-guidelines/outline-views）也说明 outline/list 能在 split view 前导区域呈现层级文本内容。基于这些合同，Session Rail 采用：

- 会话主行动作仍然是“打开会话”；重命名/移动/归档/删除放右键菜单和行尾更多菜单。
- 默认显示可管理会话；只读/外部观察会话进入“只读”标签，不污染主工作列表。
- 文件夹范围明确区分“全部会话”和“未分组”，不再把 `all` 当成未分组，避免用户以为会话丢失。
- 可管理判定使用 typed permissions：`writable`、`canSend`、`canDelete`、`canReset`，而不是依赖某一个可能缺失的旧字段。
- 会话运行目标是会话元数据的一部分，支持后续编辑 Agent / 模型 / 工作目录 / 权限模式；保存后从下一次发送生效，避免用户只能创建时选择、后续无法切换。新建会话默认面向 native CLI（Codex CLI），OpenClaw 作为平台运行时选项保留，避免 Chat 默认回到 OpenClaw WebChat 语义。
- Composer 文件能力分两类：本地上传继续走 `/api/chat/sessions/:key/upload` 生成 session-local `fileRef`；工作区已有文件走文件管理域 `/api/files/summary` + `/api/files/browse` 选择项目文件，并以 `workspace:<path>` / `@path` 形式进入 Chat send contract，避免 Chat 复制一套文件浏览后端。
- Native CLI runner 的 progress events 必须映射到 Chat typed stream / runtime projection / observability：assistant progress 进入临时 assistant 文本与 timeline，tool progress 进入 temporary.tool、run overlay 和 toolCards，避免 Codex / Claude Code / OpenCode 只在最终结果出现。

## 5. 实时与恢复

- 服务端 stream event 使用单调 `streamSeq`。
- WebSocket/SSE/Gateway attach 支持 `lastStreamSeq`。
- 前端短 TTL 保存 cursor，用于刷新/短断线后的窗口补偿。
- replay 是短窗口补偿，不是多人持久事件系统。

## 6. 当前不做

- 多人 room / workflow。
- 把生产频道会话默认写入 Tracevane Agent 会话。
- 浏览器直接解析 provider/model/tool raw block。
- 新建独立数据库作为官方会话真相。

## 8. 2026-06-26 重新审视：旧 OpenClaw Chat 能力与统一 Agent Chat 边界

### 8.1 现状判断

旧 Chat 的历史中心是 OpenClaw WebChat：浏览器发消息给 OpenClaw gateway，Chat 页面顺手承载 host-exec 控制、OpenClaw slash/raw 管理、OpenClaw session key 展示和平台诊断。现在产品目标已经改变：Chat 是统一 Agent 会话层，运行目标可以是 Codex CLI、Claude Code、OpenCode、OpenClaw 平台 Agent 或未来平台 Agent。因此 Chat 只能保留“会话、消息、附件、运行投影、队列、运行目标选择”这些跨 Agent 的共同语义。

### 8.2 必须删除或迁出的能力

| 旧能力/代码气味 | 为什么会混淆 | 处理结论 |
| --- | --- | --- |
| OpenClaw WebChat 专属 controls / host-exec 默认拦截 | Codex / Claude Code / OpenCode 各自有权限模型；统一 Chat 不应用 OpenClaw 的主机执行开关解释所有 Agent | 从 Chat 默认 UI、通用 contract、插件配置 schema 与验收测试删除；如果某 runtime 需要确认，做成具体工具调用确认或平台安全配置 |
| Chat 内置 OpenClaw slash-gateway/raw 管理代理 | 模型、技能、OpenClaw config、compact/patch 等不是“聊天”动作 | 迁到 Platform/OpenClaw typed API；Chat 不暴露 raw RPC |
| 固定展示 OpenClaw session key / webchat 字段作为标题 | 对用户是乱码，对其它 Agent 没意义 | 标题只用 customLabel/derivedTitle/最近消息；原始 key 放诊断详情 |
| permanent 三栏：会话列表 + 聊天 + 证据/控制 | 像管理后台，不像真实 Chat；右侧证据会让 OpenClaw 控制域常驻 | 保留 IM 式 session rail + conversation；运行详情改为按需抽屉，且命名为运行检查器 |
| Chat 后端直接依赖 OpenClaw transcript 作为唯一真相 | native CLI 没有同一种 transcript；会导致所有 Agent 被迫模拟 OpenClaw | Chat 自有 durable mirror 作为统一可读历史；adapter 只负责投递/读取运行器必要状态 |
| native runtimeSession 在切换 Agent/模型/目录后继续复用 | 会把新运行目标 resume 到旧 Codex/Claude/OpenCode 线程，属于严重串会话风险 | runtimeTarget identity 变化后清理 registry.runtimeSession，下一次发送创建新 native session |
| OpenClaw 平台 Agent、OpenClaw 原生配置、OpenClaw Channel/Bot/Binding 混在 Chat | 平台管理是 Platform 域，不是 Chat 域 | Chat 只可选择 OpenClaw 作为一种 runtimeTarget；配置管理留在平台域 |

### 8.3 后端目标形态

```text
/api/chat/*
  只提供统一 Agent Session BFF：
  - session create/list/patch/delete/reset/archive/folder
  - send/abort/queue
  - history/search/durable mirror
  - upload/resource/fileRefs
  - runtime summary / observability / typed stream

ChatRuntimeAdapter
  - native-cli: Codex CLI / Claude Code / OpenCode
  - openclaw-gateway: 兼容 OpenClaw 平台 Agent
  - future-platform: 未来平台 Agent

/api/platforms/openclaw/*
  只提供 OpenClaw 平台管理：Agent / Channel / Bot / Binding / Config / Skills / Logs / Diagnostics
```

### 8.4 前端目标形态

- Chat 首页就是可用的对话工作台，不再是介绍页或管理卡片页。
- 左侧 session rail 负责搜索、文件夹、右键菜单、新建/移动/归档/删除；不可管理/外部只读会话单独标签显示。
- 中间 conversation 是主舞台，铺满内容区域；不再显示顶栏冗余导航和页面边距。
- 新建/编辑运行目标使用弹窗：Agent、模型、工作目录、权限模式都来自可用列表或明确输入，保存后从下一轮发送生效。
- 右侧运行检查器只在用户主动打开时出现，包含概览、队列、诊断，不提供 OpenClaw 专属控制默认开关。
- 附件分两类：上传文件与工作区文件引用；工作区文件复用文件管理域 API，Chat 不再复制文件浏览后端。

### 8.5 近期代码清理顺序

1. 继续保留 `openclaw-gateway` adapter，但把它标记为兼容运行目标，不作为 Chat 默认语义。
2. 把 `session-policy`、测试 fixture 中的 `webchat` 命名逐步收敛为 legacy fixture；不要再把新功能写成 webchat 专属判断。
3. 删除已经无效的 host-exec/controls Playwright acceptance，避免后续误以为 controls 还属于 Chat。
4. 后续新增 runtime 只能接 `ChatRuntimeAdapter`；不得在 `ChatWorkbenchPage` 或 ChatService 主流程里直接分支 provider/raw OpenClaw 字段。

### 8.6 2026-06-26 Channel 命名收口

新建 Tracevane 管理会话的 session key 不再使用旧 `webchat` channel，而是使用 `agent-chat`：

```text
agent:<agentId>:agent-chat:direct:tracevane-<uuid>
```

保留 `agent:<agentId>:webchat:direct:tracevane-*` 作为 legacy 兼容，原因是历史 transcript、测试 fixture、OpenClaw plugin host context 仍可能携带旧 channel。新代码不得再把 `webchat` 当成默认 Tracevane Chat 语义；如果需要识别 Tracevane 管理会话，应使用统一 helper 判断 `agent-chat` 与 legacy `webchat`，并在 UI 上显示“Agent 会话”。

### 8.7 2026-06-26 Plugin Host 命名兼容层

插件 host context 统一新增 `isTracevaneManagedAgentChatHostContext` 作为 Agent Chat 语义入口；旧 `isTracevaneManagedWebchatHostContext` 仅保留为 deprecated alias，避免旧 OpenClaw plugin 调用断裂。日志和提示文案不得再把当前统一 Chat 称作 WebChat。

### 8.8 2026-06-26 默认运行目标

Tracevane 自管新会话即使调用方没有显式传 `runtimeTarget`，后端也默认落到 `native-cli / codex`，而不是旧 `openclaw-gateway`。历史 registry 中没有 runtimeTarget 的旧 `webchat` 会话仍按 legacy fallback 兼容 OpenClaw，避免破坏已有 transcript；新建路径不得再隐式回到 OpenClaw WebChat。

### 8.9 2026-06-26 Native CLI 附件边界

Chat 发送合同里的 `fileRefs` 是统一附件入口：本地上传文件和文件管理域选择的工作区文件都会先解析成可访问的本地路径，再作为 `ChannelConnectorInboundAttachment.localPath` 传给 native CLI adapter。这样 Codex CLI、Claude Code、OpenCode 这类本地 Agent 可以拿到真实文件路径、文件名、mime 和大小；原来的 `@path` 文本只保留为提示兼容，不再是唯一传递方式。

内联 base64 `attachments` 不直接落盘给 native CLI。它们继续作为消息资源展示；如果用户希望 CLI Agent 读取文件，必须通过上传文件或工作区文件选择生成 `fileRefs`。这样避免 Chat 后端悄悄创建不受文件管理域追踪的临时文件，也避免后续 runtime 混用附件语义。

### 8.10 2026-06-26 Runtime 生命周期边界

Chat 的 reset / abort / delete 必须按会话自己的 `runtimeTarget.adapterKind` 路由到对应 adapter，不能因为历史实现默认走 OpenClaw Gateway。Native CLI 会话 reset 只清理 Tracevane durable mirror、运行投影、队列和 native resume session；delete 只删除 Tracevane 本地会话记录与镜像，不发送 `sessions.reset` / `sessions.delete` 到 OpenClaw Gateway。OpenClaw Gateway adapter 继续保留这些 RPC 作为兼容运行目标的实现细节。

这个边界避免 Codex / Claude Code / OpenCode 等本地 CLI 会话在重置或删除时误触 OpenClaw 平台会话，也避免 native resume id 在 reset 后继续复用导致串会话。

### 8.11 2026-06-26 Native CLI 创建不依赖 OpenClaw Agent Catalog

统一 Agent Chat 的 native CLI 会话不能依赖 OpenClaw `agents.list` 才能创建。Codex CLI、Claude Code、OpenCode 是 Tracevane 本地运行器能力，默认 `native-cli / codex` 新会话即使没有 OpenClaw agent catalog 也必须可创建。只有用户显式选择 `openclaw-gateway` 兼容运行目标时，才需要校验 OpenClaw agent 是否存在。

这个约束避免 Chat 在没有安装或没有配置 OpenClaw 平台时失去基本对话能力，也保证“平台域管理 OpenClaw、Chat 域管理统一 Agent 会话”的边界继续成立。

### 8.12 2026-06-26 前端运行目标可用性提示

Chat 新建/编辑运行目标时，native CLI 选项必须显示本机 CLI 可用性，而不是只展示静态卡片。前端复用 `/api/terminal/status` 的二进制检测结果：Codex CLI、Claude Code、OpenCode 分别映射到 `codex`、`claude`、`opencode` binary id，并在运行目标卡片上显示“可用 / 未安装 / 检测中 / 未知”。这不阻止创建会话，因为后端仍是最终执行边界，但用户在选择 Agent 前能看到缺失依赖，避免把 CLI 代理安装问题误认为 Chat 或模型网关问题。

模型下拉继续复用模型网关模型列表；如果列表接口失败，前端明确提示会使用模型网关默认路由，而不是让用户以为保存失败。

### 8.13 2026-06-26 再审视：实时桥接也必须按运行目标隔离

这次复查发现一个仍然带旧 OpenClaw WebChat 思维的后端耦合点：前端打开 Chat SSE / WebSocket stream 或插件调用 `attachGatewayClient` 时，服务端会默认建立 OpenClaw Gateway session bridge。对 `native-cli` 会话来说这是错误的：Codex CLI、Claude Code、OpenCode 的实时状态来自 Tracevane 自有 run projection / durable mirror / typed stream，而不是 OpenClaw 的 `sessions.subscribe` 或 `sessions.messages.subscribe`。

结论与约束：

- Chat 可以保留 OpenClaw bridge，但它只能服务 `runtimeTarget.adapterKind === "openclaw-gateway"` 的兼容会话。
- `native-cli` 会话打开前端 stream 时不得连接 OpenClaw Gateway，不得发起 `sessions.subscribe`，避免无意义连接、错误降级、把 native 会话误判为 OpenClaw 会话。
- WebSocket / SSE / attach 三个入口都必须先解析会话，再通过统一 helper 判定是否需要 OpenClaw bridge。
- 未来新增平台 adapter 也不得复用 OpenClaw bridge；只能通过自己的 `ChatRuntimeAdapter` 和 typed stream 投影接入。

代码边界：

- `createCurrentChatRuntimeAdapter()` 仍是 raw OpenClaw RPC 的唯一边界。
- `shouldUseOpenClawGatewayBridge(session)` 是 stream/attach 是否可连接 OpenClaw bridge 的唯一判定。
- `RuntimeInspector` 只显示运行详情/队列/诊断，不恢复旧 host-exec controls 或 OpenClaw slash 管理。

| Chat stream 默认 OpenClaw bridge | 已修正为仅 OpenClaw runtime 启用 | 保留回归测试，防止 native-cli 被旧 bridge 污染 |
| `openclaw-gateway` adapter | 仍需要兼容 OpenClaw 平台 Agent | 保留，但只作为 adapter，不作为 Chat 默认模型 |
| `gateway-auth` / `gateway-request` / `session-bridge-manager` | 仍被 OpenClaw adapter 与平台兼容使用 | 不删除；迁到更明显的 `openclaw-runtime` 子目录可作为后续整理 |
| `webchat` 字段和测试 fixture | 仍用于历史 transcript / legacy session 兼容 | 不再新增；逐步改名为 legacy fixture，用户界面不展示 raw webchat 语义 |
| `sessionKey` 原始 key | 技术诊断必需 | 不作为主标题；只放诊断/复制详情 |

| 文件浏览后端 | Chat 不自建 | 继续复用文件管理域 API，Chat 只发送 fileRefs |

### 8.15 目标设计：Chat 是 Agent 会话层，不是平台管理层

用户视角应该是：

1. 在左侧会话列表/文件夹里选择或创建一个会话。
2. 在创建/编辑会话时选择运行目标：Codex CLI、Claude Code、OpenCode、OpenClaw 平台 Agent 或未来平台 Agent。
3. 在对话区发送文本、上传文件或引用工作区文件。
4. 需要排查时打开运行详情，查看队列、工具调用、诊断。
5. 平台原生管理（OpenClaw Agent/Channel/Bot/Binding/Config/Skills/Logs）回到平台域，不在 Chat 里出现。

因此后续删除/重构的判断标准很简单：

- 如果能力回答的是“怎么和 Agent 对话”，留在 Chat。
- 如果能力回答的是“OpenClaw 怎么配置/审批/管理”，迁到 Platform/OpenClaw。
- 如果能力回答的是“模型怎么路由”，迁到 Model Gateway。
- 如果能力回答的是“IM 怎么接入/投递”，迁到 IM Channels。
- 如果能力回答的是“文件怎么管理/编辑/预览”，迁到 Workspace/File Manager；Chat 只引用文件。

### 8.16 2026-06-26 删除旧 host-exec smoke 入口

`smoke:chat:host-exec` 指向已经不存在的 `tests/chat/host-exec.acceptance.py`，而且语义上属于旧 OpenClaw WebChat 的主机执行控制页签，不符合统一 Agent Chat 的目标。该脚本入口已从根 `package.json` 删除，避免后续 CI/人工验证继续把 host-exec 当成 Chat 必备能力。

统一 Chat 后续只验证通用能力：会话、发送、附件、队列、运行投影、运行目标选择、native CLI adapter 与 OpenClaw adapter 边界。具体命令执行审批应由运行器/审批域/平台安全配置负责，不再通过 Chat 专属 host-exec smoke 证明。

### 8.17 2026-06-26 Legacy webchat 标题收敛

历史 `agent:<agentId>:webchat:*` session key 仍然需要被识别为 legacy compatible session，但 UI 主标题不能再显示“旧 Web 会话”或 WebChat 术语。标题统一显示为“Agent 会话（兼容）”，原始 key 和 legacy channel 只保留在诊断/调试信息里，避免用户把统一 Chat 理解成 OpenClaw WebChat 的新皮肤。

### 8.18 2026-06-26 Composer 文件引用状态机

Composer 的文件能力不再只是“上传成功后显示一个文件名”。为了让 Chat 真正面向 Codex / Claude Code / OpenCode 这类本地 CLI Agent，附件池现在按状态管理：

- `uploading`：用户选择文件后立即进入附件池，发送按钮锁定，避免文件还没落盘就把消息发给 Agent。
- `ready`：上传文件或工作区文件都统一转成结构化 `fileRefs`，发送时只提交 ready fileRefs。
- `failed`：上传失败会保留可见失败项和错误原因，用户必须移除或重新选择，避免误以为文件已经交给 Agent。
- 用户移除上传中的文件时，后续迟到的上传成功/失败结果会被忽略，不再重新污染当前 Composer。

UI 上继续提供上传文件与 `@ 工作区文件` 两个入口；工作区文件仍复用文件管理域 API，Chat 不复制文件浏览后端。视觉上用 `上传 / 工作区 / 上传中 / 失败` 标签区分来源和状态，发送合同只保留干净的 `ChatSendFileRef[]`，不把 Composer UI 状态字段传给后端。

### 8.19 2026-06-26 再审视：native CLI 运行生命周期不能是假 abort

旧 Chat 直接面向 OpenClaw WebChat，因此 `abort/reset/delete` 可以默认转发到 OpenClaw Gateway。统一 Agent Chat 后，Codex CLI / Claude Code / OpenCode 这些 native CLI 会话必须有自己的运行生命周期；否则前端点“停止”会显示成功，但真实 CLI 进程仍在跑，属于比 UI 混淆更严重的后端能力缺口。

本轮代码边界：

- native CLI adapter 在发送前创建 per-session `AbortController`，并把 `signal` 传给 Channel Connector runner。
- `/api/chat/sessions/:key/abort` 对 native CLI 会话只取消当前 native runner，不再调用 OpenClaw `chat.abort`。
- native active run 只在内存登记当前 `runId` 和 `AbortController`；runner 返回后立即清理，避免 stale abort 误伤下一轮。
- reset/delete 仍保持 native 内部清理：清 durable mirror、queue、run projection、native resume session，不打 OpenClaw Gateway。

这项能力保留在 Chat，因为它回答的是“当前 Agent 会话能否停止/恢复到干净状态”，是所有运行目标的共同生命周期，不是 OpenClaw 平台管理功能。

### 8.20 前后端删除/保留判断的最新清单

| 能力/代码 | 当前判断 | 后续动作 |
| --- | --- | --- |
| Chat session/list/create/patch/delete/reset/archive/folder | 通用 Agent 会话能力 | 保留；继续让 session rail 通过右键/更多菜单管理 |
| send/abort/queue/runtime projection/history/search/upload/fileRefs | 通用对话与运行能力 | 保留；native CLI 与 OpenClaw 都必须经 adapter |
| `openclaw-gateway` adapter 与 gateway bridge | 兼容 OpenClaw 平台 Agent 的适配层 | 保留但隔离；仅 `adapterKind=openclaw-gateway` 使用 |
| `gateway-auth`、`gateway-request`、`session-bridge-manager` | OpenClaw adapter 私有基础设施 | 不在 UI 暴露；后续可移动到 `chat/openclaw-runtime/` 降低误用概率 |
| `webchat` session key / fixture | 历史兼容 | 不新增；UI 只显示“Agent 会话（兼容）”，raw key 进诊断 |
| host-exec controls / per-session controls / approvals 面板 | 旧 OpenClaw WebChat 安全控制 | 从 Chat 删除；若未来需要工具确认，放具体 runtime/tool 或审批域 |
| slash-gateway/raw OpenClaw RPC | 平台管理 | 从 Chat 删除；归 Platform/OpenClaw typed API |

| 文件浏览/编辑/预览 | Workspace/File Manager 能力 | 不进 Chat；Chat 只引用文件或上传文件 |

最终设计一句话：Chat 只管理“人与 Agent 会话”的通用体验；OpenClaw、IM、Gateway、File Manager 都是被 Chat 消费或旁路管理的域，不再反向把自己的管理 UI 塞回 Chat。

### 8.21 2026-06-26 会话列表不暴露 raw/unknown 调试态

统一 Agent Chat 的会话列表是用户工作台，不是 session key / transport frame 调试面板。前端必须把运行状态、来源和标题转成用户能理解的中文语义：

- runtime `unknown` 不作为主 badge 展示；兜底文案是“状态未同步”，并且仅在需要解释时出现。
- source 优先显示 `originLabel`；没有时按 runtime adapter 显示“Codex CLI / Claude Code CLI / OpenCode CLI / OpenClaw 平台”等语义。
- `webchat`、`agent-chat`、`dashboard` 这类历史/技术 channel 只能作为兼容判断，不直接显示给用户。
- 会话搜索也使用同一套 readable label，避免用户必须搜索 raw channel/debug token 才能找到会话。

这项属于 Chat 前端核心体验：如果列表继续显示 `unknown`、`webchat` 或 raw key，用户会自然认为当前页面仍是旧 OpenClaw WebChat 调试页，而不是可管理的统一 Agent 会话。

### 8.22 2026-06-26 运行目标标签不暴露 agent id

Chat 会同时面对 Codex CLI、Claude Code、OpenCode、OpenClaw 平台 Agent 和未来 runtime。前端顶部 Header 与会话列表二级信息不能直接显示 `codex`、`claude-code`、`opencode`、`main` 这类实现 id；这些字段是 routing/runtime identity，不是用户语言。

本轮收口：

- 新增 `runtimeAgentLabel()` 作为运行目标的人类可读标签入口。
- native CLI 显示为 `Codex CLI`、`Claude Code CLI`、`OpenCode CLI`。
- OpenClaw adapter 显示为 `OpenClaw 平台 Agent`。
- 列表和顶部 Header 均复用该 helper；后续新增 runtime 也必须先补 helper 再进入 UI。

这继续强化边界：Chat 可以管理各种 Agent 会话，但不会把某个运行器的内部 id 当成统一产品语言。

### 8.23 2026-06-26 Claude Code native Chat 发送链路回归

Codex native Chat 已有发送、附件、tool progress 与 resume session 覆盖，但统一 Agent Chat 的目标不是只支持 Codex。Claude Code 作为同级本地 CLI runtime，也必须证明：

- Chat 创建的 `native-cli / claude-code` 会话会进入 Channel Connector runner，而不是 OpenClaw `chat.send`。
- 模型、权限模式、工作目录、模型网关 base URL 会正确翻译到 Claude Code CLI process request。
- 工作区 `fileRefs` 会解析为本地 staged path，并出现在 Claude Code stdin 的附件摘要中。
- Claude Code 输出的 `session_id` 会保存到 Chat registry 的 `runtimeSession.agentNativeSessionId`，供后续 resume 使用。
- assistant 最终回复写入 Tracevane durable mirror/history，而不是依赖 OpenClaw transcript。

对应回归测试：`tests/chat/service.session-actions.test.mjs` 的 `Claude Code chat sessions send through the native runner with model, permission, and file refs`。这条测试是 Chat 支持多 CLI Agent 的后端证据之一；后续 OpenCode 也应按同样方式补覆盖。

### 8.24 2026-06-26 OpenCode native Chat 发送链路回归

OpenCode 不能再被 Chat 当成 OpenClaw WebChat 的变体，也不能只停留在“配置页能选 OpenCode”。统一 Agent Chat 的最低后端合同是：创建 `native-cli / opencode` 会话后，发送消息必须进入 OpenCode CLI runner，并通过模型网关 provider 配置使用用户选择的模型。

本轮补齐的验证点：

- Chat 发送 OpenCode 会话时调用 `opencode run --format json`，不调用 OpenClaw `chat.send`。
- 用户选择的模型会转换为 OpenCode provider model：`tracevane-gateway/<model>`，例如 `tracevane-gateway/glm-5.2`。
- 工作目录通过 `--dir` 指向会话配置目录；工作区 `fileRefs` 会被解析成本地文件路径并进入发送内容。
- OpenCode 运行时配置通过 `XDG_CONFIG_HOME / XDG_DATA_HOME` 隔离到 Chat runtime 目录，网关端点通过 `TRACEVANE_GATEWAY_ENDPOINT` 与生成的 opencode config 注入。
- OpenCode JSONL 输出中的 `sessionID` 会保存到 `runtimeSession.agentNativeSessionId`，后续可以 resume；assistant 文本写入 Tracevane durable mirror/history。

对应回归测试：`tests/chat/service.session-actions.test.mjs` 的 `OpenCode chat sessions send through the native runner with gateway model and workspace file refs`。至此 Chat 后端对 Codex、Claude Code、OpenCode 三个 native CLI runtime 都有发送链路证据，OpenClaw 相关 RPC 继续只作为 `openclaw-gateway` adapter 的兼容路径存在。

### 8.25 2026-06-26 会话管理视图状态进入 URL

统一 Agent Chat 的会话列表已经承担文件夹、归档、只读隔离、搜索和运行目标管理。如果这些状态只存在 React 本地 state，用户刷新、返回或分享链接后会回到默认列表，等同于“能管理但不可持续使用”。这不符合真实 Chat 产品的会话管理预期。

本轮前端收口：

- `chatView` 记录可管理 / 只读 / 归档 / 全部筛选；默认可管理时不写入，保持 URL 简洁。
- `chatFolder` 记录当前文件夹或未分组范围；文件夹 id 可直达同一会话目录。
- `chatQ` 记录当前搜索词；刷新后保留用户正在定位的会话集合。
- Session rail 会保留既有 `session` 参数，不覆盖当前选中会话。
- 如果 URL 指向已经删除或不存在的文件夹，前端自动回到全部会话，避免用户被困在空白目录。

这项能力仍属于 Chat，而不是 Workspace 或 Platform：它管理的是“Agent 会话索引”的可用性。后续如果加入批量移动、跨文件夹拖拽或不可管理 legacy 标签页，也应该继续复用这套 URL 状态，而不是重新做独立全局状态。

### 8.26 2026-06-26 运行详情改为通用运行目标，不恢复旧控制面板

运行详情抽屉必须帮助用户理解“这个会话正在用哪个 Agent / 模型 / 目录 / 权限运行”，而不是恢复旧 OpenClaw WebChat 的证据/控制侧栏。旧文案里的 `代理 main`、`网关` 对 native CLI 和未来平台 Agent 都不够准确，也容易让用户以为这里仍是 OpenClaw 专属控制面板。

本轮前端收口：

- 概览区标题从“运行时”改为“运行目标”。
- 主字段显示运行器、来源、模型、目录、权限、状态、活跃运行、可发送、事件连接、最后事件。
- 运行器与来源复用 Chat 的人类可读 label：Codex CLI / Claude Code CLI / OpenCode CLI / OpenClaw 平台 Agent 等。
- 抽屉说明从“网关问题”改为“连接问题”，不把所有 Agent 都解释成 OpenClaw Gateway。
- 继续保留队列与诊断；不恢复 `控制` 页签，不恢复 host-exec 泛化开关。重置仍必须弹窗确认。

这让运行详情成为所有 runtime adapter 的排障面板，而不是某一个平台的控制台。

### 8.27 2026-06-26 OpenClaw Gateway 私有后端进入 openclaw-runtime 边界

Chat 后端根目录不能继续平铺 `gateway-auth`、`gateway-request`、`session-bridge-manager` 这类旧 OpenClaw Gateway 私有模块。它们不是通用 Agent Chat 能力，而是 `openclaw-gateway` adapter 和 OpenClaw 平台管理代理的实现细节。如果继续放在 Chat 根目录，后续新增 Codex / Claude Code / OpenCode / 其它平台 runtime 时很容易误以为这些模块是通用发送、鉴权或 stream 基础设施。

本轮后端收口：

- 新增 `apps/api/modules/chat/openclaw-runtime/` 作为 OpenClaw Gateway 私有边界。
- 迁移 `gateway-auth.ts`、`gateway-request.ts`、`session-bridge-manager.ts` 到该目录。
- ChatService 仍可通过 `createCurrentChatRuntimeAdapter()` 使用 OpenClaw adapter，但 raw gateway request / bridge 不再暴露为 Chat 根模块。
- Platform/OpenClaw gateway proxy 显式从 `chat/openclaw-runtime/gateway-request` 引用，表达“这是 OpenClaw 平台兼容通道”，不是通用 Chat API。
- 相关测试改用新路径；`runtime-adapter-boundary` 继续保证 native-cli stream 不启动 OpenClaw bridge。

这不是删除 OpenClaw 兼容，而是把旧 WebChat/Gateway 代码从默认 Chat 心智中隔离出来。后续若继续整理，可以把 OpenClaw adapter 的创建逻辑也从 `service.ts` 拆出到同一边界，但不得破坏 native-cli 的默认路径。

### 8.28 2026-06-26 OpenClaw ChatRuntimeAdapter 工厂抽离

上一轮已经把 OpenClaw Gateway auth/request/bridge 文件迁入 `chat/openclaw-runtime/`，但 `ChatService.createCurrentChatRuntimeAdapter()` 里仍直接内联了 OpenClaw adapter 的 `chat.send`、`chat.abort`、`sessions.reset`、`sessions.delete`、`sessions.list`、`chat.history` raw RPC 细节。这会让 `service.ts` 看起来仍然拥有 OpenClaw Gateway 协议，而不是只做通用 Agent Chat 编排。

本轮后端收口：

- 新增 `chat/openclaw-runtime/adapter.ts`。
- `createOpenClawGatewayChatRuntimeAdapter()` 负责 OpenClaw Gateway ChatRuntimeAdapter 的所有 raw RPC 归一化。
- `ChatService` 只做 runtime 选择：`native-cli` 走 native adapter，其它兼容 OpenClaw 的会话走 OpenClaw adapter 工厂。
- session bridge 生命周期仍由 `ChatService` 管理，但以 `requestViaSessionBridge` callback 注入 OpenClaw adapter，避免这次重构扩大 websocket 生命周期风险。
- 边界测试更新为：`service.ts` 不直接调用 `requestGateway()`，OpenClaw raw RPC 只能出现在 `openclaw-runtime/adapter.ts`。

后续如果要继续拆分，可把 `shouldUseOpenClawGatewayBridge()` 与 bridge lifecycle 也迁入 OpenClaw runtime manager；但当前阶段先保持连接订阅生命周期稳定，避免影响 native CLI stream。

### 8.29 2026-06-26 Composer 文件选择器显式复用文件管理域 roots

Chat Composer 的“@ 工作区文件”不能只硬编码项目根，也不能假装任意文件管理 root 都能直接作为 Agent 上下文发送。后端当前稳定资源合同仍是 `workspace:` / `uploads:` / `tracevane-file:`，其中 Composer 的结构化 `fileRefs` 面向当前 Agent 工作区，因此前端必须把文件管理域的多 root 浏览能力和 Agent 可发送能力区分清楚。

本轮前端收口：

- 文件选择器继续复用文件管理域 API：`useFilesSummaryQuery()` 获取 roots，`useFilesBrowseQuery()` 浏览目录。
- 当文件管理域返回多个 root 时，Composer 显示 root 下拉，用户可以切换查看不同根目录。
- 只有 Agent 项目工作区 root 会允许点击文件并生成 `workspace:<path>` fileRef。
- 其它 root 可浏览但禁用“附加到消息”，并显示中文说明：当前 Agent 只能直接附加项目工作区文件。
- 切换 root 会重置目录路径，避免把一个 root 的相对路径错误套到另一个 root。
- 继续保留上传文件路径走 `/api/chat/sessions/:key/upload`，上传资源仍生成 `uploads:` ref。

这一步避免了“看起来支持所有文件根，实际后端解析不了”的假能力；后续如果要让任意 root 成为 Agent 上下文，需要先扩展后端资源 ref 合同，再放开前端发送。

### 8.30 2026-06-26 Chat 文件引用升级为 Files root 合同

上一轮为了避免假能力，Composer 只允许项目工作区 root 生成 `workspace:` fileRef；这保证了稳定，但也让 Chat 仍然像旧 OpenClaw WebChat：只能围绕当前 workspace 附件工作，不能真正复用 Tracevane 文件管理域。新的目标是让 Chat 面向所有 CLI Agent / 平台 Agent，而不是只服务 OpenClaw agent，因此文件引用必须成为运行时中立合同。

本轮研究与边界确认：

- 本地现状检查：文件管理域 `apps/api/modules/files/service.ts` 已有 root 发现、root 内路径规范化、realpath 越界防护、文件/目录类型校验；不应在 Chat 里复制一套不一致的路径安全逻辑。
- 本地现状检查：Chat media bridge 之前只会把结构化 `fileRefs.relativePath` 解析成 workspace/uploads/local path，且会丢弃前端传入的 `resourceRef`，所以非 workspace root 即使前端放开也会在后端丢失语义。
- 设计选择：新增 `files:<rootId>:<path>` 作为 Files 域 root 引用；`workspace:` / `uploads:` 继续保持兼容，避免破坏已有上传和工作区消息。
- 拒绝方案：不把任意绝对路径直接塞进 `tracevane-file:` 给 Composer；那会绕开 Files 域 root 权限和越界防护，也会把 Chat 重新拉回平台/本机细节耦合。

本轮前后端收口：

- `lib/tracevane-resource-refs.ts` 新增 `buildTracevaneFilesResourceRef()` 与 `parseTracevaneFilesResourceRef()`，统一表达 `files:<rootId>:<path>`。
- `lib/tracevane-markdown-media.ts` 支持解析 Markdown/stream 中的 `files:` 资源引用，后续 assistant 输出或显式引用也能复用同一合同。
- 文件管理域导出 `resolveFilesServiceExistingFilePath()`，Chat 只调用这个窄接口解析 Files root 文件，不访问或复制内部 root 实现细节。
- Chat media bridge 的 `normalizeSendFileRefs()` 保留前端传入的 `files:` resourceRef，并把它转换为安全 root/path；`buildSendResources()` 与 native CLI attachments 都能解析成真实本机文件。
- Composer 文件选择器现在对非项目 root 不再禁用发送：项目工作区仍生成 `workspace:<path>`；其它 root 生成 `files:<rootId>:<path>`，由后端发送时安全解析。
- UI 文案改为“项目工作区使用 workspace:，其它文件根使用 files: 并由后端安全解析”，避免继续给用户“只能附加工作区文件”的错误认知。

回归证据：

- `npm run build:api`
- `npm run build:web`
- `node --test tests/chat/media-bridge.test.mjs tests/chat/chat-composer.test.mjs tests/chat/conversation-view-source.test.mjs tests/chat/service.session-actions.test.mjs`

剩余风险：CLI Agent 对附件的最终消费方式仍依赖各 runner。当前 native runner 已能拿到 `localPath` attachment；但 OpenClaw Gateway 兼容路径里的 `@relativePath` 文本仍主要适配 workspace 语义。后续继续推进时，应把 Gateway 兼容路径和 native CLI 路径的附件提示文本拆开，避免 OpenClaw 旧协议影响通用 Chat 发送体验。

### 8.31 2026-06-26 Native CLI 发送文本不再继承 OpenClaw `@file` 前缀

Chat 发送链路以前只有一份 `transportText`：只要用户附加文件，就先把 fileRef 展平成 `@relative/path` 前缀，再把同一段文本交给所有 runtime。这个设计来自 OpenClaw Gateway/WebChat 的历史协议，但对 Codex CLI、Claude Code、OpenCode 这类 native CLI 并不合理：native runner 已经有结构化 attachments 和本机 `localPath`，继续把 `@file` 混入用户正文，会让模型误以为这是用户手写路径，也会让 `files:<rootId>:<path>` 的新合同退化回 workspace 相对路径。

本轮后端收口：

- `compileOpenClawGatewayMessageText()` 只服务 OpenClaw Gateway 兼容 adapter，继续保留 `@path` 前缀，避免破坏旧 Gateway `chat.send`。
- 新增 `compileRuntimeMessageText(adapterKind, text, fileRefs)`：
  - `native-cli` 直接发送用户/Composer 正文；附件由 `ChannelConnectorInboundAttachment.localPath` 和 attachment summary 进入 Agent prompt/native image/file args。
  - OpenClaw Gateway 兼容路径继续生成旧 `@path\n---\ntext`。
- `shadowStore.transportText` 改为保存实际 runtime message text：native 会话不再把 `@file` 污染进本地 shadow；OpenClaw 兼容会话仍保留历史 transport 文本以便匹配旧 transcript。
- native CLI 回归测试增加反向断言：Codex、Claude Code、OpenCode 的 stdin/args 中不应出现 `@notes.md`、`@brief.txt`、`@opencode.md` 这类旧 Gateway 前缀，但仍必须包含 attachment summary 的 `local: <absolute path>`。

这一步把“兼容 OpenClaw 旧协议”和“通用 Agent Chat native 运行时”进一步分开。后续继续删除旧 Chat 能力时，应优先检查是否还有 OpenClaw 专属语义泄漏到 native/default code path。

### 8.32 2026-06-26 恢复旧 Tracevane 会话时默认回到 native CLI，而不是 OpenClaw Gateway

本轮审计发现，新建 Tracevane 管理会话已经默认使用 `agent-chat` channel 和 `native-cli / codex` runtime，但从 registry 或 local transcript 恢复旧 Tracevane 会话时，如果旧记录没有 `runtimeTarget`，`normalizeChatSessionRuntimeTarget()` 仍会使用通用 fallback：`openclaw-gateway / <agentId>`。这会导致重启或恢复后，一部分本该属于 Tracevane 统一 Agent Chat 的会话被误恢复成旧 OpenClaw Gateway 会话，后续发送、reset/delete 也可能走错 adapter。

本轮后端收口：

- `normalizeChatSessionRuntimeTarget()` 支持显式 fallback target，不再把“没有 adapterKind”硬编码为 OpenClaw Gateway。
- 新增 `normalizeTracevaneManagedRuntimeTarget()`：Tracevane 管理会话缺失 runtimeTarget 时统一恢复为 `native-cli / codex`。
- `buildTracevaneManagedRowFromRegistry()` 使用 native fallback，保证 registry 旧记录不会因为缺字段退回 OpenClaw Gateway。
- `mapLocalSessionRow()` 对 `tracevane_managed` kind 使用同样 native fallback；只读外部/系统会话仍保留通用 OpenClaw fallback。
- legacy `agent:<id>:webchat:direct:tracevane-*` key 继续识别为 Tracevane 管理会话，channel 保持 `webchat` 以便历史数据匹配，但 runtime fallback 也改为 native CLI。

这一步不是删除 legacy key，而是防止 legacy key/缺字段把新 Chat 拉回旧 OpenClaw adapter。后续继续清理时，可以逐步把前端显示文案从 legacy channel 隐藏到“历史来源”里，但不要批量重写已有 session key。

### 8.33 2026-06-26 前端来源标签隐藏旧 WebChat / raw origin

Chat 的会话列表和顶部详情会通过 `sessionSourceLabel()` 展示来源。虽然后端已经把新会话默认切到 `agent-chat` 和 native CLI，但恢复的 Tracevane 管理会话仍可能带着旧 `source.originLabel = Tracevane managed` 或 legacy `source.channel = webchat`。如果前端优先显示这些 raw 字段，用户看到的仍是英文 debug label 或旧 WebChat 兼容语义，和“统一 Agent Chat”目标不一致。

本轮前端收口：

- `sessionSourceLabel()` 对 `tracevane_managed` kind 先返回产品语义 label，再考虑 raw origin/channel。
- 新会话显示为 `Tracevane Agent 会话`。
- legacy `webchat` key 显示为 `Tracevane Agent 会话（历史兼容）`，不再把 `webchat` 或 `Tracevane managed` 当主来源暴露给用户。
- 外部观察会话仍可显示真实 originLabel / channel，避免把第三方来源都误标成 Tracevane。
- 源码契约测试锁定：Tracevane-managed source label 必须在 `originLabel` 之前解析，防止后续又退回 raw debug 字段。

这一步只改显示语义，不改历史 key，不影响 transcript、organizer、shadow store 的兼容性。后续如果要进一步优化，可以在详情抽屉中把“历史兼容”放到 secondary meta，而主列表只显示统一名称。
