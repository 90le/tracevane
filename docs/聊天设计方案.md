# OpenClaw Studio Chat Design

> 状态：Architecture Hardened + Runtime Active
> 更新：2026-06-01

本文档只回答一件事：

`OpenClaw Studio` 当前阶段应该如何把 `chat` 设计成可落地、可验证、且不会阻塞未来 `workflow / room / project / community / multi-agent` 平台能力。

不在本文档范围内：

- 重写 chat 代码
- 重写 OpenClaw Gateway 协议

## 2. 当前方案是否合理

### 直接证据

- 官方 `Control UI / WebChat` 都是 Gateway 原始协议直接消费方
- `Clawith` 则是自建 DB / 消息 / WebSocket 栈
- `Studio` 现有工程不是官方控制台源码，也不是独立 SaaS 系统，而是插件壳内的本地管理工作台

### 已验证事实

- Gateway 原始 transport 能跑通
- CLI fallback 不稳定
- 当前 `Studio` 已有 chat typed contract 和前端消费面

### 推断

- 当前 chat 方向总体仍然合理：
  - 继续以 `openclaw` 原生 session 为真相
  - 不自建平行 DB
  - 不让浏览器直连 Gateway
- 但原文档里的表述还不够硬：
  - “最小 WS bridge”太模糊
  - session 策略还没有上升到单独 policy 层
  - `deliver: false` 的防线描述还不够强

### 最终决策

- 当前 chat 方向是合理的
- `Backend typed adapter / BFF`、`Secure Session Policy`、`Typed Contract` 已进入实现主线
- 后续重点从“能否落地”转为：
  - 保持统一后的 `chat` 目录边界不再漂移
  - 更完整的开放前 UX 验收
  - 更严格的全站写路径安全收口

## 3. 推荐 / 备选 / 禁止 方案

这里给出当前冻结结论。

### 推荐方案

`Studio BFF / Typed Adapter`

形态：

- 浏览器只连接 `Studio`
- `Studio` 后端代持 Gateway token
- `Studio` 后端维持 Gateway WS 连接
- `Studio` 后端把 Gateway 原始 RPC 和事件帧转换为 `Studio` 自己的 typed contract
- 前端不直接消费 Gateway 原始协议

推荐原因：

- 与 `Studio` 现有架构最适配
- 不形成双真相
- 安全边界最清晰
- 允许当前 text-first chat 落地
- 同时为未来 `event/activity layer` 预留边界

### 备选方案

`官方直连式`

只作为以下场景的备选：

- 官方一方产品
- 独立调试工具
- 极小范围的内部诊断页

不适合作为 `Studio` 主方案的原因：

- 浏览器会直接面对 Gateway 认证与原始协议
- 前端将被迫理解 `connect.challenge`、device auth、raw event frame
- 不利于为未来平台层抽象出稳定 contract
- 和 `Studio` 现有“模块化 API + 管理工作台”架构不一致

### 禁止方案

`第三方自建存储式`

禁止原因：

- 会形成 `openclaw session` 与 `Studio session/db` 双真相
- 与 `openclaw` 原生 session / routing / delivery / token accounting 脱节
- 与当前 `Studio` 的 Vue/插件壳/本地管理定位不匹配
- 会把未来 `workflow / room / project / community` 建在错误的数据基座上

## 4. 当前还没有收口的风险

### 4.1 “最小 WS bridge”过于模糊

### 直接证据

- 历史文档中曾有“最小 WS bridge”的表述
- 当前代码已经形成 `session-bridge-manager.ts`、`gateway-request.ts`、`agent-event-mapper.ts`、`transcript.ts`、`observability.ts` 等边界，但 `service.ts` 仍是主要复杂度集中点

### 推断

- 这会导致后续实现时把：
  - Gateway 握手
  - raw frame
  - session policy
  - 前端 contract
  混在一起

### 最终决策

- 文档层面继续统一称为：
  - `Studio backend typed adapter / BFF`
- 前端默认禁止看到 Gateway 原始事件帧
- 允许的唯一例外：
  - 明确标记的 debug / diagnostics 视图
  - 且也应经 `Studio` 后端转发，不是浏览器直连 Gateway

### 4.2 sessionKey 策略过于脆弱

### 直接证据

- 本轮 spike 使用的隔离 key 是：
  - `agent:<agentId>:webchat:direct:studio-spike-<timestamp>`
- 它确实可用，但目前只是命名约定

### 已验证事实

- 这个 key 形态可稳定创建、发送、读取、重置、删除

### 推断

- 命名约定本身不足以承担长期安全与产品边界
- 必须新增一层 `Secure Session Policy`

### 最终决策

必须引入 session policy / helper 层，至少固化：

- session classification：
  - `studio_managed`
  - `observed_external`
  - `system_internal`
- write policy：
  - MVP 只允许 `studio_managed`
- source metadata：
  - `source = studio`
  - `channel = webchat`
  - `surface = studio-chat`
- label / title 约定：
  - 由后端统一生成，不让前端拼 key 语义
- key format：
  - 只允许后端 helper 知道
  - 前端不应依赖 key 格式做业务判断

前端真正应该拿到的是：

- `sessionKey`
- `kind`
- `source`
- `writable`
- `label`
- `derivedTitle`
- `deliveryContext`

而不是对 key 前缀做逻辑分支。

### 4.3 `deliver: false` 不能作为唯一保险

### 已验证事实

- `deliver: false` 已在测试 session 上真实生效
- 但本轮没有对真实生产绑定 session 做危险实测

### 推断

- `deliver: false` 只能算第二层保险
- 第一层保险必须是 session write policy

### 最终决策

- 第一层保险：
  - MVP 默认只允许写 `Studio` 自管 session
- 第二层保险：
  - 默认 `deliver: false`
- 第三层保险：
  - 生产 session 默认只读观察
  - 不进入 MVP 默认可写范围

也就是说：

- 不能把“有 `deliver: false`”当成“生产 session 可随便写”

### 4.4 chat same-origin 边界必须持续保持

### 直接证据

- `apps/api/server.ts` 会识别 chat API path，并对不允许的 origin 返回 `auth_failure`
- `apps/api/core/http.ts` 仍保留通用 CORS helper，非 chat 写路径后续还需要继续收口

### 推断

- 如果 chat 退回 `CORS *`，任意网页都可能对本机 `Studio` 发起跨域请求
- 一旦 `Studio` 后端代持 Gateway token，这就会把本地 chat 控制面暴露成危险入口

### 最终决策

chat 边界必须保持 same-origin：

- `GET/POST /api/chat/*`
- `WS /ws/chat`

最低要求：

- same-origin only
- 开发态只允许 Vite 本地代理和 loopback 例外

推荐扩展：

- 逐步把整个 `Studio` 写路径都收口为 same-origin

如果不收口，直接后果是：

- 本机浏览器上下文中的恶意站点可借 `Studio` 后端代持的 Gateway token 驱动 chat 行为

### 4.5 事件模型需要为未来平台演进预留边界

### 直接证据

- 当前已验证：
  - `chat` 事件能覆盖 text-first MVP
  - `agent` 事件已通过 `Studio /ws/chat` 同链路拿到 fresh live `agent_tool_call / agent_tool_result`
- `edict` 的价值在于统一 activity stream 与可观测，而非聊天本身

### 推断

- 当前不应把未来 `workflow / room / project / community` 的事件模型混进 chat MVP
- 但现在必须预留一个更高层的 `Activity` 概念

### 最终决策

- 当前 MVP：
  - 只消费 `chat` 事件
- Phase 2：
  - 加入 `agent` 事件
- 更长期：
  - 由后端 adapter 统一抽象出更高层 `event/activity envelope`
  - 供 chat、workflow、room、project timeline、community feed 共享

## 5. chat 在长期平台中的层级

## 5.1 分层判断

用户提出的 4 层模型总体合理，但需要一个补充：

- `approval / review / audit` 更像跨层治理平面，而不是 chat 子层

推荐长期分层：

1. `Chat Core`
   - direct session
   - session rail
   - history / send / abort / reset
2. `Event / Activity Layer`
   - activity envelope
   - run lifecycle
   - agent output
   - timeline normalization
3. `Collaboration Surface`
   - room
   - plaza / feed
   - agent posts / comments / blog / article
   - mixed human-agent discussion surfaces
4. `Project / Workflow Layer`
   - project entity
   - project conversation
   - workflow
   - checkpoints / deliverables / task orchestration
5. `Governance Plane`（跨层）
   - approval
   - review
   - audit

## 5.2 当前阶段只做到哪一层

### 最终决策

- 当前 chat 阶段只做到：
  - `Chat Core`
  - 并在边界上预留未来 `Event / Activity Layer`
- 当前必须明确延后：
  - `Collaboration Surface`
  - `Project / Workflow Layer`
  - `Governance Plane` 的业务化实现

## 6. 未来能力分类

### 6.1 属于 chat 核心

- `chat`
- `direct session`

### 6.2 与 chat 邻近，但不能进入 chat MVP

- `project conversation`
- `project timeline`
- `multi-agent event stream`

这些能力依赖 chat 与 event layer，但都不应直接混入当前 MVP。

### 6.3 不属于 chat，应该进入未来独立层

- `room`
- `workflow`
- `agent posts`
- `agent comments`
- `agent blog / article`
- `plaza / feed`
- `approval / review / audit`

## 7. 当前阶段冻结结论

### 已验证事实

- `Studio` 可以建立在 `openclaw` 原生 session / Gateway transport 之上
- 当前最稳妥的长期方向不是官方直连，也不是第三方 DB
- `Studio backend typed adapter / BFF` 是当前最稳的主方案

### 最终决策

- 当前 chat 方案合理，但必须从“最小 WS bridge”升级为“typed adapter / BFF + secure session policy”
- 未来能力中，只有 `Chat Core` 可以进入当前范围
- 其它一切 collaboration / workflow / project / community 能力必须明确延后
- 当前 chat MVP 绝不能提前混入：
  - `room`
  - `workflow`
  - `project conversation`
  - `plaza/feed/post/comment/blog`
  - `approval/review/audit`

## 8. 统一后的 Chat 前端结构

当前前端已经统一到一个目录：

- `apps/web-vue/src/features/chat/**`
  - `api.ts`：Studio Chat HTTP/SSE API client
  - `session-ref.ts`：route/session 引用编码
  - `storage.ts`：最近 agent/session 记忆
  - `ChatShellPage.vue`：主聊天壳

当前结论：

- Chat 前端不再保留版本化 sibling 目录
- 后续迁移必须先锁定：
  - route 行为不变
  - `features/chat/api.ts` contract 不破坏
  - `tests/chat/*` 与 `tests/system/studio-web-chat-*` 继续覆盖
  - 不重新引入旧 wrapper / 新版本目录双轨
