# Tracevane 富消息使用说明

> 状态：Active Reference
> 更新：2026-06-26

## 1. 适用范围

本文说明 Tracevane Chat / Channel 消息中富内容的推荐表达方式。它不定义平台上传协议；平台文件收发由 Channel daemon transport 处理。

## 2. 默认选择

优先使用 Markdown：

- 文本说明
- 代码块
- 表格
- 图片/文件引用
- 简单结构化结果

只有在需要强结构化投递时使用 `tracevane_delivery` 或 Channel 文件 manifest。

## 3. Markdown 文件引用

常用 URI：

- `files:<rootId>:<path>` Tracevane Files API 根目录内资源；新 Chat Composer 附件的默认合同。
- `workspace:` 工作目录内资源。
- `uploads:` 历史 Tracevane 上传资源；只用于兼容旧消息/旧富消息，不作为新上传入口。
- `tracevane-file:` Tracevane 资源解析后的文件。

推荐写法：

```md
请查看文件：[report.md](files:project-root:report.md)

![preview](files:project-root:image-1.png)
```

原则：

- 显示引用和 AI 输入引用要一致。
- 不把本地绝对路径直接暴露给前端。
- 新的目录加载、上传、预览/下载统一通过 Files API，不再通过 Chat 专属上传/资源解析端点。
- 大文件只传引用，不把内容塞进消息正文。

## 4. Channel 出站文件

IM 渠道出站文件使用 `tracevane-channel-files` manifest：

```text
```tracevane-channel-files
[{"path":"workspace/TOOLS.md","name":"TOOLS.md","caption":"工具说明"}]
```
```

daemon 会：

- 剥离 manifest，避免污染最终回复。
- 校验路径、权限和大小。
- 按 Feishu/Octo 等平台上传发送。
- 尽量保留原始文件名。

Agent 不应调用旧外部桥接命令发送文件。

## 5. `tracevane_delivery`

`tracevane_delivery` 仅用于需要明确投递结构的场景，例如多附件、目标通道、显示 hint 或后续可操作 payload。

当前私聊主线优先使用 Markdown + manifest，避免把简单回复做成复杂工作流。

## 6. 流式行为

- Markdown 文本可以流式显示。
- 文件 manifest 只在最终回复阶段处理。
- 工具过程、思考和过程回复由 Channel/Chat progress renderer 管理，不应写进最终 Markdown 正文。

## 7. 常见失败

- 引用路径不在允许根目录内。
- 文件名被模型改写，daemon 找不到文件。
- 模型把 manifest 包在普通代码块外层，导致解析失败。
- 平台大小限制或媒体类型限制。
- 非视觉模型收到图片后进行视觉推断；应改为询问用户或切换 vision 模型。

## 8. 推荐口令

要求 Agent 发送文件时，建议说明：

```text
如需发送文件，请先在工作目录生成文件，然后在最终回复末尾输出 tracevane-channel-files manifest。
不要调用外部桥接命令；Tracevane 会负责上传和发送。
```
