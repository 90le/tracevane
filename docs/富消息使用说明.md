# OpenClaw Studio 富消息使用说明

## 1. 适用范围

这份说明只针对：

- OpenClaw Studio
- 当前 Studio 管理的 WebChat 会话

不保证对所有 channel、所有客户端、所有 OpenClaw 运行场景都生效。

---

## 2. 结论先看

当前策略是：

1. 普通富消息：优先用 assistant Markdown
2. 复杂富消息：再退到 `studio_delivery`

可以直接理解成：

- Markdown 是主通道
- `studio_delivery` 是兜底通道

---

## 3. 什么情况下用 Markdown

下面这些情况，优先用普通 assistant Markdown 富消息：

1. 一两张图片
2. 一两个视频
3. 一两个文件
4. 正文里自然夹带图片、视频、文件
5. 常规“说明文字 + 换行图片/附件 + 补充说明”
6. 希望尽量流式更早显示出来

这类场景的推荐写法是：

```md
[结构图](workspace:graph.png "studio:break-image")
[演示视频](uploads:demo.mp4 "studio:break-video")
[安装包](studio-file:./report.pdf "studio:card")
```

为什么优先走这条：

- 不需要 tool call
- 不需要等 tool result
- token 一旦闭合，流式阶段就有机会直接升级渲染
- 对用户来说更像“流式富文本”

---

## 4. 什么情况下用 `studio_delivery`

下面这些情况，更适合走 `studio_delivery`：

1. 多资源强顺序控制
2. 明确需要 paragraph / segment 级精细编排
3. 一条回复里混合很多资源，而且结构不能出错
4. 资源不是稳定文件路径，而是 buffer / data / blob
5. Markdown 很难可靠表达最终布局

一句话判断：

如果“普通 Markdown 链接 + `studio:` title”已经能自然表达，就不要用 `studio_delivery`。

如果“必须要结构化 block 协议才能说清”，再用 `studio_delivery`。

---

## 5. Markdown 方案怎么写

### 5.1 资源 URI

推荐使用这三种显式资源引用：

#### `workspace:`

相对当前 agent workspace。

```md
[结构图](workspace:graph.png "studio:break-image")
```

#### `uploads:`

相对当前 workspace 的 `uploads/` 目录。

```md
[演示视频](uploads:demo.mp4 "studio:break-video")
```

#### `studio-file:`

显式本地文件引用，可相对，可绝对，但必须通过后端 allowed roots 校验。

```md
[安装包](studio-file:./report.pdf "studio:inline-chip")
[安装包](studio-file:/home/binbin/.openclaw/workspace-ops/report.pdf "studio:card")
```

### 5.2 display hint

支持这些 `studio:` 显示方式：

- `studio:inline-image`
- `studio:inline-video`
- `studio:inline-chip`
- `studio:break-image`
- `studio:break-video`
- `studio:break-chip`
- `studio:card`

推荐理解：

- `break-*`：换行显示，适合日常默认
- `inline-*`：句内小型引用
- `card`：独立资源卡片

### 5.3 推荐默认写法

日常优先：

- 图片：`break-image`
- 视频：`break-video`
- 文件：`break-chip` 或 `card`

只有明确需要句内引用时，再用：

- `inline-image`
- `inline-video`
- `inline-chip`

## 6. 如何让 AI 优先用 Markdown

最稳的提示方式是直接明确要求：

1. 用普通 assistant Markdown 富消息返回
2. 不要只输出路径
3. 资源引用优先用 `workspace:` / `uploads:` / `studio-file:`
4. 不要调用工具，除非 Markdown 无法可靠表达

### 6.1 可直接复制的提示模板

#### 模板 A：普通图文附件

```text
请用普通 Markdown 富消息返回，不要只输出路径。
图片、视频、文件请优先使用 workspace:/uploads:/studio-file: 引用。
图片默认用 studio:break-image，视频用 studio:break-video，文件用 studio:break-chip 或 studio:card。
只有在 Markdown 无法可靠表达时才使用 studio_delivery。
```

#### 模板 B：指定资源和展示方式

```text
请直接在当前聊天里用普通 Markdown 富消息返回。
把结构图作为换行图片嵌入，使用 workspace:graph.png 和 studio:break-image。
把演示视频作为换行视频嵌入，使用 uploads:demo.mp4 和 studio:break-video。
把安装包作为卡片返回，使用 studio-file:./report.pdf 和 studio:card。
不要只写路径，不要先走工具。
```

#### 模板 C：句内引用

```text
请用普通 Markdown 富消息回复。
正文里句内引用图片时用 studio:inline-image，句内引用视频用 studio:inline-video，句内引用文件用 studio:inline-chip。
资源引用优先使用 workspace:/uploads:/studio-file:。
```

---

## 7. 如何让 AI 改用 `studio_delivery`

如果你知道这次布局明显更复杂，可以直接明确要求 fallback：

```text
这次请不要用普通 Markdown 富消息。
请改用 studio_delivery，因为我需要更强的多资源结构化排版。
```

或者更具体：

```text
请使用 studio_delivery。
我需要严格的 paragraph / segment 级顺序控制，并且这条回复包含多段文字、多个资源卡片和句内资源混排。
```

---

## 8. 选择建议

### 用 Markdown

适合：

- 轻量
- 自然
- 快速
- 流式尽早可见

### 用 `studio_delivery`

适合：

- 强结构
- 高可靠布局控制
- 复杂混排
- 非文件路径型资源

### 简单决策表

| 场景 | 推荐 |
|---|---|
| 一张图 + 一段说明 | Markdown |
| 一个视频 + 一个文件 | Markdown |
| 几个资源的普通换行展示 | Markdown |
| 多段 paragraph 精准编排 | `studio_delivery` |
| 资源来自 buffer/data/blob | `studio_delivery` |
| 担心 Markdown 表达不稳 | `studio_delivery` |

## 9. 为什么浏览器能显示本地资源

浏览器并不会直接访问：

- `workspace:graph.png`
- `uploads:demo.mp4`
- `studio-file:/abs/path/report.pdf`

真实流程是：

1. 模型输出普通 assistant Markdown
2. Studio 后端识别显式资源 URI 和 `studio:` title
3. 后端把本地资源编译成 signed `/api/chat/sessions/.../media/...` URL
4. 前端只渲染这个已签名的可访问 URL

所以：

- 浏览器不需要直接拿本地路径
- 前端不做路径猜测
- 本地文件访问边界仍在后端控制

---

## 10. 流式行为说明

Markdown 主通道现在尽量支持流式预览：

1. 当 assistant stream 中已经出现完整的 Markdown 富消息 token
2. 且命中了 `studio:` + `workspace:` / `uploads:` / `studio-file:`
3. 后端会尝试在 `delta` 阶段做低成本编译
4. 编译成功后，stream bubble 就可以提前显示富消息

注意：

- token 未闭合时不会误升级
- `final` 到来后会按同一条消息收口
- 不应出现双份消息

## 11. 常见失败原因

### 没有升级成富消息

常见原因：

1. 没写 `studio:` title
2. 用了不支持的 display
3. 资源引用没用 `workspace:` / `uploads:` / `studio-file:`
4. token 还没闭合
5. 文件不存在
6. `studio-file:` 指向了不在 allowed roots 里的路径

### AI 还是走了纯文本

常见原因：

1. 提示不够明确
2. 当前问题本身没有明显“需要资源回传”
3. 模型判断这是纯文本回答更合适

建议直接补一句：

```text
请不要只输出路径或纯文本说明，直接用普通 Markdown 富消息把资源发到当前聊天里。
```

### AI 还是走了 `studio_delivery`

这通常说明模型认为：

- 这次排版更复杂
- 或普通 Markdown 不够稳

如果你想强制优先 Markdown，可以明确说：

```text
优先使用普通 assistant Markdown 富消息，不要调用 studio_delivery，除非 Markdown 无法可靠表达。
```

---

## 12. 推荐的最终口令

如果你只想记一条，建议用这句：

```text
请在当前 Studio 聊天里优先用普通 Markdown 富消息返回，资源引用使用 workspace:/uploads:/studio-file:，并配合 studio:break-image / studio:break-video / studio:break-chip / studio:card；只有在 Markdown 无法可靠表达时才改用 studio_delivery。
```
