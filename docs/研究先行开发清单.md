# 研究先行开发清单

## 2026-07-12 三系统当前用户守护服务

- 范围：Model Gateway、Channel Connectors、OpenClaw Recovery 的统一会话托管与持久化生命周期，以及 Windows、macOS、Linux 的当前用户原生 supervisor 契约。
- 外部参考：
  - [Microsoft `schtasks` 命令总览](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/schtasks)明确 `/create`、`/delete`、`/end`、`/query`、`/run` 的职责，且本机任务默认以当前用户权限运行；[Microsoft `schtasks /query` 文档](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/schtasks-query)明确 `/TN` 查询精确任务、`/HResult` 令进程退出码使用 HRESULT 格式。本实现以命令、目标任务名与稳定退出码分类任务不存在等状态，本地化 stdout/stderr 仅保留为已脱敏诊断，不驱动生命周期分支。
  - [Task Scheduler XML schema](https://learn.microsoft.com/en-us/windows/win32/taskschd/task-scheduler-schema)是注册任务的 XML 契约；[`RestartOnFailure`](https://learn.microsoft.com/en-us/windows/win32/taskschd/taskschedulerschema-restartonfailure-settingstype-element)规定任务失败后的外层恢复；[`Count`](https://learn.microsoft.com/en-us/windows/win32/taskschd/taskschedulerschema-count-restarttype-element)基于 `unsignedByte`，因此生成模板使用合法上限 `255`，而不是超范围计数。
  - [Apple Creating Launch Daemons and Agents](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/CreatingLaunchdJobs.html)明确用户个人 `~/Library/LaunchAgents`、`ProgramArguments` 数组与 `KeepAlive` 的角色，并明确用户登出时会向该用户的 agents 发送 `SIGTERM`；因此 `RunAtLoad + KeepAlive` 只描述登录 GUI 会话内行为，不承诺登出后继续存活。
  - [systemd unit 文档](https://www.freedesktop.org/software/systemd/man/255/systemd.unit.html)列出 `~/.config/systemd/user` 当前用户 unit 搜索路径；[systemd service 文档](https://www.freedesktop.org/software/systemd/man/255/systemd.service.html)定义 `Restart=on-failure`；[`systemctl`](https://www.freedesktop.org/software/systemd/man/255/systemctl.html)明确 `--user` 连接调用者的用户服务管理器；[`loginctl`](https://www.freedesktop.org/software/systemd/man/latest/loginctl.html)明确 `enable-linger` 会在开机创建并在登出后保留用户管理器。本实现不自动改变 linger 策略。
  - [Microsoft WSL systemd 文档](https://learn.microsoft.com/en-us/windows/wsl/systemd)明确 WSL 可启用自己的 systemd，但 systemd 服务不会让 WSL 实例保持存活；WSL 因而按独立 Linux 环境处理，不复用 Windows Task Scheduler 注册。
- 本次决策：
  - 三个 daemon 域共享同一状态、动作、错误码、模板生成、固定命令执行和 owner 切换边界；域内 wrapper 只保留配置与兼容响应映射，不再各自实现原生 supervisor。
  - Session 模式由 Tracevane API 进程直接拥有每域唯一子进程，意外退出最多重启 3 次；standalone API 的正常信号与已处理 fatal shutdown 会先停止 owned children，但强杀、宿主崩溃或断电不保证 cleanup。该模式不写 template、不注册系统服务。Persistent 模式必须显式安装，且仅写当前用户目录和当前用户服务管理器，不要求管理员/root，也不是 machine-wide service。
  - `status/start/stop/restart/install/repair/uninstall` 使用统一生命周期。`install` 原子写 template、注册/enable、启动并验证 readiness；显式 `repair` 可用于已安装服务并执行重写、重注册、重启，且是 `stale-config` 或 `runtime-not-ready` 的必需恢复路径；`uninstall` 先停止并注销/disable，再删除生成 template，但保留业务配置、运行状态与日志。
  - Session 与 Persistent 切换前必须停止并验证另一 owner；同一 daemon 不主动运行两个 owner。Node 路径、构建 entry、checkout/工作目录、配置路径或 template 变化都会改变 fingerprint 并产生 `stale-config`，由 repair 收口。
  - 生成模板包含平台服务管理器所需的 identity、trigger 与 recovery 元数据；其中 launch payload 只引用固定 Node、entry、工作目录、配置路径和显式 `--config` token。secrets、代理凭证与浏览器提供的任意 command/args/cwd/env 均不进入 template、进程参数、结构化结果或日志证据。
  - Windows 使用当前用户登录触发、`InteractiveToken`、`LeastPrivilege` 与 `MultipleInstancesPolicy=IgnoreNew`。Task 内的 Tracevane watchdog 以固定 1 秒退避恢复 daemon；Task Scheduler 的 `RestartOnFailure`（`PT1M`、`Count=255`）是 watchdog 自身失败时的外层恢复。生命周期使用 `/Create`、`/Run`、`/End`、`/Delete`、`/Query /HResult`。
  - macOS template 位于 `~/Library/LaunchAgents`，使用 `gui/$UID` 域和 `bootstrap/enable/kickstart/bootout/print`，plist 含 `RunAtLoad + KeepAlive`；这不改变用户登出语义。
  - Linux template 位于 `~/.config/systemd/user`，使用 `systemctl --user daemon-reload/enable/start/stop/restart/disable/is-active/is-enabled` 与 `Restart=on-failure`。卸载在删除 unit 后再次 `daemon-reload`；不自动执行 `loginctl enable-linger`，因此不承诺登出后或无用户管理器时存活。
  - WSL 是独立 Linux runtime。Windows 与 WSL/Linux 之间不共享 `node_modules`、构建产物、runtime metadata 或 supervisor 注册；依赖、构建和 manager 操作必须在实际 owner 环境完成。
- 验证入口：
  - `npm run test:supervisor`：构建 API 后运行共享 contract、platform plan、command runner、session、service manager、watchdog 与 live harness 的非破坏性测试。
  - 原生 live smoke 必须显式 opt-in，并使用 PID 唯一名称在当前用户范围临时注册，执行 preview → install → status → start → crash recovery → restart → stop → uninstall，最终验证任务/unit/plist、template 与 fixture 进程均已清理。Windows PowerShell 使用 `$env:TRACEVANE_WINDOWS_SUPERVISOR_LIVE = "1"; npm run smoke:supervisor:windows`；macOS 使用 `TRACEVANE_MACOS_SUPERVISOR_LIVE=1 npm run smoke:supervisor:macos`；Linux 使用 `TRACEVANE_LINUX_SUPERVISOR_LIVE=1 npm run smoke:supervisor:linux`。运行前先执行 `npm run build:api`；非匹配平台或未设置 flag 时测试明确 `SKIP`。

## 2026-07-11 三系统本地开发启动器

- 范围：Windows、macOS、Linux 的本地 API + Web 开发启动、受管进程清理、端口选择和原生依赖环境边界。
- 外部参考：
  - [Node.js `child_process` 官方文档](https://nodejs.org/docs/latest-v22.x/api/child_process.html)：成功创建子进程后才触发 `spawn`；POSIX 的 `detached` 子进程成为新进程组/会话 leader；后台存活还要求 `unref()` 和与父进程解耦的 stdio。文档同时明确 PID 复用可能把信号送给另一进程，因此本地 metadata 只能缩小风险，不能证明 OS 级创建身份。
  - [Microsoft `taskkill` 官方文档](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/taskkill)：`/PID` 指定单个 PID，`/T` 包含其子进程，`/F` 强制结束；本启动器不使用 image name 或端口 owner 作为授权依据。
  - [Node.js globals 官方文档](https://nodejs.org/docs/latest-v22.x/api/globals.html)：`AbortSignal.timeout()` 提供单次请求上限；本实现再叠加统一绝对 deadline，避免多次请求超时累计突破启动预算。
  - [npm `ci` 官方文档](https://docs.npmjs.com/cli/v11/commands/npm-ci/)：依赖 package lock 做冻结安装；含原生 optional dependency 的 `node_modules` 必须在实际运行平台安装。
  - [Vite CLI 官方文档](https://vite.dev/guide/cli)：开发服务器原生提供 `--host`、`--port`、`--strictPort` 与 `--force` 等参数，可由 Node runner 以参数数组调用，不需要 Bash 环境变量展开。
  - [Playwright 浏览器官方文档](https://playwright.dev/docs/browsers)：Playwright 版本对应受支持的浏览器二进制；浏览器 smoke runner 应使用已安装的匹配 Chromium，缺失时显式执行 `npx playwright install chromium`，而不是静默依赖某台机器的固定 Chrome 路径。
  - Node.js 的 `net` API 可用 loopback listen 探测端口，不依赖外部端口/进程扫描命令。
- 本次决策：
  - `dev:restart` 与 `dev:fresh` 改为 Node 原生 `scripts/dev-runtime.mjs`，不再要求 Bash、WSL、tmux、setsid、pgrep、lsof 或 fuser。
  - PID 文件改为 JSON，至少记录 `version/pid/target/mode/resolvedRoot/runToken/startedAt`；schema、target、mode、checkout root 任一不匹配时只做 stale cleanup，不授权发送信号。只有精确 PID/tree 已确认退出后才删除有效 metadata；停止失败时保留 metadata 并中止 restart。
  - Windows 仅使用 `taskkill.exe /PID <pid> /T /F`，spawn error、null status 或非零 status 一律视为 cleanup 未确认并保留 metadata。macOS/Linux supervisor 与 worker 分别是独立进程组 leader；用 `kill(-pid, 0)` probe 整个 PGID（`EPERM` 仍表示存活），leader 先退出也继续等待/升级整个 group，且只有初始 PGID probe 为 `ESRCH` 才退回正 PID。
  - 若固定端口属于未登记进程，launcher 明确失败并要求改端口，不发现或终止端口 owner。
  - supervisor 停止时先禁止 worker restart，再终止 worker group/tree，并等待直接 child 的 `close` 完成 reap；metadata publish 或 post-ack immediate exit 的 cleanup 若失败，以 `AggregateError` 同时暴露原始错误和 cleanup 错误，且不提前 `unref()` 或删除有效 metadata。
  - npm 调用优先使用当前 `process.execPath + npm_execpath`，直接 Node 启动只尝试已知 `npm-cli.js` 路径；不使用 `shell: true`，CJK、空格与端口始终保持数组参数边界。
  - readiness 使用绝对 deadline、短 fetch timeout 与 supervisor liveness，并在读取状态后 cancel 每个 response body；backend 通过自身 health，frontend 同时通过 root 和代理 health。任一启动/超时/信号错误都按创建逆序 rollback，本轮未完整成功前不发布 env 文件。
  - `resolvedRoot/runToken` 防止跨 checkout metadata 混用，但不能消除 OS PID 在检查与发送信号之间被复用的残余竞态；当前不引入跨平台进程注册框架。
  - 同一工作目录一次只用一个原生 Node/npm 平台；Windows 与 WSL/Linux 切换时删除根与 `apps/web` 的 `node_modules` 后重新执行 `npm ci`。
- 本地验证：
  - `node --test tests/system/dev-runtime-script.test.mjs`
  - `node --check scripts/dev-runtime.mjs`
  - `npm run typecheck`
  - Windows 原生 Node：用隔离端口启动 `npm run dev:restart`，确认 backend、frontend root、frontend proxy 均为 2xx；`node scripts/dev-runtime.mjs stop` 后 supervisor/worker PID 均不存在且端口可重新 bind。
  - Windows 原生安全性：用单独记录 PID 的无关 HTTP fixture 占用 Web 端口，确认 launcher 明确失败而 fixture 仍存活并保持响应，再仅按 fixture PID 清理。
  - WSL Ubuntu-24.04：运行同一 focused suite，真实验证 leader 已退出但 descendant 仍占 PGID 的 metadata stop，以及 supervisor worker/descendant 的 stop + reap。

## 2026-07-09 Model Gateway 三协议与附件验收收口

- 范围：Model Gateway 路由、OpenAI Responses / Chat Completions / Anthropic Messages 协议互转、Codex / Claude Code / OpenCode CLI agent 验收、工具流、MCP、附件和多模态降级。
- 外部参考：
  - OpenAI Responses API `POST /responses` 支持文本、图片输入、JSON 输出、自定义代码和 web/file search 等工具。
  - OpenAI Chat Completions 是基于 messages 的响应接口；OpenAI 文档建议新项目优先考虑 Responses。
  - OpenAI file inputs 在 Responses 中通过 `input_file` 的 base64、Files API file id 或外部 URL 表达；本地路径不是直接上传。
  - OpenAI audio transcription 接收实际音频文件对象；只给本地文件名不等于上传。
  - Anthropic Messages streaming 使用 SSE，覆盖文本、tool use 和 extended thinking delta。
  - Claude Code CLI 的 `stream-json`、`--mcp-config`、permission prompt 选项是 Channel Connector 驱动边界。
  - OpenCode CLI 的 `opencode run`、MCP servers 和 custom tools 是 OpenCode agent 验收边界。
- 本次决策：
  - “100%支持”定义为每个协议/agent/工具/附件格子都有 `native`、`adapted-lossless`、`adapted-degraded` 或 `unsupported-explicit` 状态与证据；不允许发布时存在 `unknown`。
  - 附件必须区分 `path-handoff` 和 `direct-upload`。本地路径交给 CLI agent 只证明 agent 有机会读取文件，不证明 Gateway 模型已收到文件内容。
  - 非视觉模型收到图片、视频默认按降级或 explicit unsupported 处理；音频只有直接 audio endpoint 文件上传才声称 direct support。
  - secrets 文件 0600 断言必须在支持 Unix chmod 的测试根执行；WSL 指向 Windows 临时目录时 `chmod(0600)` 可能被 drvfs 显示为 `0777`。
  - live smoke 的 `fetch failed` 必须返回脱敏 `diagnostics.network`，至少包含上游/代理连接失败的 cause code、地址端口和代理来源；代理凭据不能出现在响应、日志或 health lastError 中。
  - provider circuit 由 smoke/test 自动更新，但恢复动作必须走 `POST /api/model-gateway/providers/:providerId/health/reset`，避免手动编辑 registry 破坏审计。
  - protocol matrix runner 负责生成 JSON/Markdown 验收报告，把 route、tool、MCP/compatibility、附件和降级边界放在同一份 release 证据里。
- 本地验证：
  - `npm run typecheck:api`
  - `npm run build:api`
  - `node --test tests/system/model-gateway-service.test.mjs tests/system/model-gateway-protocol-matrix-smoke-script.test.mjs`
