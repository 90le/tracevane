# 研究先行开发清单

## 2026-07-11 三系统本地开发启动器

- 范围：Windows、macOS、Linux 的本地 API + Web 开发启动、受管进程清理、端口选择和原生依赖环境边界。
- 外部参考：
  - Node.js `child_process.spawn()` 是稳定 API；Windows 的 `.cmd` 文件不能按 Unix 可执行文件直接启动，应通过 Node 的 npm JavaScript CLI 或 `cmd.exe` 处理。
  - Node.js 的 `net` API 可用本地 listen 探测端口，不依赖 `lsof`、`fuser` 等 Unix 命令。
  - npm `ci` 以 lockfile 做干净、冻结的安装；含原生 optional dependency 的 `node_modules` 必须在实际运行平台安装。
- 本次决策：
  - `dev:restart` 与 `dev:fresh` 改为 Node 原生 `scripts/dev-runtime.mjs`，不再要求 Bash、WSL、tmux、setsid、pgrep、lsof 或 fuser。
  - 仅终止由该启动器 PID 文件登记的进程：Windows 使用 `taskkill /T /F`，macOS/Linux 使用独立进程组；若前端端口属于未登记的进程，则明确失败并要求用户改端口，避免误杀。
  - `npm.cmd` 在 Windows 上通过 `npm_execpath` 和当前 Node 解释器启动，避免 shell 字符串和平台特有参数解析。
  - 同一工作目录一次只用一个原生 Node/npm 平台；Windows 与 WSL/Linux 切换时删除根与 `apps/web` 的 `node_modules` 后重新执行 `npm ci`。
- 本地验证：
  - `node --check scripts/dev-runtime.mjs`
  - Windows 原生 Node：`TRACEVANE_WEB_PORT=5180 npm run dev:restart`，确认 API `3761` 和 Web `5180` 健康就绪。

## 2026-07-10 消息接入全面重构

- 范围：Channel Connectors 全部页面、飞书/Octo 账号创建、账号与路由配置模型、连接状态、热重载与故障恢复。
- 本地参考：
  - `types/channel-connectors.ts` 记录 v3 对象边界、数据面和前后端共享契约。
  - `types/channel-connectors.ts` 是账号、工作区、分发策略、标准入站、计划影响和运行状态的共享线协议来源。
- 外部参考：
  - 飞书官方 `@larksuiteoapi/node-sdk` 的 `registerApp` 基于 OAuth 2.0 Device Authorization Grant，返回验证 URL、过期时间和最终凭据，二维码渲染由调用方负责。
  - OpenClaw 当前飞书向导支持手动/二维码两条路径，并把多账号和 Agent bindings 分开；交互式添加账号时再询问是否绑定 Agent。
  - OpenClaw 飞书排障要求分别验证应用发布、`im.message.receive_v1`、WebSocket、权限和 gateway 状态。
  - cc-connect 飞书文档把机器人能力、权限、事件订阅、卡片回调、版本发布和长连接列为独立配置/排障步骤。
  - Octo 目前没有可公开验证的外部配置规范；以本地 `/v1/bot/register` transport、返回字段和系统测试作为当前契约。
- 本次决策：
  - v3 采用 `ChannelAccount + DeliveryTarget + DeliveryPolicy`；默认目标是一等字段，来源规则只选择 target 并收紧策略，不再覆盖 Agent、目录或凭据。
  - 一个 enabled 账号最多一个物理连接；v3 daemon 直接从账号、target 和 policy 构造运行时映射，该映射不是持久化或可编辑产品对象。
  - 配置使用 plan/apply、revision、备份和 last-known-good；账号连接参数差量重连，resolver/target 更新不重连平台且不取消进行中 turn。
  - 入站标准化后经过持久去重记录、账号队列、确定性 resolver、稳定会话身份、工作区协调器和 Reply Outbox。
  - 账号可用状态按“已保存、已连接、首条真实入站、回复投递”分层，不再用 enabled、HTTP 200 或长连接在线替代完整可用。
  - 顶层路由页退出；来源例外收进账号详情，Agent 工作区作为可被多账号复用的一等页面。
- 本地验证：
  - `npm run typecheck:api`、`npm run typecheck:web`
  - v3 路由、账号连接、入站、工作区队列、Reply Outbox 和会话身份系统测试。
  - Channel Connectors service、web contract、真实 daemon 启动与单账号差量重连测试。
  - 1440x1000、845x834、390x844 浏览器截图和交互 smoke；密钥显隐、飞书二维码与横向溢出检查。

## 2026-07-09 Model Gateway 用量统计收窄

- 范围：Model Gateway `/api/model-gateway/usage` 响应、模型路由“用量”页面、对应系统测试。
- 本地参考：
  - `apps/api/modules/model-gateway/service.ts` 现有 usage ledger 是本地 runtime 请求账本，不是 provider billing API。
  - `types/model-gateway.ts` 是前后端 `/usage` contract 的共享类型来源。
  - `apps/web/src/features/model-gateway/views/UsageView.tsx` 是用户可见用量页面。
- 外部参考：
  - New API 将网关定位为含 usage analytics / cost accounting 的统一网关；其 `service/text_quota.go` 在 upstream usage 缺失时使用估算 prompt tokens 构造合成 usage，并在日志中保存 prompt/completion token。
  - OpenAI usage 返回 `prompt_tokens`、`completion_tokens`、`total_tokens`，缓存细节在 `usage.prompt_tokens_details.cached_tokens`。
  - Anthropic Messages API 响应包含 usage 计数字段；Claude prompt caching 场景也通过 usage 暴露 cache read / creation token。
- 本次决策：
  - `/usage` 返回请求次数、总 token 消耗、输入 token 消耗、输出 token 消耗，并按模型和日期聚合；默认查询最近一周，`range=all` 查询全部，`range=custom` 按指定日期范围筛选。
  - 用量页面以仪表盘头部、模型排行图、每日趋势图和统计口径面板重排；模型排行支持按次数、总 token 消耗、输入消耗、输出消耗排序，底部重复表格删除。
  - 用量页面默认显示最近一周，支持切换全部或指定日期筛选；筛选参数只回显 `range/dateFrom/dateTo`，不回显 gateway key 等敏感查询。
  - 每条请求记录都必须有 usage 对象。入账优先使用 provider 返回的 usage；provider 不返回 usage 时，用请求体估算 input tokens，成功响应再用响应体或流式 JSON payload 估算 output tokens，失败/无响应请求的 output tokens 记 0，避免出现无法统计的空 usage。
  - 删除用量页面和 `/usage` 响应中的 metered 覆盖率、缓存证据、provider/appScope breakdown、read window、ledger path、延迟等扩展统计。
  - 运行时内部 request log / usage summary 继续保留缓存、total token、延迟等诊断字段，避免影响网关转发、健康诊断和协议兼容测试。
- 本地验证：
  - `npm run typecheck:web`
  - `npm run typecheck:api`
  - `node --test tests/system/web-model-gateway.test.mjs`
  - `npm run build:api`
  - `node --test --test-name-pattern "usage ledger summarizes|estimates token usage|estimates input usage|daemon writes runtime metadata" tests/system/model-gateway-service.test.mjs`
  - `git diff --check`

## 2026-07-09 Model Gateway 三协议与附件验收收口

- 范围：Model Gateway 路由、OpenAI Responses / Chat Completions / Anthropic Messages 协议互转、Codex / Claude Code / OpenCode CLI agent 验收、工具流、MCP、附件和多模态降级。
- 外部参考：
  - OpenAI Responses API `POST /responses` 支持文本、图片输入、JSON 输出、自定义代码和 web/file search 等工具。
  - OpenAI Chat Completions 是基于 messages 的响应接口；OpenAI 文档建议新项目优先考虑 Responses。
  - OpenAI file inputs 在 Responses 中通过 `input_file` 的 base64、Files API file id 或外部 URL 表达；本地路径不是直接上传。
  - OpenAI audio transcription 接收实际音频文件对象；只给本地文件名不等于上传。
  - Anthropic Messages streaming 使用 SSE，覆盖文本、tool use 和 extended thinking delta。
  - Claude Code CLI 的 `stream-json`、`--mcp-config`、permission prompt 选项是 Channel Connector 驱动边界。
  - OpenCode CLI 的 `opencode run`、MCP servers 和 custom tools 是 OpenCode agent 验收边界。
- 本次决策：
  - “100%支持”定义为每个协议/agent/工具/附件格子都有 `native`、`adapted-lossless`、`adapted-degraded` 或 `unsupported-explicit` 状态与证据；不允许发布时存在 `unknown`。
  - 附件必须区分 `path-handoff` 和 `direct-upload`。本地路径交给 CLI agent 只证明 agent 有机会读取文件，不证明 Gateway 模型已收到文件内容。
  - 非视觉模型收到图片、视频默认按降级或 explicit unsupported 处理；音频只有直接 audio endpoint 文件上传才声称 direct support。
  - secrets 文件 0600 断言必须在支持 Unix chmod 的测试根执行；WSL 指向 Windows 临时目录时 `chmod(0600)` 可能被 drvfs 显示为 `0777`。
  - live smoke 的 `fetch failed` 必须返回脱敏 `diagnostics.network`，至少包含上游/代理连接失败的 cause code、地址端口和代理来源；代理凭据不能出现在响应、日志或 health lastError 中。
  - provider circuit 由 smoke/test 自动更新，但恢复动作必须走 `POST /api/model-gateway/providers/:providerId/health/reset`，避免手动编辑 registry 破坏审计。
  - protocol matrix runner 负责生成 JSON/Markdown 验收报告，把 route、tool、MCP/compatibility、附件和降级边界放在同一份 release 证据里。
- 本地验证：
  - `npm run typecheck:api`
  - `npm run build:api`
  - `node --test tests/system/model-gateway-service.test.mjs tests/system/model-gateway-protocol-matrix-smoke-script.test.mjs`
