## 2026-07-12 Windows 守护服务无窗口与确定性状态

- 范围：Model Gateway、Channel Connectors、OpenClaw Recovery 共享 Windows current-user Task Scheduler supervisor 的启动窗口、任务状态与健康确认；不改变 macOS launchd、Linux systemd、业务配置或 Codex 本地配置。
- 外部参考：
  - [Windows PowerShell 5.1 命令行契约](https://learn.microsoft.com/zh-cn/powershell/module/microsoft.powershell.core/about/about_powershell_exe?view=powershell-5.1)定义 `-NoProfile`、`-NonInteractive`、`-WindowStyle Hidden` 与 `-EncodedCommand`；Windows 计划任务用隐藏、非交互 PowerShell 会话同步托管既有 watchdog，避免 Task Scheduler 直接启动 console-subsystem `node.exe`。
  - [Task Scheduler `TASK_STATE` 枚举](https://learn.microsoft.com/en-us/windows/win32/api/taskschd/ne-taskschd-task_state)给出稳定数值：Unknown=0、Disabled=1、Queued=2、Ready=3、Running=4；状态分支读取该数值与 `Enabled`，不解析本地化命令文本。
  - [Task Scheduler `Hidden` 设置](https://learn.microsoft.com/en-us/windows/win32/taskschd/taskschedulerschema-hidden-settingstype-element)只控制任务是否默认显示在管理 UI，不控制进程窗口，因此不用于解决弹窗。
  - [Task Scheduler LogonType](https://learn.microsoft.com/en-us/windows/win32/taskschd/taskschedulerschema-logontype-simpletype)明确 S4U 不保存密码但不能访问网络或加密文件；网关与消息接入继续使用 `InteractiveToken + LeastPrivilege`，不以牺牲运行能力换取无窗口。
  - [Task Scheduler 错误与成功常量](https://learn.microsoft.com/en-us/windows/win32/taskschd/task-scheduler-error-and-success-constants)明确禁用任务无法运行；Start/ensure-running 因此先执行 `/Change /ENABLE`。
  - [Windows Job Objects](https://learn.microsoft.com/en-us/windows/win32/procthread/job-objects)明确关联进程的子进程默认继承同一 Job，`JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` 会在最后一个句柄关闭时终止完整关联树；Windows 8 / Server 2012 起支持嵌套 Job。
  - [`AssignProcessToJobObject`](https://learn.microsoft.com/en-us/windows/win32/api/jobapi2/nf-jobapi2-assignprocesstojobobject)建议先以 `CREATE_SUSPENDED` 创建进程再分配 Job；[`CreateProcessW`](https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessw)要求可变 command-line buffer，最大 32,767 字符；[`CREATE_NO_WINDOW`](https://learn.microsoft.com/en-us/windows/win32/procthread/process-creation-flags)用于无控制台窗口启动 console 进程。
  - [`OpenProcess`](https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-openprocess)可用 `SYNCHRONIZE` 权限取得宿主进程等待句柄；[`WaitForMultipleObjects`](https://learn.microsoft.com/en-us/windows/win32/api/synchapi/nf-synchapi-waitformultipleobjects)可同时等待 daemon 与 watchdog；[`TerminateJobObject`](https://learn.microsoft.com/en-us/windows/win32/api/jobapi2/nf-jobapi2-terminatejobobject)用于 watchdog 异常消失时立即终止该代完整 Job 进程树。
- 本次决策：
  - Windows task action 改为隐藏 Windows PowerShell 宿主；PowerShell 5.1 的旧 native argv binder 会丢空参数并改变引号/尾随反斜杠，所以宿主只传一个 Base64(JSON) payload，由 watchdog 还原 daemon entry 与完整 argv。
  - Task Scheduler `/End` 实测只终止 PowerShell action，不会自动清理 Node 后代。watchdog 接收并监测宿主 PID；每代 daemon 由隐藏 PowerShell + Win32 Job Object runner 以 suspended 状态分配到 `KILL_ON_JOB_CLOSE` Job 后再恢复。runner 同时等待 daemon 与 watchdog：daemon 根退出时先清空后代再允许重启，watchdog 被强杀或崩溃时也立即清空该代完整进程树，避免孤儿与重复守护。
  - stop/restart/repair/uninstall 与 session takeover 对 Running/Queued 任务采取 fail-closed：必须同时确认旧 runtime PID 已消失、健康端点已退出、Task Scheduler 复查已非 Running/Queued；runtime PID 缺失或损坏不能作为“已停止”证据，且修复在确认旧 owner 停止前不覆盖 task template。
  - Task Scheduler `/Run` 返回不等于 daemon 已就绪；启动证明在 8 秒绝对窗口内持续复核 runtime PID 存活、带 PID 归属的 bounded readiness 与原生 Running，三项同时成立才成功。只读 status 使用同一证明但不等待过渡窗口。Ready/Disabled/Queued 但 runtime PID 仍存活时显示 degraded 并阻断重复启动，而不是 stopped 或再次 `/Run`。
  - 本地 XML 缺失不再等价于任务未安装：Windows 始终复查原生 Task Scheduler。只有稳定 task-not-found 证据才允许幂等清理；原生任务仍存在时继续支持 status/stop/uninstall。repair 在无法读取旧 XML、因而没有可信回滚材料时会在停止任何 owner 前 fail-closed，避免修复失败反而中断现有服务。
  - 原生 Task 已不存在但 runtime PID 仍存活时，先用 session supervisor 的精确 PID 归因。不可归因的进程显示 degraded，并阻断 persistent/session takeover；可归因的 session owner 迁移到 persistent 前也必须证明旧 PID 与健康端点都已退出，不能仅凭 `session.stop()` 返回值注册第二 owner。
  - 已安装 Task 处于 Ready/Disabled 而共享 runtimePath 指向合法 session PID 时，同样先做 session 归因：persistent status 保持 stopped，session start/ensure 幂等，切回 persistent 才停止并证明 session 完整退出。persistent stop/uninstall 不把该 session PID 误认成 Task 后代。
  - 已安装 Task 处于 Queued 且 runtimePath 仍属于 session 时不能只做幂等返回：先 `/End`，再仅以原生 Task 状态证明已离开 Queued/Running；不要求合法 session PID 或端点退出。反向 persistent takeover 则先停 session，再排空 Queued Task，最后启动唯一 persistent owner。
  - stale XML 会让原生 Queued 在统一状态中表现为 `stale-config + active=null`；是否需要排空 Task 因此复用 `scheduledTaskMayOwnRuntime`，而不是只比较 `state=starting`。`/End` 前后都重读共享 runtimePath：仍是合法 session PID 时仅证明原生任务 inactive；若 Queued→Running 期间替换为新的 Task PID，保留 session 的路径等待新 PID 退出但不要求共享端点下线，persistent takeover/repair 则同时证明新 PID、端点与原生任务都已退出后才注册或启动。
  - 只有原生 Queued 或 stale-config 掩盖的 Queued 能把 replacement PID 归因给 Task；Ready/Disabled 状态在 session 归因后出现不同的 live PID 时 fail-closed，不停止合法 session，也不执行 `/Run`。persistent takeover 在 `session.stop` 前后都重读共享 runtimePath，session restart 还在新 owner 启动前复核，防止任意 live PID 利用竞态绕过 endpoint 或 `/End` 证明，也避免在已经识别冲突时无必要中断健康 session。
  - Windows install/repair 拆分 registration 与 activation。`/Create` 成功而 `/Change` 或 `/Run` 失败时保留新原生定义与新 XML 一致，并报告 degraded；fresh `/Create` 失败后仅在原生复查仍为 task-not-found 时删除 XML；active repair 的模板写入、注册、激活或 readiness 任一失败都用旧 XML 重新注册并恢复旧 owner，不再只回滚文件。
  - Windows status 通过固定 Task Scheduler COM probe 输出 UTF-8 JSON；COM 错误 HRESULT 延续现有 task-not-found/permission 分类，避免中文 Windows CP936 输出造成乱码或状态分支漂移。
  - Disabled/Ready 映射 stopped，Queued 映射 starting，Running 再以 daemon health 与 session owner 复核确认 running/degraded，Unknown、空输出或损坏证据映射受保护的 unknown。session owner 的健康端口不得把 Ready 的 persistent task 误判为 running。

## 2026-07-12 三系统当前用户守护服务

- 范围：Model Gateway、Channel Connectors、OpenClaw Recovery 的统一会话托管与持久化生命周期，以及 Windows、macOS、Linux 的当前用户原生 supervisor 契约。

  - [Task Scheduler XML schema](https://learn.microsoft.com/en-us/windows/win32/taskschd/task-scheduler-schema)是注册任务的 XML 契约；[`RestartOnFailure`](https://learn.microsoft.com/en-us/windows/win32/taskschd/taskschedulerschema-restartonfailure-settingstype-element)规定任务失败后的外层恢复；[`Count`](https://learn.microsoft.com/en-us/windows/win32/taskschd/taskschedulerschema-count-restarttype-element)基于 `unsignedByte`，因此生成模板使用合法上限 `255`，而不是超范围计数。
  - [Apple Creating Launch Daemons and Agents](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/CreatingLaunchdJobs.html)明确用户个人 `~/Library/LaunchAgents`、`ProgramArguments` 数组与 `KeepAlive` 的角色，并明确用户登出时会向该用户的 agents 发送 `SIGTERM`；因此 `RunAtLoad + KeepAlive` 只描述登录 GUI 会话内行为，不承诺登出后继续存活。
  - [systemd unit 文档](https://www.freedesktop.org/software/systemd/man/255/systemd.unit.html)列出 `~/.config/systemd/user` 当前用户 unit 搜索路径；[systemd service 文档](https://www.freedesktop.org/software/systemd/man/255/systemd.service.html)定义 `Restart=on-failure`；[`systemctl`](https://www.freedesktop.org/software/systemd/man/255/systemctl.html)明确 `--user` 连接调用者的用户服务管理器；[`loginctl`](https://www.freedesktop.org/software/systemd/man/latest/loginctl.html)明确 `enable-linger` 会在开机创建并在登出后保留用户管理器。本实现不自动改变 linger 策略。
  - [Microsoft WSL systemd 文档](https://learn.microsoft.com/en-us/windows/wsl/systemd)明确 WSL 可启用自己的 systemd，但 systemd 服务不会让 WSL 实例保持存活；WSL 因而按独立 Linux 环境处理，不复用 Windows Task Scheduler 注册。
- 本次决策：
  - 三个 daemon 域共享同一状态、动作、错误码、模板生成、固定命令执行和 owner 切换边界；域内 wrapper 只保留配置与兼容响应映射，不再各自实现原生 supervisor。
  - Session 模式由 Tracevane API 进程直接拥有每域唯一子进程，意外退出最多重启 3 次；standalone API 的正常信号与已处理 fatal shutdown 会先停止 owned children，但强杀、宿主崩溃或断电不保证 cleanup。该模式不写 template、不注册系统服务。Persistent 模式必须显式安装，且仅写当前用户目录和当前用户服务管理器，不要求管理员/root，也不是 machine-wide service。
  - `status/start/stop/restart/install/repair/uninstall` 使用统一生命周期。`install` 原子写 template、注册/enable、启动并验证 readiness；显式 `repair` 可用于已安装服务并执行重写、重注册、重启，且是 `stale-config` 或 `runtime-not-ready` 的必需恢复路径；`uninstall` 先停止并注销/disable，再删除生成 template，但保留业务配置、运行状态与日志。
  - Session 与 Persistent 切换前必须停止并验证另一 owner；同一 daemon 不主动运行两个 owner。Node 路径、构建 entry、checkout/工作目录、配置路径或 template 变化都会改变 fingerprint 并产生 `stale-config`，由 repair 收口。
  - 生成模板包含平台服务管理器所需的 identity、trigger 与 recovery 元数据；其中 launch payload 只引用固定 Node、entry、工作目录、配置路径和显式 `--config` token。secrets、代理凭证与浏览器提供的任意 command/args/cwd/env 均不进入 template、进程参数、结构化结果或日志证据。
  - Windows 使用当前用户登录触发、`InteractiveToken`、`LeastPrivilege` 与 `MultipleInstancesPolicy=IgnoreNew`。Task 内的 Tracevane watchdog 以固定 1 秒退避恢复 daemon；Task Scheduler 的 `RestartOnFailure`（`PT1M`、`Count=255`）是 watchdog 自身失败时的外层恢复。生命周期使用 `/Create`、`/Run`、`/End`、`/Delete`、`/Query /HResult`。
  - macOS template 位于 `~/Library/LaunchAgents`，使用 `gui/$UID` 域和 `bootstrap/enable/kickstart/bootout/print`，plist 含 `RunAtLoad + KeepAlive`；这不改变用户登出语义。
  - Linux template 位于 `~/.config/systemd/user`，使用 `systemctl --user daemon-reload/enable/start/stop/restart/disable/is-active/is-enabled` 与 `Restart=on-failure`。卸载在删除 unit 后再次 `daemon-reload`；不自动执行 `loginctl enable-linger`，因此不承诺登出后或无用户管理器时存活。
  - WSL 是独立 Linux runtime。Windows 与 WSL/Linux 之间不共享 `node_modules`、构建产物、runtime metadata 或 supervisor 注册；依赖、构建和 manager 操作必须在实际 owner 环境完成。

## 2026-07-09 Model Gateway 用量统计收窄

- 范围：Model Gateway `/api/model-gateway/usage` 响应、模型路由“用量”页面、对应系统测试。
- 本地参考：
  - `apps/api/modules/model-gateway/service.ts` 现有 usage ledger 是本地 runtime 请求账本，不是 provider billing API。
  - `types/model-gateway.ts` 是前后端 `/usage` contract 的共享类型来源。
  - `apps/web/src/features/model-gateway/views/UsageView.tsx` 是用户可见用量页面。
- 外部参考：
  - New API 将网关定位为含 usage analytics / cost accounting 的统一网关；其 `service/text_quota.go` 在 upstream usage 缺失时使用估算 prompt tokens 构造合成 usage，并在日志中保存 prompt/completion token。
  - OpenAI usage 返回 `prompt_tokens`、`completion_tokens`、`total_tokens`，缓存细节在 `usage.prompt_tokens_details.cached_tokens`。
  - Anthropic Messages API 响应包含 usage 计数字段；Claude prompt caching 场景也通过 usage 暴露 cache read / creation token。
- 本次决策：
  - `/usage` 返回请求次数、总 token 消耗、输入 token 消耗、输出 token 消耗，并按模型和日期聚合；默认查询最近一周，`range=all` 查询全部，`range=custom` 按指定日期范围筛选。
  - 用量页面以仪表盘头部、模型排行图、每日趋势图和统计口径面板重排；模型排行支持按次数、总 token 消耗、输入消耗、输出消耗排序，底部重复表格删除。
  - 用量页面默认显示最近一周，支持切换全部或指定日期筛选；筛选参数只回显 `range/dateFrom/dateTo`，不回显 gateway key 等敏感查询。
  - 每条请求记录都必须有 usage 对象。入账优先使用 provider 返回的 usage；provider 不返回 usage 时，用请求体估算 input tokens，成功响应再用响应体或流式 JSON payload 估算 output tokens，失败/无响应请求的 output tokens 记 0，避免出现无法统计的空 usage。
  - 删除用量页面和 `/usage` 响应中的 metered 覆盖率、缓存证据、provider/appScope breakdown、read window、ledger path、延迟等扩展统计。
  - 运行时内部 request log / usage summary 继续保留缓存、total token、延迟等诊断字段，避免影响网关转发、健康诊断和协议兼容测试。

  - `npm run build:api`

## 2026-07-09 Model Gateway 三协议与附件验收收口

- 范围：Model Gateway 路由、OpenAI Responses / Chat Completions / Anthropic Messages 协议互转、Codex / Claude Code / OpenCode CLI agent 验收、工具流、MCP、附件和多模态降级。
- 外部参考：
  - OpenAI Responses API `POST /responses` 支持文本、图片输入、JSON 输出、自定义代码和 web/file search 等工具。
  - OpenAI Chat Completions 是基于 messages 的响应接口；OpenAI 文档建议新项目优先考虑 Responses。
  - OpenAI file inputs 在 Responses 中通过 `input_file` 的 base64、Files API file id 或外部 URL 表达；本地路径不是直接上传。
  - OpenAI audio transcription 接收实际音频文件对象；只给本地文件名不等于上传。
  - Anthropic Messages streaming 使用 SSE，覆盖文本、tool use 和 extended thinking delta。
  - Claude Code CLI 的 `stream-json`、`--mcp-config`、permission prompt 选项是 Channel Connector 驱动边界。
  - OpenCode CLI 的 `opencode run`、MCP servers 和 custom tools 是 OpenCode agent 验收边界。
- 本次决策：
  - “100%支持”定义为每个协议/agent/工具/附件格子都有 `native`、`adapted-lossless`、`adapted-degraded` 或 `unsupported-explicit` 状态与证据；不允许发布时存在 `unknown`。
  - 附件必须区分 `path-handoff` 和 `direct-upload`。本地路径交给 CLI agent 只证明 agent 有机会读取文件，不证明 Gateway 模型已收到文件内容。
  - 非视觉模型收到图片、视频默认按降级或 explicit unsupported 处理；音频只有直接 audio endpoint 文件上传才声称 direct support。
  - secrets 文件 0600 断言必须在支持 Unix chmod 的测试根执行；WSL 指向 Windows 临时目录时 `chmod(0600)` 可能被 drvfs 显示为 `0777`。
  - live smoke 的 `fetch failed` 必须返回脱敏 `diagnostics.network`，至少包含上游/代理连接失败的 cause code、地址端口和代理来源；代理凭据不能出现在响应、日志或 health lastError 中。
  - provider circuit 由 smoke/test 自动更新，但恢复动作必须走 `POST /api/model-gateway/providers/:providerId/health/reset`，避免手动编辑 registry 破坏审计。
  - protocol matrix runner 负责生成 JSON/Markdown 验收报告，把 route、tool、MCP/compatibility、附件和降级边界放在同一份 release 证据里。
- 本地验证：
  - `npm run typecheck:api`
  - `npm run build:api`
  - `node --test tests/system/model-gateway-service.test.mjs tests/system/model-gateway-protocol-matrix-smoke-script.test.mjs`
