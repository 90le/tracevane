# Tracevane 界面设计守则

> 状态：Active（设计宪法，原型重设计时以此为准）
> 更新：2026-06-30
> 说明：历史 HTML 原型已删除；当前以运行中的 React 前端、`apps/web/src/design/theme.css` token、本守则和 `docs/ide-code-editor-solution/` 作为界面约束。

本文是 Tracevane **全部页面**的统一设计宪法，不是某一个页面的规范。Tracevane 是本地优先的 AI Agent 控制工作台，围绕模型、Provider、CLI Agent、IM 渠道、文件、工具、审批、长任务、自愈和证据链组织页面。所有页面共享同一套应用壳、配色、布局骨架与交互模式，只在领域约束上各自区分。

## 0. 2026-06-30 IDE / Code Editor 方案修正

当前设计最高优先级从笼统的 “Workspace-first Industrial Studio” 收敛为 **文件管理器在线编辑器 + 独立 IDE 工作台** 两种产品形态，并共享编辑内核。旧 Aurora 仍作为 token 和整体气质参考，但代码编辑器、终端、表格、日志、diff 等信息密集区域必须优先实色、清晰边界、低阴影、低装饰。

硬边界：

- **文件管理器** 管目录、文件列表、上传下载、新建、删除、重命名、移动、复制、搜索、回收站、权限与文件属性；它可以通过事件打开在线编辑器或独立 IDE，但不直接操作 Monaco model。
- **文件管理器在线编辑器** 是快速文本编辑能力：Monaco、多 Tab、保存/全部保存、dirty 状态、关闭确认、搜索替换、跳转行、字体大小、主题入口和状态栏。它不承载终端、Git、LSP、调试、复杂 docking layout 或插件系统。
- **独立 IDE 工作台** 是项目级开发入口：Activity Bar、Side Bar、Editor Area、Panel Area、Status Bar、可拆分编辑组、Tab 拖拽、布局保存/恢复/重置，并分阶段接入 Terminal、Problems、Output、Git、LSP、任务运行和 Debug。
- **共享内核** 只能放文件读写、Workspace 上下文、EditorService、MonacoModelService、DirtyState、SaveService、LanguageResolver、Search/Diff/Permission 等底层能力；不共享页面容器、Docking 布局、IDE Activity Bar、底部终端面板或命令面板组织。
- Terminal 是独立 IDE / Workspace 的 shell/process 工具，不是 CLI Agents。CLI Agents 只管 Codex/Claude/OpenCode runtime readiness 和 Agent Runs。
- `/files` 不再定义为只读 Evidence/Artifacts；它是文件管理器入口。Evidence/Artifacts 可以作为独立领域或文件管理器中的筛选/集合视图存在，不能篡改文件管理器的主对象。
- 移动端不是压缩桌面多栏：文件管理器使用列表/属性/操作 Sheet；在线编辑器使用全屏编辑面板；独立 IDE 使用 Explorer / Edit / Terminal / Problems / Agent 的单栏模式切换，并为所有 hover/右键动作提供触控替代。

具体执行以 `docs/ide-code-editor-solution/` 为准；旧 Workspace 相关设计文档只作为历史背景，凡与该方案冲突时以后者为准。

## 1. 设计方向

Tracevane 当前采用 **Aurora** 作为视觉基线：柔和整体背景 + 极淡极光氛围层、抬升的实色面板、克制的强调色、清晰的字号字重层级。浅色默认，深色近黑有真实纵深，避免纯黑纯白。它保留 macOS Material 的分层节奏与精致感，但不默认重毛玻璃；信息密集区一律实色。

**不可动摇的准绳：美观服务于清晰度，装饰服务于信息。**

Tracevane 本质是本地 Agent 工作流控制台——重数据、重操作、重状态判断。因此：

- **遵循 Aurora 外壳**：柔和画布 + 极淡氛围、扁平分组侧栏、轻顶栏、实色工作面、细分隔线、低阴影。
- **不无脑照搬 macOS 所有元素**：不画交通灯、不做纯营销装饰、不堆与本项目无关的原生 chrome。每个材质/圆角/阴影都要能回答"它帮用户看清了什么或做对了什么"。
- **装饰随信息密度退让**：终端、表格、日志、代码这类信息密集区，材质退后、圆角收紧、阴影压低；空旷的仪表盘/空状态才允许材质与阴影更舒展。
- **氛围不喧宾夺主**：极光氛围层仅做空间纵深，禁止离散光球、禁止单一色相铺满整页。

## 2. 产品气质

安静、清晰、可扫描、可长时间使用，同时具备高级质感。优先服务重复操作、状态判断和恢复路径，而非视觉表演。所有页面共享这一气质，不因页面不同而分裂出多套视觉语言。

## 3. 材质与分层

桌面层（最底）放柔和整体背景，浅色不是纯白，深色不是纯黑。背景只提供空间感，不承担装饰主角。

三种材质层级，按"是否浮动"递进，**全项目统一**：

| 材质                 | 用途                                            | 模糊                               | 透明度       |
| -------------------- | ----------------------------------------------- | ---------------------------------- | ------------ |
| **thin（薄）**       | 侧栏、检视器等常驻面板                          | `blur(30px) saturate(180%)`        | 较低，透桌面 |
| **thick（厚）**      | 主舞台、输入区                                  | `blur(40px) saturate(180%)`        | 中等         |
| **floating（浮动）** | 工具卡片、代码块、弹层、菜单、tooltip、右键菜单 | `blur(24px) saturate(200%)` 或实色 | 高，近不透   |

Aurora 映射：

- 侧栏和顶栏使用 thin，但玻璃感收敛，优先细边界和低阴影。
- 主舞台使用 thick 或实色面板，信息密集页面优先实色。
- 弹层、菜单、命令面板才使用 floating。

规则：

- 浮动层级越高，圆角越大、阴影越深、透明度越高。
- 同屏材质层级不超过 3 层，避免"层叠迷宫"。
- 信息密集区（表格、终端、日志、配置编辑器）用 thin 或实色，不用 floating——避免毛玻璃糊掉文字。

## 4. 圆角与阴影

圆角按"材料越浮越大"递进，全项目统一：

| 对象                           | 圆角     |
| ------------------------------ | -------- |
| 控件、按钮、输入框             | 8px      |
| 常驻面板                       | 12px     |
| 卡片、代码块、工具块、表格容器 | 14px     |
| 浮层、弹窗、菜单、右键菜单     | 16px     |
| 胶囊（pill、tag、segmented）   | 完全圆角 |

阴影用多层柔和 elevation，禁止硬边投影。默认三档：

- `shadow-1`（近）：常驻面板。`0 1px 2px` 近投影 + 半像素 hairline。
- `shadow-2`（浮）：卡片、代码块、工具块、输入区、展开条。
- `shadow-3`（弹）：弹窗、菜单、右键菜单、command palette。

## 5. 配色与主题

- 提供深色与浅色双主题，两者都要打磨到位。**所有页面两套主题都必须验证**。
- 深色使用舒适蓝灰 / graphite 背景，不使用全屏纯黑，避免眼疲劳。
- 浅色使用柔和 off-white / blue-gray 画布，不使用刺眼纯白。
- primary 用 Tracevane 蓝，teal 用作连接、自愈和恢复辅助 accent。语义色：成功绿、警告橙、危险红、信息青。
- 文字层级四档：主文 / 次文 / 辅助文 / 占位，对比度在两套主题下都要清晰可读。
- hairline（发丝分隔线）和 fill（hover 底色）在深浅两套下分别调对比度。

## 6. 统一工作台骨架

全项目共享一套工作台外壳，保证导航、密度、交互一致：

- **全局导航（单栏，thin material）**：Aurora 单栏分组导航，图标 + 文字同列显示。推荐分组为总览 / 工作台（文件管理器、独立 IDE）/ 运行 / 连接 / 证据 / 系统，当前页主色高亮（左侧色条 + soft 底）。**全局导航不用 VS Code 式活动栏**；仅独立 IDE 页面内部可以使用 Activity Bar。导航可整体折叠成纯图标条（窄屏/专注模式），折叠态才用 tooltip，弹出方向固定向右且留足留白，绝不遮挡内容。
- **命令面板（⌘K）**：全局快速跳转，搜索页面名或命令。键盘可达，是导航的补充而非替代。
- **页面顶栏（轻量工作区工具条）**：左侧面包屑/页面标题，右侧全局命令、证据篮、刷新/检查、主题等少量动作。不画交通灯。
- **中央主舞台（实色或 thick material）**：自适应主区，承载当前页核心内容。
- **右侧检视器（thin material，可选）**：需要时挂载，可收缩。不是每页都有。
- 高频操作在主区，低频/危险操作放进更多菜单、右键菜单、折叠区或确认弹窗。
- **列表项操作走右键菜单（context menu）**，不要常驻按钮挤占列表——符合桌面级控制台交互预期。
- 不使用大面积 hero、营销装饰渐变、浮动卡片堆叠。
- 卡片只用于重复 item、工具面板或 modal。

> 注意区分两级导航：**全局导航**（单栏，切换产品核心页面）与 **独立 IDE 内部 Activity Bar**（Explorer/Search/Git/Run/Extensions/Settings）是不同的东西，不要混用。文件管理器在线编辑器不使用 Activity Bar。

## 7. 信息架构：层次化，禁止堆叠

**这是 Tracevane 现状最大的毛病，也是重构的核心目标。**

现状：很多领域把全部功能堆进一个"主界面"——ConfigEditorPage 一个文件 67 个分区块、ModelGatewayControlPage 4287 行、ChatShellPage 7410 行。靠双栏/三栏硬塞，所有配置、列表、详情、状态、操作平铺在一屏。结果是认知过载、找东西靠滚、改一处怕动全局。

原则：**分层次设计，而不是堆更多栏。**

### 7.1 三层信息模型

每个领域页面应能清楚拆出三层，而不是把三层揉成一屏：

1. **主对象层（primary）**：这个页面存在的理由。是某个列表（Agent/技能/任务）、某个工作流（会话/终端）、或某个核心状态（网关 routing）。主舞台只承载这一层。
2. **关联层（related）**：主对象的从属信息——详情、绑定、Profile、运行日志。通过选中主对象后展开，或进入子页面，而不是常驻平铺。
3. **配置/低频层（config）**：不常改的设置、高级选项、诊断。收进抽屉、折叠区、设置子页，默认不可见。

### 7.2 反模式（明确禁止）

- **主界面堆一切**：把列表 + 详情 + 多组配置 + 状态面板 + 操作区全塞进一个 ControlPage 主界面，靠双栏/三栏切分。→ 拆。
- **常驻平铺所有分区**：67 个 section 平铺在一个文件，用户靠滚动和记忆找功能。→ 按三层分层，低频收起。
- **用更多分栏代替分层**：东西多了就加第四栏、第五栏。栏数 ≠ 层次，只是更挤。→ 分栏上限三栏，超出的内容下沉到子页面/抽屉。
- **平铺 vs 折叠错配**：高频路径藏进折叠区，低频配置却常驻占位。→ 高频可见、低频折叠。

### 7.3 落地手段

| 情况                   | 手段                                                       |
| ---------------------- | ---------------------------------------------------------- |
| 主对象 + 从属详情      | master-detail：左列表选中，右详情（Terminal/Files/Skills） |
| 同一主对象的多个维度   | tabs 分维度，而非全平铺（Agent: 列表/文档/绑定/高级/会话） |
| 深层配置 / 低频设置    | 子页面或抽屉（drawer），从主界面跳转，不常驻               |
| 高级/诊断细节          | 折叠区，默认收起，按需展开                                 |
| 临时操作               | 右键菜单、command palette、确认弹窗                        |
| 主界面确实需要多区共存 | 先问"能否拆成子页面/抽屉"，能拆就拆；确实必须共存才用三栏  |

### 7.4 文件管理器、在线编辑器与独立 IDE 的层次规则

这三者必须从信息架构上拆开，避免重新做成一个固定三栏 IDE 页面：

| 形态 | 主对象层 | 关联层 | 低频/配置层 | 禁止事项 |
| ---- | -------- | ------ | ------------ | -------- |
| 文件管理器 | 目录、文件列表、路径、选择与文件操作 | 属性、上传任务、操作历史、搜索结果、回收站 | 显示偏好、索引管理、权限说明 | 不直接持有 Monaco model；不把终端/Git/LSP塞进文件列表 |
| 在线编辑器 | 当前打开文件与编辑 Tab | 状态栏、搜索替换、跳转行、保存状态 | 字体大小、主题、编码/只读提示 | 不引入 Dockview；不承载项目级终端、Git、调试 |
| 独立 IDE | Editor Area 与项目工作台布局 | Explorer、Search、Problems、Output、Terminal、Status Bar | Git/LSP/Tasks/Debug 的阶段性配置 | 不手写固定三栏；不把文件管理器弹窗升级成完整 IDE |

验收时先问：用户是在“管理文件”、 “快速改一个文件”，还是“做项目级开发”。答案不同，入口和布局必须不同。

### 7.5 验证标准

判断一个页面是否层次清晰：

- 打开页面，**3 秒内**能说出"这页的主对象是什么"。
- 高频操作路径 ≤ 2 步可见，不用滚动、不用展开多个折叠区。
- 低频配置默认不可见，但能被找到（抽屉/子页面入口明确）。
- 主舞台只承载一个主对象层；关联层和配置层不与之平铺竞争空间。
- 单个页面组件目标行数 < 400（Shell 壳），超出的内容说明该下沉到子组件/子页面/store。

## 8. 组件规则

- 工具按钮优先图标 + tooltip。
- 模式选择用 segmented control / tabs。
- 二值设置用 toggle/checkbox。
- 数值用 slider/stepper/input。
- 列表项尺寸稳定，不因 hover、状态文本或动态图标抖动。
- 文本必须在移动和桌面视口内可读，不允许溢出遮挡。
- **代码块**：区分行内 code 与代码 block。block 用 floating 材质卡片，带语言标签和复制按钮，语法高亮底色清晰。
- focus 状态用 ring：`box-shadow: 0 0 0 3px accent-soft`，深浅两套都可见。

### 8.1 交互模式选择（硬约束）

不同动作用恰当的交互模式，**禁止一律用右侧抽屉/Sheet**（会误导设计与实现）：

- 切换 / 设默认 / 启停：行内控件（toggle、segmented，就地生效）。
- 单字段编辑：行内编辑（点击变输入框，回车保存）。
- 一组配置编辑：右侧 Drawer 表单或子页面，带保存条。
- 危险 / 不可逆（删除、apply、rollback、写凭据、重启服务）：Dialog 二次确认，说明影响。
- 只读详情 / 证据 / 日志 / diff：Sheet（右侧滑入）。
- 同级展开更多：accordion / 折叠区。
- 列表选中：就地更新右侧检视器，不弹层。
- 即时结果（smoke、test、保存）：Toast。
- 跨页面：路由跳转 + 命令面板。

详表以运行中的 React 组件和本守则为准。

### 8.2 交互模式收敛

交互模式数量本身也是成本。全项目交互收敛为有限集，**禁止为单个页面发明新模式**：toggle / segmented、行内编辑、Sheet（只读详情）、Drawer（表单编辑）、Dialog（危险确认）、Toast（即时反馈）、折叠区（同级更多）。超出此集的交互（如多步向导 wizard）默认拆成多个 Drawer 或子页面，不再引入新的覆盖层类型。

## 9. IDE / Code Editor 专项守则

### 9.1 产品形态

- Tracevane 同时支持两种编辑形态：文件管理器在线编辑器与独立 IDE 工作台。两者共享内核，但 UI 容器、布局系统和能力边界分离。
- 文件管理器在线编辑器可以是 Modal、Drawer、Fullscreen modal、独立路由或新浏览器标签页；第一版优先全屏弹窗/独立页面，保证轻量、可靠、快速。
- 独立 IDE 必须是可变工作台，不是固定页面。第一版也要为拖拽、拆分、持久化和 reset 留出真实布局能力。
- 不允许“先写死三栏，以后再改 Docking”。固定三栏会直接破坏 Tab 拖拽、多编辑组、Panel 切换和布局恢复。

### 9.2 布局与导航

- 独立 IDE 桌面结构：Menu/Top Bar + Activity Bar + Side Bar + Editor Dock + Panel Area + Status Bar，可增加 Auxiliary Bar，但不能让全局导航和 IDE Activity Bar 混在一起。
- Editor Area 支持多 Tab、多编辑组、横向/纵向拆分、Tab 拖动排序、拖到其他组、关闭当前/其他/右侧/已保存、预览 Tab、固定 Tab。
- Panel Area 第一版至少预留 Terminal / Problems / Output 切换；如果真实 PTY 未完成，Terminal 不伪造执行能力，只能作为占位或输出面板。
- 布局状态必须有 version、workspaceId、activeActivity、sidebar 展开态、panel 展开态、editor groups、tabs、active tab、sizes 和通用偏好；恢复失败 fallback 到默认布局，并提供“重置布局”。
- 移动端使用 mode nav / action sheet / fullscreen panel，不尝试显示完整 docking；所有右键、hover、拖拽关键动作必须有触控替代。

### 9.3 编辑器内核

- Monaco 只解决代码编辑，不解决 IDE 布局；Dockview / docking layout 只解决工作台布局，不负责文件读写；xterm.js 只解决终端 UI，不执行命令。
- 一个文件对应一个稳定 Monaco model，URI 采用 `file:///workspace/{workspaceId}/{path}` 一类规则。React state/store 不保存完整文件文本，只保存 Tab、dirty 元数据、view state 和 layout 元数据。
- 打开文件必须走 EditorService；读写文件必须走 FileService / SaveService；业务组件不得随意 create/dispose Monaco model。
- SaveService 必须携带 version/mtime 或等价冲突检测信息；保存冲突、外部修改、文件删除、只读权限、大文件截断都要有明确 UI 状态。
- 搜索替换、跳转行、快捷键、命令面板统一走 CommandService / ShortcutService，避免多个组件重复绑定 Ctrl/Cmd+S、Ctrl/Cmd+P 等快捷键。

### 9.4 文件与终端安全

- 前端只传 `workspaceId/rootId + relative path`，不得拼接绝对路径；后端必须 root guard，拒绝路径逃逸。
- 文件管理器与 IDE 共用文件 API，但权限在前后端都要体现：前端禁用不可用按钮，后端仍必须拒绝。
- 终端默认属于受限 workspace runtime；优先容器、远程或受限 cwd，不默认暴露宿主机任意 shell。
- 终端 session 必须支持 create/input/resize/kill/output 订阅；关闭终端或卸载组件时要清理 xterm 和后端 session。

### 9.5 阶段与验收

实施顺序固定为：M1 文件管理器在线编辑器 → M2 独立 IDE 基础工作台 → M3 终端和任务运行 → M4 高级编辑与文件同步 → M5 LSP/Git/调试。

- M1 验收：双击/动作打开文本文件、Monaco 编辑、多 Tab、保存/全部保存、dirty 标识、关闭确认、搜索替换、跳转行、字体大小、主题入口、状态栏、只读/大文件/不可编辑提示。
- M2 验收：独立 IDE 路由、Explorer、Dockview 编辑区、多编辑组、Panel 框架、Status Bar、布局保存/恢复/重置、刷新后恢复打开 Tab。
- M3 验收：xterm 前端、PTY/WebSocket 后端、workspace cwd、resize、kill、多终端、安全边界和任务运行入口。
- M4 验收：文件 watcher、外部修改提示、冲突处理、Diff、全局搜索、Problems/Output 基础数据流。
- M5 验收：单语言 LSP 起步、diagnostics 到 Problems、Git status/diff，stage/commit/debug 继续分阶段。

任何阶段完成声明都必须包含：桌面与移动可用性、深浅主题、键盘路径、触控替代、路径安全、保存冲突、布局恢复/重置和相关 smoke/typecheck 证据。
