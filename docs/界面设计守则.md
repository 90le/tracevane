# Tracevane 界面设计守则

> 状态：Active（设计宪法，原型重设计时以此为准）
> 更新：2026-06-21
> 说明：历史 HTML 原型已删除；当前以运行中的 React 前端、`apps/web/src/design/theme.css` token 和本守则作为界面约束。

本文是 Tracevane **全部页面**的统一设计宪法，不是某一个页面的规范。Tracevane 是本地优先的 AI Agent 控制工作台，围绕模型、Provider、CLI Agent、IM 渠道、文件、工具、审批、长任务、自愈和证据链组织页面。所有页面共享同一套应用壳、配色、布局骨架与交互模式，只在领域约束上各自区分。

## 0. 2026-06-25 Workspace-first 修正

当前设计最高优先级已经从“多领域管理台”升级为 **Workspace-first Industrial Studio**：PC 端是专业多栏本地 Agent Workspace，手机端是单栏任务控制器。旧 Aurora 仍作为 token 和整体气质参考，但 Workspace 的编辑器、终端、表格、日志、diff 等信息密集区域必须优先实色、清晰边界、低阴影、低装饰。

硬边界：

- Workspace 管文件、编辑、预览、终端、Git、证据和 Agent handoff。
- Terminal 是 Workspace 的 shell/process 工具，不是 CLI Agents。
- CLI Agents 管 Codex/Claude/OpenCode runtime readiness 和 Agent Runs，不管理普通 PTY。
- `/files` 如果保留，应是 Evidence/Artifacts，不再作为只读“文件管理器”。
- 移动端不是压缩桌面多栏，而是 Files / Edit / Terminal / Agent / Evidence 的单栏模式切换。

具体执行以 `Workspace设计文档.md`、`Workspace前端架构.md`、`文件管理设计.md`、`终端能力设计.md` 为准。

## 1. 设计方向

Tracevane 当前采用 **Aurora** 作为视觉基线：柔和整体背景 + 极淡极光氛围层、抬升的实色面板、克制的强调色、清晰的字号字重层级。浅色默认，深色近黑有真实纵深，避免纯黑纯白。它保留 macOS Material 的分层节奏与精致感，但不默认重毛玻璃；信息密集区一律实色。

**不可动摇的准绳：美观服务于清晰度，装饰服务于信息。**

Tracevane 本质是本地 Agent 工作流控制台——重数据、重操作、重状态判断。因此：

- **遵循 Aurora 外壳**：柔和画布 + 极淡氛围、扁平分组侧栏、轻顶栏、实色工作面、细分隔线、低阴影。
- **不无脑照搬 macOS 所有元素**：不画交通灯、不做纯营销装饰、不堆与本项目无关的原生 chrome。每个材质/圆角/阴影都要能回答"它帮用户看清了什么或做对了什么"。
- **装饰随信息密度退让**：终端、表格、日志、代码这类信息密集区，材质退后、圆角收紧、阴影压低；空旷的仪表盘/空状态才允许材质与阴影更舒展。
- **氛围不喧宾夺主**：极光氛围层仅做空间纵深，禁止离散光球、禁止单一色相铺满整页。

## 2. 产品气质

安静、清晰、可扫描、可长时间使用，同时具备高级质感。优先服务重复操作、状态判断和恢复路径，而非视觉表演。所有页面共享这一气质，不因页面不同而分裂出多套视觉语言。

## 3. 材质与分层

桌面层（最底）放柔和整体背景，浅色不是纯白，深色不是纯黑。背景只提供空间感，不承担装饰主角。

三种材质层级，按"是否浮动"递进，**全项目统一**：

| 材质                 | 用途                                            | 模糊                               | 透明度       |
| -------------------- | ----------------------------------------------- | ---------------------------------- | ------------ |
| **thin（薄）**       | 侧栏、检视器等常驻面板                          | `blur(30px) saturate(180%)`        | 较低，透桌面 |
| **thick（厚）**      | 主舞台、输入区                                  | `blur(40px) saturate(180%)`        | 中等         |
| **floating（浮动）** | 工具卡片、代码块、弹层、菜单、tooltip、右键菜单 | `blur(24px) saturate(200%)` 或实色 | 高，近不透   |

Aurora 映射：

- 侧栏和顶栏使用 thin，但玻璃感收敛，优先细边界和低阴影。
- 主舞台使用 thick 或实色面板，信息密集页面优先实色。
- 弹层、菜单、命令面板才使用 floating。

规则：

- 浮动层级越高，圆角越大、阴影越深、透明度越高。
- 同屏材质层级不超过 3 层，避免"层叠迷宫"。
- 信息密集区（表格、终端、日志、配置编辑器）用 thin 或实色，不用 floating——避免毛玻璃糊掉文字。

## 4. 圆角与阴影

圆角按"材料越浮越大"递进，全项目统一：

| 对象                           | 圆角     |
| ------------------------------ | -------- |
| 控件、按钮、输入框             | 8px      |
| 常驻面板                       | 12px     |
| 卡片、代码块、工具块、表格容器 | 14px     |
| 浮层、弹窗、菜单、右键菜单     | 16px     |
| 胶囊（pill、tag、segmented）   | 完全圆角 |

阴影用多层柔和 elevation，禁止硬边投影。默认三档：

- `shadow-1`（近）：常驻面板。`0 1px 2px` 近投影 + 半像素 hairline。
- `shadow-2`（浮）：卡片、代码块、工具块、输入区、展开条。
- `shadow-3`（弹）：弹窗、菜单、右键菜单、command palette。

## 5. 配色与主题

- 提供深色与浅色双主题，两者都要打磨到位。**所有页面两套主题都必须验证**。
- 深色使用舒适蓝灰 / graphite 背景，不使用全屏纯黑，避免眼疲劳。
- 浅色使用柔和 off-white / blue-gray 画布，不使用刺眼纯白。
- primary 用 Tracevane 蓝，teal 用作连接、自愈和恢复辅助 accent。语义色：成功绿、警告橙、危险红、信息青。
- 文字层级四档：主文 / 次文 / 辅助文 / 占位，对比度在两套主题下都要清晰可读。
- hairline（发丝分隔线）和 fill（hover 底色）在深浅两套下分别调对比度。

## 6. 统一工作台骨架

全项目共享一套工作台外壳，保证导航、密度、交互一致：

- **全局导航（单栏，thin material）**：Aurora 单栏分组导航，图标 + 文字同列显示。推荐分组为总览 / 运行 / 连接 / 证据 / 系统，当前页主色高亮（左侧色条 + soft 底）。**不用 VS Code 式全局图标活动栏**；仅 IDE 类页面内部可以使用活动栏。导航可整体折叠成纯图标条（窄屏/专注模式），折叠态才用 tooltip，弹出方向固定向右且留足留白，绝不遮挡内容。
- **命令面板（⌘K）**：全局快速跳转，搜索页面名或命令。键盘可达，是导航的补充而非替代。
- **页面顶栏（轻量工作区工具条）**：左侧面包屑/页面标题，右侧全局命令、证据篮、刷新/检查、主题等少量动作。不画交通灯。
- **中央主舞台（实色或 thick material）**：自适应主区，承载当前页核心内容。
- **右侧检视器（thin material，可选）**：需要时挂载，可收缩。不是每页都有。
- 高频操作在主区，低频/危险操作放进更多菜单、右键菜单、折叠区或确认弹窗。
- **列表项操作走右键菜单（context menu）**，不要常驻按钮挤占列表——符合桌面级控制台交互预期。
- 不使用大面积 hero、营销装饰渐变、浮动卡片堆叠。
- 卡片只用于重复 item、工具面板或 modal。

> 注意区分两级导航：**全局导航**（单栏，切换产品核心页面）与 **IDE 类页面的内部活动栏**（如工作区 IDE 的资源/搜索/Git/调试子面板切换）是不同的东西，不要混用。

## 7. 信息架构：层次化，禁止堆叠

**这是 Tracevane 现状最大的毛病，也是重构的核心目标。**

现状：很多领域把全部功能堆进一个"主界面"——ConfigEditorPage 一个文件 67 个分区块、ModelGatewayControlPage 4287 行、ChatShellPage 7410 行。靠双栏/三栏硬塞，所有配置、列表、详情、状态、操作平铺在一屏。结果是认知过载、找东西靠滚、改一处怕动全局。

原则：**分层次设计，而不是堆更多栏。**

### 7.1 三层信息模型

每个领域页面应能清楚拆出三层，而不是把三层揉成一屏：

1. **主对象层（primary）**：这个页面存在的理由。是某个列表（Agent/技能/任务）、某个工作流（会话/终端）、或某个核心状态（网关 routing）。主舞台只承载这一层。
2. **关联层（related）**：主对象的从属信息——详情、绑定、Profile、运行日志。通过选中主对象后展开，或进入子页面，而不是常驻平铺。
3. **配置/低频层（config）**：不常改的设置、高级选项、诊断。收进抽屉、折叠区、设置子页，默认不可见。

### 7.2 反模式（明确禁止）

- **主界面堆一切**：把列表 + 详情 + 多组配置 + 状态面板 + 操作区全塞进一个 ControlPage 主界面，靠双栏/三栏切分。→ 拆。
- **常驻平铺所有分区**：67 个 section 平铺在一个文件，用户靠滚动和记忆找功能。→ 按三层分层，低频收起。
- **用更多分栏代替分层**：东西多了就加第四栏、第五栏。栏数 ≠ 层次，只是更挤。→ 分栏上限三栏，超出的内容下沉到子页面/抽屉。
- **平铺 vs 折叠错配**：高频路径藏进折叠区，低频配置却常驻占位。→ 高频可见、低频折叠。

### 7.3 落地手段

| 情况                   | 手段                                                       |
| ---------------------- | ---------------------------------------------------------- |
| 主对象 + 从属详情      | master-detail：左列表选中，右详情（Terminal/Files/Skills） |
| 同一主对象的多个维度   | tabs 分维度，而非全平铺（Agent: 列表/文档/绑定/高级/会话） |
| 深层配置 / 低频设置    | 子页面或抽屉（drawer），从主界面跳转，不常驻               |
| 高级/诊断细节          | 折叠区，默认收起，按需展开                                 |
| 临时操作               | 右键菜单、command palette、确认弹窗                        |
| 主界面确实需要多区共存 | 先问"能否拆成子页面/抽屉"，能拆就拆；确实必须共存才用三栏  |

### 7.4 验证标准

判断一个页面是否层次清晰：

- 打开页面，**3 秒内**能说出"这页的主对象是什么"。
- 高频操作路径 ≤ 2 步可见，不用滚动、不用展开多个折叠区。
- 低频配置默认不可见，但能被找到（抽屉/子页面入口明确）。
- 主舞台只承载一个主对象层；关联层和配置层不与之平铺竞争空间。
- 单个页面组件目标行数 < 400（Shell 壳），超出的内容说明该下沉到子组件/子页面/store。

## 8. 组件规则

- 工具按钮优先图标 + tooltip。
- 模式选择用 segmented control / tabs。
- 二值设置用 toggle/checkbox。
- 数值用 slider/stepper/input。
- 列表项尺寸稳定，不因 hover、状态文本或动态图标抖动。
- 文本必须在移动和桌面视口内可读，不允许溢出遮挡。
- **代码块**：区分行内 code 与代码 block。block 用 floating 材质卡片，带语言标签和复制按钮，语法高亮底色清晰。
- focus 状态用 ring：`box-shadow: 0 0 0 3px accent-soft`，深浅两套都可见。

### 8.1 交互模式选择（硬约束）

不同动作用恰当的交互模式，**禁止一律用右侧抽屉/Sheet**（会误导设计与实现）：

- 切换 / 设默认 / 启停：行内控件（toggle、segmented，就地生效）。
- 单字段编辑：行内编辑（点击变输入框，回车保存）。
- 一组配置编辑：右侧 Drawer 表单或子页面，带保存条。
- 危险 / 不可逆（删除、apply、rollback、写凭据、重启服务）：Dialog 二次确认，说明影响。
- 只读详情 / 证据 / 日志 / diff：Sheet（右侧滑入）。
- 同级展开更多：accordion / 折叠区。
- 列表选中：就地更新右侧检视器，不弹层。
- 即时结果（smoke、test、保存）：Toast。
- 跨页面：路由跳转 + 命令面板。

详表以运行中的 React 组件和本守则为准。

### 8.2 交互模式收敛

交互模式数量本身也是成本。全项目交互收敛为有限集，**禁止为单个页面发明新模式**：toggle / segmented、行内编辑、Sheet（只读详情）、Drawer（表单编辑）、Dialog（危险确认）、Toast（即时反馈）、折叠区（同级更多）。超出此集的交互（如多步向导 wizard）默认拆成多个 Drawer 或子页面，不再引入新的覆盖层类型。

## 9. 领域约束

每个领域页面共用上述骨架与材质，只在内容与信息组织上各有侧重。

### 总览

- **仪表盘**：任务优先控制舱，只回答“现在能不能工作、下一步做什么、从哪里继续”。主信号围绕模型 / Provider / endpoint、Agent / session / task、文件 / 工具 / 审批 / 消息、自愈和证据链。

### 运行

- **会话任务**：Agent / session / task 的运行、等待、失败、完成状态是主对象。日志、工具调用、消息和证据进入详情或检视器，避免把聊天、日志和配置平铺在同一页。
- **工作区 IDE**：核心工作面，不是单一终端页。布局含内部活动栏、文件树、多 tab 编辑器 / 预览区、底部终端 / 问题 / 输出 / 端口面板、Git 和证据检视器。信息极密集，材质退后，编辑器和终端本体用实色。
- **Agent Chat**：不是通用聊天产品。Chat 必须携带文件、预览、diff、审批、工具调用和证据上下文，并能移入工作区 IDE。
- **长任务**：长时间 Agent run、cron、后台 daemon、恢复任务独立呈现；可暂停、恢复、重试、收集证据，不埋在聊天时间线里。

### 连接

- **模型网关**：路由的唯一真相——Provider、endpoint、协议、账号池、模型，以及把路由应用到客户端（App Connection 的 apply/rollback）。Provider/endpoint/模型管理是网关内置能力，不再单列 Providers 页。Gateway key 与敏感配置进 Drawer 或子页面。
- **IM 渠道**：私聊触发本地 CLI Agent 的消息、文件、权限、进度和 session 状态。账号、transport、Agent profile 和 runtime 分层，不做通用频道 CRUD。
- **外部连接**：MCP / 外部服务授权 / 第三方集成，作为“连接”面的补充。

> 域边界：**运行**管“Agent 怎么跑”，**连接**管“流量与外部如何接通”。两族页面互相 cross-link：CLI Agents 涉及路由时跳转模型网关，模型网关涉及 Agent 实例时跳转 CLI Agents。严禁在两处提供同一职责的可编辑入口。

### 证据

- **文件证据**：工作区文件、IM 附件、Agent 生成物、预览截图、日志和 diff 证据统一组织。默认行 / 表格视图，详情进入检视器。
- **审批**：文件写入、命令执行、凭据访问、修复写入都必须显示风险、diff 或 preview，再允许批准。
- **可观测性**：Session traces、工具调用、审批、Provider 错误、token / cost、验证证据按 session + toolCallId 合并和复盘。

### 系统

- **自愈守护**：OpenClaw / Tracevane 配置、服务、凭据、插件漂移的检测与修复。所有修复必须 backup、preview、apply、verify、rollback。
- **安全配置**：只保留 Tracevane 必需的兼容配置、密钥引用和危险写入预览。不复制完整 OpenClaw 管理后台表单。

> 不恢复 Dreaming、旧插件管理、旧模型链路诊断矩阵、安装修复复杂页面或通用 OpenClaw CRUD。群聊/管理类能力不作为当前主线，不在 UI 上制造复杂入口。

## 10. 验收检查

- 桌面和移动视口无重叠、无横向页面溢出。
- 工作台骨架的三栏/次级栏可折叠，折叠后主区可用、状态保留。
- 长文本、长路径、长 token、宽表格不撑破容器。
- 列表项右键菜单可用，不靠常驻按钮。
- 深浅双主题下，**每个页面**的文字对比度、hairline、hover 状态均清晰。
- 全项目视觉语言统一（材质/圆角/配色/交互），无页面间割裂。
- 按钮组不会堆成一排不可读低频操作。
- 每个页面通过层次化验证（§7.4）：主对象明确、高频路径 ≤2 步、低频已收起、无主界面堆一切。
- 页面没有已停止演进的旧模型链路用户可见残留。
- 真实浏览器 smoke 或截图检查覆盖主要交互面。

### 2026-06-26 移动端顶栏动作收敛

移动端/窄屏顶栏采用渐进披露：主栏只保留导航开关、页面标题和一个“快速操作”入口。搜索/跳转、配色、明暗主题等低频或全局动作进入右侧 Sheet；桌面大屏可以直接展示这些快捷入口。禁止在移动顶部继续堆叠多个图标按钮导致标题、面包屑和页面内容被挤压。

### 2026-06-26 文件管理器移动端搜索密度

文件管理器移动端区分两类搜索：当前目录过滤属于高频浏览工具，可保留在导航区域；跨文件名/内容搜索属于重型能力，默认折叠为一行“文件搜索”入口，展开后显示输入、选项和结果。桌面端可以默认展开重型搜索。禁止在手机首屏同时铺开路径栏、过滤框、统计卡、内容搜索、搜索选项和历史面板，导致真正的文件列表被挤出首屏。

### 2026-06-26 文件管理器移动端路径栏压缩

移动端文件管理器顶部只保留任务必要信息：根选择、当前目录标题、上传/更多、当前位置摘要、可编辑路径、当前目录过滤和隐藏文件图标。长根说明、产品说明、常用/最近路径和完整面包屑默认不铺开；完整面包屑通过当前位置 summary 展开。桌面端继续显示完整导航。移动端首屏优先保证文件列表可见，不允许辅助导航把列表整体挤到首屏之外。

### 2026-06-26 移动端列表工具区折叠

移动端列表页应优先展示对象本身。列表密度、视图布局、列配置、加载更多等属于列表设置，不应常驻铺满屏幕；默认折叠成一行摘要，展开后再显示。全局移动顶栏只保留导航、当前页标题和一个快速操作入口；面包屑、搜索/跳转、主题/配色等进入页面内导航或快速操作 Sheet。桌面端可继续展示完整工具栏和列表页脚。

### 2026-06-26 移动端文件预览编辑工具栏

文件预览/编辑弹窗在移动端默认全屏，顶部只保留文件身份、保存和“更多”。属性、历史版本、打开预览、下载、差异审阅等辅助操作进入移动“更多”区；桌面端可以常驻显示。编辑/预览正文必须占据 `minmax(0,1fr)` 主区域并在弹窗内部滚动，不允许被工具栏挤压成不可用高度。

### 2026-06-26 文档工作台移动端模式选择

源码/编辑、预览、边写边预览、预览时编辑是互斥视图模式。移动端使用单个“视图”选择器承载这些模式，避免长文本 segmented control 横向滚动；桌面端继续使用 segmented control 以保持 PC 级效率。任何模式切换都必须保持同一文件上下文，不另开窗口、不拆分为多个文件标签。

### 移动端顶部栏按钮预算（2026-06-26）

- App 全局移动顶部栏只保留导航入口和当前页面标题；搜索、主题、配色等全局次级动作放进移动导航抽屉。
- 领域页面不要在标题区堆叠主按钮和更多按钮。文件管理器的上传、新建、视图切换、刷新放在路径/搜索后的折叠“文件操作”区。
- 若后续新增移动端动作，先判断是否属于当前内容的高频动作；低频动作默认进入折叠区或上下文菜单，不回流到顶部栏。

### 文件预览媒体工具栏（2026-06-26）

图片/视频预览属于内容优先界面。移动端只常驻文件名、缩放比例和更多按钮，缩放/适应/打开等操作进入折叠工具区；桌面端可常驻工具按钮。媒体画布必须在预览弹窗内部滚动，放大后依靠真实布局尺寸而不是 `transform: scale()`，避免滚动区域与视觉尺寸不一致。

### PDF / 音频 / 二进制预览（2026-06-26）

PDF、音频和未知二进制文件都必须遵守“内容优先、操作收敛”的预览原则。PDF 只在预览区域内提供文件名、打开、下载，iframe 占据剩余高度；音频使用卡片播放器，原生 controls 横向铺满；未知二进制只给安全占位、元信息和打开/下载，不能尝试执行、解析或用大段说明挤压移动端屏幕。

### 前端类型验证守则（2026-06-26）

FileManager / Workspace 的 UI 改动必须同时通过 root `typecheck` 和 `typecheck:web`。凡是从菜单、索引、搜索结果等非 browse API 合成 `FileEntrySummary`，必须补齐权限、owner、hidden、text/image 能力等字段；不能用缺字段对象驱动预览、属性、批量操作。可选布尔值进入 UI badge/能力标签前必须显式归一，避免移动端/桌面端在 undefined 状态下产生不一致渲染。

### 2026-06-26 窄屏/平板顶部按钮预算

`<1280px` 统一按窄屏信息架构处理：App 顶栏只保留导航和页面标题；全局搜索、主题、配色进入导航抽屉。文件管理器的上传、新建、内容索引、回收站、刷新、最近/收藏位置不在顶部常驻，进入“文件操作”折叠区或路径/输入流。不得用 `md`/`lg` 断点提前恢复桌面工具栏，除非经过真实设备验证不会产生顶部按钮堆叠。

### 2026-06-26 Markdown/HTML 可视编辑边界

Markdown 的“预览时编辑”以块为边界：标题、段落、引用、列表直接编辑文字；代码和 Mermaid 编辑 fence 内容；表格使用单元格输入控件并保持表格渲染语义；HTML 片段只开放可见文字节点编辑。复杂 HTML、属性、脚本、SVG/Math、嵌套交互结构必须保留“源码”兜底，不允许为了看似 WYSIWYG 而牺牲安全和可预期保存。

### 2026-06-26 Markdown 表格可视编辑

Markdown 表格编辑必须保持“表格就是表格”的信息结构：桌面和移动端都使用真实 table + 内部滚动承接宽度，行列工具可 sticky，但不能把宽表挤压成不可读卡片。可视编辑只支持 GFM 能表达的单元格、行列和 left/center/right 对齐；涉及合并单元格、复杂嵌套或 HTML 表格时必须回到源码兜底。

## 2026-06-26 移动端顶部与资源块编辑补充约束

- 移动端顶栏最多承担“打开导航 + 当前页面标题”。搜索、配色、主题、批量文件动作等二级能力不得挤在顶部横向排列，应进入导航抽屉、折叠动作区或当前内容区的上下文菜单。
- 文件管理器移动端不得重复显示 AppShell 标题和页面内大标题；内容区第一屏优先给路径、搜索和列表。
- 移动端文件根目录切换不能因删 header 丢失，应放入“文件操作”折叠区；桌面端继续保留完整 header 与横向操作组。
- Markdown 图片/媒体可视编辑应在资源块内部完成：路径输入、拖拽/粘贴替换、复制路径、打开资源、资源选择器和预览属于同一个块，不额外堆顶层按钮。
- 资源块编辑器只改文档引用，不执行上传或移动；上传、批量、断点续传仍归文件管理器上传管理器。
