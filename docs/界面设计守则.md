# OpenClaw Studio UI Surface Guardrails

> 状态：Active
> 更新：2026-03-19

本文档定义 `OpenClaw Studio` 后续页面的共享布局约束，目的只有一个：

- 避免未来继续默认落回“双栏/三栏卡片矩阵 + 信息一锅端”的页面习惯。

## 1. 默认禁令

以下页面 **默认禁止** 做成固定双栏/三栏卡片矩阵：

- chat 入口/主聊天壳
- chat 私聊会话页
- 任何 transcript-first 页面
- 任何输入驱动/连续操作驱动页面

原因：

- 这类页面有明显的主任务
- 用户注意力必须集中在主区域
- 卡片矩阵会把页面切碎

补充：

- IM shell 允许存在一个固定 session list
- 但不允许再把 diagnostics / runtime / usage / timeline 与 conversation 做成同权并排面板

## 2. 什么时候可以用多栏

允许多栏，但前提是主次区域明确，而且多栏不能是默认展开状态：

- chat workbench / debug 面
- dashboard 总览
- system 诊断页

即使允许多栏，也必须满足：

- 有唯一主区域
- 次级区域通过 tabs / collapse / drawer / segmented controls 渐进展开
- 不允许 3 个同权重面板同时抢注意力
- 不允许把 navigator / inspector 做成默认常驻侧栏

## 3. 主区域 / 次区域 / 折叠区原则

### 主区域

适合承载：

- transcript
- 核心内容流
- 用户当前最频繁交互对象

### 次区域

适合承载：

- 最近会话
- 轻量上下文
- 次级状态

### 折叠区 / tabs

适合承载：

- diagnostics
- runtime 明细
- lifecycle
- tool cards
- usage
- timeline

## 4. Cards 适合与不适合的场景

### 适合

- 独立 summary
- dashboard 指标
- system 诊断块
- 可折叠 inspector 内容

### 不适合

- 聊天入口页主骨架
- transcript 主体
- 消息流主容器
- 需要持续浏览/滚动/输入的主区域

## 5. Chat / Transcript 特殊规则

- transcript 必须是绝对主区域
- composer 必须紧邻 transcript，而不是被边栏打断
- 正式私聊页禁止“session browser + diagnostics pane + thread”这类三向同权结构
- IM 主聊天壳允许固定 session list，但 conversation 必须始终是唯一主舞台
- diagnostics / runtime / transport 术语不得占主舞台
- 正式私聊页只允许轻量 inline 状态
- 完整 tool cards / timeline / diagnostics 继续留在 workbench/debug

## 6. Debugging Surface 规则

debug surface 可以复杂，但不能混乱。

要求：

- transcript 仍应是主要区域之一
- diagnostics / lifecycle / tool cards / usage / timeline 通过 tabs、drawer 或 collapse 分层
- 不允许默认固定三栏
- 不允许把所有调试块直接垂直堆满整页
- progressive disclosure 优先于“默认全部展开”

## 7. Mobile Guardrails

- mobile 下优先单列
- 次级面板应下沉、折叠或切换，而不是强行并排
- transcript 与 composer 不能被多个 side panel 挤碎
- 不能把桌面三栏简单压成移动端的一列卡片长墙
- drawer / tabs / accordion 优先于多列压缩

## 8. 页面类型建议

### Chat

- `/chat`
  - IM shell
  - 固定 session list + conversation 主区
- `/chat/s/:sessionRef`
  - 同一个 Chat Shell 的 deep-link
  - 不应再派生出另一套重复 surface

### Workbench

- 同一个 Chat Shell 的 inspect mode
- inspector 必须 tabs/collapse 化

### Dashboard

- 可以多卡片
- 但必须有明确分组和摘要优先级

### System

- 可以多面板
- 但必须按健康/风险/原始诊断分层

### Project

- 不应默认照搬 dashboard 卡片矩阵
- 应先明确是 timeline-first、workspace-first、还是 entity-first
