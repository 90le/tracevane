# OpenClaw Studio Chat Contract

> 状态：Phase 2 baseline active
> 更新：2026-03-19

本文档定义 `Studio` 自己的 chat typed contract。

目标：

- 为后续 `SA-08D Frontend Shell` 提供稳定可消费的 contract
- 明确 `Studio` 前后端之间的资源模型、API 合同、stream event 合同、错误语义
- 明确哪些字段属于 `Studio` contract
- 明确哪些字段只是 Gateway raw source，不允许直接外泄到前端

不在本文档范围内：

- 正式聊天功能实现
- UI 设计
- workflow / room / project / community 功能

## 1. 设计原则

### 直接证据

- 官方 Gateway / Control UI / WebChat 使用的底层方法和事件是：
  - `chat.history`
  - `chat.send`
  - `chat.abort`
  - `chat.inject`
  - `sessions.list`
  - `sessions.patch`
  - `sessions.delete`
  - `chat` 事件
  - `agent` 事件
- 本机 transport spike 已验证：
  - `chat.send` 非阻塞 ack
  - `chat.abort` 可只用 `sessionKey`
  - `chat.history` 存在 size-bound 截断
  - `agent` 事件存在，且 fresh live `agent_tool_call / agent_tool_result` 已通过 `Studio /ws/chat` 同链路验证

### 最终决策

- 前端不直接消费 Gateway raw frame
- 前端不直接理解 `chat.send/chat.history/chat.abort` 的官方 raw payload
- `Studio` 后端必须把 Gateway 语义映射为自己的 typed contract
- MVP 合同只覆盖：
  - `chat` 事件
  - text-first history / send / abort / reset / create-session
- `agent` 事件现在只允许以 `Studio` 归一化后的 observability contract 进入前端
- 前端仍不直接消费 raw `agent.data.*`

## 2. 资源模型

### 2.1 `ChatSessionKind`

```ts
type ChatSessionKind =
  | "studio_managed"
  | "observed_external"
  | "system_internal";
```

含义：

- `studio_managed`
  - `Studio` 创建和管理的安全 chat session
- `observed_external`
  - 来自现有 Gateway session store，但不属于 `Studio` 自管范围
- `system_internal`
  - cron / heartbeat / internal run / system-only session

### 2.2 `ChatSessionRow`

`Studio` 前端 session rail 的基础对象。

建议字段：

```ts
interface ChatSessionRow {
  key: string;
  agentId: string;
  sessionId: string | null;
  kind: ChatSessionKind;
  label: string;
  derivedTitle: string | null;
  lastMessagePreview: string | null;
  updatedAt: string | null;
  source: {
    source: "studio" | "external" | "system";
    channel: string | null;
    surface: string | null;
    originLabel: string | null;
  };
  deliveryContext: {
    channel: string | null;
    accountId: string | null;
    to: string | null;
    threadId: string | null;
  };
  permissions: {
    writable: boolean;
    canSend: boolean;
    canAbort: boolean;
    canReset: boolean;
    canDelete: boolean;
    canInject: boolean;
    visibleInFrontend: boolean;
    visibleInMvpRail: boolean;
  };
  runtime: ChatRuntimeState;
}
```

`sessionId` 语义：

- `studio_managed` draft session 在 `Studio` 创建时就分配一个本地 `sessionId`
- 该 `sessionId` 会持久化到 `Studio` registry，刷新恢复时不再丢失
- 当 Gateway 后续返回更权威的 `sessionId` 时，`Studio` 会用 Gateway 值回填 registry
- 因此当前 contract 继续保留 `sessionId`，但其来源可能是：
  - draft-local
  - gateway-authoritative

### 2.3 `ChatMessageItem`

MVP 只保留 text-first、已归一化后的消息形态。

```ts
type ChatMessageRole = "user" | "assistant" | "system" | "tool" | "unknown";

interface ChatMessageItem {
  id: string;
  role: ChatMessageRole;
  text: string;
  createdAt: string | null;
  source: "history" | "stream" | "inject";
  runId: string | null;
  truncated: boolean;
  omitted: boolean;
  aborted: boolean;
  stopReason: string | null;
}
```

设计原因：

- 前端不应处理 Gateway 原始 content array
- 前端不应解析 provider/model/tool-specific raw blocks
- 这些属于后端 adapter 的归一化职责

### 2.4 `ChatHistoryPayload`

```ts
interface ChatHistoryPayload {
  checkedAt: string;
  session: ChatSessionRow;
  messages: ChatMessageItem[];
  runtime: ChatRuntimeState;
  diagnostics: ChatDiagnostics;
}
```

### 2.5 `ChatSendRequest`

```ts
interface ChatSendRequest {
  text: string;
  clientRequestId?: string;
  thinking?: string | null;
}
```

设计原因：

- 前端发送 `clientRequestId`
- 后端再把它映射成 Gateway `idempotencyKey`
- 前端不直接暴露 Gateway raw parameter 语义

### 2.6 `ChatSendAck`

```ts
type ChatSendStatus =
  | "started"
  | "duplicate_in_flight"
  | "duplicate_completed";

interface ChatSendAck {
  accepted: boolean;
  sessionKey: string;
  sessionId: string | null;
  requestId: string;
  runId: string;
  status: ChatSendStatus;
  runtime: ChatRuntimeState;
}
```

映射关系：

- Gateway `started` -> `started`
- Gateway `in_flight` -> `duplicate_in_flight`
- Gateway `ok` -> `duplicate_completed`

### 2.12 `ChatObservabilityState`

```ts
type ChatAgentLifecyclePhase = "start" | "end" | "error";
type ChatToolStatus = "running" | "completed" | "error";
type ChatActivityKind = "lifecycle" | "assistant" | "tool_call" | "tool_result" | "usage";

interface ChatUsageSummary {
  inputTokens: number;
  outputTokens: number;
  totalTokens: number;
  cacheReadTokens: number;
  cacheWriteTokens: number;
  costUsd: number | null;
}

interface ChatLifecycleSignal {
  phase: ChatAgentLifecyclePhase;
  runId: string | null;
  emittedAt: string;
  errorMessage: string | null;
}

interface ChatToolCard {
  toolCallId: string;
  runId: string | null;
  name: string;
  status: ChatToolStatus;
  startedAt: string | null;
  updatedAt: string | null;
  argsPreview: string | null;
  resultPreview: string | null;
  isError: boolean;
}

interface ChatActivityItem {
  id: string;
  kind: ChatActivityKind;
  runId: string | null;
  toolCallId: string | null;
  emittedAt: string;
  title: string;
  detail: string | null;
  level: "info" | "success" | "warning" | "error";
}

interface ChatObservabilityState {
  lifecycle: ChatLifecycleSignal | null;
  toolCards: ChatToolCard[];
  usage: ChatUsageSummary | null;
  timeline: ChatActivityItem[];
}
```

说明：

- `toolCards / usage / timeline` 只服务当前 chat session
- 它们不是 workflow / project / room timeline
- 当前 `usage` 优先级：
  - 先用 live `final.usage`，如果存在
  - 再用 `chat.history` 中 assistant message 的 usage 作为权威基线回填/覆盖
- 当前不把 Gateway raw usage 对象直接外泄给前端

### 2.13 `ChatStreamEvent`

```ts
type ChatStreamEvent =
  | {
      kind: "ack";
      sessionKey: string;
      runId: string;
      requestId: string;
      emittedAt: string;
      status: ChatSendStatus;
      runtime: ChatRuntimeState;
    }
  | {
      kind: "delta";
      sessionKey: string;
      runId: string;
      emittedAt: string;
      textDelta: string;
      accumulatedText: string;
    }
  | {
      kind: "final";
      sessionKey: string;
      runId: string;
      emittedAt: string;
      message: ChatMessageItem;
      runtime: ChatRuntimeState;
      usage: ChatUsageSummary | null;
    }
  | {
      kind: "aborted";
      sessionKey: string;
      runId: string;
      emittedAt: string;
      stopReason: string | null;
      partialMessage: ChatMessageItem | null;
      runtime: ChatRuntimeState;
    }
  | {
      kind: "error";
      sessionKey: string;
      runId: string | null;
      emittedAt: string;
      error: ChatContractError;
      runtime: ChatRuntimeState;
    }
  | {
      kind: "runtime";
      sessionKey: string;
      runId: string | null;
      emittedAt: string;
      runtime: ChatRuntimeState;
    }
  | {
      kind: "agent_lifecycle";
      sessionKey: string;
      runId: string | null;
      emittedAt: string;
      lifecycle: ChatLifecycleSignal;
    }
  | {
      kind: "agent_assistant";
      sessionKey: string;
      runId: string;
      emittedAt: string;
      textPreview: string;
      deltaText: string | null;
    }
  | {
      kind: "agent_tool_call";
      sessionKey: string;
      runId: string | null;
      emittedAt: string;
      tool: ChatToolCard;
    }
  | {
      kind: "agent_tool_result";
      sessionKey: string;
      runId: string | null;
      emittedAt: string;
      partial: boolean;
      tool: ChatToolCard;
    };
```

已验证事实：

- live `agent` raw stream 至少稳定出现：
  - `lifecycle`
  - `assistant`
- fresh new run 现在已经通过 `Studio /ws/chat` 同链路观察到：
  - `agent_tool_call`
  - `agent_tool_result`
- 当前 transport 已对齐为：
  - `studio_managed` session 通过同一条持久 Gateway bridge 处理 `chat.send`
  - 同一条 bridge 继续接收 `chat` / `agent` 事件
- 官方机制对齐点：
  - Gateway tool event 需要 `TOOL_EVENTS` capability
  - recipient registration 发生在发起 `chat.send` 的连接上
  - `Studio` 已修正原先的 split transport 断点
- 但 `chat.history` / 本地 transcript 中存在真实的 `toolCall / toolResult / usage` 数据，所以 Phase 2 UI 已经可以对真实历史 session 展示 tool cards / usage / timeline

## 3. API Contract

## 3.1 `GET /api/chat/agents/:agentId/sessions`

返回建议：

```ts
interface ChatSessionsPayload {
  checkedAt: string;
  agentId: string;
  sessions: ChatSessionRow[];
  diagnostics: ChatDiagnostics;
}
```

用途：

- session rail
- 只显示 `visibleInFrontend=true`
- MVP rail 只展示 `visibleInMvpRail=true`

## 3.2 `GET /api/chat/sessions/:sessionKey/history`

返回：

- `ChatHistoryPayload`

用途：

- transcript pane
- chat 页面恢复
- history truncation 告警
- observability baseline：
  - tool cards
  - usage summary
  - current-session activity strip

## 3.3 `POST /api/chat/agents/:agentId/sessions`

请求：

- `ChatCreateSessionRequest`

建议返回：

```ts
interface ChatCreateSessionResponse {
  ok: boolean;
  session: ChatSessionRow;
  runtime: ChatRuntimeState;
}
```

## 3.4 `POST /api/chat/sessions/:sessionKey/send`

请求：

- `ChatSendRequest`

返回：

- `ChatSendAck`

说明：

- 前端只管 `clientRequestId`
- 后端负责映射 Gateway `idempotencyKey`

## 3.5 `POST /api/chat/sessions/:sessionKey/abort`

返回：

- `ChatAbortResponse`

## 3.6 `POST /api/chat/sessions/:sessionKey/reset`

返回：

- `ChatResetResponse`

## 3.7 `GET /api/chat/health`

返回建议：

- `ChatDiagnostics`

用途：

- chat 页面顶部运行态
- Gateway down / auth / same-origin 提示

## 3.8 `WS /ws/chat`

事件输出：

- `ChatStreamEvent`

要求：

- 只输出 `Studio` 自己的 stream event
- 不直接透传 Gateway raw frame

## 4. MVP / Phase 2 事件边界

### MVP contract

- `ack`
- `delta`
- `final`
- `aborted`
- `error`
- `runtime`

### Phase 2 预留

- `agent_lifecycle`
- `agent_assistant`
- `agent_tool_call`
- `agent_tool_result`
- `usage` 作为 `ChatUsageSummary` 进入 history / final，而不是 raw usage 对象
- future `event/activity layer` envelope

### 当前前端绝对不应该看到的 Gateway raw 字段

- `seq`
- Gateway 原始 `state`
- Gateway 原始 `message` 结构
- Gateway 原始 `usage` 对象
- Gateway 原始 `error.details`
- `connect.challenge`
- raw `agent.data.*`
- raw provider/model/tool-specific payload

这些都只能作为后端 adapter 的输入，不是前端 contract。

## 5. 错误语义

### 5.1 `ChatContractError`

```ts
type ChatContractErrorCode =
  | "gateway_down"
  | "auth_failure"
  | "session_not_found"
  | "session_not_writable"
  | "history_truncated"
  | "duplicate_in_flight"
  | "no_active_run"
  | "invalid_request"
  | "internal_error";

interface ChatContractError {
  code: ChatContractErrorCode;
  message: string;
  retryable: boolean;
  source: "studio" | "gateway";
}
```

### 5.2 错误映射规则

- Gateway connect/transport error
  - `gateway_down`
- Gateway auth / scope / pairing / device auth error
  - `auth_failure`
- `session not found`
  - `session_not_found`
- 被 `Secure Session Policy` 拒绝写入
  - `session_not_writable`
- history 被截断
  - 不是 hard error
  - 进入 `ChatDiagnostics.historyTruncated = true`
  - 必要时产生 `history_truncated` warning
- duplicate in-flight send
  - 优先表达为 `ChatSendAck.status = "duplicate_in_flight"`
  - 不作为 hard error
- abort when no active run
  - `ChatAbortResponse.hadActiveRun = false`
  - 必要时附带 `no_active_run`
- schema / params / empty text
  - `invalid_request`
- 其它未分类异常
  - `internal_error`

## 6. Phase 1 / Phase 2 边界

### Phase 1

- session rail contract
- history contract
- send/abort/reset/create contract
- `chat` stream contract
- error mapping
- secure session policy 对前端暴露的权限语义

### Phase 2

- `agent` event-derived contract
- tool cards
- timeline / activity
- usage / observability

## 7. 不允许发生的事

- 前端不能依赖 Gateway raw frame
- 前端不能自己拼 Gateway RPC 参数语义
- 前端不能直接拿官方 `chat.send/chat.history/chat.abort` raw payload 当稳定 contract
- 前端不能依赖 `sessionKey` 前缀推断权限、来源和类型

## 8. 当前冻结结论

- `Studio` chat typed contract 必须独立于 Gateway raw protocol
- text-first 主链路继续维持独立，不因 observability 返工
- `agent` 事件现在只以最小 observability contract 进入前端
- workflow / room / project / community 仍然不进入 chat 页面
