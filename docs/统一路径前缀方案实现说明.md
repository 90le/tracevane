# 统一路径前缀方案实现说明（历史与现状）

## 文档定位

这份文档不再表示“当前正式交付方案”。

截至 2026-03-31，它只用于记录：

1. 旧 `--base-path /x/studio/` 方案为什么废弃
2. 当前已经完成到什么程度
3. 后续要继续做单端口模式，下一步该收口什么

客户安装与正式支持边界请看：

1. [DEPLOY.md](/home/binbin/.openclaw/extensions/openclaw-studio/DEPLOY.md)
2. [反向代理部署指南.md](/home/binbin/.openclaw/extensions/openclaw-studio/docs/反向代理部署指南.md)

## 旧方案做了什么

旧方案的核心思路是：

1. 打包时传入 `--base-path /x/studio/`
2. 前端构建产物带上 `/x/studio/` 前缀
3. 后端尝试裁剪 `/x/studio` 前缀
4. 期望客户只配一个 `/x/studio/` 入口，就同时覆盖页面、API 和 WebSocket

这个思路并不荒谬，但实现层面没有真正闭环。

## 旧方案为什么废弃

### 1. 打包脚本承担了错误的职责

旧 `pack.sh` 同时做了：

1. 传入构建期 `STUDIO_BASE_PATH`
2. 给包名带路径后缀
3. 用 `sed` 修改 `index.html`
4. 注入 `window.__OPENCLAW_STUDIO_API_BASE__`

这会把“部署路径”这种运行时事实错误地固化在打包阶段。

因此，当前 `pack.sh` 已经移除了旧的：

```bash
--base-path /x/studio/
```

## 2. 前端页面、API、WebSocket 没有统一协议

旧方案最大的问题不是页面，而是传输协议。

当时实际存在三套彼此独立的约定：

1. 前端 router base path
2. API 请求路径
3. WebSocket 请求路径

这三者没有共用同一份运行时配置，所以“单一路径前缀”在文档上成立，在代码里并没有真正成立。

## 3. 反代层和后端都可能处理前缀

旧设计同时允许：

1. 反代层去掉 `/x/studio`
2. 后端再去掉一次 `/x/studio`

这两层逻辑如果同时生效，会让排查非常困难。

## 4. 旧测试不能证明方案成立

旧 `test-base-path.sh` 只覆盖局部 URL 请求，不是真正的浏览器链路验收。

它已经不能代表当前代码状态，因此已移除。

## 当前已完成的替代基础

旧方案虽然废弃，但并没有白做。

当前已经完成的替代基础是：

1. `standalone + gateway` 双暴露配置
2. 后端运行时注入 `window.__OPENCLAW_STUDIO_RUNTIME__`
3. 前端 router / API / WS base path 改为优先读取运行时配置
4. Studio 页面和 HTTP API 已可挂到 OpenClaw Gateway

换句话说，**旧的 build-time path 方案已经开始被 runtime transport 方案替代**。

## 当前还没完成的部分

### 1. Gateway-native realtime 还没完成

截至 2026-03-31，Gateway 模式只完成了：

1. 页面挂载
2. HTTP API 挂载

还没有完成：

1. 聊天 realtime
2. 终端 realtime

所以现在仍然不能把它对客户写成“单端口完整可用”。

### 2. 单端口模式还缺完整验收

后续至少还需要补：

1. Gateway 模式下的聊天实时流
2. Gateway 模式下的终端实时流
3. 浏览器级联调
4. 客户侧安装与升级回归

## 当前结论

截至 2026-03-31：

1. 旧 `--base-path` 方案已经废弃
2. `pack.sh` 不再支持旧打包参数
3. 当前替代方向是 runtime transport + Gateway-native 挂载
4. 这个方向已完成 HTTP foundation，但还没有完成 realtime

## 后续正确方向

如果继续推进单端口方案，正确方向不是恢复旧 `--base-path`，而是：

1. 保留 `standalone :3760` 作为正式稳定模式
2. 继续把 `Gateway-native` realtime 补齐
3. 让页面、HTTP、realtime 都共享同一份运行时 transport 配置
4. 等全部完成后，再决定是否恢复“客户单端口正式支持”的口径
