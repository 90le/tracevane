    - 当前实现不强依赖宿主私有 task registry，而是通过 `sessions.list`、当前会话活跃 `runId`、以及当前会话子代理树，合成出“Studio 可见后台任务”摘要
    - 会显示主会话是否仍在运行、当前活跃 run、子代理任务数量与状态分布，以及最近几条可见子代理任务
    - 这样在 `standalone + SSE`、单口 gateway、或宿主侧无额外 task RPC 的场景下，仍能给出稳定可用的任务视图
  - `/context` 已转为 Studio 本地命令
    - 支持 `/context`、`/context list`、`/context detail`、`/context json`、`/context help`
    - 现在会汇总当前会话 key、Agent、可见消息数、待发送队列长度、活跃 run、当前 session 模型/思考/发送策略、子代理数量和 usage 摘要
    - 明确标注这是“Studio 当前 surface 可见上下文摘要”，不伪造宿主内部的精确 prompt budget / bootstrap 报告
  - `/whoami / session / focus / unfocus` 已改为 Studio 本地显式反馈
    - 这四类命令在官方频道侧依赖 sender identity、focused binding 或 thread binding 语义
    - Studio 直连 chat 不具备这套 surface 语义时，不再回退宿主发送链路制造含糊报错，而是直接告诉用户当前命令为何不适用、应该改用哪种 Studio 交互
  - Slash 发送链路继续收口
    - 当前 slash 目录已不再残留前端 realtime-only `send` 模式命令
    - 需要保留宿主原生命令语义的 slash，现统一改走 `Studio backend -> slash-gateway -> chat.send`

    - `/redirect` 与 `/steer` 已补专门的中英文状态 copy，其他 slash 仍沿用通用反馈文案，维持最小必要复杂度
    - `/steer` / `/redirect` 当前会话分支已消除一次重复 `sessions.list` RPC，减少不必要的 Gateway 往返
  - `/models` 已转为 Studio 本地只读命令
    - 通过 `models.list` 直接读取 Gateway 允许的模型目录
    - 优先按 provider 分组展示；如果用户传入 provider 名称，会收窄到该 provider 的模型列表
    - 这一步继续遵循“先补命令支持，再决定是否需要额外状态”的边界，不额外引入新的命令面板
  - `/tools` 已转为 Studio 本地只读命令
    - 通过 `tools.effective` 直接读取当前 session 的运行时有效工具，而不是再把命令文本发回宿主对话
    - 支持 `compact / verbose` 两种轻量展示模式，保持 slash 输出仍然是“可扫读的摘要”，不把 chat 页做成完整工具面板副本
  - Slash 分发已支持 `hybrid` 模式

  - `/config show/get` 已接入 Studio 本地只读查询
    - 通过 `config.get` 返回配置快照摘要
    - 通过 `config.schema.lookup` 辅助解析路径，并显示配置路径对应的值或子项数量
    - 当前保持轻量输出，不在 chat 里展开完整配置编辑器

    - 通过 `config.get` 读取 `plugins.entries / plugins.installs / plugins.load.paths`

    - `enable/disable` 仍保留宿主 slash 原生命令语义，不在 Studio 前端伪造写操作

    - `set/unset` 继续回退到宿主 slash，保持和官方命令行为一致
  - `/subagents` 已接入 Studio 混合分发

    - `info <id>` 会输出子代理状态、模型和父会话摘要

    - 传入 `<name> [input]` 时，仍回退宿主 slash 保持真实技能执行语义，不在 Studio 前端伪造一个技能运行器
    - 这样 `slash` 菜单和帮助面板可以给出更像官方 WebUI 的可发现性，同时不破坏宿主技能系统的上下文注入方式

    - 传入真正的 queue override 参数时，仍回退宿主 slash 处理；这是刻意设计，因为当前 Gateway `sessions.patch` 并未公开 queue mode / debounce / cap / drop 的稳定 patch surface
    - 这样可以避免 Studio 前端伪造一个看似可配、实际不准确的本地 queue 设置面
