## 1. 总体状态

OpenClaw Studio 已从早期恢复阶段进入真实工作台迭代：

- Config、Skills、Terminal、Channels、Cron、Agents、System 已具备真实读写和运行态能力。

- Room / Workflow 暂缓。

- Live 附件验收脚本已加固：`--require-inbound-image`、`--require-inbound-video`、`--require-staged-files` 会验证附件类型和本地 staged 路径存在；Feishu/Octo 入站文件、Feishu/Octo 入站图片和 Feishu 入站视频已通过只读日志 smoke。

- 用户确认 Feishu/Octo 最新手动验收全部通过，包括 Feishu 发文件、权限审批、Octo 收文件并返回路径；`hello-live.txt` 和 `studio-greeting.txt` 临时文件已删除。
- 图片自动切视觉模型默认关闭，需在平台 binding 或 IM `/vision` 显式开启；binding 可设置默认 fallback 模型，IM 会话可通过 `/vision` 命令和 Feishu 菜单临时开启、关闭、指定模型或回到 binding 默认。开启后失败会回退原模型的附件说明/本地路径模式。
- Codex 图片异常排查结果：当前 Feishu 私聊 override 实际为 `gpt-5.4-mini`；已修正 Codex `exec resume` 图片 option 顺序，并让 Codex app-server result 保留视觉 args，便于日志确认图片是否进入模型。
- Gateway 已修复 Responses -> Chat-compatible provider 的 `input_image` 映射；`gpt-5.4-mini` / `gpt-5.5` 受控图片 smoke 和 `codex exec --image` + `gpt-5.4-mini` 已通过。当前没有 `gpt-5.5-mini` 模型；`claude-opus-4-6` 走 `mlamp` Chat-compatible 图片请求仍返回 400，需按 provider smoke 结果标记能力。
- Provider Center 已停止按模型名推断 vision；`gpt-*`、`claude-*` 这类名称只有在用户勾选、上游显式能力元数据或图片 smoke 通过后确认时才会进入图片能力。Smoke 页已提供独立图片 smoke，失败只提示协议/endpoint/模型不匹配，不写回 vision。
- Codex / Claude Code / OpenCode 图片链路已用受控三色方块真实 runner smoke 验证通过；非视觉模型会收到附件说明和本地路径，不传 native 图片、不允许猜图。
- 视频附件不再由 Studio 预抽帧或转图片；Studio 只负责接收、staging 和把本地路径交给 Agent，由 Agent CLI/模型或工具按任务自行处理视频。
- Octo 媒体消息带 `content/caption` 时会保留用户任务文本，不再被 `[image]` / `[video]` 占位吞掉。
- 同 session FIFO 已补可恢复 pending store：已入队但尚未启动的 Agent 消息会落盘，daemon 重启后重放；daemon/API/UI 已展示 pending 数量、待恢复记录和最近 replay/drop/fail 事件；Octo restart 回归已通过；Feishu 24h live 已证明同进程 FIFO 排队顺序执行，真实 IM 排队重启 replay 场景待复验。
- 真实 CLI thinking smoke 已完成：Claude Code 2.1.86 当前不输出 `thinking` item；OpenCode 1.17.0 在支持 reasoning 的模型上可输出 `reasoning` part；状态、菜单和前端已区分 parser/live 支持。
- 下一步聚焦 Agent runner 成熟度：补 OpenCode 中间过程回复真实 IM event-log 样本、Feishu durable queue live 复验；Octo 出站文件自动事件证据和 `/stop` 回归/live 证据已完成。
