# OpenClaw Studio 当前进展

> 状态：Active Snapshot
> 更新：2026-06-02

本文档是当前仓库状态摘要，面向后续开发接力。更细的产品原则见 `docs/产品需求.md`，架构边界见 `docs/系统架构.md`。

## 1. 总体状态

项目已从早期恢复阶段进入稳定迭代阶段：

- 管理域已经形成真实 API + Vue 工作台
- Chat 已从规划进入运行态
- Terminal、System、Channels、Agents、Skills、Cron、Config 等核心域具备真实读写能力
- Dashboard、Codex Stack、Files、Plugins、Dreaming 已有接入，但仍需要持续打磨信息架构和验收链路
- Room / Workflow 暂缓

## 3. Chat 当前状态

Chat 已经不是“尚未实现”的规划项。当前能力包括：

- `/api/chat/health`
- `/api/chat/bootstrap`
- organizer / folders / session assignment
- sessions list / create / patch / delete / reset
- history page / dates / search
- stream
- queue
- controls
- slash gateway
- send / abort
- upload
- resource resolve / media resolve
- 单人模式正在按 IM 会话产品体验收口：长历史滚动锚点、虚拟窗口、消息列可读宽度、输入区固定宽度和附件池都纳入稳定性验证

架构结论仍然是：

- Studio 后端做 typed adapter / BFF
- 浏览器不直连 OpenClaw Gateway
- Studio 自管 `studio_managed` 会话可写
- `observed_external` 默认只读
- 不引入第三方平行 DB 会话模型

前端当前已统一到一个目录：

- `features/chat/api.ts`：Studio Chat API client
- `features/chat/session-ref.ts`：route/session 引用 helper
- `features/chat/ChatShellPage.vue`：当前真实 Chat Shell
- `features/chat/*`：SessionList、ConversationPane、Composer、Inspector、Record Browser、Slash UI、富消息渲染等主要 UI

后续应继续保持单目录边界，避免重新出现版本化前端目录。

2026-06-01 单人模式性能/布局基线：

- `ConversationPane` 的长会话虚拟窗口滚动更新已按 animation frame 合并，减少 scroll 事件上的响应式重算
- timeline 日期分隔和高度估算已缓存为布局行，避免滚动期间反复解析历史消息内容
- 虚拟窗口已改为基于累计高度索引的二分查找；滚动时不再线性扫描全部消息/overlay 行，长历史 PC 滚动路径更接近 IM 连续会话要求
- timeline 文本高度估算改为无复制扫描：虚拟窗口布局不再为每条长消息执行 `trim()`、`split()` 和多次正则 `match()`，超长文本会按 soft line 估算提前封顶，降低长历史加载和滚动窗口重算时的 CPU/内存放大
- timeline 测量缓存会随当前 timeline item ids 裁剪；同一会话内 history window 替换、过滤或 reset 后，不再保留已经离开当前时间线的高度缓存、DOM ref 和 ResizeObserver 观测关系，降低长时间使用后的内存与 observer 压力
- 会话 viewport 快照和 prepend 稳定恢复读取可见锚点时，只扫描当前虚拟窗口范围内的 timeline item，再用 DOM message anchor 兜底；长历史中保存/恢复视口不再每次遍历完整 timeline
- timeline item 与 thread 容器的 `ResizeObserver` 回调已合并到同一个 animation frame；图片、Markdown 预览、工具卡片等异步高度变化时，一帧内只做一次 viewport sync、sentinel reconnect、测量和底部修复判断
- Chat 会话级 viewport 快照已覆盖切换、浏览器返回和刷新：翻阅历史时 Shell 低频保存可见 `msg-*` 锚点，优先选择已经进入视口的第一条消息、再用部分可见消息兜底；刷新后用 anchor window 重新打开历史位置；`ConversationPane` 仍保留 SPA 切换锚点恢复和底部距离兜底；默认贴底查看最新消息的会话不保存快照
- 桌面消息流和 composer 收敛到 IM 可读宽度，保留富文本、Markdown、tool card 和文件渲染能力
- 桌面 composer/footer 增加总高度上限，展开的待发送队列在 inline rail 内独立滚动，避免大草稿、多附件和多条队列同时存在时挤没消息阅读区；`smoke:chat:matrix` 已加入 PC composer 压力验收，覆盖 8 条待发送队列、超长草稿和 8 个附件同时存在的场景
- `smoke:chat:matrix` 已加入真实浏览器富文本布局验收，构造宽 Markdown 表格、长代码块、inline HTML 和 SVG，确保这些内容只在自身容器内横向滚动，不撑宽 PC chat pane / thread / assistant bubble
- 长消息 / 重 Markdown 的 deferred 占位摘要改为有界生成：占位卡只扫描到预览所需长度并折叠空白，不再为了未进入视口的超长正文执行整段 `replace(/\s+/g)`，降低长历史里大量占位 bubble 的初始化成本
- grouped message 的离屏 bubble 摘要增加整体长度上限，并按循环短路生成；连续多条消息合并成一个离屏占位时，不再先为所有 message 分配 preview 数组和完整 join 字符串
- 重 Markdown 懒渲染判定改为无复制扫描：空白检测和去首尾空白后的行数阈值不再通过 `trim()` / `split()` 分配大字符串或行数组，避免长历史中未进入视口的 Markdown 正文在判定阶段先消耗一次高内存路径
- 普通 Markdown 消息首帧直接走正文分支，只有真正重 Markdown 且未进入 ready 状态时才显示 deferred 卡片；避免轻量消息 mounted 前短暂闪占位卡，减少普通 IM 消息滚入视口时的布局抖动
- 桌面 composer 已显式使用 `overflow-wrap:anywhere` 处理超长无空格命令、路径和 token；PC composer 压力验收会断言输入区/外层 composer 不产生横向溢出
- 翻阅历史时的“返回最新”按钮会显示短窗口内新增消息数量，并带有 aria label，避免实时消息到达时用户只能看到一个无上下文箭头
- 新消息数量按尾部 message / overlay 身份去重；同一条 AI 流式回复或同一个工具过程的多次 partial 更新只计 1 条，避免历史阅读时未读数虚高
- `smoke:chat:heavy-history` 已覆盖长历史初始贴底、向上翻页、无重复 history 请求、无 after 请求误触发、无 console error

- 如果前端从本地恢复的旧 cursor 遇到后端重启后的低序号事件，会允许一次低序号纠偏，避免长期携带无效高 cursor

- 当前实现是单人模式的短窗口实时补偿，不是最终多人/群聊的持久事件总线；多用户阶段仍需要 participant、room/conversation store、持久 cursor 和权限模型
