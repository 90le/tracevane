# Studio Chat Markdown/HTML/SVG 混合渲染方案

日期：2026-03-27

## 结论

推荐采用 **方案 A：保留现有 `marked + DOMPurify + MarkdownBlock.vue` 架构，增量支持 Markdown 正文中的原生 HTML/SVG 混合渲染**。

不建议为了这个需求迁移到新解析库。当前项目已经具备：

- Markdown 渲染主链路
- HTML/SVG/Mermaid fenced code block 预览能力
- 全局/会话级开关能力
- HTML iframe 预览、SVG 安全清洗、Mermaid 全屏预览

当前缺的不是库，而是产品策略和一层渲染规则：

- 现在把 Markdown 里的原生 HTML 强制转义了
- SVG 只在 ```svg 代码块里走预览链路
- 开关只覆盖 code fence preview，没有覆盖“正文原生标签渲染”

因此最佳做法不是重写，而是 **在现有渲染器上补 raw HTML/SVG mixed render**，同时保留现有 code fence preview 作为复杂内容的高隔离兜底。

## 当前实现现状

### 1. 前端渲染入口

消息正文最终都进入 `MarkdownBlock.vue`：

- `display-adapter.ts` 会把消息正文整理成 `type: 'markdown'` 的显示块
- `MessageBubble.vue` 对 markdown block 统一使用 `MarkdownBlock`

关键位置：

- `apps/web-vue/src/features/chat-v2/display-adapter.ts`
- `apps/web-vue/src/features/chat-v2/MessageBubble.vue`
- `apps/web-vue/src/features/chat-v2/MarkdownBlock.vue`

### 2. 当前 Markdown 主链路

`MarkdownBlock.vue` 的核心是：

- `renderChatMarkdownResult(props.source, { interactive: true })`
- 把结果 `v-html` 注入聊天气泡
- 渲染完成后，再扫描 `.chat-live-preview-block` / `.chat-mermaid-block`
- 对 `mermaid/html/svg` fenced code block 做二次增强

其中：

- Mermaid -> 前端 `mermaid.render(...)`
- HTML -> `iframe srcdoc`
- SVG -> 直接 sanitize 后内联注入

### 3. 当前为何“不能 Markdown 和 HTML 混合渲染”

根因在这里：

```ts
markdownRenderer.html = ({ text }: Tokens.HTML | Tokens.Tag) => escapeHtml(text);
```

也就是说，`marked` 已经识别出原生 HTML token，但项目主动把它转义成文本了。

这说明：

- 不是 `marked` 做不到
- 不是聊天流式链路做不到
- 是当前 renderer 策略明确关闭了 raw HTML render

### 4. 当前为何“SVG 只能代码块预览”

SVG 的现有逻辑只在 `renderer.code` 里识别：

- ```svg -> `renderSvgPreviewBlock(...)`
- 然后在 `MarkdownBlock.vue` 里通过 `sanitizeSvgPreviewMarkup(...)` 落地

所以正文里的 `<svg>...</svg>` 现在不会走任何 SVG 渲染特化逻辑。

### 5. 开关体系其实已经具备

`inline-preview-preferences.ts` 已经有：

- `mermaid`
- `html`
- `svg`

并且支持：

- 全局默认
- 会话覆盖
- 监听变化并热更新

`ConversationPane.vue` 也已经有完整设置 UI。

这意味着本次功能不需要重新发明配置系统，只需要决定：

- 复用现有 `html/svg` 开关
- 还是新增“raw html/raw svg”开关

## 后端是否是阻塞点

不是。

当前后端 `assistant-markdown-media.ts` / `media-bridge.ts` 的职责主要是：

- 在 Markdown 文本里重写 studio 媒体引用
- 且显式跳过 fenced code block / inline code

它并不负责 Markdown -> HTML 渲染，也没有阻止正文里的原生 HTML/SVG。

因此这个需求 **可以前端优先落地**，后端最多只需要保持现状。

## 本地验证结论

我已经在当前依赖版本下验证了两件事：

### 1. `marked` 本身支持 mixed render

当 `renderer.html` 返回原文而不是 `escapeHtml(text)` 时，Markdown 与 HTML 可以直接混排。

### 2. `marked` 对 inline SVG 是逐标签 token 化

例如：

```md
Text <svg ...><circle .../></svg> end
```

会被拆成：

- `<svg ...>`
- `<circle .../>`
- `</svg>`

这意味着实现 `html/svg` 双开关时，不能只看“整块 `<svg>` 字符串”，需要在 renderer 中维护一个轻量的 `svgDepth` 上下文，或者使用更细的 token 级判定。

## 可选方案对比

## 方案 A：保留 `marked`，增量支持 raw HTML/SVG mixed render

### 做法

1. 保留现有 `marked` 解析器、媒体 token、自定义 code renderer
2. 修改 `markdownRenderer.html`
3. 新增“允许 raw HTML / raw SVG”的渲染选项
4. 根据开关决定：
   - 允许：输出原始 token，再走 DOMPurify
   - 禁止：继续转义显示
5. 继续保留 `html/svg/mermaid` fenced code block preview

### 优点

- 改动最小，复用现有架构
- 不破坏现有 Mermaid/HTML/SVG 代码块预览
- 能满足“非代码块实时渲染”
- 不引入新依赖，不增加 bundle 和维护成本
- 与现有全局/会话开关天然兼容

### 缺点

- 需要更谨慎的 sanitizer 分层
- 原生 HTML 直接进聊天 DOM，会有样式污染风险
- 原生 HTML 的“全屏放大”能力不如 code fence preview 自然

### 综合判断

这是最合适方案。

## 方案 B：不开放 raw HTML，仅把 html/svg/mermaid 代码块做成“近乎无边框”

### 做法

- 保留当前 fenced code block preview 机制
- 把头部、工具栏、边框压到最弱
- hover 时才显示按钮
- 保留点击放大/全屏/复制源码

### 优点

- 安全边界最清晰
- HTML 仍在 iframe 中，隔离最好
- 实现风险最小

### 缺点

- 不能满足“Markdown 里直接写 HTML 标签就渲染”
- 仍然要求模型输出 fenced block
- 流式期间必须等 fence 结构成型，实时感较差

### 综合判断

可以作为 fallback，但不应该作为主方案。

## 方案 C：迁移到 `markdown-it` 或 `remark/rehype`

### `markdown-it`

它支持 `html: true`，可以开启 Markdown 中的原生 HTML。

问题在于你们当前已经深度定制了：

- code block 渲染
- Studio 媒体 token
- copy button
- Mermaid/HTML/SVG preview block
- DOMPurify 安全规则

迁移成本高，但收益有限。

### `remark/rehype + rehype-raw`

这条链路的好处是 AST 控制更细，插件生态更强。

但对当前项目而言是明显过度设计：

- 需要重写 renderer 逻辑
- 需要把当前字符串模板体系迁到 AST/rehype
- 前端改造面更大

### 综合判断

除非后续明确要做：

- 更复杂的 Markdown AST 扩展
- 统一 SSR 渲染
- 更重的富文本编辑/转换能力

否则不建议为了这次需求迁库。

## 推荐方案细化

## 推荐决策

采用 **“双轨制”**：

### 轨道 1：正文 mixed render

用于：

- 轻量 HTML 片段
- 内联或块级 SVG
- 需要流式立即看到效果的内容

特点：

- 不需要代码块
- 与 Markdown 正文自然混排
- 直接在聊天 DOM 中渲染

### 轨道 2：现有 fenced code preview

用于：

- 独立 HTML 页面片段
- 需要 iframe 隔离的内容
- 需要全屏放大的 SVG / Mermaid / HTML
- 需要保留源码折叠与复制的内容

特点：

- 有明显结构边界
- HTML 运行在 `iframe srcdoc`
- 全屏预览能力天然更强

### 产品策略

建议向模型和用户同时支持两种写法：

1. 简单片段优先 raw HTML/SVG
2. 复杂 demo、完整页面、需要脚本或样式隔离时，继续输出 fenced code block

这样不会把一个机制强行用于所有场景。

## 具体技术设计

### 1. 扩展 `renderChatMarkdownResult` 选项

当前只有：

```ts
export interface ChatMarkdownRenderOptions {
  interactive?: boolean;
}
```

建议改为：

```ts
export interface ChatMarkdownRenderOptions {
  interactive?: boolean;
  allowRawHtml?: boolean;
  allowRawSvg?: boolean;
}
```

并把 cache key 一起纳入：

- `interactive`
- `allowRawHtml`
- `allowRawSvg`

否则切换开关时会命中错误缓存。

### 2. 在 `MarkdownBlock.vue` 中把现有开关接入 Markdown 主渲染

当前 `inlinePreviewPrefs` 已经有：

- `html`
- `svg`

建议直接复用：

- `html` 同时控制：
  - raw HTML tag render
  - ```html fenced preview
- `svg` 同时控制：
  - raw SVG tag render
  - ```svg fenced preview

这样 UI 不膨胀，用户心智也简单。

对应调用改成类似：

```ts
renderChatMarkdownResult(props.source, {
  interactive: true,
  allowRawHtml: inlinePreviewPrefs.value.html,
  allowRawSvg: inlinePreviewPrefs.value.svg,
});
```

### 3. 重写 `markdownRenderer.html` 的判定逻辑

不能再一刀切 `escapeHtml(text)`。

建议做一个专门函数：

```ts
renderRawHtmlToken(tokenText, state, options)
```

需要解决两个问题：

#### 问题 A：普通 HTML

例如：

```md
Hello <span class="pill">world</span>
```

当 `allowRawHtml = true` 时直接渲染；
关闭时转义显示。

#### 问题 B：SVG 是逐标签 token

例如：

```md
<svg><circle /></svg>
```

`marked` 会拆成多个 `html` token。

所以建议在一次 parse 周期内维护：

- `svgDepth`

规则：

- 遇到 `<svg ...>`：`svgDepth += 1`
- 遇到 `</svg>`：`svgDepth -= 1`
- `svgDepth > 0` 时，其间 token 统一按 SVG token 对待

这样可以正确决定：

- 走 raw SVG
- 还是转义成文本

### 4. sanitizer 必须拆层，不能继续“一套名单跑天下”

这是本次设计里最重要的一点。

当前 `sanitizeOptions` 允许了很多 HTML tag，并且允许全局 `style` attr。

在 raw HTML 被真正开放之前，这个影响不大，因为 HTML 已经被 escape。
一旦开放 raw HTML，`style`、复杂块级标签、SVG/foreignObject 都会进入真实 DOM。

建议拆成三类 sanitizer：

#### A. Markdown 最终包裹 sanitizer

用于最终 `renderedHtml` 总清洗。

特点：

- 普通 Markdown 标签白名单
- 默认不放开全局 SVG 标签
- 默认不对普通 HTML 放开 `<style>`

#### B. Raw HTML fragment sanitizer

用于正文里的原生 HTML。

建议：

- 允许常见结构标签：`div/span/p/section/article/details/summary/table/...`
- 允许安全链接和图片
- 尽量 **不要** 对普通 raw HTML 放开 `<style>` 和 `iframe`
- `style` attribute 建议默认关闭或最小化

#### C. Raw SVG fragment sanitizer

用于正文里的 `<svg>`

建议复用现有 `sanitizeSvgPreviewMarkup(...)` 思路，但单独命名，例如：

- `sanitizeInlineSvgMarkup(...)`

必要时允许：

- SVG profile
- `foreignObject`
- `dominant-baseline`
- 少量 `style`

但这里需要注意：

- raw SVG 一旦直接进入聊天 DOM，`foreignObject` 的影响范围比 code fence preview 更大
- 如果希望最稳，可以在第一期对 raw SVG 先不开放 `foreignObject`
- 复杂 SVG 继续推荐用 ```svg code fence

### 5. raw HTML 和 raw SVG 的 UX 边界

建议第一期这样处理：

- raw HTML / raw SVG 直接渲染
- 不额外加工具栏
- 不自动提供全屏按钮

原因：

- 这是“正文内容”的一部分，不应都长成 preview card
- 强行加 toolbar 会破坏正文阅读

但保留一条增强线：

- 对“独占一行的 block-level raw SVG”可在第二期加 hover affordance
- 点击后复用现有 `livePreview` modal

### 6. HTML 复杂场景继续保留 iframe code fence

这一点必须明确：

- raw HTML mixed render 适合小片段
- 复杂完整页面、需要隔离样式、需要脚本环境、需要稳定尺寸测量的内容
  继续用 ```html code fence 更合适

也就是说：

- mixed render 不是替代 iframe preview
- mixed render 是补充一个“轻量、即时”的渲染轨道

## 开关设计建议

## 建议直接复用现有 `html/svg` 开关

原因：

- 用户心智简单
- UI 不需要扩 5 个开关
- 配置持久化逻辑不用重写

语义定义为：

- `HTML = On`
  - raw HTML tags 可渲染
  - ```html preview 可渲染
- `HTML = Off`
  - raw HTML tags 显示为文本
  - ```html 退回高亮代码块

- `SVG = On`
  - raw `<svg>` 可渲染
  - ```svg preview 可渲染
- `SVG = Off`
  - raw SVG 显示为文本
  - ```svg 退回高亮代码块

`Mermaid` 继续只控制 fenced preview 即可。

## UI 文案建议

当前文案是“控制默认内联预览；关闭后回退为高亮代码块。”

建议改成更准确的表达，例如：

- `控制 HTML 渲染：正文中的 HTML 标签与 HTML 代码块预览。关闭后标签按文本显示，代码块回退为高亮源码。`
- `控制 SVG 渲染：正文中的 SVG 标签与 SVG 代码块预览。关闭后标签按文本显示，代码块回退为高亮源码。`

因为混合渲染启用后，它不再只是“inline preview”。

## 分阶段实施计划

## Phase 1：最小可用版本

目标：先把 mixed render 做通，不改整体产品心智。

### 前端代码改动

1. `apps/web-vue/src/features/chat/markdown.ts`
   - 扩展 render options
   - 重写 cache key
   - 重写 `markdownRenderer.html`
   - 增加 raw HTML / raw SVG sanitizer
   - 最终 sanitize 规则拆层

2. `apps/web-vue/src/features/chat-v2/MarkdownBlock.vue`
   - 用当前会话生效的 `html/svg` 开关驱动 `renderChatMarkdownResult`
   - 开关切换时触发重新渲染

3. `apps/web-vue/src/features/chat-v2/ConversationPane.vue`
   - 更新文案，说明它控制的是“渲染”而不是只有“预览”

4. `apps/web-vue/src/features/chat-v2/inline-preview-preferences.ts`
   - 类型不一定要改，语义文档要改

### 测试项

1. Markdown + `<span>` 混排，HTML 开启时渲染，关闭时转义
2. Markdown + block `<div>` 渲染
3. Markdown + `<svg>` 渲染，SVG 开启时显示，关闭时转义
4. ```html / ```svg / ```mermaid 现有行为不回退
5. 全局/会话开关切换即时生效
6. 流式追加未完成 HTML/SVG 时不报错
7. `script` / `onload` / `javascript:` 等危险内容被去除

## Phase 2：体验增强

### 可选增强

1. 对独占块级 raw SVG 增加点击放大
2. 对 fenced preview block 做“弱边框/弱工具栏/hover 才显示”的样式
3. 给 raw HTML block 增加轻量容器类，避免贴边过紧
4. 增加模型提示词规范：
   - 小片段用 raw HTML/SVG
   - 完整 demo 用 fenced code block

## Phase 3：策略收紧或放宽

视安全评估结果决定：

1. raw SVG 是否放开 `foreignObject`
2. 普通 raw HTML 是否放开 `style` attr
3. 是否要引入更细粒度的 raw-html/raw-svg 独立开关

## 为什么我不建议“一上来就迁库”

因为这次需求的核心不是 parser 能力，而是：

- 当前 renderer 的策略选择
- sanitizer 分层设计
- mixed render 与 code fence preview 的边界

无论换成 `markdown-it` 还是 `remark/rehype`，这些问题都还在。

迁库只能解决“如何解析”，不能自动解决：

- 当前已有 preview block 能力复用
- 全局/会话开关复用
- HTML iframe 与 raw HTML 的产品边界
- SVG 安全范围

因此迁库不是这次最优解。

## 最终推荐

### 推荐路线

采用：

1. **现有 `marked` 保留**
2. **开放 raw HTML/SVG mixed render**
3. **保留 fenced code preview 作为复杂内容专用轨道**
4. **直接复用现有 `html/svg` 全局与会话开关**
5. **第一期不强行给 raw HTML/SVG 加工具栏；全屏能力继续以 fenced preview 为主**

### 不推荐路线

不建议优先做：

1. 只做“去边框代码块 preview”
2. 为这次需求整体迁移到 `markdown-it`
3. 为这次需求整体迁移到 unified/remark/rehype

## 建议的实施顺序

1. 先做 `markdown.ts` 的 raw HTML/SVG renderer 与 sanitizer 分层
2. 再把 `MarkdownBlock.vue` 与会话开关打通
3. 再补设置文案
4. 最后决定是否做 raw SVG 点击放大和 code fence 弱边框样式
