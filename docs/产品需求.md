# Tracevane Product Requirements

> Status: active baseline
> Updated: 2026-06-25

## 1. Product goal

Tracevane is a local-first AI Agent connectivity and runtime control layer.

It should let a user safely answer and operate:

- Which model/provider/account is each Agent or client using?
- How do Codex, Claude Code, OpenCode, OpenClaw, local Chat and IM-triggered tasks connect to the model gateway?
- Which Agent Run is currently running, idle, failed, delivered or waiting?
- Which IM account/bot/binding receives messages and where are replies delivered?
- Which logs, tool calls, terminal sessions, files, usage and errors prove what happened?
- Which daemon or local config needs repair?

## 2. Current product reality

| Area | Current state | Product judgment |
| --- | --- | --- |
| Model Gateway | High completion. Provider/model/account/protocol/app-connection/usage flow is usable and should be stabilized. | Continue hardening, not redesigning. |
| IM Channels | Mostly read-only/control-light. It does not yet provide a full account/bot/binding CRUD and test workflow. | Needs P0 implementation work. |
| CLI Agents | Runtime projection exists; first reduction removed primary Persona and terminal-management tabs. | Continue de-OpenClaw/de-terminal refocus. |
| Chat / Workspace IDE / Recovery / Platform | Support surfaces for local work, evidence, substrate support, integrations and repair. | Keep as supporting pillars, not competing core domains. |
| External / Long Tasks legacy pages | Read-only aggregation and supervised-work projection. | Do not expose as primary domains; keep as compatibility deep-links or support-page drilldowns. |

## 3. Core domains

### Model Gateway

Owns provider and model connectivity:

- Provider registry, endpoint profiles, API keys/secret refs.
- Codex account provider and account pool.
- Model catalog, aliases, context/output budgets and capabilities.
- Protocol adaptation among OpenAI Responses, Chat-compatible and Anthropic Messages.
- Routing, fallback, health, circuit breaker, usage and cache evidence.
- Client config writing for Codex, Claude Code, OpenCode and OpenClaw.
- Gateway daemon/service state.

### IM Channels

Owns IM platform connectivity:

- Platform account and bot credentials.
- Bot/app id, app secret, token, verification token, encryption key, webhook URL, long-connection state and platform-specific fields.
- Bindings from IM user/group/channel/thread to Agent runtime settings.
- Inbound message handling, command surface, attachment staging, outbound reply delivery and delivery logs.
- Channel daemon/service state.

### CLI Agents

Owns local Agent CLI runtime and the primary Agent Runs operating surface:

- Codex, Claude Code and OpenCode install/version/config readiness.
- Launch/resume/stop boundaries for local CLI sessions.
- Agent Runs projection from terminal, IM and Chat runtime sources.
- Terminal-sourced evidence links back to IDE / terminal instead of owning terminal management.
- Runtime evidence: process/session id, model/provider when known, workspace, errors, output and links back to source evidence.

### Support and compatibility surfaces

These are not core domains:

- External integration evidence: read-only MCP / Skills / app-connection / transport evidence. It lives under Platform/support navigation or deep links and must not advertise write capability unless a dedicated backend contract exists.
- Task supervision (`/long-tasks` compatibility route): cross-domain supervised-work projection for active/waiting/attention states. CLI Agents remains the primary Agent Runs workbench; IM Channels and Recovery own their domain-specific controls.

CLI Agents must not become generic OpenClaw agent/profile administration. OpenClaw is a supported substrate and can appear as platform/recovery context, not as the center of CLI Agents.

## 4. Non-goals

- Do not duplicate the official OpenClaw admin UI.
- Do not build generic chatbot features unrelated to Agent runtime evidence.
- Do not merge provider secrets, IM bot tokens and PTY controls into one unsafe service.
- Do not fake support for IM writes, CLI launches, cache hits or context management.
- Do not preserve stale docs that contradict current architecture.

## 5. Near-term priorities

### P0

1. Make IM Channels usable, not just visible: account CRUD, bot credential forms, binding CRUD, validation, smoke tests and delivery evidence.
2. Refocus CLI Agents around Codex / Claude Code / OpenCode runtime: remove OpenClaw generic persona/admin concepts and keep terminal management in IDE.
3. Keep Model Gateway stable: do not regress Codex account providers, app connections, route checks, usage/cache explanation or daemon restart flow.

### P1

- Expand Agent Runs lifecycle and evidence fields.
- Add clear IM platform templates for Feishu, WeCom/企业微信 and Octo.
- Improve route/usage diagnostics with real ledger samples.

### P2

- Consider a higher-level “Agent Connectivity” narrative only after IM and CLI runtime are genuinely usable.
- Do not add new top-level domains for read-only aggregation; fold them into Platform, Dashboard, or the owning core domain.
