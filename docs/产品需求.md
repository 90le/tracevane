# Tracevane PRD

> Status: active baseline
> Updated: 2026-06-18
> Naming: Tracevane is the current product name; public launch still requires registrar and trademark preflight.

## 1. Product Goal

Build a local-first AI Agent control workbench for people who run real agent workflows across model providers, CLI Agents, IM channels and OpenClaw runtime.

The product must help users answer:

- Which model/provider/endpoint is being used?
- Which agent/session/task is running, waiting, failed or complete?
- Which files, tools, approvals and messages were involved?
- Which daemon, runtime config or credential issue needs repair?
- How can a user safely connect Codex, Claude Code, OpenCode, OpenClaw and IM channels without editing scattered config by hand?

## 2. Product Boundary

Core:

- Model Gateway
- Channel Connectors
- CLI Agent Workbench
- System Guard / Recovery
- Observability and evidence
- Task-first Dashboard
- Tracevane-specific Chat and Files

Support:

- Config compatibility and safe writes
- OpenClaw runtime integration
- Terminal maintenance surface

Delegate:

- Generic OpenClaw CRUD already covered well by official OpenClaw Web UI
- Generic self-hosted chat UI features already covered by Open WebUI / LibreChat-style products
- Generic visual workflow/agent builder features already covered by Dify/n8n/Copilot Studio/Oracle AI Agent Studio-style products

Retired:

- Dreaming / memory management UI and BFF
- Legacy plugin management UI and BFF
- Old complex Gateway usage archive/cost/pagination surface

## 3. Target Users

- Local-first developers running Codex, Claude Code, OpenCode and OpenClaw together.
- Users who want IM private chat to trigger local CLI Agents with file, image, permission and progress support.
- Users who need a model gateway to bridge provider protocols and client protocols.
- Operators who need recovery, config validation and evidence before trusting long-running agent work.

## 4. Product Principles

- Runtime-agnostic first: OpenClaw is supported, but not the whole product.
- Official contracts first: provider, SDK, CLI, channel and API behavior must be researched before implementation.
- Structured events over TUI parsing: use official session/SDK/hook/event surfaces when available.
- One active task must not be misclassified: avoid both false finish and infinite wait.
- UI must be task-first and understandable; avoid dumping every field on one page.
- High-risk writes need preview, validation, backup and rollback where possible.
- Unsupported is better than fake support.
- Do not claim public naming clearance until registrar and trademark preflight is complete.

## 5. Domains

| Domain | Classification | Direction |
| --- | --- | --- |
| Model Gateway | Core | Provider routing, protocol adaptation, App Connections, usage and health. |
| Channel Connectors | Core | IM to CLI Agent runtime, files, progress, permissions and sessions. |
| CLI Agent Workbench | Core | Codex/Claude Code/OpenCode/OpenClaw profiles, model binding, workdir, permissions and native sessions. |
| System Guard / Recovery | Core | OpenClaw config compatibility, service repair, daemon status and secret audit support. |
| Observability | Core | Session traces, tool calls, approvals, provider errors, tokens, cost and verification evidence. |
| Dashboard | Core redesign | Task-first operations cockpit, not a generic status clone. |
| Chat | Support/core hybrid | Agent workbench chat with evidence and controls, not a generic chat clone. |
| Files | Support | Artifact and attachment inspector, not a full file manager. |
| Config | Support | Compatibility, diff, backup and critical guided settings only. |
| Agents / Channels | Support | Tracevane-specific profiles and bindings; delegate broad OpenClaw CRUD. |
| Skills | Retire candidate | Keep only if it becomes cross-Agent capability/workflow packaging. |
| Terminal | Support | Debug/maintenance only. |
| Dreaming / Plugins management | Retired | Do not restore without a new PRD. |

## 6. Success Criteria

- Gateway can prove current provider/protocol support with local and live smoke where required.
- Channel Connectors can reliably classify running, waiting, failed and completed work for Codex, Claude Code and OpenCode.
- Recovery can detect and repair supported OpenClaw runtime/config/service breakage without unsafe rewrites.
- The UI exposes fewer, clearer pages with drill-in detail instead of one-page field dumping.
- Current docs describe product boundaries in less than one reading pass.
- Naming remains conflict-aware; no crowded market name is adopted by accident.

## 7. Current Non-Goals

- Rebuilding official OpenClaw Web UI.
- Building a generic no-code AI workflow builder.
- Building a generic Open WebUI/LibreChat competitor.
- Supporting every OpenClaw setting as a Tracevane form field.
- Restoring Dreaming or legacy plugin management.
- Claiming public name clearance without the naming gate.
