# OpenClaw Studio PRD

> 状态：Management Stable + Chat Runtime Active
> 更新：2026-06-01

## 1. 目标

OpenClaw Studio 是 `openclaw` 的管理控制台，用来替代手工编辑 JSON、散落的 CLI 和记忆性运维操作。

当前阶段继续以管理域稳定为底座，同时把 `chat` 作为正式运行面推进：后端 typed adapter / BFF、会话策略、前端 Chat Shell、富消息渲染和回归测试都已经进入主线。

## 2. 当前范围

当前正式纳入：

- `config`
- `skills`
- `terminal`
- `channels`
- `cron`
- `agents`
- `system`
- `chat`

当前已接入但仍需持续打磨：

- `dashboard`
- `codex-stack`
- `dreaming`
- `files`
- `plugins`

当前暂缓：

- `room`
- `workflow`

## 3. 目标用户

- 单机自托管 OpenClaw 的个人用户
- 管理多 Agent、多频道、多技能、多 cron 任务的重度用户
- 需要并行开发、快速定位状态和进行低风险运维的开发者

## 4. 产品原则

- 配置优先界面化，不鼓励直接改 JSON
- 页面按领域拆分，不把多个域重新揉进一个大页
- 共享类型先行，前后端语义保持一致
- 写操作保守，优先保证配置合法和可回滚
- 高风险能力必须给出清晰错误态和确认路径
- `chat` 继续遵守 `types/`、`apps/api/modules/<domain>`、`apps/web-vue/src/features/<domain>` 边界
- 浏览器不直接消费 Gateway raw protocol；面向前端的是 Studio typed contract

### 5.9 Chat

职责：

- 以 `Studio` 工作台形式管理 `openclaw` 会话聊天
- 基于现有 Gateway / session 语义提供真实历史、发送、停止、重置和会话切换
- 先提供 `Studio` 自管的安全 chat 工作区，不直接把生产频道会话改成默认写入目标

当前状态：

- 已完成方案重评估，详细设计见 `docs/聊天设计方案.md`
- 已完成 typed contract，详见 `docs/聊天契约.md`
- 已完成 session policy，详见 `docs/聊天会话策略.md`
- 已完成开放门槛梳理，详见 `docs/聊天开放门槛.md`
- 后端已有 `apps/api/modules/chat/**`、`types/chat.ts` 和 `/api/chat/*` 路由
- 前端已有 `/chat` 主壳，当前主实现集中在 `apps/web-vue/src/features/chat/**`
- 不采用 `Clawith` 的 React/FastAPI/数据库会话结构，继续遵守 `插件壳 + 模块化 API + Vue 管理工作台`

## 6. 功能边界

必须具备：

- 插件能稳定加载
- 各领域页面路由清晰
- 关键配置和管理对象支持真实读写
- 错误态、空态、成功态清晰可见
- 支持并行子代理开发，不争抢同一模块目录

当前不做：

- `workflow` 图形编排
- 复杂 `room` 协作界面
- 默认把生产频道会话作为 `Studio` chat 的写入目标
- 新建 `Clawith` 式平行数据库会话模型
- 浏览器直连 OpenClaw Gateway 并消费 raw frame

## 7. 成功标准

当前阶段成功标准：

- `config / skills / terminal / channels / cron / agents` 六个管理域可真实使用
- 常见管理操作不再依赖手写 `openclaw.json`
- `dashboard / system` 能消费真实后端摘要
- `chat` 的范围、方案、阶段和边界在文档中冻结，详见 `docs/聊天设计方案.md`
- `chat` 主链路具备 create / history / send / stream / abort / reset / search / organizer / upload / rich media 基础能力
- 文档、代码结构和当前实现状态保持一致

## 8. 当前结论

Studio 现在的正确方向已经固定：

1. 管理域保持稳定、可读写、可回滚
2. `chat` 以 Studio typed adapter / BFF 为主线继续收口
3. `chat` 前端后续应收敛 `chat` 与 `chat` 命名边界，避免长期双轨
4. `dashboard / codex-stack / files / plugins / dreaming` 继续按真实运维工作台推进
5. `room / workflow` 继续后置
