# Platform 域目标与设计

> Status: active target
> Updated: 2026-06-25

## 1. 定位

Platform 域不是“卡片式摘要页”，也不是 Tracevane 的第四个日常工作流。它是 **第三方平台管理域**：管理 Tracevane 依赖、连接、嵌入或需要展示运行证据的外部平台。

OpenClaw 是第一个平台，但 Platform 不能只按 OpenClaw 特例设计。未来平台可以包括 Feishu、WeCom、GitHub、MCP server、enterprise SSO、artifact store 或其它宿主/集成平台。

核心判断：

- 一级导航只保留 `平台`。
- `/platforms` 是平台目录和平台健康总览。
- 每个平台拥有自己的二级、三级、四级页面。
- 能写入的页面必须有明确后端契约、权限边界、确认流和结果证据。
- 只读能力必须明确标注为证据，不得伪装成已完成管理能力。

## 3. Navigation model

Platform needs nested navigation, not a flat card wall.

### Global navigation

Global sidebar only exposes:

```text
系统 / 平台
```

### Platform directory

`/platforms` shows platform rows or a dense platform list:

- OpenClaw
- Feishu / WeCom / GitHub / MCP 等未来平台占位或真实接入
- Status, capabilities, last check, owner links

Current rule: only real third-party platforms enter the primary directory.
Do not render Tracevane owner-domain handoff panels or compatibility-link panels
on the Platform homepage. Model Gateway, IM Channels, CLI Agents and IDE already
have first-class navigation. Legacy aggregation routes (`/external`,
`/long-tasks`) are redirect-only compatibility links; do not keep standalone
frontend feature folders for them.

### Platform shell

Every `/platforms/:platformId/*` page should have:

- Breadcrumb: `平台 / OpenClaw / Skills / skill-id`
- Platform identity rail/header: platform name, status, version/account/tenant, last check
- Platform-local navigation: Overview, Guard, Config, Agents, Skills, Channels, Bindings, Services, Logs, Diagnostics
- Main work area: list/detail/settings/evidence
- Owner-domain handoffs: Model Gateway / IM Channels / CLI Agents / IDE links when the work belongs elsewhere

Breadcrumb accessibility should follow WAI-ARIA breadcrumb guidance: use a navigation landmark and mark the current item. Platform-local navigation can use tabs/viewbar for sibling sections or a disclosure/tree/list pattern when sections become deeply nested.

## 4. OpenClaw ownership

OpenClaw platform pages own OpenClaw-native concepts:

| Section | Owns | Must not own |
| --- | --- | --- |
| Overview | Runtime identity, version, counts, platform health, important cross-links. | Provider editing, IM route editing, CLI run control. |
| Guard | OpenClaw gateway probe, recovery daemon lifecycle, repairs, backups, restore evidence. | Tracevane business-data restore. |
| Config | OpenClaw config inspection, validation, diff, backup-backed safe edits when backend exists. | Raw unsafe config editing without validation/backup. |
| Agents | OpenClaw agent definitions/profiles and platform metadata. | Runtime Agent Runs; those belong to CLI Agents. |
| Skills | OpenClaw skills registry, enablement/config evidence and safe toggles when backend exists. | Arbitrary plugin install unless verified and guarded. |
| Channels | OpenClaw channel definitions and platform-side channel evidence. | Tracevane IM account/binding CRUD; that belongs to IM Channels. |
| Bindings | OpenClaw bindings as platform config evidence/management when backed by OpenClaw config contract. | IM delivery/session control. |
| Services | OpenClaw service/gateway/control UI/daemon status and lifecycle with confirmations. | Generic OS service manager. |
| Logs | OpenClaw/platform events and command evidence. | Replacing domain-specific logs. |
| Diagnostics | Doctor, config validation, bootstrap, security and install diagnostics. | Fabricated health checks. |

## 5. Relationship to existing Tracevane domains

Platform can show relationships, but writes must return to the owner domain unless the write is truly platform-owned.

- Model provider/API key/model routing -> Model Gateway.
- IM bot/account/route/session delivery -> IM Channels.
- Codex/Claude/OpenCode runtime and Agent Runs -> CLI Agents.
- Terminal/PTY/file workspace -> IDE / Files.
- OpenClaw agent/skill/channel/config/service -> Platform / OpenClaw.

Example: OpenClaw `channels` page may show a channel that is used by Tracevane IM Channels, but the IM credential/binding editor remains in IM Channels. The OpenClaw page links to the IM owner instead of duplicating the form.

## 6. Backend integration strategy

Do not invent platform write capability in the frontend. Implement by capability tier:

1. **Read evidence**: use existing `agents`, `channels`, `skills`, `config`, `system`, `openclaw-recovery`, dashboard summaries.
2. **Safe diagnostics**: status, validate, doctor/probe, logs, events.
3. **Guarded writes**: edits/toggles/service actions only with typed backend route, diff/backup where relevant, confirmation dialog and command/result evidence.
4. **Destructive actions**: require explicit confirmation, backup/rollback when possible and visible audit trail.

Existing backend areas to evaluate for OpenClaw platform pages:

- `apps/api/modules/agents`
- `apps/api/modules/channels`
- `apps/api/modules/skills`
- `apps/api/modules/config`
- `apps/api/modules/system`
- `apps/api/modules/openclaw-recovery`
- Dashboard summary counts for agents/channels/skills/config/bootstrap

## 7. Frontend design rules

Platform pages are operational workbenches:

- Use dense list/detail and settings/evidence layouts, not marketing cards.
- Use rows, tables, definition lists, command evidence, diff panels and right-side Sheets for details.
- Mobile must collapse nested nav into a selector/disclosure without horizontal page overflow.
- Breadcrumb must remain visible near the top of platform subpages.
- Details can be third/fourth-level routes or Sheets depending on whether the detail is shareable/bookmarkable.
- Empty states must say which backend contract is missing or which owner domain should be used.

## 8. Implementation phases

### Phase 1 — IA shell

- Status: implemented in the current workspace baseline.
- `docs/平台域目标与设计.md` is the authority.
- Platform shell primitives are present: `PlatformBreadcrumb`, OpenClaw section registry and platform-local navigation.
- `/recovery` and `/runtime-admin` redirect to `/platforms/openclaw/guard`; `/platforms/openclaw/recovery` is kept as a compatibility alias through section normalization.
- Existing OpenClaw summary and guard remain functional.

### Phase 2 — OpenClaw read surfaces

- Status: implemented as read/evidence surfaces.
- OpenClaw sections for config, agents, skills, channels, bindings, services, logs and diagnostics use existing read APIs (`config`, `agents`, `skills`, `channels`, `system`, `openclaw-recovery`).
- Read-only sections must keep owner-domain handoff text visible until a typed guarded write contract exists.
- System tests cover route registry, breadcrumb, read API binding and owner-boundary language.

### Phase 3 — Guarded writes

- For each section, add only verified write actions with backend contracts.
- Config edits require validation + diff + backup + result evidence.
- Service/repair actions require confirmations and command evidence.

### Phase 4 — Additional platforms

- Generalize platform registry for Feishu, WeCom, GitHub, MCP or other third-party platforms.
- Add platform capability descriptors so pages can show available read/write/test/diagnostic actions honestly.

## 9. Non-goals

- Do not make Platform the owner of Model Gateway, IM Channels or CLI Agents workflows.
- Do not expose raw secrets or unmasked tokens.
- Do not create fake add/edit/delete buttons without backend contracts.
- Do not delete compatibility routes during the refactor.
- Do not implement arbitrary OpenClaw plugin marketplace install/update without research and guarded backend support.
