# OpenClaw Studio 前端重构方案

> 状态：方案（待评审） · 作者：前端架构审计 + 2026 生态调研 · 日期：2026-06-17
> 关联：[当前进展](./当前进展.md) · [界面设计守则](./界面设计守则.md) · [系统架构](./系统架构.md)

---

## 0. 一句话结论

**保留 Vue 3，做一次有纪律的渐进式重构**：先立状态层（Pinia）和样式地基（Tailwind v4），再从最大的上帝组件 `ChatShellPage.vue`（7410 行 / 270 函数）开刀拆解，最后收敛 14 个重复的 `*-workspace.css`。**不重写到 React/Solid/Svelte**，除非团队明确愿意承担小众框架的招聘成本且对性能有硬需求。

完整论证见本文 §1（现状）、§2（框架决策）、§3（落地路线）。

---

## 1. 现状诊断（带证据）

前端位于 `apps/web-vue`，Vue 3 + vue-router + `@nuxt/ui`，约 **9.7 万行**。后端 API（`apps/api`）稳定，问题集中在前端。

### 1.1 四个结构性硬伤

| # | 问题 | 证据 |
|---|---|---|
| 1 | **上帝组件** | `ChatShellPage.vue` = 7410 行，单个 `<script setup>` 内 60+ 顶层 ref、270 个函数、41 个 import。同类还有 `ConfigEditorPage`(4430)、`ModelGatewayControlPage`(4279)、`ConversationPane`(3619) |
| 2 | **零状态管理** | 全仓 `grep`：**0 个 Pinia、0 个 Vuex、0 个 provide/inject**。所有跨组件状态堆在页面局部 ref 里，靠 props/events 透传 |
| 3 | **CSS 全局裸奔** | **0 个 `<style scoped>`**；38 个全局 `.css`；14 个几乎相同的 `*-workspace.css`（合计 ~2.1 万行）逐 feature 复制粘贴同一套工作台壳子；`style.css` 单文件 2453 行塞满所有 token |
| 4 | **引入却不用 UI 库** | `@nuxt/ui` 已注册，但全仓仅 **1 处** `<UApp>`；所有按钮/对话框/抽屉手搓 `<button class="...">` + 全局 CSS；drawer/dialog/overlay 每个 feature 各重写一遍（20+ 文件） |

### 1.2 一个被低估的好底子

现状并非全黑。仓库**已经出现半成品的提取模式**——`ChatShellPage` 虽臃肿，但已把部分状态推进了独立模块：

```
chat-session-runtime-machine.ts   1562 行  ← 会话运行状态机
session-list-view-model.ts         318 行  ← 会话列表视图模型
chat-runtime-view-model.ts         256 行
chat-session-scroll-state.ts       369 行  ← 滚动锚点
chat-record-browser-state.ts       142 行
terminal-workspace-state.ts        620 行
system-event-store.ts               94 行
```

**这意味着重构不是从零发明架构，而是把已有的 view-model / machine 模式统一升级、外推到全部 feature。** 这大大降低了风险和工期估算。

## 2. 框架与组件库决策（基于 2026 调研）

调研结论（对抗式验证后 7 条存活 / 18 条被否，详见 deep-research 报告）：

### 2.1 框架：保留 Vue 3，不重写

针对本项目的真实负载（万行表格、WebSocket 终端流、流式聊天事件），框架实测：

| 操作 | React | Solid | Svelte |
|---|---|---|---|
| 更新 1000 行 | 49.6ms | **10.6ms** | 11.0ms |
| 几何均值 slowdown | **5.09（最差）** | 1.13 | 1.16 |

> 数据源：krausest/js-framework-benchmark（Chrome 148 / 2026-06），黄金基准。

**但选 React/Solid/Svelte 重写的代价：**
- 招聘/生态：Stack Overflow 2025 显示 Node 开发者最想用的是 **React/Next/Vue**；Solid 满意度第一但使用率仅 ~10%（小众，招聘贵）。Qwik 明确"人才池更小"。
- 本项目是**消费独立 HTTP/WS API 的纯客户端 SPA**（见 `index.ts`：`registerHttpRoute` 3760 端口 + gateway `/studio` 路径），SSR/RSC 优势**对它毫无意义**。
- 重写会丢弃现存的、被 acceptance 测试（`tests/chat/*.acceptance.py` 数十个）反复验证过的行为。

**结论：性能差距真实存在，但当前 Vue 3 应用并未报告性能瓶颈；为"可能更快"而承担整盘重写的招聘、工期、回归风险，不划算。** 性能留作触发条件：若未来 Chat/Terminal 出现实测卡顿，再单独评估迁 Solid（届时已有干净的 store/组件边界，迁移成本更低）。

> ⚠️ 调研存疑项：Vue Vapor Mode 的 GA 状态不确定（"Vue 3.6 稳定"被否决，VueSchool 措辞为"2026 可能落地"）。**不要赌 Vapor**。

### 2.2 组件库：替换 `@nuxt/ui` → Reka UI + shadcn-vue + Tailwind v4

- `@nuxt/ui` 引入却几乎没用，是**负债不是资产**（占依赖体积、却还要手搓一切）。
- 2026 React 生态的事实标准是 **shadcn/ui**（113.6k stars）= "CLI 复制进仓库的组件层"，底层 Radix UI 或 Base UI 原语 + Tailwind。
- shadcn/ui 是 **React-only**。Vue 路径的对应物：**Reka UI**（Radix-Vue 继任者）+ **shadcn-vue** + Tailwind v4。

> ⚠️ 开放风险（调研未验证）：Reka UI / shadcn-vue 的 2026 成熟度、star 走势、生产采纳率**没有存活的确认论点**。**Phase 0 必须先做一个 spike**：在最小一个页面上跑通 Reka UI + shadcn-vue，确认组件覆盖（Combobox 虚拟化、DataTable、Drawer）满足本项目需求，再全面铺开。

### 2.3 CSS：迁移到 Tailwind v4（无论选哪个组件库都做）

- Tailwind v4（2025-01 GA）：Rust Oxide 引擎，构建快 10–100×；原生 cascade layers、CSS 变量、容器查询；CSS-first `@theme` 配置。
- 框架无关，直接消灭 38 个全局 CSS + 14 个重复 `*-workspace.css` 这笔债务。
- 调研里 **CSS-in-JS / token 迁移 / feature-sliced 的具体做法全部没有存活的确认论点**（被限流打掉验证，非被证伪）。本方案用工程界公认做法：**先建 token、逐 surface 迁移、新旧并存**。

## 3. 落地路线（4 个 Phase，可灰度、可回滚）

每个 Phase 都**独立可上线**，不依赖后续 Phase。建议每个 Phase 走 `feature` 分支 + 对应 acceptance 测试把关。

### Phase 0 · 基础设施 spike（约 1–2 天，CC 辅助 ~1h）

**目标**：验证技术选型可行，不赌空。

1. 在 `DashboardView`（最小、最简单的页面）上跑通：Tailwind v4 + Reka UI + shadcn-vue。
2. 确认覆盖：DataTable 虚拟化（Terminal 万行表用）、Drawer、Dialog、Combobox。
3. 建立 `tokens.css`（从现有 `style.css` 的 `--studio-*` / `--mono-*` 抽出）。
4. **产出一份 go/no-go 评估**。若 Reka UI 覆盖不足，回退到 PrimeVue 或真正启用 `@nuxt/ui`。

> 完成判据：一个页面用新栈，视觉与旧版一致，`npm run dev:web:smoke` + 对应 acceptance 通过。

### Phase 1 · 立状态层（最高优先级，解 God Component 的前提）

**目标**：引入 Pinia，把分散的 view-model/machine 收编为正式 store。

1. 引入 Pinia（`apps/web-vue/package.json` + `main.ts` 注册）。
2. **不改行为**，只搬运。把现有这些模块升级为 Pinia store，对应关系：

   | 现有模块 | → Pinia store | 职责 |
   |---|---|---|
   | `chat-session-runtime-machine.ts` | `useChatRuntimeStore` | 会话运行状态机、WebSocket/SSE 连接、事件流分发 |
   | `session-list-view-model.ts` | `useSessionListStore` | 会话列表、筛选、organizer |
   | `chat-session-scroll-state.ts` | `useChatScrollStore` | 滚动锚点、视口快照 |
   | `chat-record-browser-state.ts` | `useChatHistoryStore` | 历史分页、上下文窗口加载 |

3. 服务器状态引入 **TanStack Query for Vue**（调研未验证，但 VueQuery 是公认默认），统一 `loadAgents/loadHealth/loadSessions` 这类请求的缓存/失效，消灭手写的 `xxxLoading` ref 三连。

> 完成判据：`ChatShellPage` 的 60+ ref 下沉到 store，页面只保留 UI 状态（drawer 开关等）。行为零回归（acceptance 全绿）。

### Phase 2 · 拆解 ChatShellPage.vue（7410 行 → ~8 个组件 + stores）

**目标**：消灭最大风险源。按 `ChatShellPage` 内天然的 270 个函数聚类，拆成：

```
ChatShellPage.vue (壳，<300行，只做布局编排)
├─ SessionListPanel.vue        ← 已存在，对接 useSessionListStore
├─ ConversationPane.vue        ← 已存在(3619行)，继续拆
│   ├─ MessageList.vue
│   ├─ MessageBubble.vue       ← 已存在
│   └─ ToolStreamRenderer.vue
├─ ComposerBar.vue             ← 已存在，对接 useChatRuntimeStore
├─ InspectorPanel.vue          ← 已存在
├─ QueuedMessageRail.vue       ← 已存在
└─ ChatConnectionManager.ts    ← connectChatSocket/connectGatewayClient/startGatewayHeartbeat
                                  提为独立连接层（非组件），由 store 驱动
```

**策略**：先把 `connect*/heartbeat/handleStream*` 这批连接与事件分发函数（~20 个）提到 `useChatRuntimeStore`，这些是最该脱离 UI 的。每提一批，跑一遍 `tests/chat/*.acceptance.py`。

> 完成判据：`ChatShellPage` < 300 行；其余同类 God Page（`ConfigEditorPage`、`ModelGatewayControlPage`）按同模板后续处理。

### Phase 3 · 收敛样式 + 替换 UI 库

**目标**：消灭 CSS 全局裸奔与组件重复。

1. 抽公共组件到 `src/shared/components/`（现仅 5 个）：`StudioDrawer`、`StudioDialog`、`StudioEmptyState`、`StudioDataTable`、`StudioButton`。统一替换 20+ 处各 feature 自造的 drawer/dialog。
2. 把 14 个 `*-workspace.css` 合并为**一个参数化的 `StudioWorkspace` 布局组件** + scoped 样式，消除 ~2.1 万行重复。
3. 逐 feature 把全局 CSS 迁到 Tailwind v4 utility + scoped，用 cascade layers 隔离。
4. 移除 `@nuxt/ui` 依赖。

> 完成判据：0 个全局裸奔样式；新组件覆盖率 ≥90%；依赖瘦身。

## 4. 风险与回滚

| 风险 | 应对 |
|---|---|
| Reka UI 成熟度不足 | Phase 0 spike 先验证，不通过则回退 PrimeVue / 启用 `@nuxt/ui` |
| ChatShellPage 拆解引入行为回归 | 每个 Phase 走 acceptance 测试把关；保留旧文件直到新组件全绿再删 |
| Tailwind 全量迁移阻塞业务 | **token 先行 + 逐 surface 迁移**，新旧 CSS 并存，不做一次性大 PR |
| Pinia 迁移工作量大 | Phase 1 只搬运不改行为，复用现有 view-model 模块，风险最低 |
| 团队对 Solid/Svelte 的执念 | 性能优势真实但非当前瓶颈；设为触发条件，留干净边界以便将来单 feature 迁移 |

---

## 5. 不做什么（明确排除）

- **不重写到 React / Next.js**：性能最差、无 SSR 红利、招聘虽易但无独特优势。
- **不重写到 Solid / Svelte**：性能与 DX 更优，但招聘贵、重写代价大，且无实测性能瓶颈。留作触发条件。
- **不赌 Vue Vapor Mode**：GA 状态存疑。
- **不做一次性大爆炸式重构**：每个 Phase 独立可上线、可回滚。
