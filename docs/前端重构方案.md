# Tracevane 前端重构方案

> 状态：**V8 原型基线已收敛，仍未进入落代码阶段**
> 更新：2026-06-18
> 关联：[产品需求](./产品需求.md) · [系统架构](./系统架构.md) · [界面设计守则](./界面设计守则.md) · [V8 设计基线](./prototypes/tracevane-v8-design-baseline.md)

## 1. 当前结论

Tracevane 前端的大方向已经从“修旧 OpenClaw 管理后台”调整为：

> **本地优先的 AI Agent 控制工作台。**

当前视觉与应用壳基线是：

- `docs/prototypes/dashboard-v3-balanced-workbench-theme-v8-durable-shell.html`

当前技术方向是：

- React + TypeScript + Vite
- shadcn/ui + Tailwind CSS
- Radix primitives / lucide-react
- TanStack Query / TanStack Table
- xterm.js、Monaco 或 CodeMirror 只用于终端、编辑器等专业工作台表面

当前阶段仍然只做独立 HTML/CSS/JS 原型。不要修改 `apps/web-vue`，不要创建 React 前端壳，确认页面原型和信息架构后再进入实现讨论。

## 2. 已撤回路线

此前走过“macOS 材质派 + 直接落代码”路线，并尝试过 `apps/web-vue-next`。该路线已撤回，作为教训保留：

- 不在视觉方向未确认时直接写框架代码。
- 不把单一美学路线提前固化成工程实现。
- 不把旧前端路由、旧模块主页或后端模块名当作新产品信息架构。
- 不把 OpenClaw 管理后台、Dreaming、旧插件管理、旧模型链路诊断矩阵恢复为 Tracevane 主线。

旧 Vue 渐进重构、Reka UI / shadcn-vue、Pinia 拆分等方案已不再是当前目标路线。它们只作为历史背景存在，不作为新实现依据。

## 3. 产品边界

新前端围绕这些核心问题组织：

- 当前到底用了哪个模型、Provider、endpoint？
- 哪个 Agent、session、task 正在运行、等待、失败或完成？
- 过程中用了哪些文件、工具、审批、消息和证据？
- 为什么 IM / CLI / Gateway 卡住或失败？
- 配置、服务、凭据、插件漂移能否安全检测和修复？
- Codex / Claude Code / OpenCode / OpenClaw / IM 渠道能否稳定接起来，而不是手改一堆配置？

一句话：让本地 AI Agent 工作流可连接、可观察、可恢复、可证明。

## 4. 页面组织

全局导航采用 V8 的左侧分组导航。建议页面组：

| 组 | 页面方向 | 页面形态 |
| --- | --- | --- |
| 总览 | 仪表盘 | Summary |
| 运行 | 会话任务、工作区 IDE、Agent Chat、长任务 | Workbench / Status Console |
| 连接 | CLI Agents、模型网关、Providers、IM 渠道 | List Detail / Status Console |
| 证据 | 文件证据、审批、可观测性 | Rows + Inspector |
| 系统 | 自愈守护、安全配置 | Status Console / Settings |

页面设计必须遵守层次化：主对象只在主舞台，关联信息进入 Inspector / Sheet / Drawer / 子页面，低频配置默认不可见。

## 5. V8 设计基线

V8 durable shell 是后续页面的应用壳参考：

- 左侧分组导航，不使用全局 VS Code 活动栏。
- 顶栏是轻量工作区工具条，不做营销 hero。
- 浅色不是纯白，深色不是纯黑。
- 主舞台优先实色面板、细边界和低阴影。
- 信息密集区用行、表格、工具条、终端、编辑器和检视器，不做卡片墙。
- 卡片只用于少量摘要、重复对象或弹层。
- 命令面板是导航补充，不替代主导航。
- 移动端侧栏变抽屉，主舞台优先。

## 6. 原型阶段交付

先继续完善 HTML 原型，不进入实现：

1. 保留 V8 HTML 作为当前唯一视觉基线。
2. 从 V8 延展关键页面：仪表盘、工作区 IDE、会话任务 / Agent Chat、模型网关、CLI Agents、IM 渠道、文件证据、审批、自愈守护。
3. 每个页面必须有深浅主题、响应式布局和基础交互。
4. 原型界面中文为主；库名、命令、协议名和路径保留英文。
5. 每个新原型记录到 `docs/prototypes/INDEX.md`。
6. 不连接后端；必要聚合数据可先设计为前端假数据，后续再补 API。

## 7. 实现阶段草案

只有当页面原型确认后，才进入 React 实现。建议落地顺序：

1. 新建独立 React 前端目录，不改 `apps/web-vue`。
2. 运行 `shadcn init`，生成 `components.json` 后再按页面需要引入组件。
3. 建立 Tailwind v4 token：颜色、圆角、阴影、字体、状态色、深浅主题。
4. 先实现 V8 AppShell：Sidebar、Topbar、Command、Sheet、Drawer、Theme。
5. 用 TanStack Query 封装 API 服务端状态，用 TanStack Table 承载对象列表。
6. 对终端、编辑器、预览、diff 和证据链使用专用引擎和独立性能验证。
7. 用 API 合同测试、页面 smoke 和关键流程回归保护旧功能迁移。

## 8. 不做什么

- 不在原型确认前落 React 代码。
- 不继续 Vue 渐进重构作为目标路线。
- 不把 shadcn/ui 官网 dashboard 皮肤原样搬进产品。
- 不恢复 V2-V7 旧原型方向。
- 不恢复 V9 全页面尝试。
- 不恢复 Dreaming、旧插件管理、通用 OpenClaw CRUD 或旧模型链路诊断矩阵。
- 不把所有配置、日志、详情和操作塞进一个主页。

## 9. 验证标准

原型阶段：

- `docs/prototypes/` 只保留当前有效原型和决策文档。
- 文档中没有互相矛盾的 Vue / React 实现结论。
- V8 是唯一当前视觉基线。
- 深浅主题、响应式、导航折叠、命令面板、Sheet / Drawer 等交互可运行。

实现阶段：

- 新 React 前端能独立启动、构建和测试。
- 只消费现有后端 API，新增聚合 API 需先记录设计原因。
- 关键页面通过真实浏览器 smoke。
- API 合同、类型检查、构建和关键回归测试通过。
