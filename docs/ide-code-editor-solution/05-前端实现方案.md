# 前端实现方案

## 1. 前端目标

前端需要同时支持：

```txt
- 文件管理器在线编辑器
- 独立 IDE 工作台
```

但两者不能互相耦合。正确做法是共享代码编辑内核，不共享页面布局。

```

IDE docking layout：

```

如果项目已有状态管理、请求库、图标库，优先用现有方案。

## 3.1 视觉主题适配层

前端主题实现必须先服从 `apps/web/src/design/theme.css`、`docs/界面设计守则.md`、`DESIGN.md` 和 `14-视觉主题与设计系统适配.md`，再适配 Monaco / xterm / Dockview 的 API。不要在业务组件内散落第三方默认主题或硬编码色值。

建议按“集中读取 CSS variables + 薄 adapter”组织：

```txt
apps/web/src/shared/theme
├─ cssVars.ts                 # 读取当前 light/dark/palette 下的 CSS variables
├─ monacoThemeAdapter.ts      # 注册/更新 Monaco theme
├─ xtermThemeAdapter.ts       # 输出 xterm ITheme / ANSI palette
├─ dockviewThemeAdapter.ts    # 输出 Dockview class/token 映射
└─ semanticTheme.ts           # Diff / Problems / Output / Git / Diagnostics 状态色映射
```

如果现有项目已有 shared theme 工具，应优先扩展现有工具，不新建平行主题系统。

适配规则：

```txt
- Monaco、xterm、Dockview、Diff、Problems、Output 都从 CSS variables 派生颜色。
- React 组件优先使用 CSS class + var(--token)，只有第三方 API 必须静态色值时才由 adapter 转换。
- theme/palette 切换后，adapter 需要重新注册或刷新第三方组件主题。
- 信息密集区使用实色 panel、清晰 border、低阴影；不使用影响可读性的透明玻璃。
- light/dark 验收必须覆盖 File Surface、Mini Explorer、IDE Workbench、Terminal、Problems、Output、Diff。
```

## 4. Monaco 接入原则

### Monaco-first 原则

Monaco 已经提供完整编辑器交互，前端不应重复实现单文件编辑器内部能力。Tracevane 的 React 层只做文件生命周期、Tab/window shell、保存安全、冲突处理和状态栏。搜索/替换、上/下一个、大小写、全词、正则、多光标、折叠、minimap、word wrap、context menu、撤销/重做等能力应优先交给 Monaco actions/options/widgets。详细策略见 `10-monaco-first-online-editor-strategy.md`。


### 不要把 Monaco 当 textarea

错误做法：

```txt
- onChange 后每次 setState 保存完整文本。
- Tab 切换时销毁并重建文本。
- 每个组件自己创建 model。
```

正确做法：

```txt
- 一个文件一个 Monaco model。
- 文件内容保存在 model 中。
- React state 只保存 path、dirty、activeTab、viewState。
- 保存时从 model.getValue() 获取文本。
```

### Model URI

建议：

```ts
function createModelUri(workspaceId: string, path: string) {
  return `file:///workspace/${workspaceId}${path.startsWith("/") ? path : `/${path}`}`;
}
```

示例：

```txt
file:///workspace/demo/src/App.tsx
```

### Language 识别

```ts
type LanguageResolver = (path: string, mime?: string) => string;
```

常见映射：

```txt
.js  -> javascript
.jsx -> javascript
.ts  -> typescript
.tsx -> typescript
.json -> json
.css -> css
.html -> html
.md -> markdown
.py -> python
.sh -> shell
.yaml -> yaml
```

## 5. EditorService 伪代码

```ts
async function openFile(path: string, options?: OpenFileOptions) {
  const existing = editorStore.findTabByPath(path);
  if (existing) {
    editorStore.activateTab(existing.id);
    return existing;
  }

  const file = await fileService.readFile({
    workspaceId: currentWorkspaceId,
    path,
  });

  const model = monacoModelService.getOrCreateModel(file);
  const tab = editorStore.addTab({
    path,
    title: getFileName(path),
    language: resolveLanguage(path),
    dirty: false,
    readonly: file.readonly,
  });

  monacoModelService.bindDirtyListener(model, tab.id);
  editorStore.activateTab(tab.id);

  return tab;
}
```

保存：

```ts
async function saveFile(tabId: string) {
  const tab = editorStore.getTab(tabId);
  if (!tab || tab.readonly) return;

  const model = monacoModelService.getModelByPath(tab.path);
  if (!model) return;

  const content = model.getValue();

  await fileService.writeFile({
    workspaceId: tab.workspaceId,
    path: tab.path,
    content,
    expectedVersion: tab.lastSavedVersion,
  });

  editorStore.markClean(tabId);
}
```

## 6. Tab 行为

### 文件管理器编辑器 Tab

M1 基础版已支持打开、激活、关闭和 dirty 标记。M1.x 继续补齐文件管理器编辑器自身的 Tab 体验：

```txt
- 打开
- 激活
- 关闭当前
- 关闭其他
- 关闭全部
- 关闭已保存
- dirty 标记
- dirty-aware 关闭确认
- 超过少量标签后自适应缩小
- 达到最小宽度后横向滚动
- 不因容量限制静默丢弃 dirty Tab
- 切换 Tab 时恢复光标、选区、滚动 viewState
```

文件管理器编辑器仍不支持跨编辑组拖拽、split right/down 或 Dockview 布局；这些属于独立 IDE。

### 独立 IDE Tab

M4 使用独立的 EditorGroup / EditorTab 生命周期，不复用文件管理器在线编辑器 TabBar shell。

```ts
type IdeOpenTarget =
  | "current-group"
  | "side-group"
  | "below-group"
  | "preview-current-group";
```

必做：

```txt
- 同一 group 内同一 rootId + path 不重复打开，只激活已有 tab。
- Explorer 单击打开 preview tab，并复用当前 group 的 preview tab。
- Explorer 双击打开 pinned tab。
- 跨编辑组拖动
- 拖动排序
- preview tab
- pinned tab
- split right
- split down
- dirty tab 自动 pinned，关闭 dirty tab 必须确认。
- rename/move/delete 已打开文件时按共享 EditorService 规则同步或保护 dirty 内容。
- close to right
- close saved
```

文件管理器在线编辑器和独立 IDE 不共用完整 Tab UI；可以共享文件 identity、Monaco model、dirty/save/close-confirm 逻辑和底层预览组件。

## 7. 快捷键设计

第一批快捷键：

```txt
Ctrl+S             保存当前文件
Ctrl+Shift+S       保存全部
Ctrl+F             当前文件搜索（Monaco 原生 Find widget）
Ctrl+H             当前文件替换（Monaco 原生 Replace widget）
Ctrl+G             跳转行（可由 Tracevane 薄入口或 Monaco action 触发）
Ctrl+W             关闭当前 Tab
Ctrl+P             快速打开文件，IDE 后续实现
Ctrl+`             切换终端，IDE 后续实现
Ctrl+\             拆分编辑器，IDE 后续实现
```

原则：

```txt
- 快捷键触发 CommandService。
- 不要在多个组件里重复绑定同一个快捷键。
- 输入框、Monaco、终端的快捷键焦点需要区分。
- 右键菜单、工具栏按钮、Tab 菜单和 Command Palette 与快捷键共用 commandId。
- UI 组件不直接处理权限、dirty 确认、保存冲突或路径同步；这些逻辑在 command handler / service 层集中处理。
```

M3/M4 轻量命令边界：

```txt
- file.open / file.save / file.saveAll / file.close
- explorer.rename / explorer.delete / explorer.copy / explorer.move / explorer.refresh
- editor.splitRight / editor.splitDown
- workbench.toggleSidebar / workbench.togglePanel / workbench.resetLayout
- terminal.new / terminal.kill 先可注册为 disabled 或 placeholder，真实执行在 M5 接入
```

## 8. 文件树 / Explorer 联动

文件树能力分三层：

```txt
File Manager 主页面
- 完整文件管理：list/grid、上传、下载、收藏、搜索、批量操作。

Online Editor Mini Explorer
- 轻量文件导航：当前目录/邻近目录树、上级、刷新、新建、重命名、删除、复制、移动。
- 点击文件打开到同一个 File Surface tab。
- 小屏默认抽屉，不挤压 Monaco。

IDE Explorer
- Workbench SideBar：ActivityBar、Dockview、多编辑组、reveal active file、Git/Problems 徽标等。
```

三者应共享 `explorer-core` 的数据、命令和状态模型，并尽量共享 `explorer-ui` 基础树节点、加载/空/错误状态、基础菜单项等 primitives；不共享完整产品容器 UI。

```ts
type ExplorerOpenFile = (input: {
  rootId: string;
  path: string;
  target: "online-editor" | "ide-current-group" | "ide-side-group";
}) => void;
```

重要边界：Explorer 只发出打开/文件操作意图，不直接创建 Monaco model；真实打开仍交给 EditorService/File Surface。不要把 File Manager、Mini Explorer、IDE Explorer 做成一个靠大量 `mode`/`enableX` 参数切换的大组件。

文件操作同步规则：

```txt
- FileManager、Online Editor Mini Explorer、IDE Explorer 复用同一 FileService/fileOperations。
- rename / move 成功后触发 EditorService.onPathRenamed，所有命中的 Tab 更新 path/title，dirty 内容保留。
- delete 成功后触发 EditorService.onPathDeleted；clean Tab 可关闭或标记 deleted，dirty Tab 必须保留内容并提示另存为/重新创建/放弃。
- 目录 move/delete 对所有命中子路径的打开 Tab 批量处理。
- M3/M4 先保证本应用内操作同步；外部 watcher 事件后续复用同一套 onPathRenamed/onPathDeleted 规则。
```

保存冲突与 watcher 规则：

```txt
- changed + clean：提示 reload 或自动 reload，并更新 lastSaved token。
- changed + dirty：进入 compare / overwrite / reload / cancel，不能静默覆盖。
- deleted：复用 onPathDeleted；dirty Tab 保留内容并标记 deleted。
- renamed：只有 watcher 提供 oldPath/newPath 才复用 onPathRenamed；delete+create 不猜测 rename。
- save：必须携带 expected version/mtime/hash 或当前接口支持的 expectedModifiedAt/expectedSize；409 进入同一冲突流程。
```

## 9. 布局实现

### 文件管理器在线编辑器

可用普通布局，不用 Dockview：

```txt
Header
Body
├─ Optional Mini Explorer（桌面侧栏 / 小屏抽屉）
└─ EditorArea
   ├─ Tabs
   ├─ FileSurfacePanel（Monaco 或媒体预览）
   └─ StatusBar
```

Mini Explorer 的打开状态属于在线编辑器壳层偏好，不应复用或污染 IDE SideBar layout state。

### 独立 IDE

使用 Dockview：

```txt
ActivityBar + SideBar 可以用常规布局。
Editor Area 和 Panel Area 用 Dockview 或 Dockview + SplitView。
```

状态：

```ts
type PanelId =
  | "terminal"
  | "problems"
  | "output"
  | "debug-console";

type PanelPlacement = "bottom" | "right";

type WorkbenchPanelState = {
  placement: PanelPlacement; // M4 only implements bottom; right is reserved.
  visible: boolean;
  collapsed: boolean;
  size: number;
  maximized: boolean;
  activePanelId: PanelId;
};

type IdeLayoutState = {
  layoutVersion: number;
  sideBar: {
    placement: "left" | "right"; // M4 only implements left; right is reserved.
    visible: boolean;
    collapsed: boolean;
    width: number;
  };
  secondarySideBar?: {
    visible: boolean;
    collapsed: boolean;
    width: number;
  };
  panel: WorkbenchPanelState;
  editor: {
    dockviewLayout: unknown;
    activeGroupId?: string;
    activeTabId?: string;
  };
  viewPlacements: WorkbenchViewPlacement[];
  activeActivity: ActivityId;
};

type LayoutMigration = {
  from: number;
  to: number;
  migrate: (state: unknown) => unknown;
};
```

布局持久化规则：

```txt
- M4 持久化 SideBar、Panel、Editor dockviewLayout、active group/tab、viewPlacements、Explorer 展开/选中状态。
- M4 不持久化真实 terminal session、terminal scrollback、LSP/Git/Debug runtime 状态。
- user 级保存主题、字体、快捷键、通用布局偏好。
- workspace 级保存打开文件、编辑组、Explorer、Panel、viewPlacements。
- session 级只保存当前 focus、hover、临时 selection 等可丢弃状态。
- 恢复失败必须 fallback 默认布局；workbench.resetLayout 只重置布局，不丢弃 dirty 编辑内容。
- 每次 layoutVersion 变化必须补 migration，不能直接丢旧状态。
```

M4 可以先只实现默认布局：

```txt
- SideBar placement = left
- Panel placement = bottom
- Editor Area 支持 split right/down
- Panel 支持显示/隐藏、折叠/展开、调高、最大化/恢复
- Panel 固定 Tab：Terminal / Problems / Output / Debug Console
```

M4 必做交互：

```txt
- Explorer 显示/隐藏、折叠/展开、调宽。
- Explorer 打开文件到当前编辑组。
- Editor 多 Tab、多编辑组、Split Right、Split Down、Tab 跨组拖拽。
- Panel 显示/隐藏、折叠/展开、调高、最大化/恢复。
- Terminal / Problems / Output / Debug Console tab 切换。
- Panel 状态保存 activePanelId、visible、collapsed、size、maximized；placement 只使用 bottom 但保留 right。
- Terminal / Problems / Output / Debug Console 在 M4 只展示空状态或占位，不暗示真实运行能力。
- 布局保存、恢复、重置。
```

M4 不做但状态预留：

```txt
- SideBar placement = right。
- Panel placement = right。
- Terminal 真实 xterm/PTY 接入。
- Problems diagnostics、Output channel、Debug Console 数据接入。
- SecondarySideBar 完整交互。
- Workbench View 在 PrimarySideBar / SecondarySideBar / Panel 之间移动。
- Workbench View 排序、显隐和重置位置。
- Terminal split/group。
- View 任意 docking。
- 真实 PTY 终端。
```

但状态模型和组件边界必须为最终自由布局预留，不能把 Explorer、Panel 或 Terminal 写死为不可移动的固定 DOM 结构。

长期目标：

```txt
- Explorer 可移动到右侧。
- Panel / Terminal 可在 bottom/right 间切换。
- Terminal 支持 split pane 和 terminal group。
- Problems / Output / Debug Console 等 View 可参与 dockable view 管理。
- Secondary SideBar 可启用。
```

Workbench View 模型：

```ts
type WorkbenchViewId =
  | "explorer"
  | "search"
  | "source-control"
  | "run"
  | "extensions"
  | "outline"
  | "terminal"
  | "problems"
  | "output"
  | "debug-console";

type WorkbenchViewContainerId =
  | "primary-sidebar"
  | "secondary-sidebar"
  | "panel";

type WorkbenchViewPlacement = {
  viewId: WorkbenchViewId;
  containerId: WorkbenchViewContainerId;
  order: number;
  visible: boolean;
  collapsed?: boolean;
};
```

M4 可以只创建默认 placements：

```txt
- explorer -> primary-sidebar
- terminal / problems / output / debug-console -> panel
```

但不要把这些 View 的位置写死到组件内部；后续 SecondarySideBar、View 移动、排序、隐藏和 Reset View Locations 都应基于同一模型扩展。

持久化：

```txt
- localStorage 可用于个人偏好。
- 服务端可用于 workspace layout。
- 布局 state 需要 version。
- 版本不兼容时 reset。
```

## 10. 终端前端

M5 只做 Real Terminal Foundation：

```txt
TerminalView
├─ TerminalTabs
├─ XtermHost
└─ TerminalToolbar
```

关键点：

```txt
- xterm 初始化后 attach WebSocket。
- 面板 resize 后调用 fitAddon.fit()。
- fit 后把 cols/rows 发给后端。
- 关闭 terminal 时通知后端 kill session。
- 组件卸载时 dispose xterm 实例。
```

终端布局演进：

```txt
M5：Real Terminal Foundation
- TerminalTabs + Active XtermHost。
- sessions + activeSessionId。
- create / input / output / resize / close / kill。
- cwd guard、shell whitelist、canOpenTerminal。
- disconnected / exited / error 状态。
- 不做 split terminal pane，不做 terminal group，不做 Panel right placement。

M5.x：Terminal Split / Group / Panel Placement
- TerminalGroup / TerminalPane。
- split right / split down。
- pane resize / focus / close / move between groups。
- terminal layout persistence。
- Terminal Panel bottom/right placement。

M6+
- Terminal View 参与更完整的 Workbench dockable view 管理。
- 可评估 Terminal 作为 editor-like tab 打开。
```

终端状态模型要从 M5 开始预留演进路径：

```ts
type TerminalSessionStatus =
  | "connecting"
  | "running"
  | "exited"
  | "error"
  | "disconnected";

type TerminalSession = {
  id: string;
  workspaceId: string;
  cwd: string;
  shell: string;
  name: string;
  status: TerminalSessionStatus;
  exitCode?: number | null;
  createdAt: string;
  updatedAt: string;

```

## 11. 性能要求

需要避免：

```txt
- 大文件内容进 React state。
- 每次输入触发整页 rerender。
- 文件树一次性加载所有目录。
- Monaco model 泄漏。
- 关闭 Tab 后 listener 未释放。
```

建议：

```txt
- 文件树懒加载。
- Monaco model 引用计数。
- 大文件阈值保护。
- Tab 和文件树组件 memo。
- watch event 合并去抖。
```

## 12. 前端验收标准

```txt
- 文件管理器可以打开在线编辑器。
- 独立 IDE 有独立路由。
- 两者共享文件读写和保存逻辑。
- 文件内容不进入全局 React state。
- 多 Tab dirty 状态准确。
- 保存失败不会错误清除 dirty。
- 重命名、删除文件能同步打开中的 Tab。
- IDE 布局可以拖拽、保存、恢复、重置。
- 终端 UI resize 后不出现尺寸错乱。
```
