# 远程代码工作台产品聚焦与长期执行机制

## 1. 文档目的

本文把 Tracevane 后续长期推进从“继续补 IDE 能力”改为“完成远程项目在线改代码主链路”。

它回答五个问题：

1. 当前项目完善到什么程度。
2. 哪些能力已经具备，哪些仍未完成。
3. 后续长期 goal 应该如何写。
4. 当用户不懂技术时，AI 应如何替用户做产品决策。
5. 如何降低开发、测试、Code Review 和 AI 执行成本。

核心结论：

> Tracevane 不应继续追求浏览器版 VS Code 全面平替，而应收敛为“远程服务器 / 面板场景下最好用的安全在线代码工作台”。

---

## 2. 当前项目完善程度

Tracevane 已经不是概念稿，也不是早期 demo。它已经完成在线编辑器、独立 Workbench、Monaco、文件预览、终端、Git、LSP、Problems/Output、多语言 provider 和 toolchain guard 的大部分工程基础。

当前阶段判断：

| 维度 | 完成度判断 | 后续方向 |
| --- | --- | --- |
| 文件管理器在线编辑器 | 较高 | 继续稳定文件打开、保存、冲突和预览 |
| Monaco 编辑能力 | 较高 | 继续复用 Monaco 原生能力，不重复造轮子 |
| File Surface / 预览 | 中高 | 优先保证图片/视频/PDF/Hex 等预览稳定、布局不挤压 |
| Mini Explorer / Explorer Core | 中高 | 继续复用共享 core，不做巨型 mode 组件 |
| 独立 IDE Workbench | 中高 | 重点打磨项目级主链路，不继续盲目扩布局花活 |
| Terminal | 中等 | 保持可用、稳定、路径插入、复制粘贴；复杂玩法后置 |
| Git | 中等偏高 | 保持 status/diff/stage/commit 最小闭环；危险操作后置 |
| LSP / 语言智能 | 中高 | 主流 Web 栈优先，长尾/重型 toolchain 降级观察 |
| Debug | 基础可用 | 不作为近期主线，不追完整 VS Code parity |
| 测试/验收体系 | 强但重 | 改为分层验证，避免每轮全量耗时 |
| 产品聚焦 | 仍偏弱 | 明确“远程改代码主链路”优先 |

一句话：

> 工程能力已经很多，现在最重要的是让“远程项目里打开、定位、修改、保存、提交”这条链路非常稳定、顺手、可信。

## 4. 当前未完成的关键问题

### 4.1 产品主线未完全收敛

目前工程能力强，但产品落点仍容易扩散。必须明确：

> Tracevane 的近期产品目标不是浏览器 VS Code，而是远程服务器/面板场景下的安全在线代码工作台。

### 4.2 远程改代码主链路仍需打磨

主链路定义：

```txt
打开项目
→ 浏览文件
→ 快速定位文件/符号/报错
→ 编辑代码
→ 安全保存
→ 处理外部修改/冲突/只读/大文件
→ 查看 diagnostics/hover/definition/symbols
→ Git diff/stage/commit
→ 必要时用终端执行简单命令
```

这个链路里任何一个高频环节不顺，都会比增加一个长尾语言 provider 更影响产品价值。

### 4.3 主流 Web 栈体验应优先于重型 toolchain

近期优先级：

1. TS / JS。
2. JSON。
3. HTML / CSS。
4. ESLint。
5. 文件搜索、符号跳转、Problems、Output。

后置：

- Rust 更深能力。
- Go 更深能力。
- Java / clangd rich interactions。
- Debug parity。

### 4.4 测试体系质量高但成本偏重

当前 typecheck、system tests、smoke、docs link check、RC matrix 都很完整，但每轮都跑全量会拖慢 AI 开发效率。应改成分层验证。

### 4.5 文档和阶段历史过重

M1-M13 记录非常完整，但 AI 每次都读全历史会浪费上下文。需要把文档分成：

- 当前主线文档。
- 历史 archive。
- parking-lot / 长期观察。
- 当前阶段执行入口。

## 5. 后续应该暂停、继续、降级的事项

### 5.1 应暂停

以下事项默认暂停，除非用户明确要求或主链路证明需要：

- 浏览器版 VS Code 全面 parity。
- Git force push / merge / rebase / 复杂 conflict flow。
- 重型 Debug parity。
- Terminal 花活和复杂视图移动。
- 长尾语言继续横向扩张。
- 为了“覆盖率”继续补 provider。

### 5.2 应继续推进

以下事项应作为主线：

- 文件打开、编辑、保存。
- 保存冲突、外部修改、dirty、readonly、大文件保护。
- 文件/文本搜索。
- Quick Open / symbols / go to definition。
- Problems / diagnostics / provider status。
- 主流 Web 栈体验。
- Git 最小闭环：status / diff / stage / commit。
- 安全边界：root guard、allowlist、trust、degraded reason。

### 5.3 应降级为长期观察

以下不删除，但不作为近期默认下一步：

- Go / Rust / Java / clangd 更深 rich interactions。
- watcher-backed symbol index 深化。
- 广义 Command Palette 扩张。
- Debug 高级能力。
- Git 高风险远程操作。
- Terminal 高级布局和 editor-like terminal tabs。

## 6. 长期 goal prompt 推荐版本

> 用于替换或新建长期 goal 时复制。当前已有 active goal 时，不要强行重置；应在下一次新 goal 或明确换目标时使用。

```text
长期推进 Tracevane 成为“远程服务器 / 面板场景下最好用的安全在线代码工作台”。

核心目标不是追求浏览器版 VS Code 全面平替，而是打穿并稳定以下远程改代码主链路：

用户在远程项目中：
1. 打开项目。
2. 浏览文件。
3. 快速定位文件 / 符号 / 报错。
4. 编辑代码。
5. 安全保存。
6. 处理冲突 / 只读 / 大文件 / 外部修改。
7. 查看 diagnostics / hover / definition / workspace symbols。
8. 使用最小 Git 闭环：status / diff / stage / commit。
9. 必要时使用终端完成简单命令。
10. 明确知道 provider / LSP / Git / Terminal 当前是否可用、为何降级。

执行原则：
- 优先稳定远程改代码主链路。
- 优先主流 Web 栈：TS / JS / JSON / HTML / CSS / ESLint。
- 暂停追求 IDE parity。
- 暂停危险 Git 深水区：force push / merge / rebase / 复杂冲突流。
- 暂停重型 Debug parity。
- 暂停长尾语言继续横向扩张。
- Go / Rust / Java / clangd 等 toolchain rich interaction 降级为长期观察项，除非它们直接服务当前主链路。
- 不新增第二套 Files / LSP / Git / Terminal API。
- 不把 File Manager Online Editor 和 IDE Workbench 合并成一个巨型组件。
- 不用一个巨型 `<Explorer mode="mini|ide|file-manager" />` 取代共享 core + 独立产品壳。
- Monaco 负责文本模型与编辑行为；React 负责 shell、metadata、commands 和生命周期。
- xterm 只负责终端 UI；命令执行必须在后端 PTY/WebSocket 且受 workspace/root/runtime guard 约束。
- Dockview 只负责布局，不拥有文件 IO、保存语义或 Monaco model 生命周期。

每轮工作方式：
1. 先判断该任务是否增强“远程改代码主链路”。
2. 如果只是 IDE parity、长尾语言或高风险花活，默认后置。
3. 每次只做一个小阶段。
4. 优先修复主链路阻塞和高频体验问题。
5. 完成后更新 `.codex/project-context.md`、`docs/ide-code-editor-solution/00-README.md`、相关阶段文档和 archive summary。
6. 使用 Lore protocol git commit。
7. 验证采用分层策略：小改跑小测试，核心链路跑 smoke，阶段收口才跑完整矩阵。

当前阶段指针规则：
- 新建长期 goal 时，objective 保持产品主线级别，不把某个短阶段写死成唯一目标。
- 每完成一个阶段，只更新仓库中的 `Current next stage` 与归档 summary；不要为每个阶段重建 runtime goal。
- 如果 runtime goal objective 仍包含旧起点，以 `.codex/ide-long-term-goal.md` 和 `.codex/project-context.md` 的当前指针为准。

当前下一阶段示例：
P1-A-5 Persistence / terminal clipboard checklist。
P0-A 已完成自动化主链路 baseline；P0-B 已完成真实远程项目路径缺口审计；P1-A 已把缺口拆为可执行 hardening 切片。P1-A-1 已清理界面里残留的工程阶段文案；P1-A-2 已完成 Explorer 真实工作流 smoke；P1-A-3 已完成 Editor edge-files 验收；P1-A-4 已完成 responsive layout 验收。下一步进入 P1-A-5 persistence / terminal clipboard checklist。
```

## 7. AI 面向小白用户的决策规则

用户不需要懂技术。AI 必须替用户做产品判断，不能只顺着功能列表堆能力。

### 7.1 做不做的判断

每个新任务先问：

| 判断问题 | 如果答案是“是” | 默认决策 |
| --- | --- | --- |
| 是否影响打开/编辑/保存？ | 是 | 优先做 |
| 是否影响查找文件/符号/报错？ | 是 | 优先做 |
| 是否影响保存安全/冲突/丢数据？ | 是 | 优先做 |
| 是否影响状态可见性/降级解释？ | 是 | 优先做 |
| 是否只是为了更像 VS Code？ | 是 | 后置 |
| 是否是长尾语言能力？ | 是 | 后置 |
| 是否是危险 Git 操作？ | 是 | 后置 |
| 是否是复杂 Debug parity？ | 是 | 后置 |
| 是否会新增第二套 API/壳层？ | 是 | 拒绝或重构为复用 |

### 7.2 做到什么程度

不要问“能不能做完整”，而问：

> 用户完成当前任务最低需要什么？

例子：

- Git：先 status/diff/stage/commit，不先 rebase/merge。
- LSP：先 TS/JS/JSON/HTML/CSS/ESLint 主链路，不先全语言。
- Terminal：先能稳定运行命令、复制粘贴、插入路径，不先复杂布局。
- Debug：近期后置，不用为了 parity 拖慢主线。

### 7.3 什么时候停止

满足以下条件就停止，不继续顺手加功能：

1. 本阶段目标已满足。
2. 主链路确实更稳定或更顺手。
3. 验证通过。
4. 文档记录清楚。
5. 未把 future-stage 能力拉进来。

### 7.4 什么叫完成

代码写完不等于完成。完成必须同时满足：

- 用户路径能跑通。
- 不白屏。
- 不误保存。
- 不丢内容。
- 状态能看懂。
- 降级有解释。
- smoke 或 system test 能证明。
- 文档记录完成边界和未做事项。
- commit 可追溯。

## 8. 产品优先级漏斗

任何新任务必须经过 5 层漏斗。

### 第 1 层：是否服务远程改代码主链路

不服务，默认不做。

### 第 2 层：是否高频

高频优先：

- 打开文件。
- 搜索文件。
- 搜索文本。
- 查看报错。
- 跳定义。
- 保存。
- Git diff。
- commit。

低频后置：

- Java 深度调试。
- Rust proc macro 配置。
- Git rebase UI。
- Terminal 高级布局。

### 第 3 层：是否降低风险

优先降低：

- 误保存风险。
- dirty 内容丢失风险。
- workspace root escape。
- provider 静默失败。
- 大文件卡死。
- 终端/文件操作误触。

### 第 4 层：是否能小步交付

大任务拆成：

1. research / plan。
2. skeleton。
3. minimal implementation。
4. smoke/system proof。
5. acceptance docs。

### 第 5 层：是否复用已有成熟能力

优先级：

1. Browser / Node / TypeScript 原生能力。
2. Tracevane 现有 service / hook / component / utility。
3. 已安装依赖。
4. 维护良好的小型依赖。
5. 最后才写自定义实现。

## 9. 开发、测试、Code Review 提效机制

### 9.1 为什么现在 AI 开发慢

主要原因：

1. 上下文太大：M1-M13 历史非常长。
2. 功能耦合多：改一个点可能影响 File Manager、IDE、Explorer、Monaco、LSP、Git、Terminal。
3. 测试矩阵重：质量高，但每轮全跑成本大。
4. 目标易漂移：问“下一步做什么”容易自然顺着技术路线扩功能。
5. Code Review 面大：代码、测试、文档、阶段状态一起改。

### 9.2 任务分类

| 类型 | 目标 | 验证 |
| --- | --- | --- |
| P0 主链路修复 | 打开/编辑/保存/Git/LSP 高频 bug | targeted smoke + typecheck |
| P1 主链路增强 | 搜索、跳转、状态可见性 | relevant smoke |
| P2 工程收口 | 文档、测试、清理 | diff check + link check |
| P3 长期探索 | Go/Rust/Java/Debug/高级 Git | docs-only plan 优先 |

### 9.3 分层验证策略

小改：

```bash
git diff --check
npm run typecheck:web
```

API / LSP 改动：

```bash
npm run typecheck:api
npm run test:system:<relevant-test>
```

IDE UI 改动：

```bash
npm run typecheck:web
npm run smoke:ide:<relevant-scenario>
```

文件管理器 / Online Editor 改动：

```bash
npm run typecheck:web
npm run smoke:file-manager:online-editor
```

阶段收口：

```bash
git diff --check
markdown link check
relevant smoke matrix
```

大版本 / RC：

```bash
full RC matrix
```

原则：

> 不要每个小改都跑完整矩阵；只有阶段收口和 release candidate 才跑重验证。

### 9.4 Code Review 三问

每个 PR / commit 只先问三件事：

1. 这个改动是否服务远程改代码主链路？
2. 是否破坏已有边界或引入第二套 API/壳层？
3. 是否有与风险匹配的验证证据？

三问通过，再看代码细节。

### 9.5 文档提效

以后文档分层：

- `00-README.md`：当前状态、主入口、阶段列表。
- `15-远程代码工作台产品聚焦与长期执行机制.md`：产品取舍和长期 goal。
- `07/08`：当前阶段技术路线、验收和风险。
- `archive/*`：历史阶段详情。
- `chat-history.md`：聊天级产品判断记录，可作为非正式来源。

避免每次为了一个小阶段重复改大量历史段落。
