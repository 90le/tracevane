# 实施阶段、验收与风险

## 1. 总体实施策略

当前实施节奏已从最初“先编辑器、再 IDE”细化为：先完成文件管理器在线编辑器和统一 File Surface，再抽共享 Explorer Core，最后进入 IDE Workbench Layout Foundation。

```txt
已完成 M1：文件管理器在线编辑器基础版
已完成 M1.x：窗口/Tab/保存安全/状态栏/入口增强
已完成 Monaco-first cleanup：释放 Monaco 原生能力，删除重复搜索替换 UI
已完成 M2：统一 File Surface，删除旧 FilePreviewPanel 冲突
已完成 M2.x：媒体预览增强
已完成 M3：Online Editor Mini Explorer + Shared Explorer Core

下一步 M4：IDE Workbench Layout Foundation
后续 M5：Real Terminal Foundation
后续 M5.x：Terminal Split / Group / Panel Placement
后续 M6：Watcher / Search / Problems / Output
后续 M7：LSP / Git / Debug
```

原则：在线编辑器 Mini Explorer 是轻量文件导航，不是独立 IDE SideBar；但两者必须共享 Explorer Core，避免树、文件操作和路径逻辑重复开发。

视觉原则：所有阶段都必须服从 Tracevane Aurora token。Monaco、xterm、Dockview、Diff、Problems、Output 通过集中适配层映射 `theme.css`；不得用 VS Code/Terminal/第三方默认色作为产品主题。详细见 `14-视觉主题与设计系统适配.md`。

## 2.1 M1.x 文件管理器在线编辑器增强（已完成）

### 目标

在 M1 已验证基础上继续补齐文件管理器编辑器的日常编辑体验和保存安全能力。

### 范围

```txt
- 窗口最大化 / 还原 / 最小化 / 关闭
- Tab 关闭其他 / 关闭全部 / 关闭已保存
- 多 Tab 自适应缩小与横向滚动
- Tab viewState 保留：光标、选区、滚动位置
- 文件 reload / refresh
- 关闭确认：保存 / 不保存 / 取消
- 外部修改冲突检测
- compare / reload / overwrite
- 保存前 mtime/version/hash 校验
- 状态栏增强：换行符、缩进、编码、大小、权限、mtime
- 搜索替换显式选项
- light / dark / auto 主题选择
- 右键菜单、操作列、顶部按钮等编辑入口
```

### 验收标准

```txt
- 打开超过 8 个文件时，不会静默丢弃任何 dirty Tab。
- Tab 顶栏在标签增多时先缩小，达到最小宽度后横向滚动。
- 最大化、最小化、恢复、关闭均有明确且可访问的控件。
- dirty 文件关闭/刷新时可以保存、不保存或取消。
- 外部修改不会被 dirty Tab 静默覆盖。
- 用户可以比较、重新加载或显式覆盖外部修改。
- 状态栏展示的元数据来自真实后端或 Monaco 模型，不猜测。
- 新入口不会让非文本文件误入编辑器。
```

### 与 M2 的边界

M1.x 仍然只增强文件管理器在线编辑器，不引入 Dockview、多编辑组、终端、Git、LSP、Problems 或 Output。

详细执行计划见 `archive/m1x-execution-plan.md`。

## 3. M2 统一 File Surface 与媒体预览（已完成）

### 目标

消除文件管理器里的多套编辑器/预览器冲突，让所有打开入口进入同一个在线编辑器窗口。

### 范围

```txt
- Ctrl/Cmd+C/V Monaco 与文件管理器快捷键隔离
- Monaco zh-CN 本地化
- 全语言懒加载和高亮抽样验证
- 无扩展名、备份后缀、未知扩展文本文件通过有界内容样本补充语言识别；不为识别大 JSON 解析完整文件
- 双击 / 右键编辑 / 检查文件 / 操作列 / 快捷键统一进入 File Surface
- 文本/代码：Monaco
- 图片/视频/音频/PDF/二进制：同一 File Surface 预览
- 删除旧 FilePreviewPanel 和旧 previewTabs 路由
- 图片滚轮缩放、拖动、旋转、复位
- 视频/音频播放速度与跳转控制
```

### 验收标准

```txt
- 不存在两个不同的文件编辑/预览窗口。
- 文本与代码文件在 Monaco 中打开。
- 媒体/PDF/二进制在同一 File Surface 中预览或下载。
- Monaco 查找、替换、命令面板、快捷键不被文件管理器抢占。
- 无扩展名 JSON、`.json.bak.*`、`.json.pre-update`、`.json.clobbered.*` 等文件在 Monaco 中识别为 JSON；大文件语言判断保持样本有界。
- 旧 FilePreviewPanel 无运行时引用。
```

验证记录见 `archive/m2-progress.md` 和 `12-file-surface-unification-and-monaco-gap-plan.md`。

## 4. M4 IDE Workbench Layout Foundation

M4-A 已完成实现前探查与最小骨架准备，记录见 [`archive/m4-a-workbench-foundation-plan.md`](./archive/m4-a-workbench-foundation-plan.md)。M4-A 只建立 `/ide` 路由、`features/ide-workbench` 目录、ActivityBar / SideBar Explorer / EditorArea / PanelArea / StatusBar 骨架和 layoutVersion 状态模型；真实 Dockview editor split、IDE Explorer 文件操作、Terminal、LSP、Git、Debug、Problems/Output 数据接入仍按 M4-B/M4-C/M5+ 后置。


### 目标

建立独立 IDE 页面和可扩展工作台结构。M4 的验收口径是 **Workbench Layout Foundation**，不是完整 VS Code/IDE 验收。它必须交付默认左 Explorer、中 Editor、底 Panel 的可用骨架，并为最终自由布局预留区域位置、显隐、折叠、尺寸和版本。

### 范围

```txt
- /ide/:workspaceId
- Activity Bar 基础版
- Side Bar Explorer（复用 shared/explorer-core）
- Editor Area
- Dockview 接入
- 多编辑组
- Tab 拖动
- Split Right
- Split Down
- Panel Area 框架
- Status Bar
- 布局保存/恢复
- 布局重置
- SideBar / Panel placement、visible、collapsed、size、version 状态预留
```

### M4 必做 / 预留 / 不做

必做：

```txt
- 默认布局：左 ActivityBar + 左 Explorer + 中 Editor + 底 Panel + StatusBar。
- Explorer 显示/隐藏、折叠/展开、调宽。
- Explorer 加载文件树，并打开文件到当前编辑组。
- Explorer 右键菜单、toolbar 和快捷键入口统一走 CommandService。
- Editor 多 Tab、多编辑组、Split Right、Split Down、Tab 跨组拖拽。
- EditorGroup 尺寸调整、active group / active tab 状态。
- Panel 显示/隐藏、折叠/展开、调高、最大化/恢复。
- Panel tab 切换：Terminal / Problems / Output / Debug Console。
- Panel 状态保存 activePanelId、visible、collapsed、size、maximized；placement schema 预留 bottom/right，M4 只实际使用 bottom。
- Terminal / Problems / Output / Debug Console 只显示空状态或占位；文案不能暗示真实 PTY、diagnostics、日志 channel 或 debug runtime 已接入。
- 布局保存、恢复、重置；恢复失败 fallback 默认布局。
- workbench.toggleSidebar / workbench.togglePanel / workbench.resetLayout 等布局命令统一入口。
```

预留但不实现：

```txt
- SideBar placement = right。
- Panel placement = right。
- Terminal 真实 xterm/PTY、terminal split/group。
- Problems diagnostics、Output channel、Debug Console 数据接入。
- SecondarySideBar。
- Workbench View 在 PrimarySideBar / SecondarySideBar / Panel 之间移动。
- View 排序、显隐和 Reset View Locations。
- Terminal split/group。
- Panel / View 任意 docking。
```

明确不做：

```txt
- 真实 xterm + PTY 终端。
- 完整 Git / LSP / Debug。
- 插件市场。
- 复杂多窗口 popout。
- 多窗口跨浏览器窗口拖拽。
```

### 前端交付

```txt
- IdeWorkbench
- ActivityBar
- SideBar
- ExplorerView
- EditorDock
- EditorGroupView
- PanelArea
- StatusBar
- layoutStore
- viewPlacements model
```

### 后端交付

```txt
- workspace tree
- 文件操作接口复用 M1/M2/M3
- layout 持久化接口可选
```

### 验收标准

M4 验收必须围绕 IDE Workbench Layout Foundation，不要求真实终端、LSP、Git、Debug 或完整 View docking。

```txt
- IDE 是独立入口，不是文件管理器弹窗。
- 左侧资源管理器能打开文件。
- 中间支持多 Tab。
- Tab 能拖拽排序。
- 文件能左右/上下拆分到不同编辑组。
- Explorer 单击文件打开 preview tab，继续单击复用 preview tab。
- Explorer 双击文件 pin tab。
- 同一 group 重复打开同一 rootId + path 只激活已有 tab。
- dirty tab 自动 pinned，关闭 dirty tab 有确认。
- Explorer rename / move 已打开文件时，同一 rootId + path 的所有 EditorGroup Tab 同步 path/title。
- Explorer delete 已打开 dirty 文件时保留内容并标记 deleted，不静默关闭。
- Split Right / Split Down 能创建或复用对应编辑组并打开文件。
- Explorer 可折叠、可调宽。
- Panel 可折叠、可调高、可最大化/恢复。
- Terminal / Problems / Output / Debug Console 可在 Panel 中切换。
- 布局刷新后可恢复。
- 布局可重置。
- viewPlacements 至少保存 explorer 与固定 panel tabs 的默认位置。
- 布局状态包含 layoutVersion；恢复失败时 fallback 到默认布局，不能白屏。
- layoutVersion 变化必须提供 migration。
- workbench.resetLayout 只重置 workspace 布局，不删除主题/字体/快捷键，也不丢 dirty 内容。
- CommandService 命令入口有效：file.open/save/saveAll/close、explorer.rename/delete、editor.splitRight/splitDown、workbench.toggleSidebar/togglePanel/resetLayout。
- 保存冲突或外部修改不会静默覆盖 dirty 内容。
- 当前实现可以不支持 Explorer 移到右侧或 Terminal split，但不得把这些未来能力写死阻断。
- ActivityBar、SideBar、Editor Dock、Dockview drop/resize 状态、Panel、StatusBar 在 light/dark 下均映射 Aurora token。
```

M4 验收不要求：

```txt
- 真实 xterm + PTY 终端。
- Terminal split / terminal group。
- Problems 真实 diagnostics。
- Output 真实日志 channel。
- Debug Console 真实调试运行时。
- LSP。
- Git。
- Debug。
- 右侧 Panel。
- Explorer 移到右侧。
- SecondarySideBar。
- 全 View docking / View Movement。
```

## 6. M6 Watcher / Search / Problems / Output

### 目标

提升编辑器和 IDE 的工程可用性。M6 做 watcher/search/problems/output 的数据与面板基础，不做 LSP/Git/Debug。

### 范围

```txt
- DiffEditor
- 全局搜索
- 文件 watcher
- 外部修改提示
- compare / reload / overwrite / cancel
- Problems 面板数据基础
- Output channel / log 基础
- Markdown 预览可选
- JSON/YAML 格式化可选
```

M6 不做：

```txt
- LSP Gateway。
- 真实 LSP diagnostics 接入。
- Git status/diff/stage/commit。
- Debug Adapter Protocol。
```

M6 的 Problems 可以先展示结构化问题数据，例如搜索/任务/自定义校验结果；真实 LSP diagnostics 到 M7 接入。

### 验收标准

```txt
- 可以查看两个文件或文件版本 diff。
- 全局搜索可以跳转到文件行列。
- 外部修改文件时前端能收到提示。
- changed + clean 可 reload；changed + dirty 进入 compare / overwrite / reload / cancel。
- deleted + dirty 保留内容并标记 deleted。
- renamed 只有 watcher 提供 oldPath/newPath 时才迁移路径；delete+create 不猜测 rename。
- 保存 409 file_write_conflict 进入同一冲突流程，dirty 文件不会被静默覆盖。
- Problems 面板可以展示结构化问题数据，但不要求来自 LSP diagnostics。
- Output 可以展示 channel/log 基础内容。
- Diff、Problems severity、Output channel/log 使用项目语义色 token，在 light/dark 下可扫描。
```

### 范围


## 8. 风险清单与防偏移原则

| 风险 | 表现 | 影响 | 应对 |
|---|---|---|---|
| M3 Online Editor Mini Explorer 膨胀成完整 IDE Explorer | 在线编辑器里出现 ActivityBar、Dockview、终端、Git/LSP 入口 | 在线编辑器变重，边界混乱 | M3 验收写明不做 IDE Workbench；只做轻量导航 + Shared Explorer Core |
| 一个大 Explorer 组件靠 `mode` 参数支撑所有场景 | `mode=mini/ide/file-manager` 后继续堆 enable flags | 组件复杂、难测试、后续 IDE 受限 | 共享 `explorer-core` + `explorer-ui` primitives，产品 shell 分离 |
| M4 IDE Workbench Layout Foundation 被做成完整 IDE | 提前追真实终端、LSP、Git、Debug、全 View docking | M4 无法收敛，基础布局不稳 | M4 验收只认 IDE Workbench Layout Foundation；不做事项写入验收 |
| M5 一上来做 terminal split/docking | split/group/placement 先行，真实 PTY 生命周期不稳 | 终端安全、断线、kill、resize 不可靠 | M5 只做 Real Terminal Foundation；split/group/placement 放 M5.x |
| M6 提前接 LSP/Git/Debug | watcher/search/problems/output 未闭环就接复杂外部协议 | 阶段失焦，冲突和问题面板基础不稳 | M6 只做 Watcher/Search/Problems/Output；LSP/Git/Debug 放 M7 分段 |
| 第三方组件默认主题割裂 | Monaco/xterm/Dockview/Problems/Output 出现 VS Code 默认蓝、纯黑终端或随机色 | 深浅主题不一致，产品像拼装页 | 建立集中 ThemeAdapter，从 `theme.css` CSS variables 派生颜色；每阶段验证 File Surface/IDE/Terminal/Problems/Output 的 light/dark |
| UI 组件绕过 CommandService 互相调用 | Explorer 直接调用 Editor/Panel，快捷键和菜单各写逻辑 | 行为不一致，dirty/权限/冲突散落 | 快捷键、菜单、toolbar、Tab 菜单统一走 CommandService |
| 文件管理器和编辑器状态混乱 | 重命名、移动、删除、保存不同步 | 数据丢失、Tab 错乱 | 统一 CommandService、FileService、EditorService 和 onPathRenamed/onPathDeleted 事件 |
| 文件内容进 React state | 大文件卡顿、输入延迟 | 编辑体验差 | 内容只放 Monaco model |
| 终端安全风险 | 用户可执行任意命令 | 高危 | 容器隔离、cwd guard、shell whitelist、canOpenTerminal |
| 路径逃逸 | 访问 workspace 外文件 | 高危 | 后端 root guard |
| 保存冲突 | 外部修改被覆盖 | 数据丢失 | expected token + 409 conflict + compare/overwrite/reload/cancel；dirty 内容优先保护 |
| LSP 复杂度失控 | 多语言服务同时接入 | 进度失控 | M7.1 先接单语言 diagnostics，再逐步扩展 |
| 布局状态不可迁移 | 版本升级后 layout 崩溃 | 用户无法打开 IDE | layoutVersion + migration + fallback 默认布局 + resetLayout |
| 早期布局写死 | Explorer/Terminal 后续不能移动或拆分 | 需要大规模重构 | M4 即预留 placement/visibility/size/collapsed/version/viewPlacements |
| Monaco model 泄漏 | 打开关闭大量文件后内存升高 | 性能问题 | model 引用计数和 dispose |

防偏移原则：

```txt
- 每个阶段都必须写清楚“不做什么”，并进入验收口径。
- 共享 core，不共享臃肿产品容器；产品 shell 独立适配。
- 用户入口统一走 CommandService，权限、dirty、冲突、路径同步集中处理。
- 布局状态必须有 layoutVersion、migration、fallback 和 resetLayout。
- 任何 rename/move/delete/save/conflict 都优先保护 dirty 内容，不静默丢失。
- 阶段性交付可以降低当期范围，但不能写死最终 IDE 自由布局能力。
- 交互语义可参考 VS Code，视觉主题必须服从 Tracevane Aurora token；第三方默认主题必须经过 adapter。
```

## 9. MVP 定义

如果只做最小可用版本，建议 MVP 是：

```txt
MVP-A：文件管理器在线编辑器
- 双击打开文件
- Monaco 编辑
- 多 Tab
- 保存/全部保存
- dirty 确认
- 搜索替换
- 状态栏

MVP-B：独立 IDE 骨架
- /ide/:workspaceId
- 左侧 Explorer
- 中间 Monaco 多 Tab
- Dockview 编辑区
- 底部 Panel 框架
- 布局保存恢复
```

终端不建议放进第一个 MVP，除非后端 PTY 和安全边界已经准备好。

## 11. 上线检查清单

上线前至少检查：

```txt
- 所有文件路径都走后端校验。
- 未保存文件不会静默丢失。
- 保存冲突不会静默覆盖。
- 大文件不会拖垮页面。
- 关闭 Tab 会释放不需要的 model。
- IDE 布局异常可以重置。
- 终端权限受控。
- Monaco、xterm、Dockview、Diff、Problems、Output 在 light/dark 下都通过 token 适配且可读。
- 删除、重命名、覆盖等危险操作有确认。
- 错误信息能让用户知道下一步怎么处理。
```

## 12. 最终推荐

建议立项时拆成两个 Epic：

```txt
Epic 1：文件管理器在线编辑器
- 目标是快速、可靠、多标签编辑文件。
- 不承担完整 IDE。

Epic 2：独立 IDE 工作台
- 目标是项目级开发工作台。
- 第一版就引入 docking layout。
- 逐步接终端、LSP、Git、任务运行。
```

这样既满足文件管理器的即时编辑需求，也为后续主流 IDE 级体验留出正确架构。最终产品目标是完整 IDE 级自由布局；阶段性交付只是降低当期范围，不降低最终能力边界。
