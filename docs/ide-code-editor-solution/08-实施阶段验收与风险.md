## 8. 风险清单与防偏移原则

| 风险 | 表现 | 影响 | 应对 |
|---|---|---|---|
| M3 Online Editor Mini Explorer 膨胀成完整 IDE Explorer | 在线编辑器里出现 ActivityBar、Dockview、终端、Git/LSP 入口 | 在线编辑器变重，边界混乱 | M3 验收写明不做 IDE Workbench；只做轻量导航 + Shared Explorer Core |
| 一个大 Explorer 组件靠 `mode` 参数支撑所有场景 | `mode=mini/ide/file-manager` 后继续堆 enable flags | 组件复杂、难测试、后续 IDE 受限 | 共享 `explorer-core` + `explorer-ui` primitives，产品 shell 分离 |
| M4 IDE Workbench Layout Foundation 被做成完整 IDE | 提前追真实终端、LSP、Git、Debug、全 View docking | M4 无法收敛，基础布局不稳 | M4 验收只认 IDE Workbench Layout Foundation；不做事项写入验收 |
| M5 一上来做 terminal split/docking | split/group/placement 先行，真实 PTY 生命周期不稳 | 终端安全、断线、kill、resize 不可靠 | M5 只做 Real Terminal Foundation；split/group/placement 放 M5.x |
| M6 提前接 LSP/Git/Debug | watcher/search/problems/output 未闭环就接复杂外部协议 | 阶段失焦，冲突和问题面板基础不稳 | M6 只做 Watcher/Search/Problems/Output；LSP/Git/Debug 放 M7 分段 |
| M6 前没有真实 IDE Editor | Search/Problems/Watcher 没有稳定 Monaco model、dirty、tab、viewState 承载对象 | M6 功能只能挂在 placeholder 上，后续返工大 | M5.y 先完成 IDE Editor Foundation，再进入 M6 |
| 第三方组件默认主题割裂 | Monaco/xterm/Dockview/Problems/Output 出现 VS Code 默认蓝、纯黑终端或随机色 | 深浅主题不一致，产品像拼装页 | 建立集中 ThemeAdapter，从 `theme.css` CSS variables 派生颜色；每阶段验证 File Surface/IDE/Terminal/Problems/Output 的 light/dark |
| UI 组件绕过 CommandService 互相调用 | Explorer 直接调用 Editor/Panel，快捷键和菜单各写逻辑 | 行为不一致，dirty/权限/冲突散落 | 快捷键、菜单、toolbar、Tab 菜单统一走 CommandService |
| 文件管理器和编辑器状态混乱 | 重命名、移动、删除、保存不同步 | 数据丢失、Tab 错乱 | 统一 CommandService、FileService、EditorService 和 onPathRenamed/onPathDeleted 事件 |
| 文件内容进 React state | 大文件卡顿、输入延迟 | 编辑体验差 | 内容只放 Monaco model |
| 终端安全风险 | 用户可执行任意命令 | 高危 | 容器隔离、cwd guard、shell whitelist、canOpenTerminal |
| 路径逃逸 | 访问 workspace 外文件 | 高危 | 后端 root guard |
| 保存冲突 | 外部修改被覆盖 | 数据丢失 | expected token + 409 conflict + compare/overwrite/reload/cancel；dirty 内容优先保护 |
| LSP 复杂度失控 | 多语言服务同时接入 | 进度失控 | M7.1 先接单语言 diagnostics，再逐步扩展 |
| 布局状态不可迁移 | 版本升级后 layout 崩溃 | 用户无法打开 IDE | layoutVersion + migration + fallback 默认布局 + resetLayout |
| 早期布局写死 | Explorer/Terminal 后续不能移动或拆分 | 需要大规模重构 | M4 即预留 placement/visibility/size/collapsed/version/viewPlacements |
| Monaco model 泄漏 | 打开关闭大量文件后内存升高 | 性能问题 | model 引用计数和 dispose |

防偏移原则：

```txt
- 每个阶段都必须写清楚“不做什么”，并进入验收口径。
- 共享 core，不共享臃肿产品容器；产品 shell 独立适配。
- 用户入口统一走 CommandService，权限、dirty、冲突、路径同步集中处理。
- 布局状态必须有 layoutVersion、migration、fallback 和 resetLayout。
- 任何 rename/move/delete/save/conflict 都优先保护 dirty 内容，不静默丢失。
- 阶段性交付可以降低当期范围，但不能写死最终 IDE 自由布局能力。
- 交互语义可参考 VS Code，视觉主题必须服从 Tracevane Aurora token；第三方默认主题必须经过 adapter。
```

## 12. 最终推荐

建议立项时拆成两个 Epic：

```txt
Epic 1：文件管理器在线编辑器
- 目标是快速、可靠、多标签编辑文件。
- 不承担完整 IDE。

Epic 2：独立 IDE 工作台
- 目标是项目级开发工作台。
- 第一版就引入 docking layout。
- 逐步接终端、LSP、Git、任务运行。
```

这样既满足文件管理器的即时编辑需求，也为后续主流 IDE 级体验留出正确架构。最终产品目标是完整 IDE 级自由布局；阶段性交付只是降低当期范围，不降低最终能力边界。
