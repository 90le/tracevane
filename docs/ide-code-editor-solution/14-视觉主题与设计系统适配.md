# 视觉主题与设计系统适配

## 1. 目的

本文件把文件管理器在线编辑器、File Surface、Mini Explorer 和独立 IDE Workbench 的视觉主题约束收敛到同一套 Tracevane Aurora 设计系统，避免实现时因为 Monaco、xterm、Dockview 或 VS Code 参考行为而引入第二套配色。

结论：**IDE 行为可以参考 VS Code / Cursor / JetBrains，但视觉实现必须服从 Tracevane 项目主题。**深色、浅色、auto 跟随项目主题都必须可用；任何阶段验收都不能只验证默认浅色或第三方组件默认主题。

## 2. Source of truth 顺序

视觉与主题决策按以下顺序取准：

1. `apps/web/src/design/theme.css`：唯一 token 源，定义 light/dark 下的颜色、文字、边界、阴影、圆角、字体和语义色。
2. `docs/界面设计守则.md`：跨产品界面宪法，定义 Aurora 气质、信息密集区规则和 IDE / Editor 主题硬约束。
3. `DESIGN.md`：当前仓库的产品设计契约，定义 File Manager、Online Editor、IDE Workbench 的视觉原则和验收预期。
4. 本文件：把 IDE 专项组件映射到 token，并作为 M3+ 视觉验收清单。

如果本文与上游 source of truth 冲突，先更新 `theme.css` / `docs/界面设计守则.md` / `DESIGN.md`，再同步本文。

## 3. 硬约束

- 不允许在业务组件里散落硬编码 VS Code、Terminal、Dockview 默认色或随机 hex 色值。
- 不允许为 Monaco、xterm、Dockview、diff、Problems、Output 另建一套独立主题系统。
- 第三方组件必须使用集中适配层，从 CSS variables 读取 token 后映射成其需要的静态颜色值。
- File Manager、Online Editor、Mini Explorer、IDE Workbench 必须共享 Aurora 视觉语言：柔和画布、实色面板、清晰边界、克制阴影。
- 代码、终端、日志、表格、diff、Problems/Output 等信息密集区优先实色背景和清晰分隔线，不使用会降低可读性的毛玻璃或高透明背景。
- 每个阶段的完成声明必须包含 light/dark 或 auto 主题下的截图、smoke、人工检查或明确验证说明。

## 4. Token 映射表

| 用途 | 首选 token | 说明 |
|---|---|---|
| 应用背景 / workbench 外层 | `--bg`, `--canvas` | 页面与 IDE workbench 的最底层背景 |
| 常驻面板 / sidebar / editor chrome | `--panel`, `--panel-2`, `--panel-3` | SideBar、Panel、Tab bar、Toolbar、File Surface chrome |
| 主文字 / 强文字 | `--ink`, `--ink-strong` | 编辑器 chrome、列表、Tab、状态栏 |
| 弱文字 / 辅助提示 | `--muted`, `--subtle` | placeholder、路径片段、只读说明、空状态 |
| 分隔线 / resize handle | `--line`, `--line-2` | 面板边界、Tab 分隔、Dockview group border |
| 当前项 / focus / active | `--primary`, `--primary-soft`, `--primary-line`, `--ring` | 活动 Tab、选中树节点、drop target、focus ring |
| 成功 / 警告 / 错误 / 信息 | `--green`, `--amber`, `--red`, `--teal` 及 soft 变体 | Problems、diagnostics、Git 状态、保存状态、Diff 标记 |
| 圆角 / 阴影 | `--radius-*`, `--shadow-*` | 按 `docs/界面设计守则.md` 的密度规则使用 |
| 字体 | `--font-sans`, `--font-mono` | UI 用 sans，代码/路径/终端/日志用 mono |

## 5. 适配层要求

### 5.1 MonacoThemeAdapter

Monaco 主题必须由 CSS variables 派生：

- editor background：`--canvas` 或信息密集区专用 `--panel`。
- gutter / line number / minimap：`--panel-2`、`--muted`、`--subtle`。
- current line / selection / find match：`--primary-soft`、`--primary-line` 或可读的 token 派生色。
- diagnostics：错误、警告、信息分别映射项目语义色。
- 不允许直接复制 VS Code Dark+ / Light+ 配色作为业务主题。

### 5.2 XtermThemeAdapter

xterm 只负责终端渲染，主题应由 Aurora token 派生：

- background：`--canvas` 或 `--panel`。
- foreground：`--ink`。
- cursor / selection：`--primary`、`--primary-soft`。
- ANSI 16 色必须在 light/dark 下分别调校对比度，不能只沿用终端默认黑/白/蓝。
- terminal 断线、退出、错误、只读等状态使用语义色 token，而不是内嵌随机红黄绿。

### 5.3 DockviewThemeAdapter

Dockview / workbench layout 的视觉状态必须映射到 Aurora：

- active group border、active tab、drop target 使用 `--primary` / `--primary-line`。
- group background、tab bar、inactive panel 使用 `--panel` / `--panel-2`。
- split handle、resize handle 使用 `--line` / `--line-2`，hover 时可用 `--primary-line`。
- 不保留第三方默认蓝、纯黑、纯白或高饱和灰。

### 5.4 Diff / Problems / Output / Debug Console

- Diff 新增、删除、修改使用语义色 soft background + 清晰边界，文字仍使用 `--ink`，不能靠低对比度彩底承载阅读。
- Problems severity：error/warn/info/hint 映射 `--red` / `--amber` / `--teal` / `--muted`。
- Output channel / Debug Console 使用实色 panel、mono 字体、清晰行高；错误行和警告行只使用语义 token 强调。

## 6. 表面级要求

### File Manager / File Surface / Online Editor

- File Surface chrome、Tab strip、状态栏、搜索/跳转/字体/主题入口使用项目控件和 token。
- Online Editor 的 `auto / light / dark` 选择只改变编辑体验偏好，不创建第二套视觉系统。
- Mini Explorer 必须与文件管理器列表和 IDE Explorer 共享树节点基础视觉，但壳层密度按在线编辑器轻量场景收敛。

### IDE Workbench

- ActivityBar、Primary SideBar、EditorArea、PanelArea、StatusBar 是 IDE 内部布局，不替代全局导航。
- 默认布局可以像 VS Code，但视觉不使用 VS Code 主题。Explorer、Editor、Terminal、Problems、Output 必须看起来属于 Tracevane。
- SideBar/Panel 可折叠、可调宽高、可移动的交互反馈使用 `--primary-line` / `--ring`，不要用第三方默认 drop indicator。

### Terminal / Problems / Output

- Terminal、Problems、Output 在 M4 可以是占位，但占位也必须使用最终 token 和密度规则，不能用临时灰块或第三方默认面板。
- M5 接真实终端时，xterm theme 是验收项；M5.x terminal split/group 的 pane border、focus、resize 同样从 token 派生。

## 7. 阶段集成验收

| 阶段 | 主题/视觉验收 |
|---|---|
| M3 Mini Explorer | Mini Explorer 在 light/dark 下可读；树节点、菜单、文件操作状态复用 Aurora token；小屏抽屉不遮挡 Monaco 基本编辑。 |
| M4 Workbench Layout Foundation | ActivityBar、SideBar、Editor Dock、Panel、StatusBar、Dockview drop/resize 状态均映射 token；light/dark 下布局恢复后仍一致。 |
| M5 Real Terminal Foundation | xterm 背景、前景、cursor、selection、ANSI 色在 light/dark 下可读；退出/断线/错误状态使用语义 token。 |
| M5.x Terminal Split / Placement | terminal pane focus、split border、resize handle、bottom/right Panel placement 在 light/dark 下清晰。 |
| M6 Watcher / Search / Problems / Output | Diff、Problems severity、Output channel/log 使用语义 token；错误/警告在两套主题下可扫描。 |
| M7 LSP / Git / Debug | diagnostics、Git 状态、debug console、breakpoint/active frame 等状态不引入外部 IDE 默认色，统一走 token。 |

## 8. AI / 开发实现防偏移清单

实现前先检查：

```txt
- 是否从 theme.css 读取 token，而不是在组件内写 hex？
- 第三方组件是否有集中 adapter，而不是每个调用点各配一套主题？
- light/dark 是否都验证过 Monaco、xterm、Dockview、Diff、Problems、Output？
- 是否把 VS Code 当成交互参考，而不是视觉主题来源？
- 信息密集区是否实色、清晰、低装饰？
- theme reset / layout reset 是否不会误删用户字体、快捷键、编辑器偏好？
```

禁止示例：

```txt
- 复制 VS Code Dark+ 色表作为 Tracevane IDE 默认主题。
- 在 TerminalView.tsx 内直接写 background: '#000'。
- Dockview 默认主题没改就上线。
- Problems error 用随机红色，diff 用低对比透明绿底。
- 只检查普通管理页主题，不检查在线编辑器和独立 IDE。
```
