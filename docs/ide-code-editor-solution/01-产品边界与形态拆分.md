# 产品边界与形态拆分

## 1. 背景判断

现有系统已经有文件管理器能力，后续需要增加代码编辑器和 IDE 能力。

这里最大的风险不是 Monaco 能不能接入，而是产品边界不清导致最终做成：

- 一个固定三栏页面。
- 文件管理器和 IDE 状态相互污染。
- 弹窗编辑器被塞进终端、Git、LSP、调试等复杂能力。
- 独立 IDE 不能拖拽布局、不能拆分编辑组、不能恢复用户自定义布局。
- 后期想接近 VS Code / Cursor / Visual Studio 类产品时，前期架构推倒重来。

因此方案必须从产品形态上拆开。

## 2. 推荐拆分

```txt
React 系统
├─ 文件管理器
│  └─ 在线文件编辑器
│     ├─ 弹窗/抽屉/全屏弹窗
│     ├─ Monaco
│     ├─ 多 Tab
│     └─ 保存/搜索/替换/跳转行
│
└─ 独立 IDE 工作台
   ├─ Activity Bar
   ├─ Side Bar
   ├─ Editor Area
   ├─ Panel Area
   ├─ Status Bar
   ├─ Terminal
   ├─ Problems
   ├─ Output
   └─ 后续 LSP/Git/Tasks/Debug
```

## 3. 形态一：文件管理器在线编辑器

### 定位

文件管理器在线编辑器是文件管理器的增强能力。

它解决的问题是：

- 用户在文件管理器里快速打开文本文件。
- 临时修改配置、脚本、Markdown、JSON、YAML、代码文件。
- 支持同时打开多个文件。
- 支持保存、全部保存、搜索、替换、跳转行。

它不解决的问题是：

- 完整项目开发。
- 运行命令。
- 调试。
- Git 工作流。
- 多面板自定义布局。
- 插件系统。
- 复杂语言服务。

### 典型用户路径

```txt
1. 用户进入文件管理器。
2. 双击 nginx.conf。
3. 打开在线编辑器弹窗。
4. 修改内容。
5. Ctrl+S 保存。
6. 再打开 app.js，形成第二个 Tab。
7. 全部保存或关闭弹窗。
8. 如有未保存内容，提示确认。
```

### 适合的 UI 容器

可以由项目自己的 UI/UX 决定：

```txt
- Modal
- Drawer
- Fullscreen modal
- 独立路由页面
- 新浏览器标签页
```

但不建议在这个形态里加入复杂 docking layout。

## 4. 形态二：独立 IDE 工作台

### 定位

独立 IDE 是一个项目级开发工作台。

它解决的问题是：

- 用户打开一个 workspace。
- 左侧查看项目文件。
- 中间编辑多个文件。
- 支持多编辑组和 Tab 拖拽。
- 底部运行终端。
- 查看 Problems、Output、Debug Console。
- 后续接入 Git、LSP、运行任务、调试。

### 典型用户路径

```txt
1. 用户进入 /ide/:workspaceId。
2. 左侧资源管理器加载项目目录。
3. 用户打开 src/App.tsx。
4. 再打开 package.json。
5. 将 package.json 拖到右侧形成第二个编辑组。
6. 打开底部终端运行 npm run dev。
7. 查看输出或问题面板。
8. 关闭页面后，下次恢复布局、Tab 和终端面板状态。
```

### 关键能力

独立 IDE 不应该是固定页面，而应该是可变工作台：

```txt
- 左侧面板可折叠
- 底部面板可调整高度
- 编辑组可横向/纵向拆分
- Tab 可拖动排序
- Tab 可拖到其他编辑组
- Panel 可切换 Terminal / Problems / Output
- 布局可保存
- 布局可恢复
- 布局可重置
```

## 5. 两者共享能力

文件管理器在线编辑器、在线编辑器 Mini Explorer 和独立 IDE 应共享底层能力，但不共享完整页面布局。

```txt
共享：
- FileService / 文件 API hooks
- WorkspaceService / rootId + path 上下文
- EditorService / File Surface 打开意图
- MonacoEditorWrapper / Monaco model 管理
- LanguageResolver / FileIconResolver / 文件类型判断
- DirtyState / SaveService / 冲突检测 token
- Explorer Core：目录读取、树节点模型、展开状态、排序、过滤、文件操作命令
- Explorer UI primitives：基础树、树节点、加载/空/错误状态、基础菜单项
- SearchService / DiffService / PermissionGuard 的薄适配

不共享：
- 文件管理器主页面布局
- 在线编辑器弹窗/抽屉布局
- 独立 IDE Workbench 布局系统
- IDE Activity Bar
- IDE Dockview 多编辑组
- 底部终端面板
- IDE 级 Command Palette / Git / Run / Extensions 面板
```

共享层的原则是“数据、命令、状态模型和基础 UI primitives 复用；容器、布局、产品形态隔离”。不要用一个带大量 `mode` / `enableX` 参数的大 Explorer 组件同时承担文件管理器、在线编辑器和独立 IDE。

## 6. 边界建议

### 文件管理器

文件管理器保留文件系统管理能力：

```txt
- 目录树
- 文件列表
- 上传
- 下载
- 新建文件
- 新建目录
- 删除
- 重命名
- 移动
- 复制
- 搜索
- 权限
- 压缩/解压
```

文件管理器只通过事件打开编辑器：

```ts
openFileEditor({ workspaceId, path })
openIde({ workspaceId, initialPath })
```

不要让文件管理器直接操作 Monaco model。

### 文件管理器在线编辑器

文件编辑器只负责快速编辑：

```txt
- 打开文件
- 编辑文件
- 多 Tab
- 保存
- 全部保存
- 搜索替换
- 跳转行
- 修改提示
- 状态栏
```

M1.x 可以继续增强文件管理器在线编辑器，但仍然属于这个边界：

```txt
- 窗口最大化 / 最小化 / 关闭
- Tab 关闭其他 / 关闭全部 / 关闭已保存
- 多 Tab 自适应缩小与横向滚动
- Tab 光标 / 选区 / 滚动 viewState 保留
- 文件 reload
- 外部修改 compare / reload / overwrite
- 状态栏元数据增强
- 搜索替换选项增强
- 主题和字体偏好
- 更多文件管理器编辑入口
```

这些增强不能越界成独立 IDE：不加入 Dockview、多编辑组、终端、Git、LSP、Problems 或 Output。

### 可选 Mini Explorer 边界

在线编辑器可以增加一个轻量 **Mini Explorer**，用于在编辑器窗口内快速切换附近文件。它属于文件管理器在线编辑器的小增强，不等于完整 IDE 资源管理器。

```txt
可以做：
- 当前目录 / 邻近目录浏览
- 树型展开与折叠
- 返回上级、刷新、新建文件/目录
- 重命名、删除、复制、移动等常见文件操作入口
- 点击文件打开到同一个 File Surface tab
- 可折叠、可收起、小屏抽屉化

不做：
- ActivityBar
- Dockview
- 多编辑组
- 终端 / Git / LSP / Problems / Output
- VS Code 级 workspace 管理
```

Mini Explorer 应复用共享 Explorer Core 和可组合 Explorer UI primitives，而不是复制 FileManagerList 或未来 IDE Explorer 的逻辑。详细计划见 `13-mini-explorer-shared-explorer-plan.md`。

### 独立 IDE

独立 IDE 负责项目级工作台：

```txt
- 可拖拽布局
- 多编辑组
- 终端
- 输出
- Problems
- Git
- 任务运行
- 语言服务
- 状态栏
- 命令体系
```

独立 IDE 的最终目标是完整 IDE 级自由布局，而不是固定左中下三栏：

```txt
- Explorer / SideBar：可折叠、调宽、移动到右侧、恢复位置。
- Editor：多编辑组、左右/上下拆分、Tab 跨组拖拽、编辑组最大化/恢复。
- Terminal：多终端 Tab、split pane、terminal group、底部/右侧 placement、布局恢复。
- Problems / Output / Debug Console：Panel 内切换，长期参与 dockable view 管理。
- Primary / Secondary SideBar：长期支持 View 在侧栏、副侧栏、Panel 间移动。
```

Explorer、Search、Source Control、Run、Extensions、Outline、Terminal、Problems、Output、Debug Console 应统一视为 Workbench View。阶段性交付不能写死最终布局。M4 可以只交付默认布局和编辑器 split，但 layout state 必须预留 placement、visible、collapsed、size、version 和 viewPlacements。

IDE EditorArea 必须使用独立的 EditorGroup / EditorTab 生命周期：Explorer 单击文件进入 preview tab，双击或编辑后 pinned；同一 group 内同一 `rootId + path` 不重复打开；dirty tab 自动 pinned 且关闭必须确认。它与文件管理器在线编辑器共享底层 FileService、Monaco model 和 dirty/save 逻辑，但不共享 FileOnlineEditorDialog 窗口壳或 TabBar 状态。

## 7. 不建议的做法

### 不建议一：把文件管理器弹窗做成完整 IDE

问题：

```txt
- 弹窗复杂度过高。
- 文件管理器职责膨胀。
- 用户只是快速改文件时体验变重。
- 后期终端和 Git 安全边界难处理。
```

### 不建议二：用固定 flex 布局做独立 IDE

问题：

```txt
- 后期无法拆分编辑组。
- Tab 不能跨组拖拽。
- 底部和侧边栏状态很难持久化。
- 布局一复杂就会出现重绘、尺寸、焦点、快捷键问题。
```

### 不建议三：文件内容直接存在 React state

问题：

```txt
- 大文件卡顿。
- 频繁 setState。
- undo/redo 和 Monaco 内部状态不一致。
- 多 Tab 下状态难维护。
```

## 8. 方案口径

可以对内这样描述：

> 本方案将代码编辑能力拆成“文件管理器在线编辑器”和“独立 IDE 工作台”两种入口。前者面向快速文件编辑，提供 Monaco 多标签、保存、搜索替换、跳转行、状态栏和可选 Mini Explorer；后者面向项目级开发，使用 docking layout 构建可拖拽、可拆分、可持久化的 IDE 工作台，逐步集成终端、Problems、Output、Git、LSP 和任务运行。两者共享文件服务、Explorer Core、编辑器模型和保存机制，但保持布局和产品边界独立，避免固定化 IDE 页面导致后续扩展困难。
