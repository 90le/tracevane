# 产品边界与形态拆分

## 1. 背景判断

现有系统已经有文件管理器能力，后续需要增加代码编辑器和 IDE 能力。

这里最大的风险不是 Monaco 能不能接入，而是产品边界不清导致最终做成：

- 一个固定三栏页面。
- 文件管理器和 IDE 状态相互污染。
- 弹窗编辑器被塞进终端、Git、LSP、调试等复杂能力。
- 独立 IDE 不能拖拽布局、不能拆分编辑组、不能恢复用户自定义布局。
- 后期想接近 VS Code / Cursor / Visual Studio 类产品时，前期架构推倒重来。

因此方案必须从产品形态上拆开。

## 2. 推荐拆分

```txt
React 系统
├─ 文件管理器
│  └─ 在线文件编辑器
│     ├─ 弹窗/抽屉/全屏弹窗
│     ├─ Monaco
│     ├─ 多 Tab
│     └─ 保存/搜索/替换/跳转行
│
└─ 独立 IDE 工作台
   ├─ Activity Bar
   ├─ Side Bar
   ├─ Editor Area
   ├─ Panel Area
   ├─ Status Bar
   ├─ Terminal
   ├─ Problems
   ├─ Output
   └─ 后续 LSP/Git/Tasks/Debug
```

## 3. 形态一：文件管理器在线编辑器

### 定位

文件管理器在线编辑器是文件管理器的增强能力。

它解决的问题是：

- 用户在文件管理器里快速打开文本文件。
- 临时修改配置、脚本、Markdown、JSON、YAML、代码文件。
- 支持同时打开多个文件。
- 支持保存、全部保存、搜索、替换、跳转行。

它不解决的问题是：

- 完整项目开发。
- 运行命令。
- 调试。
- Git 工作流。
- 多面板自定义布局。
- 插件系统。
- 复杂语言服务。

### 典型用户路径

```txt
1. 用户进入文件管理器。
2. 双击 nginx.conf。
3. 打开在线编辑器弹窗。
4. 修改内容。
5. Ctrl+S 保存。
6. 再打开 app.js，形成第二个 Tab。
7. 全部保存或关闭弹窗。
8. 如有未保存内容，提示确认。
```

### 适合的 UI 容器

可以由项目自己的 UI/UX 决定：

```txt
- Modal
- Drawer
- Fullscreen modal
- 独立路由页面
- 新浏览器标签页
```

但不建议在这个形态里加入复杂 docking layout。

## 4. 形态二：独立 IDE 工作台

### 定位

独立 IDE 是一个项目级开发工作台。

它解决的问题是：

- 用户打开一个 workspace。
- 左侧查看项目文件。
- 中间编辑多个文件。
- 支持多编辑组和 Tab 拖拽。
- 底部运行终端。
- 查看 Problems、Output、Debug Console。
- 后续接入 Git、LSP、运行任务、调试。

### 典型用户路径

```txt
1. 用户进入 /ide/:workspaceId。
2. 左侧资源管理器加载项目目录。
3. 用户打开 src/App.tsx。
4. 再打开 package.json。
5. 将 package.json 拖到右侧形成第二个编辑组。
6. 打开底部终端运行 npm run dev。
7. 查看输出或问题面板。
8. 关闭页面后，下次恢复布局、Tab 和终端面板状态。
```

### 关键能力

独立 IDE 不应该是固定页面，而应该是可变工作台：

```txt
- 左侧面板可折叠
- 底部面板可调整高度
- 编辑组可横向/纵向拆分
- Tab 可拖动排序
- Tab 可拖到其他编辑组
- Panel 可切换 Terminal / Problems / Output
- 布局可保存
- 布局可恢复
- 布局可重置
```

## 5. 两者共享能力

文件管理器在线编辑器和独立 IDE 可以共享底层能力：

```txt
共享：
- FileService
- WorkspaceService
- EditorService
- MonacoEditorWrapper
- LanguageResolver
- DirtyState
- SaveService
- FileIconResolver
- SearchService
- DiffService
- PermissionGuard

不共享：
- 布局系统
- 页面容器
- 左右侧面板组织方式
- 底部终端面板
- IDE Activity Bar
- IDE Command Palette
```

## 6. 边界建议

### 文件管理器

文件管理器保留文件系统管理能力：

```txt
- 目录树
- 文件列表
- 上传
- 下载
- 新建文件
- 新建目录
- 删除
- 重命名
- 移动
- 复制
- 搜索
- 权限
- 压缩/解压
```

文件管理器只通过事件打开编辑器：

```ts
openFileEditor({ workspaceId, path })
openIde({ workspaceId, initialPath })
```

不要让文件管理器直接操作 Monaco model。

### 文件管理器在线编辑器

文件编辑器只负责快速编辑：

```txt
- 打开文件
- 编辑文件
- 多 Tab
- 保存
- 全部保存
- 搜索替换
- 跳转行
- 修改提示
- 状态栏
```

### 独立 IDE

独立 IDE 负责项目级工作台：

```txt
- 可拖拽布局
- 多编辑组
- 终端
- 输出
- Problems
- Git
- 任务运行
- 语言服务
- 状态栏
- 命令体系
```

## 7. 不建议的做法

### 不建议一：把文件管理器弹窗做成完整 IDE

问题：

```txt
- 弹窗复杂度过高。
- 文件管理器职责膨胀。
- 用户只是快速改文件时体验变重。
- 后期终端和 Git 安全边界难处理。
```

### 不建议二：用固定 flex 布局做独立 IDE

问题：

```txt
- 后期无法拆分编辑组。
- Tab 不能跨组拖拽。
- 底部和侧边栏状态很难持久化。
- 布局一复杂就会出现重绘、尺寸、焦点、快捷键问题。
```

### 不建议三：文件内容直接存在 React state

问题：

```txt
- 大文件卡顿。
- 频繁 setState。
- undo/redo 和 Monaco 内部状态不一致。
- 多 Tab 下状态难维护。
```

## 8. 方案口径

可以对内这样描述：

> 本方案将代码编辑能力拆成“文件管理器在线编辑器”和“独立 IDE 工作台”两种入口。前者面向快速文件编辑，提供 Monaco 多标签、保存、搜索替换、跳转行和状态栏能力；后者面向项目级开发，使用 docking layout 构建可拖拽、可拆分、可持久化的 IDE 工作台，逐步集成终端、Problems、Output、Git、LSP 和任务运行。两者共享文件服务、编辑器模型和保存机制，但保持布局和产品边界独立，避免固定化 IDE 页面导致后续扩展困难。
