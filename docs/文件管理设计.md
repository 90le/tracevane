# 文件管理设计

> 状态：Active design
> 更新：2026-06-25

## 1. 定位

文件管理是 Workspace 的核心能力，不再是一个只读 `/files` 页面。

两种形态：

| 形态 | 用途 |
| --- | --- |
| Explorer Mode | IDE 左侧高频浏览、打开、上下文选择。 |
| File Manager Mode | 完整文件管理：表格/网格、批量操作、路径输入、上传下载、归档。 |

## 2. 与终端的硬边界

文件管理和终端不能混为一谈：

- 文件管理拥有文件生命周期：新建、重命名、复制、移动、删除、上传、下载、压缩、解压、批量选择。
- 终端只拥有进程生命周期：shell、task、dev server、logs、ports、reconnect、replay。
- 文件管理可以把当前目录作为 terminal cwd 打开；这只是 handoff，不是把文件管理交给终端。
- 终端输出可以保存为 evidence；这不是文件管理事实源。
- UI 上文件管理主任务必须在 Files/File Manager 完成，不能用“去终端执行命令”替代。

## 3. 必须能力

- 多 root 切换：root 选择器只是快捷入口，不代表只能访问固定几个目录。
- 可编辑路径栏：面包屑负责快速导航，路径栏允许直接输入文件夹路径并按 Enter 跳转。
- 全路径访问：通过 `system-root` 覆盖 `/`，用户可输入任意绝对路径；后端仍用 root + relative path 约束和真实路径校验防止越界。
- 常用/最近位置：快捷 root、收藏路径、最近成功访问路径应在地址栏附近轻量呈现，不能恢复占空间的左侧树。
- 路径补全：地址栏输入时应基于快捷 root、收藏/最近位置和当前目录已加载子目录给出轻量建议，Tab 可补全首项。
- 路径错误恢复：不存在、无权限或非目录路径必须显示可读错误，并支持返回上个可用目录。
- 树视图。
- 表格视图。
- 文件预览/编辑弹窗：双击文件或右键“预览 / 编辑”打开弹窗工作台；代码/文本支持源码编辑保存；Markdown 支持源码、渲染和边写边预览；HTML 支持源码、沙箱渲染和边写边预览；图片、视频、音频、PDF 提供内联预览；其它文件提供打开/下载入口。默认列表不被右侧常驻预览面板挤占。
- 搜索文件名/内容。
- 新建文件/目录。
- 重命名。
- 复制/移动。
- 删除。
- 批量操作结果面板：复制、移动、删除、打包、上传等完成后保留最近操作记录，显示成功/失败数量、受影响路径和错误摘要，避免只依赖 toast。
- 上传/下载。
- 拖拽上传：文件管理器主区域支持文件、批量文件、文件夹拖入；拖入后进入上传管理器，默认目标为当前目录。
- 批量选择。
- archive/unarchive。
- 大目录分页/虚拟滚动。
- 内容索引管理：查看分片/记录/容量，扫描失效、清理失效、按当前 root 受限重建索引。
- 状态栏：数量、选中、路径、分页、Git 状态。
- 移动端 action sheet。

## 4. UI 规则

PC：

```text
toolbar + path bar
left/tree or table
right optional inspector
status bar
```

Mobile：

```text
top path/search
single list/table cards
selection mode
bottom action sheet
```

不要再出现“本页只读，去 IDE 操作”的说明。

## 5. 与 Evidence 的关系

文件管理负责操作文件；Evidence 负责保存和复盘文件相关证据。

示例：

- 下载文件：文件管理动作。
- 查看某次 Agent 生成的 artifact：Evidence。
- 对 Git diff 做审查：Workspace Git/Editor。
- 查看 IM 附件：Evidence，可 reveal/open in Workspace。

## 6. 安全和确认

| 动作 | 模式 |
| --- | --- |
| 新建/重命名 | inline/dialog，失败 toast。 |
| 覆盖/移动/删除 | confirm dialog/sheet。 |
| 批量删除 | 列出 affected paths。 |
| archive/unarchive | 显示目标目录与文件数量。 |
| upload | sheet + progress + overwrite policy。 |

## 2026-06-25 同标签页预览/编辑工作台

文件管理器打开文件后不再把“编辑”和“预览”视作两个窗口。当前文件详情区域是一个同标签页工作台：

- 代码/文本：使用 Monaco 编辑器承载源码视图，提供语法高亮与编辑器内置查找能力。
- Markdown：支持源码编辑、渲染预览、边写边预览；保存仍写回同一个文件。
- HTML：支持源码编辑、sandbox 预览、边写边预览；完整 WYSIWYG 作为后续可插拔编辑器能力，而不是当前阶段引入重量依赖。
- 图片、视频、音频、PDF：在同一个文件上下文内内联预览；不支持内联编辑的二进制文件保留打开/下载入口。
- 查找/替换：文件管理器提供轻量同页查找、大小写/Regex、替换一个、全部替换；Workspace IDE 继续依赖 Monaco 的 IDE 级查找/替换与快捷键。

设计约束：预览、源码、编辑必须是同一个文件标签页内的视图模式，不能产生“打开两个窗口才完成一个文件工作流”的割裂体验。

## 2026-06-25 文件操作 Diff 预览复用方向

Workspace Search 的跨文件替换 diff 已抽为共享 `ReplaceDiffPreview`。独立文件管理器后续在批量覆盖、移动冲突、解压覆盖、操作历史回滚等高风险动作中，应复用同一套差异预览/审阅组件，而不是只给出文字确认。

## 2026-06-26 操作历史审阅

独立文件管理器的操作结果从临时 toast/最近 5 条升级为会话持久的操作历史：最多保留 50 条，写入浏览器 localStorage；面板支持按成功、部分完成、失败筛选，展开记录查看影响路径、错误详情，并可复制 JSON 记录用于排障或证据留存。该历史仍是前端 evidence，不替代后端审计日志；后续撤销/回滚应在此基础上增加明确能力边界。

## 2026-06-26 操作历史模块边界

操作历史已从 `FileManagerPage.tsx` 抽离到 `file-manager/OperationHistoryPanel.tsx`，该模块拥有：操作记录类型、创建记录、localStorage 持久化、筛选、展开详情和复制 evidence。`FileManagerPage` 只负责在文件动作完成后 push record，不再承载历史面板内部状态。后续撤销/回滚、后端审计或恢复点应优先扩展该模块，而不是把逻辑重新塞回页面主文件。

## 2026-06-26 预览编辑模块边界

文件详情、源码/渲染/边写边预览、代码高亮、同标签页查找替换、Markdown/HTML/图片/音频/视频/PDF 预览已从 `FileManagerPage.tsx` 抽离到 `file-manager/FilePreviewPanel.tsx`。主页面只负责选择文件和提供 read query；预览模块负责编辑状态、保存、渲染和格式判断。后续增强 WYSIWYG、Monaco DiffEditor、更多媒体格式或 Workspace/FileManager 复用时，应优先扩展该模块，而不是把编辑器逻辑重新塞回主页面。

## 2026-06-26 内容索引管理模块边界

内容索引管理已从 `FileManagerPage.tsx` 抽离到 `file-manager/ContentIndexManager.tsx`。该模块拥有内容索引统计、扫描失效、清理失效、重建索引、索引存储路径展示和相关 toast 结果。主页面只负责在“文件列表 / 内容索引”之间切换，不再承载索引管理细节。后续内容搜索、索引 shard 管理、索引健康度和容量策略应扩展该模块。

## 2026-06-26 共享文档预览层

独立文件管理器的文件预览/编辑弹窗复用 Workspace 的 `DocumentPreview` 渲染层，统一 Markdown、HTML、图片、视频、音频、PDF 和文本兜底预览。FileManager 只保留文件详情、下载、保存、查找替换和操作历史等域内职责，不再复制媒体类型判断。代码/文本文件现在也可以作为只读渲染预览显示，同时仍可进入源码/编辑模式。

## 2026-06-26 共享查找替换条

FileManager 的预览/编辑弹窗不再内置私有 `SearchReplaceStrip`、`countTextMatches`、`replaceText`。这些能力统一迁移到 `workspace/shared/TextSearchReplaceStrip.tsx`，作为当前文档级别的查找/替换组件。文件管理器只负责把替换结果写入当前 draft，并由保存按钮提交到后端；跨文件批量替换仍属于 Workspace Search 的确认/撤销流程。

## 2026-06-26 FileManager Chrome 模块边界

文件管理器主页面的标题区、根选择、主操作按钮、面包屑、可编辑路径栏、路径建议、收藏/最近位置、当前目录过滤和隐藏文件开关已抽到 `FileManagerChrome.tsx`。`FileManagerPage.tsx` 继续拥有服务端查询、选择状态、上传/拖拽/粘贴、文件操作、操作历史和内容索引视图编排。Chrome 组件只负责系统文件管理器的导航与顶部操作表现，避免主页面继续膨胀，也让后续路径栏/收藏/快捷位置优化有独立边界。

同时修复收藏位置持久化：新增收藏和取消收藏都会写入 localStorage，避免刷新后新增收藏丢失。

## 2026-06-26 文件列表模块边界

系统级文件管理器的列表与批量选择动作已从 `FileManagerPage.tsx` 抽离到 `file-manager/FileManagerList.tsx`。该模块拥有云面板式默认列表、列头、空态/错误态/骨架屏、分页加载、键盘上下移动、Enter 打开、Escape 清空选择、Shift 范围选择、复选框多选和批量操作条。主页面只保留数据查询、路径导航、选择状态、上传/拖拽/粘贴和文件操作编排。

设计约束：列表是文件管理器的核心内容区，不再和操作历史、预览、路径栏混在一个大组件里。后续网格视图、排序、列宽拖拽、虚拟列表或右键菜单增强，应优先扩展 `FileManagerList`，避免重新膨胀主页面。

## 2026-06-26 复用 Document Workbench

独立文件管理器的预览/编辑弹窗已切换到共享 `DocumentWorkbench`。文件管理器不再私有实现源码编辑、预览模式切换、边写边预览或查找替换，只负责提供当前文件内容、保存动作和文件操作上下文。这样可保证 FileManager 与 Workspace IDE 对“同一个文件标签内编辑/预览/源码”的产品语义一致。

## 2026-06-26 列表排序与可操作表头

文件管理器默认列表视图新增可点击表头排序：名称、大小、修改时间、类型。排序采用自然排序（数字友好、大小写不敏感），并保持目录优先，避免目录与文件混排导致路径浏览效率下降。排序状态由 `FileManagerPage` 持有，`FileManagerList` 只负责展示表头、ARIA sort 状态和排序触发，保持列表组件可复用。

## 2026-06-26 视图偏好持久化

文件管理器列表新增视图偏好持久化：列排序和列表密度写入 `tracevane:file-manager:view-preferences`。用户选择“舒适/紧凑”或点击列头改变排序后，刷新页面仍保留偏好。偏好只影响当前浏览器显示，不改变后端文件事实源；超大目录后续可扩展为服务端排序参数，但本阶段先保持前端轻量可逆。

## 2026-06-26 可配置列表列

文件管理器列表新增“列”配置入口，允许用户显示/隐藏大小、修改时间、类型列；名称列和操作列始终保留。列配置与排序、密度一起持久化到 `tracevane:file-manager:view-preferences`。当前阶段先支持显隐和稳定顺序，后续可以继续扩展列宽拖拽、列顺序拖拽和更多元数据列。

## 2026-06-26 行级操作与键盘上下文菜单

文件管理器列表新增每行操作按钮，并支持键盘打开上下文菜单：`ContextMenu` 键或 `Shift+F10` 会在当前行附近打开同一套 `FileActionsMenu`。鼠标右键、行内“…”按钮和键盘入口共享同一个操作菜单，避免出现三套不同文件操作入口。该能力服务于系统文件管理器的可达性和高频文件操作效率。

## 2026-06-26 文档工作台与 IDE 级查找升级

Workspace IDE 与独立文件管理器统一采用 `DocumentWorkbench` 作为“一个文件 = 一个标签页”的文档工作台。源码、编辑、渲染预览、边写边预览、当前文件查找/替换都只是当前标签页内的视图状态，不允许再拆成独立 Preview 标签或第二个窗口。

本轮增强：

- Monaco 编辑器继续负责代码高亮、语言识别和内置 Ctrl/⌘+F / Ctrl/⌘+H 查找替换。
- Tracevane 的同标签页查找条新增上一个/下一个、当前匹配序号、搜索结果同步到 Monaco decorations，高亮所有匹配并定位当前匹配。
- “替换一个”按当前激活匹配替换，不再总是替换第一处；“全部替换”仍保持批量替换。
- Markdown / HTML 仍保持源码、预览、边写边预览三模式；完整富文本/WYSIWYG 编辑器以后以插件化方式接入，避免当前阶段引入重型依赖。

产品约束：全局搜索、跨文件批量替换继续属于 Workspace Search 域；当前文件查找/替换属于 DocumentWorkbench。两者共享“预览差异、确认、可回滚”的交互方向，但不要把文件管理器详情面板膨胀成全局搜索页。

## 2026-06-26 复制/移动同名冲突策略

文件管理器复制/移动弹窗新增同名冲突策略：阻止、覆盖、跳过、保留两者。默认保持阻止，避免危险覆盖；用户明确选择覆盖时才向后端传 `overwrite`；选择跳过时前端记录跳过结果；选择保留两者时前端基于目标目录列表生成 `name (1).ext` 形式的新名称并传 `nextName`。

当前实现先做目标目录前端预检，pageSize 为 5000，满足常规云面板/系统文件管理器使用；超大目录的最终权威仍在后端文件操作 API，后续应把同名冲突预检下沉为服务端批量 dry-run，避免极大目录分页遗漏。

## 2026-06-26 内容索引参与 Workspace Search

内容索引库不再只服务上传免传复用和独立文件管理器统计。Workspace Search 的后端搜索现在会先读取当前 root 的内容索引分片，验证记录仍指向同一文件 size/mtime 后，优先在这些已知文件中查找文件名和内容命中；随后再用递归扫描补全目录名、未索引文件和索引未覆盖结果。

约束：索引是性能缓存，不是文件事实源。每条索引命中都必须重新 stat 校验并按当前搜索目录、递归、隐藏文件开关过滤；未命中索引时必须继续扫描补全，不能因为索引不完整而漏掉搜索结果。前端 Search 面板显示“内容索引命中/索引未命中扫描补全”作为性能可见性，不把它当作用户必须理解的数据库概念。

## 2026-06-26 复制/移动服务端预检

复制/移动在执行前新增服务端 dry-run：前端弹窗会把源路径、目标目录、冲突策略发送到 `/api/files/transfer/dry-run`，后端基于真实 root/path 解析和目标目录当前内容生成操作计划。预检结果显示就绪、覆盖、跳过、保留两者、阻塞冲突和错误数量，并列出风险项。若策略为阻止且存在同名冲突，执行按钮会被禁用，避免用户提交后才在部分项目失败。

该预检不是替代最终执行校验：执行时后端 copy/move 仍按真实文件系统状态处理。预检用于让 Web 文件管理器具备系统文件管理器/云面板级的可审阅操作体验，减少“点了才知道失败”的低级交互。

## 2026-06-26 解压服务端预检

解压操作新增 `/api/files/unarchive/dry-run`。右键菜单的“解包到…”弹窗在用户输入目标目录或切换冲突策略时，会读取归档目录并生成解压计划：就绪、覆盖、跳过、重命名、阻塞冲突和不安全/不支持条目。存在阻塞冲突或不安全条目时禁用执行按钮，避免归档解压到一半才发现目标同名或路径逃逸。

安全边界：dry-run 和实际解压都必须检查归档条目不能逃逸目标目录；tar 里的 symlink、hardlink、device 等条目继续视为不支持。dry-run 只是预检，不是事务锁；执行时仍由后端按真实文件系统状态和冲突策略处理。

## 2026-06-26 独立文件管理器页面级解压

独立 FileManager 不再只依赖行右键菜单触发解压。列表选择恰好一个后端支持的归档文件（`.zip`、`.tar`、`.tar.gz`、`.tgz`、`.tar.bz2`、`.tbz2`、`.tar.xz`、`.txz` 等）时，批量操作条显示“解压”。弹窗默认目标目录为当前目录，允许用户改目标相对路径，并提供与后端一致的冲突策略：阻止、覆盖、跳过、保留两者。

执行前通过 `/api/files/unarchive/dry-run` 做服务端预检，显示就绪、覆盖、跳过、重命名、冲突和错误条目；存在阻塞冲突或不安全条目时禁用执行。执行成功后写入 FileManager 操作历史，并刷新文件查询。该入口复用 Workspace 的文件操作 hook，但 UI 生命周期和操作历史归 FileManager 域所有。

## 2026-06-26 操作历史检索与定位

FileManager 操作历史从“最近结果列表”继续增强为可审阅工作台：顶部显示总数/成功/部分/失败统计；支持按状态和关键字搜索操作标题、影响路径、错误信息；复制按钮复制当前筛选结果；展开详情后每条影响路径可以单独复制，也可以“定位”到文件管理器当前目录并选中该路径。定位只改变前端选择和目录，不声称撤销操作；真正撤销/回滚仍需后端恢复点或事务日志支持。

## 2026-06-26 内容索引记录预览与维护

内容索引管理不再只显示聚合数字。`/api/files/content-index` 返回受限 `recordsPreview`（默认最多 200 条）和 `previewLimit`，每条包含 path、sha256、size、indexedAt、valid/stale 状态。FileManager 内容索引页支持按路径/SHA 搜索、按有效/失效筛选、复制单条记录，并可对有效记录执行“定位”跳转到文件列表对应目录。

约束：预览列表是运维辅助，不是完整数据库浏览器；统计仍覆盖全部记录，预览只取安全上限，避免大索引一次性拉爆前端。清理/重建/扫描仍通过后端动作执行，文件系统仍是唯一事实源。

### 2026-06-26 批量操作风险分层与删除确认

批量选择条采用两层结构：第一层只表达选择事实（已选数量、文件数、目录数、文件总大小、名称摘要），第二层才展示可执行操作。复制、移动、下载、打包、解压保持普通批量操作；删除被物理隔离到右侧危险操作区域，并使用红色风险语义，避免和普通文件操作混在一起。

删除弹窗新增强确认：展示待删除路径、文件/目录数量，并要求输入 `DELETE` 后才启用确认按钮。因为当前文件管理器删除后没有回收站/撤销保证，删除不能只靠一次点击确认。后续若实现回收站或可恢复删除，可以按风险等级降低确认强度；永久删除仍必须保留强确认。

### 2026-06-26 内容索引健康面板

内容索引管理从单纯统计页升级为维护控制台：顶部展示健康状态（检查中、未建立、健康、有失效项、索引较大）、失效率、检查时间和本会话维护事件。扫描、清理、重建都会留下事件证据；“复制诊断”会复制 root、记录数、分片数、有效/失效数量、容量和存储路径，方便后续排查索引性能或一致性问题。

索引记录预览仍然只展示有限条数，避免大索引一次性塞入 DOM；当记录数超过 previewLimit 时明确提示后续应升级分页 API，而不是把前端列表无限加大。文件系统始终是事实源，索引只服务同内容复用、去重判断和后续高性能搜索。

### 2026-06-26 内容索引记录分页 API

内容索引管理不再依赖 stats payload 的 `recordsPreview` 固定预览作为主列表。后端新增 `/api/files/content-index/records` 分页查询，支持 `status/query/offset/limit`，前端每页读取 100 条并提供上一页/下一页。统计接口继续负责健康摘要，记录接口负责大索引浏览；两者分离，避免大索引管理时为了翻页反复拉取完整统计或把大量记录塞进单个响应。

设计约束：当前分页是 offset/limit 的管理视图分页，适合普通索引维护；如果索引规模继续扩大到深分页风险，应按 cursor/search_after 思路升级为游标分页，而不是无限增大 offset 或一次性导出全部记录。

### 2026-06-26 文件操作事件中心

文件操作历史从“最近操作记录”继续升级为事件中心：每条记录包含开始/结束时间、耗时、项目数、成功/失败数量、影响路径和错误。面板顶部展示总量、成功/部分/失败统计和平均耗时；当前筛选结果可复制 JSON、复制错误报告或导出 CSV。旧 localStorage 记录若没有 `durationMs` 会按开始/结束时间自动补齐，避免升级后丢弃历史证据。

当前仍不宣称删除/覆盖可撤销。Material Snackbar 的 Undo 模式适合可逆动作；但本文件管理器目前删除直接走真实文件操作，没有回收站/撤销包，所以历史面板只提供审计、定位和复制证据。后续若要做撤销，必须由后端为 copy/move/rename/replace/delete 建立可验证的 rollback package。

### 2026-06-26 上传任务中心诊断增强

上传管理器继续向“任务中心”推进：侧边任务条现在显示上传速度和剩余时间；弹窗内每个文件行显示独立进度条、状态和上传速率；总进度区提供“复制诊断”，导出每个任务的文件名、目标路径、进度、速率、uploadId、chunkCount、已上传分片数、activeChunks 和错误。失败/取消任务集中显示，便于用户快速看到哪几个文件需要重新选择或恢复。

设计约束：上传任务仍以显式上传弹窗作为确认面，外部拖拽/粘贴只进入队列，不静默绕过冲突策略；任务诊断是排障证据，不包含文件内容或敏感数据。后续全局任务中心可以复用这些任务摘要字段，不应再为 Workspace 与 FileManager 各自实现一套上传状态 UI。

### 2026-06-26 IDE 级搜索选项与文件管理器同标签详情

- `/api/files/search` 新增 `caseSensitive` 与 `regex` 参数，响应回传当前搜索模式；正则无效时返回 `error` 和空结果，不抛出到页面崩溃。
- Workspace Search 面板的跨文件替换沿用当前搜索模式，确保“搜索看到的结果”和“替换预览写入的结果”语义一致。
- 独立 FileManager 的预览/编辑弹窗继续复用 `DocumentWorkbench`，并显式记录当前视图模式、未保存状态和 Ctrl/Cmd+S 保存快捷键；文件管理器不复制 Workspace 的编辑器实现。
- HTML 预览继续使用空 sandbox iframe；可视编辑只修改当前 draft，必须经保存按钮或 Ctrl/Cmd+S 写回。

### 2026-06-26 JSON/CSV 结构化预览

共享 `DocumentPreview` 新增 JSON 和 CSV/TSV 结构化预览：

- JSON：`json/jsonc/geojson/webmanifest` 使用安全 `JSON.parse()` 结构树预览，过大或解析失败时回退到源码提示/原文展示。
- CSV/TSV：按 RFC-4180 风格解析引号和转义双引号，表格预览限制行数、列数和字节数，避免大文件拖垮 UI。
- 编辑仍回到当前文件标签内的源码模式，保证结构化预览只负责阅读，不产生第二套写入语义。

### 2026-06-26 内容索引记录导出

内容索引管理页新增“导出本页 CSV”：

- 导出当前筛选、当前分页的索引记录，字段包含 `status/path/size/sha256/indexedAt`。
- 浏览器环境优先触发 CSV 下载；不可下载时降级为剪贴板复制。
- 只导出当前页，避免大索引一次性生成超大文件；全量导出后续应走后端流式任务。

### 2026-06-26 Workspace 资源管理器快捷键对齐

Workspace 左侧资源管理器补齐 PC/IDE 级上下文快捷键，但限定在资源管理器焦点内：

- `F2`：对当前键盘聚焦/选中的树节点直接打开重命名弹窗。
- `Delete` / `Backspace`：对当前活动树节点直接打开删除确认。
- `F5`：刷新文件查询缓存。

实现上，`FileTree` 向父级暴露 `onActiveEntryChange`，Workspace Explorer 维护当前活动项；`FileActionsMenu` 支持 `initialFlow`，因此键盘命令可以直接进入 rename/delete 等动作流，而不是先弹出右键菜单让用户再点一次。输入框、选择框、textarea 和 contenteditable 区域不会被这些快捷键劫持，避免和文件名编辑、路径输入、搜索以及 Monaco 编辑冲突。

这保持了当前产品的核心设计：左侧资源管理器负责选择/组织文件；右侧文件标签页才是文档工作台。源码、预览、边写边预览、可视编辑都属于同一个文件标签页的视图模式，不再恢复旧的 Editor/Preview 双窗口模型。

### 2026-06-26 Workspace 文件树键盘上下文菜单

Workspace 资源树补齐键盘打开上下文菜单：当焦点在资源管理器内且存在活动文件/目录时，`ContextMenu` 键或 `Shift+F10` 会在活动行附近打开同一套 `FileActionsMenu`。鼠标右键、键盘上下文菜单、F2/Delete 直接动作都共享同一个文件动作系统，避免为键盘用户和鼠标用户维护两套命令。

菜单定位基于当前活动树行的 DOM 边界，缺失行时才回退到固定安全坐标。该能力继续限定在 Workspace 资源管理器上下文内，不做全局监听，避免和 Monaco、搜索输入、终端快捷键冲突。

### 2026-06-26 FileManager 网格视图模式

独立文件管理器新增“列表 / 网格”视图模式。默认仍保持云面板/宝塔式细节列表，支持排序、列显隐、密度和批量操作；网格模式用于图片/目录/媒体等更适合视觉定位的场景，卡片内保留复选框、双击打开、键盘 Enter/Space、右键菜单、`ContextMenu` / `Shift+F10` 等操作入口。

设计约束：列宽只是浏览器侧视图偏好，不改变文件事实源；隐藏列时保留宽度，重新显示后恢复用户偏好；网格视图不应用列宽。当前实现避免引入完整 DataGrid/TanStack Table 依赖，复用现有列表组件，降低 bundle 和迁移成本。

设计约束：列顺序是视图偏好，不改变文件事实源；隐藏列重新显示时追加到当前顺序末尾，不强制打回默认顺序；提供 ↑/↓ 按钮作为非拖拽等价路径，避免只支持鼠标拖拽。

### 2026-06-26 FileManager 内部拖拽移动/复制

独立文件管理器支持把文件/目录拖到目录行或网格目录卡片上：默认打开“移动所选项目”弹窗；按住 Ctrl/Alt 投放则打开“复制所选项目”弹窗。该交互不会直接执行文件操作，而是复用现有复制/移动弹窗、服务端 dry-run、冲突策略和操作历史。

设计约束：拖拽是快捷入口，不是绕过确认的隐式写操作；复制/移动按钮和目标路径输入仍是非拖拽替代方式；禁止把目录移动/复制到自身或其子目录。外部 OS 文件拖入仍进入上传队列，内部文件拖拽使用自定义 drag MIME 与上传拖拽区分。

Workspace IDE 的编辑区已移除“同一文件打开第二个镜像编辑窗”的 split 状态。文件标签页是唯一文档容器；标签右侧的“源码 / 预览 / 编辑+预览 / 预览时编辑”只是当前活动文件标签的视图模式，并且活动标签内会显示当前模式徽标。

## 2026-06-26 Workspace 资源树批量复制/移动

Workspace 左侧资源树的多选操作补齐“复制 / 移动”。该能力定位为 IDE 内高频轻量文件操作：用户勾选多个文件/目录后，可以输入目标目录，选择同名冲突策略，并在提交前看到服务端 dry-run 摘要。

策略：

- 默认 `阻止`：发现同名冲突则禁用执行，避免误覆盖。
- `覆盖`：明确替换目标同名项。
- `跳过`：保留目标同名项，只处理无冲突项。
- `保留两者`：由服务端预检生成 `name (1).ext` 形式目标名，前端按预检目标执行。

设计边界：Workspace Explorer 只做快速批量复制/移动，不承载完整操作历史、CSV 导出、拖拽文件编排和索引维护；这些仍属于独立 FileManager。这样侧栏保持 IDE 导航效率，同时不把用户逼回终端执行基础文件操作。

## 2026-06-26 FileManager 文件名/内容搜索

独立文件管理器新增后端搜索面板，和当前目录列表过滤明确分层：

- 当前目录过滤：只过滤已经加载到表格/网格中的行，适合快速缩小当前目录视图。
- 文件搜索：调用 `/api/files/search`，在当前目录作用域下搜索文件名和文本内容，可递归子目录，支持大小写、正则，并沿用隐藏文件开关。
- 搜索结果展示名称/内容命中类型、snippet 高亮、内容索引命中状态；文件结果可定位并通过双击/右键进入弹窗预览编辑，目录结果可直接打开。

这使 FileManager 更接近 Google Drive / Dropbox / OneDrive 一类云服务文件管理器的搜索体验，同时保持文件系统仍是事实源，内容索引只作为性能缓存和命中可见性。

## 2026-06-26 内容索引管理按需加载

内容索引维护是文件管理器的系统级能力，但不是默认文件浏览、上传、搜索和预览的首屏路径。FileManager 现在仅在用户切换到“内容索引”视图时通过 `React.lazy` 加载 `ContentIndexManager`，并显示轻量 loading 状态。

设计约束：这只是下载时机优化，不改变索引统计、记录分页、扫描失效、清理失效、重建索引和诊断复制能力；文件系统仍是事实源，索引仍是性能缓存。

## 后续专项：Markdown 预览时编辑升级

当前“预览时编辑”还不够像思源笔记/Typora/Milkdown：部分 Markdown token 会以源文本形式露出。后续需要把它作为独立专项升级，而不是继续在 `contentEditable` 原文上小修小补。

方向：

- 继续保持一个文件标签页内的模式切换：源码、预览、编辑+预览、预览时编辑。
- 短期增强现有 Markdown live editor：图片、视频、Mermaid、代码块、表格、链接尽量渲染成块；只有当前聚焦块暴露必要 Markdown 语法。
- 中期评估 Milkdown/ProseMirror：它更接近插件化 WYSIWYG Markdown editor，可做 Mermaid、media block、主题、slash menu、撤销栈。
- 思源笔记作为交互借鉴：block-level WYSIWYG、块菜单、图表/媒体块、主题美化；但不迁移它的数据模型，Tracevane 文件管理仍以普通文件和 Markdown 源码为事实源。
- 所有重编辑器能力必须懒加载，不能进入 FileManager/Workspace 默认首屏。

## 2026-06-26 FileManager 上传任务条 / 后台进度入口

独立文件管理器的上传体验采用“上传管理器 + 后台任务条”的双层模型：上传管理器负责选择单文件/多文件/文件夹、粘贴文件、冲突策略、诊断、失败列表和完整队列；当用户关闭上传管理器但队列仍存在时，页面底部保留轻量上传任务条，显示上传进度摘要，并提供重新打开、暂停、继续、取消入口。

这样大文件、目录和断点续传任务不会因为弹窗关闭而从用户视野中消失；同时任务条复用 Workspace 已有 `UploadTaskStrip` 与上传 handle，不新增第二套状态源。FileManager 保持云服务级文件管理器心智：当前目录仍是上传默认目标，弹窗用于策略与详情，底部任务条用于后台可见性和快速控制。

## 2026-06-26 Markdown 预览时编辑块级渲染

Markdown 的“预览时编辑”从逐行 plaintext 编辑升级为块级渲染编辑：文档先切分为标题、正文、列表、引用、代码块、Mermaid、表格、图片、HTML/媒体等块；默认状态渲染每个块，用户只有点击“编辑块”才看到该块 Markdown 源码并应用回同一文件 buffer。

这不是完整思源/Milkdown 级富文本内核，但明显改善“预览时编辑仍像源码”的问题：图片、视频/HTML、Mermaid、代码块和表格默认以预览形态出现，源码暴露被限制在局部块编辑阶段。后续若接入 ProseMirror/Milkdown，应保持同一文件标签页的源码、预览、分屏、可视编辑模式，不引入第二个文件窗口。

## 2026-06-26 Markdown 图片/媒体资源解析

Markdown 预览现在把相对图片/媒体路径绑定到文件系统上下文：`docs/readme.md` 中的 `./img/a.png` 会按当前 `rootId` 解析为 `docs/img/a.png` 的 `/api/files/download` URL，而不是错误地按浏览器当前页面路径加载。绝对 root 内路径、上级目录相对路径也会归一化；外部 http(s)、data/blob 和页内锚点保持不改。

预览层还会对图片启用浏览器原生 lazy loading/async decoding，对 video/audio 强制提供 controls 和 metadata preload，并给外部链接加安全打开属性。该能力同时服务 Workspace IDE 与独立 FileManager，因为两者都复用同一个 `DocumentPreview` / `MarkdownPreview` 管线。

## 2026-06-26 内容索引分页文案收敛

内容索引管理已经进入“统计摘要 + 后端分页记录列表”模型，因此 UI 不再提示“大索引后续可升级分页 API”。健康状态在大索引场景下显示“分页管理”，说明记录列表通过后端分页查询管理大索引；只有兼容旧数据结构时才展示固定预览 fallback。

## 2026-06-26 内容索引记录搜索防抖

内容索引记录搜索采用两段状态：输入框使用 `queryDraft` 立即响应，260ms 后才同步到真正参与后端分页查询的 `query`。这样大索引下快速输入不会对 `/api/files/content-index/records` 形成每键一次的请求压力；页码只在稳定 query、状态筛选或 root 变化后重置。

## 2026-06-26 文件搜索结果上限显式化

文件搜索后端保留 250 条安全上限，防止大型 workspace 或系统 root 搜索一次性扫描/渲染过多结果。现在搜索响应会返回 `limit` 与 `truncated`：如果达到上限，Workspace Search 和独立 FileManager Search 都会明确提示用户收窄关键词或路径。跨文件替换只作用于本次返回并预览勾选的结果，不能暗示已覆盖超过上限之外的未知匹配。

## 2026-06-26 文件搜索缓存键完整性

文件搜索缓存键现在包含 `recursive` 和 `hidden`。递归搜索、隐藏文件开关、大小写和正则都属于搜索结果事实的一部分，必须进入 query key；否则 UI 开关变化可能复用旧缓存。后续搜索能力如果增加 limit、cursor、文件类型过滤或路径作用域，也必须同步进入缓存键。

## 2026-06-26 文件搜索安全 limit 参数

文件搜索 API 支持受控 `limit`：默认 250，后端最大 500，并通过 `truncated` 告诉前端是否达到上限。Workspace Search 与 FileManager Search 当前继续使用默认上限；后续若某个视图只需要轻量预览，可以传更小 limit，若进入专门搜索页再传更高 limit。所有影响搜索结果的参数，包括 limit，都必须进入 query key。

## 2026-06-26 Workspace/FileManager 搜索 limit 分层

Workspace Search 使用 250 条结果上限，符合 IDE 侧栏“快速定位 + 跨文件替换预览”的轻量定位职责；独立 FileManager Search 使用 500 条结果上限，更适合系统级文件管理器的大范围检索。两个界面都会读取后端 `truncated` 并提示用户收窄范围，避免把有限结果误认作全量结果。

## 2026-06-26 后续专项：富 Markdown / 可视编辑内核升级

用户反馈：当前 Markdown “预览时编辑”仍会在部分场景暴露大量源文本，不符合思源笔记、Typora、Obsidian/Milkdown 一类现代写作体验。该问题不应再当作小 UI bug 处理，而应进入独立的富文档渲染/编辑专项。

目标体验：

- Markdown 文件仍是事实源，源码、预览、编辑+预览、预览时编辑都在同一个文件标签页内切换，不新增第二个预览窗口。
- 预览时编辑默认展示渲染后的块：标题、列表、引用、表格、代码块、任务列表、图片、视频、音频、HTML 安全块、Mermaid/图表、数学公式等；只有当前聚焦块或显式“源码编辑块”时暴露 Markdown token。
- 图片/视频/附件路径继续按当前 root + Markdown 文件所在目录解析；粘贴图片、拖入媒体后应写入相对路径并进入上传/内容索引流程。
- Mermaid、数学公式、图表等重依赖必须懒加载，并随浅色/深色主题自动切换；渲染失败时保留可编辑源码块和错误信息。
- 主题层不能只是“能看”：Markdown 阅读区、代码块、表格、callout、媒体卡片、Mermaid SVG、错误态和编辑态都要接入 Aurora 浅/深色 token。
- 保留 IDE 能力：当前文件搜索/高亮/替换、多选、快捷键、撤销栈、源码模式 Monaco 编辑不能被 WYSIWYG 牺牲。

外部产品/开源借鉴基线：

- 思源笔记 / Protyle：借鉴 block-level WYSIWYG、块菜单、块级渲染状态、图表/媒体块和大文档编辑体验；不复制 `.sy`/kramdown 数据模型，不直接内嵌其编辑器。
- Milkdown / ProseMirror / Remark：作为中期 POC 的首选方向，验证 Markdown AST、编辑事务、插件化节点、Mermaid/media block、撤销栈和源码序列化质量。
- TOAST UI Editor：借鉴 Markdown / WYSIWYG / Live Preview 同文档多模式切换、GFM、语法高亮、scroll sync 和插件化图表能力。
- Cherry Markdown：借鉴轻量可扩展的 Web Markdown 编辑器形态、Mermaid 插件、自定义渲染 hook；如引入必须评估体积、主题隔离和 React 集成复杂度。
- GitHub Markdown / Mermaid：把 fenced `mermaid` 作为图表事实语法，不发明 Tracevane 私有图表语法；扩展只做兼容层。

技术路线：

1. P0 修复：定位当前“预览时编辑仍显示大量源文本”的具体语法缺口，优先补 GFM 表格、任务列表、链接、图片、HTML 安全块、代码块和 Mermaid 的默认渲染。
2. P1 富预览：统一 MarkdownPreview 与 MarkdownLiveEditor 的 renderer registry，让图片、视频、音频、Mermaid、代码块、表格、callout、数学公式都走同一主题/错误/懒加载策略。
3. P2 编辑内核 POC：评估 Milkdown/ProseMirror、Cherry Markdown、TOAST UI Editor 的 bundle、许可证、React 适配、源码保真、插件模型和主题成本；没有明确收益不得新增依赖。
4. P3 交互升级：实现块菜单、源码块 fallback、拖拽/粘贴媒体自动上传并写入相对路径、同标签页搜索高亮和替换安全边界。
5. 扩展 Viewer Registry：把 Markdown、HTML、Office、压缩包、日志、大文件抽样预览都当作 `DocumentWorkbench` 插件能力，避免再次散落到 Workspace/FileManager 两套实现。

验收标准：

- 常见 Markdown 文档在“预览时编辑”下不再整段暴露源码；源码只出现在正在编辑的块或无法安全渲染的块。
- Mermaid、图片、视频、表格、代码块、HTML 安全片段在 Workspace 与 FileManager 中表现一致。
- 浅色/深色主题下阅读层级、代码高亮、图表主题和媒体边框一致。
- 大 Markdown 文件不阻塞首屏；Markdown/Mermaid/WYSIWYG 内核按需加载，有明确性能测试或 smoke 证据。

## 2026-06-26 Markdown 图片语法承载音视频

Markdown 标准没有统一的视频语法，但用户在文件管理器/IDE 文档里经常用 `![描述](demo.mp4)` 或 `![描述](voice.mp3)` 记录本地媒体。预览层现在会在安全 DOM 后处理阶段识别这类 `img` 节点：若资源扩展名是常见 video/audio 格式，则替换成带 controls、metadata preload、playsinline（视频）的原生 `<video>` / `<audio>` 元素，并继续使用当前 root + Markdown 文件目录解析相对路径。

这样既不引入新的 Markdown 方言依赖，也让 FileManager 与 Workspace 在同一 `MarkdownPreview` 管线中支持“Markdown 图片、视频、音频”一体化预览；无法识别或非媒体扩展名仍按图片处理。

## 2026-06-26 Mermaid 预览主题同步

Markdown/Mermaid 预览现在接入 Tracevane Aurora 主题状态：浅色模式使用 Mermaid `default`，深色模式使用 Mermaid `dark`。主题切换时会重新生成 sanitized Markdown HTML，让已渲染 SVG 回到未处理 Mermaid 占位块，再按当前主题重新渲染，避免深色界面里仍显示浅色图表或箭头/文字对比不足。

约束：Mermaid 仍保持动态导入和 `startOnLoad: false`，只在 Markdown 预览真正包含 Mermaid 块时加载；安全级别继续使用 `strict`，不为了主题效果放宽图表安全边界。

## 2026-06-26 FileManager 底部状态栏选择摘要

独立文件管理器底部状态栏从单纯“可见/总项”升级为系统文件管理器式状态摘要：无选择时显示未选择项目；有选择时显示已选项数、文件数、目录数和选中文件总大小。批量操作条继续负责可执行动作，状态栏负责始终可见的当前位置/列表状态反馈。

设计约束：目录大小不在前端递归计算，避免大目录阻塞 UI；当前摘要只统计已加载条目中的选中文件 size。后续若需要目录总大小，应通过后端异步任务或索引能力提供，而不是在浏览器中递归扫描。

## 2026-06-26 FileManager 详情/预览面板可收起

该方案已被 2026-06-26 的弹窗式文件预览/编辑取代。当前不再常驻右侧详情/预览面板，也不再持久化 `detailsPanelOpen`；文件预览/编辑由双击或右键打开弹窗承载。

设计约束：详情/预览仍是当前文件上下文的一部分，不创建第二个文件窗口；开关只改变布局可见性，不改变 `DocumentWorkbench` 的源码、预览、编辑+预览、预览时编辑语义。后续如增加可拖拽面板宽度，应继续写入同一视图偏好，而不是新建独立布局系统。

## 2026-06-26 FileManager Space 快速预览

独立文件管理器在选中文件后支持 `Space` 快速打开预览/编辑弹窗。该快捷键对齐系统文件管理器“选中文件后按空格快速预览”的习惯，但 Tracevane 统一进入同一个 `FilePreviewDialog` 和 `DocumentWorkbench`，不再复用右侧常驻面板。

安全/交互约束：当焦点在输入框、文本域、下拉框、按钮、链接、菜单项等交互控件上时，不拦截 Space，避免破坏原生按钮激活和表单输入。Space 只改变面板可见性，不保存、不写文件、不改变选中项。


## 2026-06-26 FileManager 弹窗式文件预览/编辑

用户反馈右侧常驻详情/预览面板会挤占文件列表，不符合宝塔面板/云文件管理器的主流文件操作心智。FileManager 改为列表优先：

- 双击目录进入目录；双击文件打开居中的预览/编辑弹窗。
- 右键菜单为文件提供“预览 / 编辑”，与双击进入同一个 `FilePreviewDialog`。
- 选中文件按 Space 也打开同一个弹窗，作为键盘快捷预览入口。
- 弹窗内部继续复用 `DocumentWorkbench`，所以源码、预览、编辑+预览、预览时编辑仍是同一文件上下文的 view mode。
- 默认列表保持全宽，避免右侧常驻预览面板把云面板式文件表格压窄；文件元信息、保存、下载等动作进入弹窗上下文。

设计取舍：不复刻宝塔全部界面，只借鉴“文件列表为主、编辑器/预览以弹窗承载”的交互结构；Workspace IDE 仍保持文件标签页工作台，不受 FileManager 弹窗策略影响。

旧的“详情/预览面板可收起”方案被该弹窗模式取代：不再持久化 `detailsPanelOpen`，也不再把 `FileDetailsPanel` 常驻在列表右侧。后续如需要详情，可作为轻量属性弹窗或右键“属性”实现，而不是恢复占位侧栏。

## 2026-06-26 FileManager 弹窗编辑未保存保护

文件预览/编辑弹窗现在继承 IDE 编辑器的 dirty guard：当 `DocumentWorkbench` 产生未保存 draft 时，关闭弹窗不会直接丢弃内容，而是先打开确认对话框，用户可以继续编辑或明确选择“不保存并关闭”。同一 dirty 状态还会按需注册 `beforeunload`，只在存在未保存更改时拦截浏览器关闭/刷新，避免对普通文件浏览造成干扰。

设计约束：未保存保护属于弹窗工作台，不恢复右侧常驻预览面板；保存仍由 FileManager 调用 Files API 写回当前 root/path，保存成功后清 dirty 状态。

## 2026-06-26 FileManager 弹窗关闭支持保存并关闭

未保存关闭确认从“继续编辑 / 不保存并关闭”升级为三段式：继续编辑、保存并关闭、不保存并关闭。`FilePreviewDialog` 拥有关闭意图和确认弹窗，`FileDetailsPanel` 通过 `onSaveHandlerChange` 暴露当前文件保存函数；用户选择“保存并关闭”时，只有 Files API 写入成功才关闭弹窗，保存失败则留在当前弹窗并保留 dirty 状态。

这让 FileManager 弹窗编辑更接近桌面 IDE/文件编辑器：关闭动作本身就能完成安全保存，不需要用户先取消关闭、回到弹窗手动保存后再关闭。

## 2026-06-26 FileManager 预览编辑弹窗最大化

文件预览/编辑弹窗新增最大化/还原控制：默认仍是居中的大弹窗，适合快速预览；点击“最大化”后扩展到接近整个视口，给代码、Markdown 分屏、HTML 预览时编辑和大图片/视频预览提供足够空间。该能力借鉴 VS Code centered modal 可最大化的编辑器体验，也贴合宝塔/云面板中文件编辑常以大窗口承载的习惯。

实现约束：最大化只是弹窗尺寸状态，不把 FileManager 预览转成 Workspace 标签页，也不恢复右侧常驻预览栏；同一 `DocumentWorkbench` 继续承载源码、预览、编辑+预览和预览时编辑模式。

## 2026-06-26 FileManager 搜索结果直达预览/编辑

文件搜索结果中的文件项现在不再只是“定位到列表并等待用户再双击”。点击文件结果或“预览 / 编辑”会先定位该路径，再直接打开同一个 `FilePreviewDialog`。目录结果仍保持“打开目录”。这样搜索、列表双击、右键菜单和 Space 快捷键全部汇入同一套弹窗预览/编辑工作台，避免出现多套入口、多套预览逻辑。

对尚未加载在当前目录列表中的搜索结果，FileManager 会使用搜索结果携带的 `FileSearchResult` 作为临时 `previewTarget`，并通过 `/api/files/read` 读取真实内容；列表稍后加载到同一路径时再自然使用列表条目元数据。

## 2026-06-26 FileManager 内容索引记录直达预览/编辑

内容索引管理页的有效记录现在也接入同一套弹窗式文件工作台：用户可以从索引记录行直接“预览 / 编辑”，FileManager 会先定位文件路径，再用该记录构造临时文件条目并通过 Files read API 读取真实内容。这样内容搜索、索引维护、列表双击、右键菜单和 Space 快捷键都汇入同一个 `FilePreviewDialog`，不会因为入口不同而出现右侧常驻预览、独立预览页或第二套编辑器。

设计约束：索引不是文件事实源，只提供路径、大小和 hash 线索；打开文件时仍以当前 root/path 的 Files API 为准。失效记录只保留复制/诊断能力，不提供预览编辑入口，避免用户编辑已经不存在或已变更的对象。

## 2026-06-26 FileManager 属性弹窗

文件管理器新增右键“属性”和 `Alt+Enter` 快捷键，用轻量属性弹窗承载文件/目录元信息：名称、位置、root、类型、大小、修改时间、隐藏状态和相对路径。该入口替代旧右侧详情面板的“看元数据”用途，同时不挤占默认列表宽度，也不和预览/编辑弹窗混成一个窗口。

设计取舍：属性弹窗只展示当前 `FileEntrySummary` 与路径上下文，目录大小不在前端递归计算；复制路径是安全的低风险操作。后续如加入权限、owner、hash、MIME 或扩展属性，应继续扩展该属性弹窗或后端 metadata API，而不是恢复常驻右侧详情栏。

## 2026-06-26 上传目标目录可编辑

上传管理器现在在弹窗内显示并允许修改“上传目标目录”。工具栏/右键/拖拽仍默认使用当前目录或目标目录，但用户在真正开始上传前可以把目标调整为同 root 下的其它相对路径；上传开始后目标目录输入锁定，避免同一批任务中途改变写入位置。

该能力同时应用于独立 FileManager 和 Workspace Explorer，因为两者复用同一个 `UploadManagerDialog`。冲突策略仍保留“保留两者/覆盖/跳过/冲突时报错”，外部剪贴板快速上传继续默认“保留两者”，符合云文件管理器对重名上传的安全默认。

## 2026-06-26 拖拽到文件夹上传

独立 FileManager 支持把浏览器外部文件直接拖到列表/网格里的某个文件夹项：行/卡片会显示“上传到此处”，松开后进入同一个上传管理器，目标目录自动设置为该文件夹路径。拖到空白区域仍上传到当前目录；拖拽 FileManager 内部条目到文件夹仍保持复制/移动语义，并通过 Ctrl/Alt 切换复制。

设计约束：外部文件拖拽只做上传，内部文件拖拽只做复制/移动，两者通过 `DataTransfer.types` 明确区分，避免把本地文件误当作内部 move，或把内部 move 误当作上传。上传仍进入显式上传队列，不静默绕过冲突策略和进度管理。

## 2026-06-26 FileManager 文件剪贴板快捷键

独立 FileManager 新增文件级剪贴板：`Ctrl/⌘+C` 复制当前选择，`Ctrl/⌘+X` 剪切当前选择，`Ctrl/⌘+V` 把剪贴板项目粘贴到当前目录，并进入既有复制/移动弹窗和服务端 dry-run 预检。这样键盘流与右键“复制到/移动到”共享同一套冲突策略、覆盖/跳过/保留两者规则和操作历史，不另写一套静默文件操作。

设计约束：该剪贴板是 FileManager 内部文件剪贴板，不覆盖系统文本剪贴板；当剪贴板为空时不拦截浏览器粘贴，仍允许剪贴板文件快速上传。粘贴后仍要求通过复制/移动确认弹窗执行，避免 Ctrl+V 直接覆盖或移动文件。

## 2026-06-26 FileManager 导航与新建快捷键

独立 FileManager 继续补齐 PC 级键盘流：`Alt+↑` 跳到上级目录，`F3` 聚焦当前目录过滤框，`Ctrl/⌘+N` 新建文件，`Ctrl/⌘+Shift+N` 新建目录。它们与已有 `F2` 重命名、`Delete/Backspace` 删除、`F5` 刷新、`Ctrl/⌘+A` 全选、文件剪贴板和 Space 预览组成基础桌面文件管理器快捷键层。

设计约束：这些快捷键只在 FileManager 容器获得焦点且焦点不在输入框/按钮/链接/菜单项等可编辑或交互控件内时触发，避免破坏路径栏、搜索框、弹窗输入和编辑器原生快捷键。

## 2026-06-26 文件预览/编辑弹窗编辑器 Chrome

研究记录：aaPanel Files 文档确认 Web 服务器文件管理器以列表为主，支持右键属性、双击受支持文件进入在线查看/编辑/保存，并支持代码高亮；Webix File Manager 文档也将双击文本文件打开专用 editor、Ctrl+S 保存作为主路径。Tracevane 因此继续避免右侧常驻预览栏，把文件内容操作收敛到按需弹出的 `FilePreviewDialog`。

落地约束：
- 文件列表仍是文件管理器主屏，不把预览/编辑常驻到右侧侧栏。
- 弹窗编辑器增加类 IDE 状态栏：保存状态、当前视图模式、可编辑/只读、语言/MIME、大小、UTF-8/LF 和查找/替换/保存快捷键提示。
- 保存按钮显式标注 `Ctrl/⌘+S`，与弹窗关闭 dirty guard 共存；Esc/关闭时仍先确认未保存内容。
- 后续历史版本、差异恢复、权限/owner 等 aaPanel 级能力必须在这个弹窗/属性模型上增量扩展，不再新增第二套侧边预览实现。

## 2026-06-26 文件权限 / Owner 元数据只读展示

研究记录：aaPanel Files 文档把 PMSN/Owner 作为文件列表核心列，并在属性中查看文件/目录基本信息；Node.js `fs.Stats` 官方文档暴露 `mode`、`uid`、`gid`、`size`、`mtimeMs` 等稳定 stat 字段。当前阶段先把权限/属主作为事实元数据贯通到 Files API、FileManager 弹窗和属性弹窗，不做 chmod/chown 写操作。

落地约束：
- `FileEntrySummary` 和 `FilesReadPayload` 增加 `mode`、`permissions`、`uid`、`gid`，由后端 stat 一次性生成。
- 列表仍默认保持简洁，不强行新增高密度权限列；属性弹窗和编辑弹窗基础信息展示权限/UID/GID。
- 权限写入、递归 chmod/chown 属于高风险操作，必须后续单独做 dry-run、权限边界和确认弹窗，不和本次只读元数据混在一起。

## 2026-06-26 权限 / Owner 可选列表列

权限与 UID/GID 已接入 Files API 后，FileManager 列表不默认增加高密度列，而是把 `权限` 和 `Owner` 加入列配置菜单。用户可以像云服务/宝塔面板那样切换出权限信息，并继续拖拽调整列顺序、调整列宽、按权限或 Owner 排序；默认列仍保持 `大小 / 修改时间 / 类型`，避免普通文件浏览变臃肿。

设计边界：
- 这是只读展示列，不是权限修改入口。
- 权限列显示 symbolic permissions，tooltip 保留 octal mode。
- Owner 列先显示 `uid:gid`，不做用户名/组名解析，避免跨平台 NSS/LDAP/容器环境阻塞 UI。

## 2026-06-26 删除默认进入 root 回收站

为了让系统级文件管理器更接近宝塔/云服务面板，删除不再默认直接 `rm`：Files API 的 `deletePaths` 默认把项目移动到当前 root 的隐藏 `.tracevane-trash`，并写入 `metadata.json` 记录原路径、删除时间、类型和目标回收站路径。独立 FileManager 删除弹窗保留 `DELETE` 手动确认，并新增“永久删除”选项；未勾选时按钮文案为“移入回收站”。Workspace Explorer 的上下文删除也改为“移入回收站”语义。

设计边界：
- `.tracevane-trash` 是当前阶段的安全回收站事实源，不进入内容索引重建扫描。
- 永久删除仍可用，但必须在独立 FileManager 删除弹窗显式勾选。
- 后续需要在独立文件管理器域补“回收站管理”子页：列出 metadata、恢复到原路径、冲突策略、永久清空。当前阶段先保证默认删除不再不可逆。

## 2026-06-26 文件预览/编辑弹窗优先

文件管理器不采用右侧常驻预览栏。主列表保留完整宽度和云面板式文件管理密度；文件双击、右键“预览 / 编辑（弹窗）”、Space 快速预览才打开独立弹窗工作台。弹窗内仍是同一个文件上下文，包含源码/预览/边写边预览/预览时编辑模式；代码和普通文本默认进入源码编辑，Markdown、HTML 和媒体默认进入预览，避免打开代码文件后先看到弱价值的普通预览。

设计借鉴 aaPanel/宝塔文件管理的双击在线编辑心智，但不复刻其视觉：Tracevane 继续使用项目主题、浅深色 token、同一 DocumentWorkbench，以及更强的查找/替换和状态栏。右侧详情/预览面板不再作为 FileManager 的默认结构。

## 2026-06-26 回收站管理闭环

默认删除进入 `.tracevane-trash` 后，独立 FileManager 新增“回收站”子视图，避免“软删除但只能手动翻隐藏目录”的半成品体验：

- 后端 Files API 提供回收站列表、单项恢复、永久清理 selected/all。
- 回收站记录读取删除时写入的 `metadata.json`，显示原路径、删除时间、大小、文件/目录类型。
- 恢复支持冲突策略：保留两者（默认）、遇到同名时报错、覆盖原路径、跳过恢复。
- 永久清理只作用于 `.tracevane-trash/<entry>/` 包装目录，避免误删 root 或普通路径。
- FileManager 顶部在“文件列表 / 内容索引”之外新增“回收站”，作为文件管理域内部子页，不放回 Workspace IDE。

设计原则：回收站是安全文件操作的一部分，不是 Evidence；操作历史记录恢复和永久清理结果，便于审计当前会话。

## 2026-06-26 权限修改 dry-run

独立 FileManager 在批量操作区新增“权限”动作，补齐服务器文件管理器常见的 PMSN/Permission 能力：

- 支持 3-4 位八进制权限输入，例如 `644`、`0755`。
- 支持目录递归应用，默认对选中目录开启递归提示；执行前必须 dry-run 展示影响文件/目录数量和前若干路径的 `currentMode → nextMode`。
- dry-run 超过安全预览上限时阻止执行，要求缩小选择范围，避免一次误改海量系统文件。
- 执行结果进入操作历史；列表/属性/预览弹窗的权限字段通过 Files API 刷新。

本轮只实现 chmod；chown/owner 修改暂不做，因为 UID/GID 到用户名/组名解析、权限边界和平台差异需要单独的 dry-run 与失败恢复策略。

## 2026-06-26 弹窗编辑器专注布局

在“文件双击后弹窗预览/编辑”的基础上，FileManager 弹窗继续从“右侧详情面板风格”收敛为“在线代码编辑窗口风格”：模态态默认不再把名称、大小、权限、UID/GID 等详情表格常驻压在编辑器上方，而是只保留一条轻量命令栏，显示语言、大小、权限摘要、打开预览、下载、保存和“显示属性”。完整属性作为按需折叠区出现。

设计约束：
- 文件管理主屏仍是列表/云面板，不放右侧常驻预览。
- 弹窗打开后编辑器区域优先占满可用高度，避免元数据挤压代码阅读和查找替换。
- 属性信息不删除，只从默认可见改为按需展开；权限/Owner 等信息仍由 Files API 事实源提供。
- 继续复用 `DocumentWorkbench`，源码/预览/边写边预览/预览时编辑属于同一个弹窗文件上下文。

## 2026-06-26 弹窗编辑器可调整尺寸

FileManager 的文件预览/编辑弹窗新增可调整尺寸能力：非最大化状态下，用户可以拖拽右下角调整窗口宽高，也可以聚焦 resize handle 后用方向键微调，Shift+方向键加速调整；顶部提供“默认尺寸”用于回到系统默认。最大化仍是单独状态，适合临时进入近全屏编辑。

设计约束：
- 调整尺寸只作用于当前弹窗会话，不写入全局布局，避免文件管理器主界面被历史编辑窗口尺寸污染。
- 通过最小尺寸和 viewport 边界 clamp 防止弹窗被拖到不可用大小或超出屏幕。
- resize handle 是可聚焦按钮，保留键盘可达性；不依赖第三方拖拽布局库，避免引入额外复杂度。

## 2026-06-26 弹窗编辑器保存失败恢复

在线编辑器保存失败不再只依赖短暂 toast。保存失败后，弹窗命令栏下方保留错误条，显示失败原因和“重试保存”按钮；草稿保持 dirty，不会因为失败而清空。保存成功后错误条消失，并显示最近保存时间，状态栏同步从“未保存/保存失败”回到“已保存”。

设计约束：
- 保存失败必须留在当前编辑器上下文内可见，不能只做一次性 toast。
- 失败后保留 draft 和 beforeunload dirty guard，避免用户误关丢失内容。
- 错误条使用 alert 语义，成功保存时间使用 status 语义，保证辅助技术可感知状态变化。

## 2026-06-26 同文件替换差异预览

`DocumentWorkbench` 的当前文件查找/替换继续向 IDE 级体验收敛：单个替换仍可直接执行，但“全部替换”不再一键静默修改当前文件。用户需要先点击“预览全部”查看当前文件内前若干差异行，再点击“确认全部替换”应用到 draft。应用后文件进入未保存状态，仍需保存才写回磁盘。

设计约束：
- 当前文件替换预览复用共享 `ReplaceDiffPreview`，避免 Workspace Search 与 FileManager 弹窗各维护一套 diff UI。
- 预览只展示有限差异行，防止大文件替换时把编辑器工具栏撑爆；完整替换数量通过匹配计数提示。
- 全部替换仍只修改当前文件 draft，不越权成为跨文件批量替换；跨文件替换继续属于 Workspace Search 域。

## 2026-06-26 Markdown 任务列表预览时编辑

Markdown 预览时编辑继续向思源笔记/块级编辑方向推进：GFM task list 在块级预览中不再只是静态渲染。用户在 `预览时编辑` 模式下可以直接勾选/取消勾选任务项，系统会把对应 Markdown 源码从 `[ ]` / `[x]` 之间切换，并写入当前文件 draft；仍需保存才落盘。

设计约束：
- 只在可编辑的 Markdown list block 中启用 checkbox 回写，普通预览保持只读。
- checkbox 交互复用现有 `MarkdownPreview` DOM 装饰阶段，不新增 Markdown 编辑器依赖。
- 回写只修改当前块源码，不引入思源 `.sy` 等私有格式，保持项目文档的 Markdown 可移植性。

## 2026-06-26 Markdown 块级增删与移动

Markdown `预览时编辑` 在任务列表勾选之外继续补齐块级编辑：每个预览块提供上移、下移、新增块、编辑块、删除块。操作直接修改当前 Markdown draft，保持原文件格式；保存后才写回磁盘。新增块默认插入一个普通段落，用户可以立即通过“编辑块”改成标题、列表、代码块、Mermaid 或其它 Markdown 结构。

设计约束：
- 块操作只在 Markdown visual mode 内出现，不改变源码模式和普通预览模式。
- 移动/删除使用 HTML button，保留键盘和辅助技术可达性；上/下边界按钮禁用。
- 该能力借鉴思源等块级编辑体验，但不引入私有块存储和引用系统，避免破坏 Markdown 文件可移植性。

## 2026-06-26 Markdown 图片/媒体块替换资源

Markdown `预览时编辑` 的图片/媒体块新增“替换资源”入口。用户可以在块头直接输入相对路径或 URL，系统会改写当前块的 Markdown 图片语法或 HTML `src` 属性，并写入当前 draft；保存后才落盘。该能力用于把文件管理器中的资源路径快速接入文档，不需要切回源码模式手动找括号或 `src`。

设计约束：
- 现阶段只改引用路径，不复制/上传文件；真实上传和文件选择仍属于 FileManager 上传/资源管理能力。
- 替换资源保持 Markdown/HTML 原文结构，避免把文档转换成私有块格式。
- 后续可在此入口上接入文件管理器选择器，选择当前 root 内文件后自动生成相对路径。

## 2026-06-26 Workspace 资源树弹窗预览入口

Workspace 左侧资源管理器同步收敛为 PC 文件管理器式交互：单击只负责选中/聚焦文件或目录，双击文件打开独立文件预览/编辑弹窗；右键文件提供同一个“预览 / 编辑（弹窗）”入口。这样文件预览不再依赖右侧编辑区域，也不会把预览作为 Workspace 的常驻侧边面板。右侧 Dockview 编辑区仍保留给真正的 IDE 工作流、搜索跳转和多标签编辑；资源树中的快速查看/轻量修改优先使用弹窗工作台。

设计约束：
- 资源树单击不再误触发打开文件，降低浏览目录时打断编辑器上下文的概率。
- 双击/Enter 是显式打开；目录保持展开/收起，文件进入弹窗。
- 弹窗复用 FileManager 的 `FilePreviewDialog` 与 `DocumentWorkbench`，所以代码、Markdown、HTML、图片、视频等预览/编辑能力不分裂。
- 右键菜单保留预览/编辑入口，符合宝塔/aaPanel 类文件管理器的上下文菜单习惯。

## 2026-06-26 Markdown 资源选择器

Markdown `预览时编辑` 的图片/媒体块不再只能手动输入资源路径。替换资源面板内新增轻量资源选择器：基于当前文件的 root 和文档目录读取文件列表，展示子目录、图片或媒体候选文件；点击候选文件后自动生成相对当前 Markdown 文件的路径并填入输入框，用户确认后才改写当前块源码。

设计约束：
- 资源选择器只选择当前 root 内已有文件，不隐式上传、不复制文件、不改变文件系统。
- 自动生成相对路径，保证 Markdown 文件移动到同一仓库/目录结构时仍具备可移植性。
- 图片块只显示图片候选；媒体块显示图片、音频、视频等可嵌入资源候选，避免在大目录中展示无关文件。
- 该选择器属于文档工作台的局部能力，不替代独立 FileManager 的完整上传、复制、移动、搜索和索引管理。

## 2026-06-26 弹窗编辑器保存前差异审阅

FileManager / Workspace 资源树共用的文件预览编辑弹窗新增保存前差异审阅。文本文件被编辑后，命令栏出现“审阅差异”，点击“保存修改”也会先打开差异审阅弹窗；用户可以查看当前草稿相对磁盘内容的新增/删除/修改行，再点击“确认保存”写回 Files API。

设计约束：
- 差异审阅只比较当前弹窗草稿和本次读取的磁盘内容，不伪装成 Git diff，也不替代 Workspace Git 面板。
- 差异预览限制最大展示行数，避免超大文件把弹窗 DOM 撑爆；大文件仍可回到源码编辑器审阅。
- Ctrl/⌘+S 仍保留快速保存能力；命令栏保存按钮走审阅确认路径，兼顾效率和防误操作。
- 审阅弹窗和编辑弹窗共用同一 dirty draft，返回编辑不会丢失草稿。

## 2026-06-26 弹窗编辑器本地历史版本快照

在线编辑器新增“历史版本”入口。文本文件每次成功保存后，会在当前浏览器 localStorage 中记录保存前内容作为本地快照；同一文件最多保留 5 个，单个快照内容限制为 256KB。历史弹窗支持查看快照、删除快照、恢复到当前草稿。恢复动作不会立即写回磁盘，用户仍需审阅差异并保存。

设计约束：
- 本阶段是前端安全网，不宣称为服务器级版本库；清理浏览器数据会丢失快照。
- 快照只保存文本内容和少量元数据，不保存二进制文件，避免浏览器存储膨胀。
- 大文件不进入 localStorage 快照；后续应升级为后端版本历史或 IndexedDB/分片快照。
- 恢复到草稿后仍保留保存前 diff 审阅和保存失败恢复流程。

## 2026-06-26 弹窗编辑器服务器历史版本

FileManager / Workspace 共用的在线编辑弹窗从“仅浏览器本地快照”升级为服务器侧历史版本优先：Files API 在文本文件写回前会把当前文件内容保存到 `.tracevane/file-versions`，历史弹窗同时展示服务器版本和本地快照。服务器版本支持列表、读取、恢复到当前草稿和删除；恢复到草稿不会立即覆盖磁盘，用户仍需通过保存前差异审阅确认后写回。

设计约束：
- 历史版本属于 Files API 文件管理域，不混入 Workspace Git 历史，也不伪装成 Git commit。
- 只保存小于 1MB 的文本类文件历史，单文件最多保留 20 个服务器版本，避免索引库和工作目录膨胀。
- 版本存储放在 OpenClaw 根目录下 `.tracevane/file-versions`，内容索引重建跳过 `.tracevane`，避免把内部历史版本误当用户文件索引。
- 保存/恢复主路径优先级高于历史快照：历史写入失败不能阻塞用户保存文件。
- 本地 localStorage 快照保留为浏览器侧兜底，但 UI 标明“服务器 / 本地”来源。

## 2026-06-26 服务器历史版本内容预览

历史版本弹窗不再把服务器版本当成只能恢复的黑盒。用户选中服务器历史版本后，前端通过 Files API 读取该版本内容并直接在弹窗右侧预览；本地快照与服务器版本使用同一个预览区域。恢复仍只写入当前编辑草稿，不立即覆盖磁盘，之后继续走保存前差异审阅。

设计约束：
- 服务器版本内容按需读取，避免打开历史弹窗时一次性拉取所有版本内容。
- 读取失败在历史预览区内展示，不关闭弹窗、不破坏当前草稿。
- 恢复按钮在服务器版本内容读取中禁用，避免用户在内容尚不可见时误恢复。

## 2026-06-26 历史版本与当前草稿差异对比

历史版本弹窗新增“对比草稿”模式。用户选中服务器历史版本或本地快照后，可以在同一个弹窗内把历史内容与当前编辑草稿做轻量行级 diff；确认差异后再选择恢复到草稿。这样历史恢复从“看内容后恢复”升级为“看内容 + 看差异后恢复”，更符合在线代码编辑器的防误操作体验。

设计约束：
- diff 比较的是“历史版本内容 → 当前草稿”，不是 Git diff，不参与提交历史。
- 对比复用保存前差异审阅的 `FileSaveDiffPreview/createFileSaveDiffLines`，避免历史恢复和保存审阅形成两套视觉语言。
- 选中不同历史版本时自动回到内容预览，防止用户误以为仍在对比上一个版本。

## 2026-06-26 预览时编辑滚动与响应式收敛

文件预览/在线编辑弹窗继续收敛为“一个文件一个工作台”：源码、预览、边写边预览、预览时编辑都在同一标签/弹窗上下文中切换，不再用左右侧边预览制造重复窗口。Markdown `预览时编辑` 从双栏实验界面改为单一渲染画布：正文区域默认就是预览效果，块工具条只在悬停或聚焦当前块时浮现；普通标题、段落、引用、列表可直接在渲染块中编辑文字，复杂块仍可切换块源码。

响应式约束：
- 工作台根布局统一使用 `min-h-0 / min-w-0 / overflow-hidden`，内容区使用独立滚动容器，避免弹窗中出现“编辑器只有很小高度、无法滚动”的问题。

- 文件预览弹窗的拖拽尺寸会按当前视口收敛，手机端不会被桌面最小宽度 720px 强行撑爆。
- 图片预览内置缩放/适应按钮，缩放后走滚动容器，不让图片溢出整个弹窗。
- 设计方向参考 Milkdown/Moraya/思源这类“渲染即编辑、块工具按需出现”的产品，但当前不引入大型编辑器依赖，先用现有 Markdown 渲染和轻量 contentEditable 保持项目体积可控。

### FileManager 上传任务快照与待恢复入口（2026-06-26）

独立 FileManager 已接入与 Workspace 共享的上传任务快照工具：实时上传任务关闭弹窗后仍显示底部任务条；刷新页面后，如果存在未清理的上传任务快照，会显示“待恢复上传”，提示用户重新选择同一文件以继续后端分片 checkpoint。快照只保存文件名、状态、大小、目标路径和更新时间，不保存文件内容，也不把 localStorage 视为可靠任务事实源。

实现边界：

- `uploadTaskSnapshots.ts` 负责 Workspace 与 FileManager 共享的快照读写和 schema 校验。
- Workspace 与 FileManager 使用不同 localStorage key，避免两个域互相污染。
- 上传续传事实仍来自 `/api/files/uploads/:uploadId` 和分片 checkpoint；快照只是 UX 提示。
- 清空上传列表会同时清理当前域快照。
- 后续如需要跨页面全局任务中心，应升级为 Files 域任务服务，而不是继续扩大 localStorage 职责。

### 分片续传调度去重（2026-06-26）

真实浏览器续传 smoke 发现：当后端 checkpoint 表示 chunk 0 已上传、chunk 1 缺失时，多个并发 worker 可能在 chunk 1 完成前同时选择同一缺失分片，导致重复 PUT 同一 chunk。上传管理器已增加 `scheduledChunks` in-flight reservation，缺失分片在派发前先进入保留集合，其他 worker 会跳过已上传和已调度分片。验证方式是在浏览器中预先创建 uploadId、写入 chunk 0 和 localStorage checkpoint，再运行 `createUploadBatch` 重新选择同一文件；网络捕获必须只出现一次 `PUT /chunks/1`，完成后 checkpoint 被清理，目标文件大小正确。

### 上传重复文件名冲突策略验证（2026-06-26）

上传系统的同名文件处理固定为“显式策略 + 安全默认”：上传弹窗展示保留两者、覆盖、跳过、冲突时报错四种策略；资源管理器焦点快捷粘贴和拖拽等快速入口默认使用“保留两者”，自动生成 `name (1).ext`，避免用户无感覆盖现有文件。

行为合同：

- `skip`：目标已存在时不上传分片，不改写原文件，任务标记为跳过。
- `rename`：目标已存在时选择下一个可用后缀路径，保留原文件和新文件。
- `overwrite`：只有用户在弹窗中显式选择后才覆盖目标文件。
- `fail`：严格模式，目标已存在直接失败。

该策略参考 Google Drive 的同名上传保留两者能力、Windows Explorer 的 replace/skip/keep both 冲突模型，以及云存储 API 常见的 autorename/write mode 语义，但实现保持 Tracevane Files API 的轻量分片上传协议，不引入新的上传 SDK。

验证记录：Playwright 在真实浏览器中打开 `/#/file-manager`，在临时目录预置 `conflict.txt=original` 后分别调用上传管理器执行 `skip`、`rename`、`overwrite`。结果：`skip` 任务状态为 `skipped` 且原文件内容仍为 `original`；`rename` 生成 `conflict (1).txt` 且内容为新上传文本；`overwrite` 显式覆盖 `conflict.txt` 为新内容；网络捕获只出现两次 chunk PUT（rename 与 overwrite），证明 skip 没有上传分片。

### 多格式预览真实浏览器验证（2026-06-26）

文件弹窗预览继续采用“同一个文件上下文、多种视图模式”的模型：Markdown/HTML 默认进入预览，可切换源码、边写边预览和预览时编辑；JSON/CSV 进入源码后仍可切换结构化预览；图片、PDF、媒体通过安全下载 URL 在弹窗内部预览，不把二进制内容注入主页面。

- 路径栏、面包屑和搜索保持可用，但隐藏桌面说明、常用/最近路径和列配置，避免首屏被工具栏占满。
- 列表模式在小屏切换为三列行布局：选择框、文件名+类型/大小/修改时间、操作菜单；桌面端继续使用可调整列宽表格。
- 列表滚动容器禁止横向滚动，横向信息通过行内元信息折行展示，避免移动端出现表格横向滚动条。

后续移动端继续优化方向：把高级文件搜索改为可折叠面板/抽屉，批量操作在选择后浮动到底部，减少文件列表上方的固定高度。

## 移动端搜索与批量操作收敛（2026-06-26）

移动端不再把桌面级搜索选项和批量按钮铺在文件列表上方：

- 文件名/内容搜索保留输入框，但递归、大小写、正则和索引状态在小屏收进“搜索选项” disclosure；桌面端继续展开。
- 选中文件后，桌面端使用原批量操作条；移动端使用底部 action sheet，包含选择摘要、取消选择、复制、移动、下载、打包、权限、删除等动作。
- 删除在移动端 action sheet 中保持红色危险语义；真正删除仍必须进入强确认弹窗，不允许从 sheet 直接静默删除。
- 当前不新增底部抽屉库，先以固定底部面板完成清爽度和可达性；后续如需要拖拽手势再评估轻量 headless sheet。

## 移动端首屏信息密度（2026-06-26）

移动端文件管理器继续以“尽快看到文件列表”为原则：

- 统计区移动端只展示总数、目录/文件和已选摘要；完整目录、文件、隐藏、已选、已加载 chips 仅桌面展开。
- 操作历史移动端默认折叠，只显示最近一次操作状态；展开后才显示筛选、搜索、导出和记录列表。
- 当移动端存在选中项时，文件列表区域为底部 action sheet 预留底部空间，避免最后几行被固定操作栏遮挡。

后续仍应把内容索引、回收站等管理型视图按同样原则做移动端渐进披露，避免把桌面控制台直接塞到手机屏幕。

## 移动端预览弹窗和媒体舞台（2026-06-26）

文件预览/编辑弹窗在移动端采用全屏工作台，而不是桌面小窗口缩放：

- Dialog 内容高度使用 `100dvh`，内部只允许内容区滚动，避免弹窗自身和页面同时滚动。
- 顶部文件操作工具栏、文档工作台模式切换、媒体缩放工具栏在小屏改为横向滚动，不再多行堆叠挤压编辑器高度。
- 图片、视频默认居中并以 `object-fit: contain` 适配当前媒体视口；用户放大后由媒体滚动区承接横向/纵向溢出。
- 状态栏移动端横向滚动，桌面端可换行展示更多信息。

后续需要继续增强：图片双击/手势缩放、视频全屏/画中画能力、PDF 移动端降级打开策略，以及 Markdown/HTML 可视编辑的细粒度块编辑体验。
