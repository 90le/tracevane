# Tracevane Frontend Functional Architecture

> Status: active baseline
> Updated: 2026-06-25

## 1. Current stack

The active frontend is `apps/web`:

- React
- Vite
- Tailwind v4 / Aurora tokens
- shadcn/Radix-style local UI primitives
- React Router hash routing
- TanStack Query typed API hooks
- Feature-sliced pages under `apps/web/src/features/*`

The former Vue/raw-prototype frontend and raw prototype rendering are retired.

## 2. Frontend product rule

Each route must have one clear owner job. Cross-domain read-only summaries are allowed; cross-domain writes are not.

| Page | Primary job | Should not do |
| --- | --- | --- |
| Model Gateway | Configure and verify model/provider/protocol/client routing. | Show CLI process runtime as if Gateway owns it. |
| IM Channels | Configure IM platforms/bots/bindings and inspect IM delivery/session state. | Hide missing credential CRUD behind read-only status cards. |
| CLI Agents | Observe Codex/Claude Code/OpenCode runtime, CLI readiness, Agent Runs and evidence. | Become OpenClaw generic agent/persona admin or a terminal manager. |
| Chat | Local chat session/history/runtime controls. | Replace CLI Agents runtime workbench. |
| IDE / Files | Workspace edit, preview, terminal, Git and file evidence. | Become generic file manager without Agent context. |
| Platform | OpenClaw substrate support, platform guard/recovery, integration evidence and owner-domain links. Legacy `/recovery` redirects here. | Own daily model/IM/CLI workflows or generic OpenClaw CRUD. |
| External / Task Supervision deep-links | Compatibility/read-only drilldowns for integration evidence and supervised work. | Appear as primary navigation domains or duplicate owner-domain writes. |

## 3. Current navigation intent

Main workbench groups:

1. Overview / Dashboard — high-level state only.
2. Model Gateway — high-completion model routing surface.
3. IM Channels — must evolve from read-only to full IM setup and binding control.
4. CLI Agents — must be refocused around real CLI runtime.
5. Chat / IDE / Files — local work and evidence.
6. Platform — support, diagnostics, OpenClaw substrate guard and integration evidence.

`/external`, `/long-tasks`, and legacy `/recovery` are no longer first-class sidebar entries. They may remain routed compatibility/deep-link surfaces, but the primary navigation must direct users to Model Gateway, IM Channels, CLI Agents, IDE/Chat and Platform.

## 4. Model Gateway frontend

Current state is relatively strong. Keep:

- `overview`: route cockpit, provider health, client config risk.
- `providers` / `providercfg`: provider list and configuration.
- `models`: model catalog and budgets.
- `accounts`: account-style provider pool.
- `apps`: Codex/Claude Code/OpenCode/OpenClaw client config write/backup/rollback.
- `usage`: request/token/cache evidence with honest wording.

Do not reintroduce “Agent Cockpit” wording. Gateway owns routes, not process runtime.

## 5. IM Channels frontend target

Current problem: too much of the page is observational. The target is a real setup and operations surface:

- Local views: 概览 / 平台账号 / 绑定路由 / 会话投递 / 守护诊断.
- Platform account rows with account Drawer forms, platform-specific fields, redacted-secret warning, and verified smoke actions.
- Binding route rows with route Drawer forms, Agent profile selection, allowlist/admin/commands, and match/delivery test where supported.
- Runtime session and delivery rows with failure categories and evidence Sheets.
- Daemon/service Status Console with generated config evidence collapsed.
- Agent profiles and daemon native bindings are related evidence/selectors, not equal first-screen panels.

See `IM渠道目标与设计.md` and `IM渠道前端设计契约.md`.

## 6. CLI Agents frontend target

Current problem: it carried OpenClaw/persona and terminal-management concepts too prominently. The target is a real CLI runtime workbench:

2026-06-24 refinement: CLI Agents must not duplicate Gateway or IM configuration. It owns CLI readiness, launch handoff, Agent Runs and proven Agent terminal controls only; Gateway/IM appear as dependency links.


- Codex / Claude Code / OpenCode install and version readiness.
- Gateway connection readiness per CLI.
- Agent Runs as the primary running-work list.
- Terminal-sourced runs link back to IDE; generic terminal launch/input/delete stays in IDE.
- Runtime evidence and source links.
- OpenClaw only as substrate/platform context, not the main subject.

See `CLI代理目标与设计.md`.

## 7. External evidence and task supervision

- External evidence is not “接入” unless the page can actually create, authorize, test and delete integrations. Until then, name it as evidence/capability and link writes to owner domains.
- Long-running work is not a separate product owner. The primary Agent Run surface is CLI Agents; IM delivery/session controls remain in IM Channels; OpenClaw substrate guard controls live in Platform.
- A cross-domain supervision view may exist as a deep-link for active/waiting/attention triage, but it must not become another duplicate runtime center.

## 8. Platform frontend target

Platform is the low-frequency substrate/support workbench, not a fourth daily operations domain. Target sections:

- Overview: OpenClaw runtime identity, model gateway health, IM daemon health and integration evidence summaries.
- OpenClaw: official Control UI link-out, runtime/version/host diagnostics, masked permission evidence and owner-domain handoffs.
- Platform Guard (`/platforms/openclaw/recovery`): OpenClaw gateway probe, repair events, config backups and guarded daemon lifecycle. Legacy `/recovery` redirects here.
- Integration Evidence (`/external` compatibility): MCP / Skills / tool evidence until real create/test/delete contracts exist.

Platform must not duplicate Model Gateway provider writes, IM account/binding writes, CLI Agent Runs, terminal controls or official OpenClaw generic CRUD.


## 9. UI quality rules

- Empty states must explain what action is missing.
- “Test passed” must mean the real owning backend action succeeded.
- Dangerous writes need confirm dialogs and result evidence.
- Pages should not duplicate the same long lists across overview/detail tabs.
- If a capability is not implemented, say so directly instead of presenting it as complete.
