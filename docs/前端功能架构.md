# Tracevane Frontend Functional Architecture

> Status: active baseline
> Updated: 2026-06-24

## 1. Current stack

The active frontend is `apps/web`:

- React
- Vite
- Tailwind v4 / Aurora tokens
- shadcn/Radix-style local UI primitives
- React Router hash routing
- TanStack Query typed API hooks
- Feature-sliced pages under `apps/web/src/features/*`

The former Vue/raw-prototype frontend and raw prototype rendering are retired.

## 2. Frontend product rule

Each route must have one clear owner job. Cross-domain read-only summaries are allowed; cross-domain writes are not.

| Page | Primary job | Should not do |
| --- | --- | --- |
| Model Gateway | Configure and verify model/provider/protocol/client routing. | Show CLI process runtime as if Gateway owns it. |
| IM Channels | Configure IM platforms/bots/bindings and inspect IM delivery/session state. | Hide missing credential CRUD behind read-only status cards. |
| CLI Agents | Observe Codex/Claude Code/OpenCode runtime, CLI readiness, Agent Runs and evidence. | Become OpenClaw generic agent/persona admin or a terminal manager. |
| Chat | Local chat session/history/runtime controls. | Replace CLI Agents runtime workbench. |
| IDE / Files | Workspace edit, preview, terminal, Git and file evidence. | Become generic file manager without Agent context. |
| Recovery / Platforms | Service diagnostics, OpenClaw substrate support and repair. | Own daily model/IM/CLI workflows. |

## 3. Current navigation intent

Main workbench groups:

1. Overview / Dashboard — high-level state only.
2. Model Gateway — high-completion model routing surface.
3. IM Channels — must evolve from read-only to full IM setup and binding control.
4. CLI Agents — must be refocused around real CLI runtime.
5. Chat / IDE / Files — local work and evidence.
6. Recovery / Platforms / External — support, diagnostics and integrations.

## 4. Model Gateway frontend

Current state is relatively strong. Keep:

- `overview`: route cockpit, provider health, client config risk.
- `providers` / `providercfg`: provider list and configuration.
- `models`: model catalog and budgets.
- `accounts`: account-style provider pool.
- `apps`: Codex/Claude Code/OpenCode/OpenClaw client config write/backup/rollback.
- `usage`: request/token/cache evidence with honest wording.

Do not reintroduce “Agent Cockpit” wording. Gateway owns routes, not process runtime.

## 5. IM Channels frontend target

Current problem: too much of the page is observational. The target is a real setup and operations surface:

- Platform account list.
- New/edit/delete account forms.
- Platform-specific credential templates.
- Binding list with new/edit/delete.
- Binding test and delivery smoke.
- IM session list and session control where supported.
- Delivery logs and clear failure reasons.
- Daemon/service control with evidence.

See `IM渠道目标与设计.md`.

## 6. CLI Agents frontend target

Current problem: it carried OpenClaw/persona and terminal-management concepts too prominently. The target is a real CLI runtime workbench:

- Codex / Claude Code / OpenCode install and version readiness.
- Gateway connection readiness per CLI.
- Agent Runs as the primary running-work list.
- Terminal-sourced runs link back to IDE; generic terminal launch/input/delete stays in IDE.
- Runtime evidence and source links.
- OpenClaw only as substrate/platform context, not the main subject.

See `CLI代理目标与设计.md`.

## 7. UI quality rules

- Empty states must explain what action is missing.
- “Test passed” must mean the real owning backend action succeeded.
- Dangerous writes need confirm dialogs and result evidence.
- Pages should not duplicate the same long lists across overview/detail tabs.
- If a capability is not implemented, say so directly instead of presenting it as complete.
