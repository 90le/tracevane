# Tracevane Frontend Functional Architecture

> Status: active baseline
> Updated: 2026-06-25

## 1. Current stack

The active frontend is `apps/web`:

- React
- Vite
- Tailwind v4 / Aurora-derived tokens
- local shadcn/Radix-style primitives
- React Router hash routing
- TanStack Query typed API hooks
- feature-sliced pages

## 2. Frontend product rule

Each route must have one clear owner job. Workspace is now the owner of local project work: files, editor, preview, terminal, Git and evidence.

| Page | Primary job | Must not do |
| --- | --- | --- |
| Workspace | Local project work: files, editor, preview, terminal, Git, evidence, Agent handoff. | Own CLI Agent runtime readiness, Provider secrets, IM bot config or OpenClaw platform CRUD. |
| Model Gateway | Configure and verify model/provider/protocol/client routing. | Show terminal or Agent process lifecycle as if Gateway owns it. |
| IM Channels | Configure IM platforms/bots/bindings and inspect delivery/session state. | Hide missing credential CRUD behind read-only status cards. |
| CLI Agents | Observe Codex/Claude Code/OpenCode readiness, config state and Agent Runs. | Manage normal terminal sessions or file editor state. |
| Chat | Local chat and context capture, with handoff to Workspace. | Replace Workspace or CLI Agents. |
| Platform | Third-party platform directory and platform-specific workspace. | Own Workspace files/terminal, Gateway provider writes or IM routing writes. |
| Files / Artifacts | If kept, evidence/artifact browser only. | Pretend to be a full file manager while read-only. |

## 3. Navigation intent

Main groups should become:

1. Overview
2. Workspace
3. Runtime
4. Connections
5. Evidence
6. System

Recommended top-level entries:

- Dashboard
- Workspace
- CLI Agents
- Chat
- Model Gateway
- IM Channels
- Artifacts / Evidence Files
- Approvals
- Platform

`/ide` may redirect to `/workspace` during migration. `/files` should either redirect to Workspace file manager or be renamed to Evidence Files/Artifacts.

## 4. Workspace frontend architecture

See `Workspace前端架构.md`.

Key implementation rules:

- Use one Workspace store/context.
- Desktop/tablet/mobile layouts are separate render shells over shared state.
- File actions use one command registry across toolbar/context menu/command palette/mobile sheet.
- Terminal panel cannot import CLI Agent runtime controls.
- CLI Agents may link to Workspace evidence, but must not manage terminal UI.

## 5. Responsive requirements

Every new Workspace route must verify:

- 1440×900 desktop no overflow.
- 1024 tablet usable with collapsible panels.
- 390×844 mobile no horizontal overflow.
- Touch alternatives for every right-click/hover action.
- Terminal mobile input toolbar.
- Full-screen editor/terminal/diff on mobile.

## 6. UI quality rules

- No visible placeholder tabs for unimplemented capabilities.
- Empty states must provide real next action.
- Dangerous writes need confirm and result evidence.
- If a capability is unsupported, say so in the owning surface, not as a permanent cross-page excuse.
- Tables/logs/code must break or scroll inside their own container, never the page.
