# Studio 外部访问与单端口说明

## 文档目的

这份文档说明截至 2026-04-08，Studio 对外访问的**当前支持边界**和**单口方向**。

它不是客户安装说明。客户安装请直接使用：[DEPLOY.md](/home/binbin/.openclaw/extensions/openclaw-studio/DEPLOY.md)

## 当前结论

### 正式支持

1. `standalone` 模式
2. Studio 独立监听 `3760`
3. 客户直接访问 `3760`，或者由客户自有反代做“根路径转发到 3760”

### 单口目标的准确定义

1. 外网安全组只开放 `OpenClaw` 一个端口
2. 内网 / 本机回环不受限制
3. `3760` 可以继续在设备内部保留
4. 目标是让外部用户通过 `OpenClaw` 这一个端口，获得接近直接访问 `3760` 的体验

### 核心链路已完成，但还不能正式交付

1. `Gateway-native` 单端口模式
2. 页面和 HTTP API 已可挂到 OpenClaw Gateway
3. `chat` / `terminal` 的 realtime 已切到扩展侧 `gateway-rpc`
4. 当前剩余问题是回归和客户环境验证，不再是 transport 缺口

因此，**当前仍不能直接对客户承诺“只开放 18789 一个端口，就已经是正式支持模式”**。

## `pack.sh` 当前行为

当前 `pack.sh` 已移除旧的：

```bash
--base-path /x/studio/
```

原因：

1. 旧方案把部署路径写死在构建产物中
2. 当前代码已经改为运行时 transport 配置
3. 路径和暴露方式应该由运行时配置决定，而不是由打包参数决定

## 为什么旧 `--base-path` 方案不再继续保留

旧方案的问题是：

1. 前端页面、API、WebSocket 不是同一套协议
2. 打包脚本用 `sed` 注入 HTML，职责混乱
3. 后端和反代层都可能各自处理前缀，容易重复裁剪
4. 测试脚本 `test-base-path.sh` 不能真实覆盖浏览器链路，已经移除

所以当前已经明确收口为：

1. 正式交付继续用 `standalone :3760`
2. 单口正式方向继续做，但不再依赖宿主私有补丁

## 客户只有一个外部端口怎么办

分两种情况：

### 情况 1：客户可以自己做反代

如果客户自己能做外部入口转发，无论是 Nginx、Caddy、LB，还是其他同类设施，都可以把外部端口转到内部 `3760`。

这仍然属于当前正式支持方案。

### 情况 2：客户完全不能用外部反代，只能依赖 OpenClaw 自己

这正是当前在做的单口方向。

当前正确方向不是改宿主补私有 WebSocket 接口，而是：

1. 页面和 HTTP API 继续挂到 `/studio/*`
2. realtime 改为复用宿主已有 Gateway WS 协议
3. 插件通过 `registerGatewayMethod()` 暴露 `studio.chat.*` / `studio.terminal.*`

但截至 2026-04-08，它已经完成了：

1. Studio 页面挂载
2. HTTP API 挂载
3. `terminal` 的扩展侧 `gateway-rpc` realtime
4. `chat` 的扩展侧 `gateway-rpc` realtime

因此，这个方向**已经跑通核心链路，但还不能作为正式客户方案交付**。

## 对客户的统一口径

截至 2026-04-08：

1. 正式支持：`OpenClaw >= 2026.3.23` + `standalone :3760`
2. 不正式支持：`OpenClaw 2026.3.13`
3. 不正式支持：旧 `--base-path /x/studio/` 打包方案
4. 不正式支持：`Gateway 单端口完整可用` 的客户承诺

## 长期推荐方向

从长期稳定性看，最好的方案是：

1. 对外统一只走 `OpenClaw` Gateway 一个端口
2. Studio 仅依赖宿主公开能力：
   - `registerHttpRoute()`
   - `registerGatewayMethod()`
   - 宿主已有 Gateway WS `req/res + event`
3. `3760` 只作为本机 / 内网维护与回退入口保留

这条路比私有宿主补丁更适合长期使用。
