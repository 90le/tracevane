# OpenClaw Studio Chat Session Policy

> 状态：Active
> 更新：2026-06-12

## 1. 设计原则

- Studio 只默认写入自己创建和管理的会话。
- 外部会话可以观察，但默认不可写。
- 前端不暴露内部 session key 策略。
- 投递行为必须由后端权限和 session kind 决定。

## 2. Session Classification

| Kind | 用途 | 写入规则 |
| --- | --- | --- |
| `studio_managed` | Studio 创建的安全 Chat 会话 | 可写，可发送/停止/重置/删除 |
| `observed_external` | 外部 Gateway/OpenClaw 会话 | 默认只读，除非后端显式授权 |
| `system_internal` | cron、heartbeat、internal run | 不在普通用户 rail 显示 |

## 3. Writable Rules

- 只有 `studio_managed` 默认 `canSend=true`。
- `observed_external` 不允许前端直接注入消息。
- `system_internal` 不允许用户操作。
- delete/reset/abort 需要后端给出明确权限。

## 4. Session Metadata

前端可展示：

- label / derivedTitle
- agentId / agent display name
- lastMessagePreview
- updatedAt
- runtime state
- source summary

前端不依赖：

- Gateway raw session payload。
- provider 私有 run id。
- OpenClaw 内部 transport key。
- 未脱敏 delivery context。

## 5. Delivery Rules

- Studio Chat 默认只投递到 Studio-managed runtime。
- 不把生产 IM/channel 会话默认作为 Chat 写入目标。
- 未来如果开放外部会话写入，必须增加显式授权、审计和回滚。

## 6. 安全边界

- 浏览器只访问 same-origin Studio API。
- Gateway key、upstream key、channel token 都由后端代持。
- CORS、token 和 local loopback 规则由 API 层控制。
- 前端不能直接调用 Gateway daemon 或 Channel daemon 管理端口。

## 7. 当前结论

Chat session policy 服务单人 Studio 工作台。多人/群聊和跨端持久 cursor 是未来独立产品层，不进入当前 Chat MVP。
