# OpenClaw Studio Chat Session Policy

> 状态：Active
> 更新：2026-06-01

本文档定义 `Studio` chat 的 session classification、可写规则、默认安全策略和 same-origin 边界。

## 1. 设计原则

### 直接证据

- 本轮 spike 已验证 `agent:main:webchat:direct:studio-spike-*` 可稳定创建、发送、读取、重置、删除
- 本轮也已验证：
  - `deliver: false` 能工作
  - 但未对真实生产绑定 session 做危险实测
- 当前 `apps/api/server.ts` 已对 chat API path 做 same-origin 检查
- 当前 `apps/api/core/http.ts` 仍保留通用 CORS helper，非 chat 写路径后续还需要继续收口

### 最终决策

- 第一层保险必须是 session write policy
- 第二层保险才是 `deliver: false`
- 生产 session 默认只读观察，不属于 MVP 默认写面

## 2. Session Classification

## 2.1 `studio_managed`

来源：

- 由 `Studio` 主动创建
- 使用 `Studio` 自管 key/policy/helper
- 当前推荐 key 形态：
  - `agent:<agentId>:webchat:direct:studio-<opaqueId>`

默认策略：

- 默认可写：是
- 允许 `send`：是
- 允许 `abort`：是
- 允许 `reset`：是
- 允许 `delete`：是
- 允许 `inject`：仅后端诊断/维护使用，不进入 MVP UI
- 允许前端展示：是
- 允许进入 MVP rail：是

## 2.2 `observed_external`

来源：

- 现有 Gateway session
- 非 `Studio` 自管
- 例如：
  - `agent:main:main`
  - 真实 channel direct/group/channel session

默认策略：

- 默认可写：否
- 允许 `send`：否
- 允许 `abort`：否
- 允许 `reset`：否
- 允许 `delete`：否
- 允许 `inject`：否
- 允许前端展示：是，但应标记为只读观察
- 允许进入 MVP rail：否

## 2.3 `system_internal`

来源：

- heartbeat
- cron internal run
- system/internal session
- 未来内部 orchestration session

默认策略：

- 默认可写：否
- 允许 `send`：否
- 允许 `abort`：否
- 允许 `reset`：否
- 允许 `delete`：否
- 允许 `inject`：否
- 允许前端展示：默认否，仅在 diagnostics 场景可见
- 允许进入 MVP rail：否

## 3. Writable Rules

### 已冻结规则

- MVP 默认只允许写 `studio_managed`
- `observed_external` 默认只读
- 生产 session 不属于 MVP 默认写范围
- `deliver: false` 只是第二层保险
- 第一层保险必须是 session write policy

### 解释

为什么不能只靠 `deliver: false`：

- 它只能防止某些 delivery 行为
- 但不能替代“这个 session 根本不该被写”
- 一旦对外部 session 允许写入，即使 `deliver: false`，仍可能带来：
  - 上下文污染
  - session reset 误伤
  - internal event / abort 误操作

因此：

- “可写”必须先由 policy 决定
- `deliver: false` 只在允许写的 session 上作为额外保险

## 4. Session Key / Metadata Policy

## 4.1 哪些属于内部实现细节

前端不允许依赖：

- `sessionKey` 前缀规则
- `webchat:direct:studio-*` 的命名细节
- raw `lastChannel / lastTo / origin / deliveryContext` 的 Gateway 推断方式

这些都属于后端 helper / policy 层的实现细节。

## 4.2 前端应该拿到的 metadata

前端至少应拿到：

- `key`
- `agentId`
- `sessionId`
- `kind`
- `label`
- `derivedTitle`
- `lastMessagePreview`
- `updatedAt`
- `source`
  - `source`
  - `channel`
  - `surface`
  - `originLabel`
- `deliveryContext`
  - `channel`
  - `accountId`
  - `to`
  - `threadId`
- `permissions`
  - `writable`
  - `canSend`
  - `canAbort`
  - `canReset`
  - `canDelete`
  - `canInject`
  - `visibleInFrontend`
  - `visibleInMvpRail`

补充语义：

- `sessionId` 继续保留在前端 metadata 中
- 对 `studio_managed` 来说：
  - draft 创建时由 `Studio` 先分配本地 `sessionId`
  - registry 恢复时沿用该值
  - Gateway 后续返回更权威的 `sessionId` 时，后端会回填并覆盖 registry
- 前端不需要区分这个 `sessionId` 的来源，只把它当作会话标识字段

## 4.3 Label / Title Policy

- `label`
  - 由 `Studio` 后端统一写入或管理
- `derivedTitle`
  - 由后端读取 Gateway/history 后归一化生成
- 前端不应自行把 `sessionKey` 格式转换成标题

## 5. Delivery Rules

### 默认安全规则

- `studio_managed`
  - 默认 `deliver: false`
- `observed_external`
  - 默认只读，不允许 send，因此不存在默认 delivery
- `system_internal`
  - 默认只读，不允许 send

### MVP 结论

- MVP 不允许把生产 session 设为默认可写
- MVP 不允许把“观察到的外部 session”升级成“顺手可发消息”

## 6. Chat 安全边界

### 6.1 same-origin 规则

必须 same-origin 的路径：

- `/api/chat/*`
- `/ws/chat`

推荐后续逐步扩展到全部 `Studio` 写路径。

### 6.2 `CORS *` 收口方向

当前状态：

- chat HTTP path 已在 request handler 中识别并执行 origin 检查
- 通用 Studio HTTP helper 仍保留宽松 CORS 能力，用于开发和非 chat 路径兼容

建议实现方向：

- 生产环境：
  - chat HTTP/WS 全部 same-origin
- 开发环境：
  - 允许 Vite 本地开发门面
  - 推荐仅限：
    - `http://127.0.0.1:5176`
    - `http://localhost:5176`
  - 且最好通过 Vite dev middleware/proxy 同源代理，而不是任意 origin 放开

### 6.3 为什么浏览器不能直连 Gateway

- Gateway token 不应暴露给浏览器
- 浏览器不应处理 device auth / challenge / raw frame
- 前端不应直接绑定 Gateway raw protocol
- 否则 `Studio` 将失去 typed contract 与 future activity layer 的抽象能力

### 6.4 为什么 `Studio` 后端必须代持 Gateway token

- 当前推荐方案就是 `Studio backend typed adapter / BFF`
- 只有后端代持 token，才能：
  - 收口安全边界
  - 统一错误映射
  - 统一 session write policy
  - 屏蔽 Gateway raw frame

## 7. 当前冻结结论

- `studio_managed / observed_external / system_internal` 已正式定义
- MVP 默认只允许写 `studio_managed`
- `deliver: false` 明确只是第二层保险
- 生产 session 默认只读观察
- `/api/chat/*` 与 `/ws/chat` 必须持续保持 same-origin 边界
