# OpenClaw Studio Architecture

> 更新：2026-04-10

## 1. 当前架构结论

Studio 采用“插件壳 + 模块化 API + Vue 管理工作台”结构。

`chat` 重新纳入后仍必须遵守这套结构，不能引入平行的 React/FastAPI/数据库子系统。
同时，`chat` 必须作为未来 `event/activity`、`room`、`project/workflow`、`community` 的底层之一，而不是把这些能力提前塞进当前 MVP。

关键原则：

- 每个管理域独立目录
- 共享类型位于 `types/`
- 后端以 `apps/api/modules/<domain>` 为边界
- 前端以 `apps/web-vue/src/features/<domain>` 为边界
- 路由、壳层和公共组件集中在基础层

## 3. 数据源映射

- `config`：`openclaw.json`、`exec-approvals.json`
- `skills`：`~/.openclaw/skills`、`openclaw skills` CLI、技能市场
- `terminal`：PTY、WebSocket、CLI 探测
- `channels`：`openclaw.json`、`credentials/*`、pairing 文件
- `cron`：`~/.openclaw/cron/jobs.json`、`cron/runs/*.jsonl`
- `agents`：`openclaw.json`、`agents/<id>/sessions/`、workspace docs
- `system`：本地进程与运行时诊断
- `dashboard`：聚合以上模块摘要
- `release metadata`：源码 `package.json`、站点 `version.json` / `openclaw-studio-latest.json`、installer fallback 常量
- `chat`：
  - 主数据源：Gateway `chat.*`、`sessions.*`、`agent` 事件
  - 诊断 / fallback：`~/.openclaw/agents/<id>/sessions/sessions.json`、`*.jsonl`
  - 调试 / 手工 fallback：`openclaw gateway call <method>` 仅限诊断，不作为正式 runtime transport

## 6.1 Chat 分层位置

长期分层建议：

1. `Chat Core`
   - direct session
   - history / send / abort / reset
2. `Event / Activity Layer`
   - timeline
   - run lifecycle
   - agent output
3. `Collaboration Surface`
   - room
   - plaza / feed / posts / comments / articles
4. `Project / Workflow Layer`
   - project entity
   - workflow / checkpoints / deliverables

补充说明：

- `approval / review / audit` 应作为跨层治理平面，而不是 chat 子层
- 当前 chat 阶段只允许落 `Chat Core`
- 只在边界上为 `Event / Activity Layer` 预留抽象

## 6. 关键实现策略

- 列表与详情优先直接读真实状态；chat 列表与收流优先走 Gateway，文件只做 fallback
- 写操作优先修改官方实际状态结构，不发明平行存储
- 对 CLI 依赖不稳定的能力做 graceful fallback
- 高风险输入优先界面化选择，不鼓励自由手填
- `chat` 优先以 Gateway 为 source of truth，本地 session 文件只做诊断与 fallback
- `chat` 不直接复用 `Clawith` 的数据库会话模型，只复用交互语义和页面经验
- `chat` 默认使用 `Studio` 自管的安全会话命名空间，不默认写入生产频道会话
- `chat` 正式 runtime transport 采用 `Studio backend typed adapter / BFF`
- `chat` 对 `studio_managed` session 采用“单 session 持久 Gateway bridge”：
  - 同一条逻辑链路负责 `chat.send`
  - 同一条逻辑链路负责 `chat` 事件
  - 同一条逻辑链路负责 `agent` 事件与 `tool-events` capability
- `chat` 对“运行中继续发送”采用 session 级内存队列：
  - 队列只存在于 Studio 进程内存，不写盘
  - 刷新保留，Studio 重启清空
  - 当前 run 结束后按 FIFO 自动 flush
- `chat` 对宿主管理类 `exec/shell/bash` 采用双层显式授权：
  - 全局配置：`plugins.entries.studio.config.chat.allowHostManagementExecInStudioChat`
  - session 临时控制：`allowHostManagementExec`
  - 两层都满足才允许放行宿主管理命令
- `chat` 的 slash 命令采用“本地 / hybrid / send”三级分发：
  - 本地 / hybrid 命令统一走 `Studio` 后端 `slash-gateway`，不依赖浏览器实时 WebSocket 是否已连接
  - `Studio Chat /allowlist` 默认映射为当前 Agent 的 exec allowlist 管理，直接使用 `exec.approvals.*`
  - 频道 surface 才有意义的 `/allowlist` 参数仍回退宿主原生命令，避免在 Studio 直连会话中伪造错误语义
  - 当前 slash 目录里需要保留宿主原生命令语义的命令，不再走前端 realtime-only `send`，而是通过 `slash-gateway -> chat.send`
  - 现阶段已覆盖 `/btw / acp / debug` 这类宿主差异较大的命令，保持“由宿主执行、由 Studio 负责稳定提交”的边界
- 前端 contract 必须是 `Studio` 自己的 typed contract，不允许直接绑定 Gateway raw protocol
- `deliver: false` 只能作为第二层保险，第一层保险必须是 secure session policy
- `version / installer / landing page` 采用同一条发布元数据链：
  - 源码 `package.json.version`
  - 源码 `package.json.openclaw.install.minHostVersion`
  - `pack.sh` 生成的 `version.json / studio-version.json / openclaw-studio-latest.json`
  - installer 优先读取 bundled `package.json` 或远端 release metadata
  - landing page 优先读取同站点 `version.json`

## 7. 并行边界

允许并行：

- `types/<domain>.ts`
- `apps/api/modules/<domain>`
- `apps/web-vue/src/features/<domain>`

不要跨域改动：

- 非本域 `modules/`
- 非本域 `features/`
- 顶层壳层和全局样式，除非明确做基础层收口

## 8. 当前已知问题

- `studio` 插件仍有 plugin id mismatch warning
- 某些官方 CLI 到 Gateway 的 ws 通道偶发关闭，需要页面层 graceful fallback
- `openclaw` npm 导出面当前未公开 Gateway client helper；`chat` 接入需要由 `Studio backend typed adapter / BFF` 自己承接 Gateway WS
- `openclaw gateway call` 在当前环境中不稳定，不能作为正式 chat runtime 主链路
- `apps/api/core/http.ts` 当前仍是 `CORS *`，chat 接入前必须对 chat HTTP/WS 至少收口到 same-origin
- fresh live `agent_tool_call / agent_tool_result` 已通过 `Studio /ws/chat` 同链路验证
- 当前 usage summary 以 history-derived assistant usage 为主，live final.usage 只作为补充
