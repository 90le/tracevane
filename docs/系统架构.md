# OpenClaw Studio System Architecture

> 状态：Active
> 更新：2026-06-01

本文档记录 `openclaw-studio` 当前代码结构和运行边界。它以仓库现状为准，而不是早期恢复阶段的规划稿。

## 1. 项目定位

OpenClaw Studio 是 OpenClaw 插件系统里的 UI 扩展。它不是独立服务产品，而是本地管理控制台：

- 以 OpenClaw 原生配置、session、agent、channel、skill、cron 和 gateway 能力为真相
- 通过 Studio API 和 Vue 工作台提供可读写管理界面
- 对高风险写操作保持保守、显式和可回滚
- 对 chat 使用 Studio 后端 typed adapter / BFF，不让浏览器直接消费 Gateway raw protocol

## 2. 顶层目录

```text
openclaw-studio/
  apps/
    api/          # Studio 后端 API、服务、路由和 standalone server
    web-vue/      # Vue 3 + Vite 前端工作台
  docs/           # 产品、架构、chat、渲染和开放门槛文档
  lib/            # 插件 hook、chat display、delivery、兼容工具
  resources/      # Codex Stack 等随包资源
  scripts/        # 开发、打包、coverage/inventory、standalone 启动脚本
  tests/          # chat / system / terminal 回归测试
  types/          # 前后端共享类型
  index.ts        # OpenClaw 插件入口
```

## 3. 插件入口

`index.ts` 是 OpenClaw 看到的插件入口，主要负责：

- 声明插件 ID、名称、描述、配置 schema
- 注册 `studio_delivery` 工具
- 注册 `before_prompt_build`、`before_tool_call`、`reply_dispatch` hooks
- 根据配置注册 Gateway HTTP route 和 Gateway methods
- 根据配置启动 standalone Studio HTTP/WS server

暴露模式有两类：

- `standalone`：默认端口 `3760`，适合本地开发和直连调试
- `gateway`：默认 base path `/studio`，复用 OpenClaw Gateway 单端口入口

## 4. 后端结构

后端入口集中在 `apps/api/index.ts`：

- `createStudioConfig` / `createStandaloneStudioConfig` 解析 OpenClaw root、gateway port、web dist、transport config
- `createStudioContext` 构建所有 domain service
- `createStudioRequestHandler` / `createStudioServer` 创建 HTTP/WS 运行面

核心支撑在 `apps/api/core/`：

- `router.ts`：轻量 HTTP router
- `http.ts`：JSON/text/SSE/CORS/body 解析工具
- `context.ts`：`StudioApiContext` 和 `StudioServices`
- `state.ts`：JSON 文件读写、目录创建、OpenClaw config 读取

领域模块位于 `apps/api/modules/<domain>`：

- `agents`
- `channels`
- `chat`
- `codex-stack`
- `config`
- `cron`
- `dashboard`
- `files`
- `plugins`
- `skills`
- `system`
- `terminal`

每个领域优先保持 `service.ts` + `routes.ts` 的边界。更复杂的领域可以拆出 store、normalizer、view-model 或 adapter 文件。

## 5. 前端结构

前端是 Vue 3 + Vite：

- `apps/web-vue/src/main.ts` 创建应用
- `apps/web-vue/src/App.vue` 是统一 shell，负责 sidebar、topbar、route stage、command palette、移动端 shell
- `apps/web-vue/src/router.ts` 使用 `features/shell/route-manifest.ts` 里的路由表
- `apps/web-vue/src/shared/api.ts` 统一处理 API base、Gateway auth token、请求重试和 GET 去重
- `apps/web-vue/src/views/*View.vue` 是 top-level route wrapper
- `apps/web-vue/src/features/<domain>` 是具体领域 UI

当前主导航领域包括：

- Overview：dashboard、chat
- Operations：skills、files、codex-stack、terminal、system
- Management：agents、channels、cron、config、plugins
- System：dreaming

## 6. Chat 架构

Chat 是 Studio 当前最复杂的运行面。

后端边界：

- `apps/api/modules/chat/service.ts`
- `apps/api/modules/chat/routes.ts`
- `apps/api/modules/chat/session-policy.ts`
- `apps/api/modules/chat/session-bridge-manager.ts`
- `apps/api/modules/chat/gateway-request.ts`
- `apps/api/modules/chat/agent-event-mapper.ts`
- `apps/api/modules/chat/transcript.ts`
- `apps/api/modules/chat/observability.ts`

前端边界：

- `apps/web-vue/src/features/chat/api.ts`
- `apps/web-vue/src/features/chat/session-ref.ts`
- `apps/web-vue/src/features/chat/**`

当前设计原则：

- OpenClaw 原生 session 是真相
- Studio 不自建平行 DB 会话模型
- 浏览器只连接 Studio
- Studio 后端代持 Gateway token
- 前端只消费 Studio typed contract
- `studio_managed` 默认可写，`observed_external` 默认只读

`apps/web-vue/src/features/chat/**` 是当前唯一 Chat 前端目录。后续应保持 API、session helper、Shell、消息渲染和 inspector 在这个目录内演进，避免重新拆出版本化 sibling 目录。

## 7. 状态与持久化

状态来源主要分三类：

- OpenClaw 原生配置和运行状态，例如 `openclaw.json`
- Studio 自管状态，例如 chat organizer、session registry、terminal descriptor、system event log
- 随包资源和 release metadata，例如 Codex Stack resources、dist assets、installer metadata

默认策略：

- 能复用 OpenClaw 原生数据就不复制一份新真相
- Studio 自管状态只保存 UI/策略/恢复需要的补充信息
- 写入配置时保持结构化 JSON，避免字符串拼接式修改

## 8. 测试与验证

主要命令：

```bash
npm run typecheck
npm run test:chat
npm run test:chat:web
npm run test:system
```

测试目录：

- `tests/chat`：chat contract、service、runtime、markdown、browser smoke
- `tests/system`：管理域、shell、system、dashboard、design/layout contract
- `tests/terminal`：terminal session、manifest、ledger、route sync

开发或文档更新时至少应保证：

- README 中列出的命令真实存在
- README 中列出的文档真实存在
- PRD 状态不与代码目录和路由事实冲突
