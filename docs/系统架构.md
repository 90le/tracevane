# Tracevane System Architecture

> Status: active baseline
> Updated: 2026-06-18

## 1. Architecture Intent

The system is a local-first AI Agent control workbench. It provides a Vue frontend, Node API, independent daemons, shared contracts and repair tools for Gateway, Channel Connectors, CLI Agent workflows and OpenClaw runtime support.

OpenClaw is one runtime dependency. Tracevane must keep clear boundaries so Gateway, Channel Connectors and Recovery can continue serving even when the web UI is unavailable.

## 2. Repository Structure

```text
apps/api/        Tracevane API, service modules and daemon entry points
apps/web-vue/    Vue frontend workbench
types/           shared TypeScript contracts
docs/            current product boundaries, plans and contracts
resources/       packaged resources
scripts/         build, dev and smoke helpers
tests/           system, contract and smoke tests
```

## 3. Backend Domains

| Domain | Responsibility |
| --- | --- |
| `model-gateway` | provider registry, endpoint profiles, secret references, daemon lifecycle, adapters, routing, health, usage and App Connections |
| `channel-connectors` | Feishu/Octo transport, CLI Agent runner, persistent sessions, files, progress, permissions and daemon status |
| `system` / recovery | health, diagnostics, OpenClaw config compatibility, service repair, event log and recovery daemon |
| `chat` | typed Tracevane chat BFF, sessions, history, send/abort/reset/upload/stream and runtime adapter |
| `config` | safe OpenClaw/Tracevane config read/write, validation, preview and backup paths |
| `agents` / `channels` | Tracevane-needed profiles, bindings and runtime metadata, not broad OpenClaw CRUD |
| `terminal` | PTY/WebSocket maintenance terminal |
| `cron` | scheduled task management and run records |

Retired backend domains:

- Dreaming / memory management service
- Legacy plugin management service

## 4. Frontend Domains

Frontend code remains organized under `apps/web-vue/src/features/<domain>`.

Core surfaces:

- `model-gateway`
- `channel-connectors`
- CLI Agent profile/workbench surfaces
- `system` and `system/recovery`
- `dashboard`
- `chat`
- `files`

Support surfaces:

- `config`
- `agents`
- `channels`
- `skills` if retained as cross-Agent capability packaging
- `terminal`
- `cron`

UI rule: prefer drill-in pages, focused dialogs and child pages over dumping all configuration fields on one page.

## 5. Runtime Daemons

| Daemon | Purpose |
| --- | --- |
| Tracevane Gateway daemon | model protocol conversion, provider routing, loopback endpoint and local Gateway key |
| Channel Connectors daemon | IM long connection, Agent runner, file transfer, progress, command handling and sessions |
| OpenClaw recovery daemon | OpenClaw runtime/config/service self-healing and repair evidence |

Daemon rules:

- Linux uses `systemd --user`.
- macOS/Windows support remains a future portability track.
- Tracevane API/UI crashes must not stop Gateway or Channel Connectors.
- Repair writes must backup first and prefer official CLI/schema-compatible operations.

## 6. Data and Secret Boundaries

- Upstream provider keys stay server-side in Tracevane/OpenClaw secret stores.
- Browser clients only receive redacted previews and typed status.
- Gateway key is a local user-facing key; it is not an upstream provider key.
- Channel tokens, attachments, runtime logs and session state stay in daemon-controlled storage.
- Docs, tests and fixtures must not contain live secrets.

## 7. Contract Strategy

- Official/API/SDK/GitHub/community research is required before new provider, CLI Agent, channel or protocol support.
- Prefer structured SDK/session/hook events over terminal text.
- TUI heartbeat parsing is a fallback, never the only authority for completion.
- Unknown upstream events should be sampled, versioned and reported without crashing or faking success.
- Unsupported combinations return structured unsupported errors.

## 8. Verification Baseline

For code changes, choose checks by blast radius:

- `npm run typecheck:api`
- `npm run typecheck:web`
- `npm run build:api`
- `npm run build:web`
- targeted `node --test tests/system/...`
- targeted smoke scripts for Gateway, Channel Connectors, Recovery or UI

For docs-only changes:

- `git diff --check`
- stale/banned active-scope term check
