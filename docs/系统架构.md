# Tracevane System Architecture

> Status: active baseline
> Updated: 2026-06-25

## 1. Architecture summary

Tracevane is a local-first app composed of:

```text
apps/web React UI
  ↓ typed API/query layer
apps/api local BFF/runtime services
  ↓
local filesystem, git, PTY, preview, model gateway, IM connector daemon, platform services
```

Workspace is the core local work surface. Model Gateway, IM Channels, CLI Agents and Platform remain separate owner domains.

## 2. Major services

| Service | Owns |
| --- | --- |
| Workspace facade | Workspace state, file/git/terminal/preview/evidence/Agent handoff orchestration. |
| Files service | Root allowlist, browse, read, search, write, upload, copy/move/delete/archive. |
| Git service | Status, diff, commits, future stage/commit/revert. |
| Terminal service | PTY sessions, stream, replay, resize/input, stale cleanup. |
| Model Gateway | Provider/model/protocol/account/client config/usage. |
| Channel Connectors | IM accounts, bindings, delivery sessions, daemon. |
| CLI Agents | CLI Agent runtime readiness and Agent Runs. |
| Platform/OpenClaw | Platform-native guard/config/services/logs/diagnostics. |
| Evidence | Append-only records and artifacts from workspace/runtime flows. |

## 3. Boundary contracts

### Terminal vs CLI Agents

Terminal is local shell/process infrastructure. CLI Agents are Agent runtimes. They may exchange evidence links, but they are not the same service.

### Workspace vs Platform

Workspace edits local project files and runs local tasks. Platform manages third-party platform-native resources. OpenClaw platform config does not belong in Workspace unless opened as a file under an allowed root for explicit editing.

### Workspace vs Gateway/IM

Workspace may use model/IM status as context links, but provider secrets and IM bot credentials stay in their owner domains.

## 4. Research-first contracts

External/browser/runtime references checked for this Workspace reset on 2026-06-25:

- Monaco Editor concepts: models, URIs, editors, providers, disposal, mobile browser support warning — `https://github.com/microsoft/monaco-editor`.
- xterm.js docs and addon/API structure — `https://xtermjs.org/docs/`.
- MDN media queries and mobile-first responsive design — `https://developer.mozilla.org/en-US/docs/Learn_web_development/Core/CSS_layout/Media_queries`.
- MDN viewport/mobile viewport behavior — `https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/CSSOM_view/Viewport_concepts`.

Design conclusion:

- Monaco can inform browser editor architecture, but current Monaco docs warn mobile browsers are not supported; mobile editing must be validated with the chosen editor and can use a different interaction shell.
- xterm.js addon model supports fit/web links/webgl-style extension, but reliability depends on Tracevane's own PTY/session/replay backend.
- Responsive breakpoints must be content-driven and mobile cannot be a squeezed desktop pane layout.

## 5. Verification expectation

Any Workspace implementation must run:

- `npm run typecheck:api`
- `npm run typecheck:web`
- targeted system tests for files/git/terminal/workspace
- desktop and mobile Playwright smoke
- no-horizontal-overflow checks
