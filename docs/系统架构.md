# Tracevane System Architecture

> Status: active baseline
> Updated: 2026-06-25

## 1. Architecture intent

Tracevane is a local-first AI Agent connectivity and runtime control layer. It is composed of a React frontend, Node API modules, local daemon processes, shared TypeScript contracts, system tests and recovery helpers.

The architecture must keep credential and control boundaries clear:

- Model provider secrets stay in Model Gateway.
- IM platform tokens stay in IM Channels / Channel Connectors.
- PTY/session controls stay in Terminal / CLI Agents.
- OpenClaw support stays in platform/recovery context, not in every feature.

## 2. Repository structure

```text
apps/api/        Node API, service modules and daemon entry points
apps/web/        React + Vite + Tailwind + TanStack Query frontend
types/           shared TypeScript contracts
docs/            current product boundaries, architecture and research gates
resources/       packaged resources
scripts/         build, dev, restart and smoke helpers
tests/           system, contract and smoke tests
```

The former Vue/raw-prototype frontend is retired. Any doc still treating it as active is stale and should be removed or updated.

## 3. Backend domains

| Module | Responsibility |
| --- | --- |
| `model-gateway` | Provider registry, endpoint profiles, account providers, protocol adapters, routing, health, usage, app connections and gateway daemon. |
| `channel-connectors` | IM transports, platform/binding runtime, command surface, attachment staging, delivery, persistent Agent sessions and channel daemon. |
| `channels` | Older/basic channel account/binding config surface. Should converge into IM Channels workflows instead of becoming a parallel product domain. |
| `agents` | Tracevane Agent metadata and read-only Agent Run projection. Must not become broad OpenClaw CRUD or terminal manager. |
| `terminal` | PTY/WebSocket sessions, terminal descriptors, CLI launch command resolution and terminal evidence. |
| `chat` | Local Tracevane chat sessions, history, runtime state and stream/control contracts. |
| `files` / `git` / `system` / `openclaw-recovery` | Workspace evidence, Git state, diagnostics and repair. |

## 4. Frontend domains

| Route | Owner purpose |
| --- | --- |
| `/model-gateway` | Model/provider/protocol/route/account/app-connection/usage control. |
| `/im-channels` | IM platform account, bot credentials, binding, IM session, delivery log and channel daemon control. |
| `/cli-agents` | Codex/Claude Code/OpenCode readiness, Agent Runs and runtime evidence. Terminal operations link back to IDE. |
| `/chat` | Local chat workbench and history. |
| `/ide` / `/files` | Workspace IDE, files, preview, Git and evidence. |
| `/recovery` / `/platforms` | System guard, OpenClaw support, integration evidence and recovery. |
| `/external` | Compatibility/deep-link read-only integration evidence; not a primary owner domain. |
| `/long-tasks` | Compatibility/deep-link supervised-work projection; not a primary owner domain. |

## 5. Core data flows

### Local CLI flow

```text
User starts Codex / Claude Code / OpenCode
        ↓
CLI Agents observes runtime and Agent Run evidence
        ↓
Client calls Model Gateway
        ↓
Gateway routes to provider/account/protocol endpoint
        ↓
CLI Agents records Agent Run/evidence links
```

### IM-triggered flow

```text
IM message arrives through webhook or long connection
        ↓
IM Channels validates platform account and binding
        ↓
Channel Connectors starts/reuses CLI Agent session
        ↓
Agent uses Model Gateway for model calls
        ↓
IM Channels delivers reply and records delivery logs
        ↓
CLI Agents shows Agent Run projection and links back to IM evidence
```

### Model configuration flow

```text
Provider / Codex account / API key configured in Model Gateway
        ↓
Model catalog and active route resolved
        ↓
App Connections write Codex/Claude Code/OpenCode/OpenClaw config with backup/rollback
        ↓
CLI/IM/Chat clients consume Gateway endpoint
```

## 6. Stability rules

- Public API paths must remain backward compatible unless a migration plan exists.
- Read-only projections may aggregate across domains; writes must stay in the owning domain.
- Read-only aggregations and projections should not be promoted to primary navigation domains unless they gain a real owner contract and unique operator workflow.
- High-risk writes require validation, confirmation, evidence and rollback where possible.
- Service restarts must not delete provider/account/channel/session data.
